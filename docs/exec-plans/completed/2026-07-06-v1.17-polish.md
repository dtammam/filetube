# v1.17.0 "Polish"

## Goal

Make the v1.16.0 "Watch experience" release feel finished: root-cause and fix
the mobile horizontal-overflow bug (and its likely-shared-cause watch-page
scroll jump), remove friction from delete and autoplay, close two stuck-UI/
wrong-glyph bugs, and turn down an overly aggressive default — a tight,
cohesive bug-fix-and-polish bundle, not new architecture.

## Scope

Eight items, one cohesive v1.17.0 round, branch `feature/v1.17-polish` off
`main` (currently at v1.16.0):

### FR-1 — Mobile page-sizing / horizontal-overflow root cause (PRIORITY, anchor of the branch)

Dean, verbatim: *"some videos being sized incorrectly, or it's not even the
videos, it's like the whole page. If I pinch and zoom out it resizes it, but
then the search bar is way too small at the top. There's still something
funky. We had this nailed at one point. It was never a thing. Then it became a
thing."* This is the signature of **horizontal overflow**: some element wider
than the viewport forces mobile Safari to zoom the whole page out to fit,
shrinking everything, including the header search bar. Likely regressed with
the v1.16.0 header/search-box-on-more-shells changes or the new fixed
`#player-dock`/persistent-shell CSS.

Intent: **root-cause the exact offending element** (find what exceeds `100vw`
/ breaks the viewport), bisect to the regressing change where feasible, and
fix it **at the source** — not another `overflow-x: hidden` band-aid layered
on top of the existing ones (the codebase already carries several `min-width:
0` overflow fixes on the header/search box from prior rounds; this has
regressed more than once, so a real root cause — not a symptom patch — is the
bar). Folds in two related, older ROADMAP items that may share the same root
cause:
- "Mobile player oversized / not iOS-sized (regression)" / "Mobile player
  STILL launches wrong-size on some videos (post-v1.16.0)" — certain videos
  launch at the wrong size, forcing a scroll/resize to correct (residual after
  the v1.13.0 45vh player cap and v1.15.0 mobile-scale work; may be a
  per-video aspect-ratio interaction with the mobile height cap, or the
  v1.16.0 persistent-player-host CSS).
- "Mobile UI scale polish" — the home page's sort-controls row (heading +
  "Newest first" dropdown + the v1.14.0 "Shuffle again" button) overflows on
  mobile (the shuffle button is cut off), and video cards read large/zoomed.

Dean's on-device pass is the arbiter for this item.

### FR-2 — Watch-page scroll-jump (likely shares FR-1's root cause)

From ROADMAP: on certain videos, scrolling UP makes the whole bottom panel
jump to the middle of the screen; scrolling down snaps it back; a page
refresh makes it behave. Intermittent/video-dependent, almost certainly a
sticky/position-interaction CSS or layout/scroll-position bug on the watch
page. **This is hypothesized to share a root cause with FR-1** (same
watch-page layout/scroll family, same era of regression) — Design must
investigate the two together before treating them as two separate fixes; if
one root-cause fix closes both, that should be documented rather than
producing two independent patches.

### FR-3 — Quicker delete flow

(a) Remove the "File deleted successfully." success alert entirely from the
existing watch-page delete flow (`watch.js`'s `deleteBtn` handler, after
`DELETE /api/videos/:id` returns `{success:true}`) — it's pure friction. A
subtle, non-blocking, auto-dismissing toast is acceptable; nothing is also
acceptable. **Note:** the existing pre-delete confirmation step
(`showConfirmModal('Confirm Permanent Deletion', ...)`) is a legitimate
are-you-sure guard, not the friction Dean is describing, and stays as-is —
only the POST-delete success alert is in scope for removal.

(b) Add a trash-can affordance directly ON the home/library cards (a
currently delete-less surface — `public/js/main.js` has no delete affordance
today). Tapping it **arms** it (shows an inline "Are you sure?" / re-reveals
state) without deleting; a **second tap** on the armed control deletes.
No full modal for this path — YouTube-native quick-delete feel — but it still
guards against an accidental single-tap delete.

Preserve, unchanged: the existing read-only-mount graceful-delete behavior
(`DELETE /api/videos/:id`'s `EROFS`/`EACCES` → `409` + `removeAnyway` opt-in
re-request path in `server.js`) and the delete endpoint's existing behavior/
security posture. Note for Design: the client today has **no UI at all** for
the `removeAnyway` follow-up (a pre-existing gap, not introduced by this
round) — closing that gap is optional/nice-to-have if it falls out naturally
of touching the delete UI, not a hard requirement of FR-3.

### FR-4 — Autoplay on the watch page + refinements

Dean didn't know how to turn autoplay on — it's a buried Settings-page-only
toggle (`autoplay-next-check` in `public/setup.html`/`public/js/setup.js`,
shipped v1.16.0). Three refinements, same underlying `autoplayNext` server
setting (`db.settings`, already exists):

(a) **Surface a visible autoplay toggle on the watch page**, near the
prev/next controls (YouTube-style), reading/writing the SAME persisted
`autoplayNext` setting — toggling on either surface (watch page or Settings)
must stay in sync.

(b) **Skip the resume-playback overlay on an autoplay-advanced next video.**
Today, `player.js`'s `handleResumePlayback` shows the "Resume at…" overlay
any time `savedProgress > 5`, with no distinction for how playback arrived at
that video. Dean: *"assume resume playback of the next thing, and that's fine
for when autoplay is on."* When autoplay-next (FR-3/T3's existing
`handleAutoplayNext`) advances to a new video, that video should just start
playing — no interrupting overlay — even if it has saved progress. A normal,
non-autoplay navigation to a video with saved progress must still show the
resume overlay exactly as today (no regression).

(c) **Reset the live player position to 0 on `'ended'`**, in addition to the
existing v1.16.0 C2 behavior that already resets the *saved/persisted*
progress to 0 on `'ended'` (`player.js`'s `wireHostListeners`,
`saveProgressToServer(0)`). Today only the server-side progress record is
reset; the live `<video>` element's own `currentTime` is left wherever
playback ended. Dean: *"when a video is completed, does it reset to the
beginning without the user having to start over manually?"* This applies to
every completed video (not conditioned on the autoplay setting) so replay is
one tap and the poster frame is clean.

Autoplay stays OFF by default; desktop is unencumbered; no change to the
existing desktop-only initial-load autoplay posture.

### FR-5 — PWA force-close doesn't stop playback + awkward resume

Dean: force-closing the PWA while something was playing didn't stop
playback; reopening resumed "a little awkward." This is an iOS PWA lifecycle
gap: background-audio + the Media Session API keep the playback session
alive past the app's visible closure, and the subsequent resume drops back
mid-stream rather than through the clean resume-overlay re-entry. Intent:
pause **and** persist progress on `pagehide` / `visibilitychange` going
hidden / `freeze` (the app backgrounding or closing), so audio does not
outlive the app; on reopen, the resume overlay is the re-entry point (not an
awkward auto-mid-play). Dean's on-device iOS-PWA pass is the specific
arbiter — this is not fully unit-testable given iOS's background-audio
lifecycle quirks.

### FR-6 — Stuck one-off download modal

From ROADMAP/Dean: opening the header/main-menu one-off download modal
(`injectOneOffDownloadButtonIfEnabled` / `buildOneOffModal` in
`public/js/common.js`) and tapping the backdrop **without** submitting a URL
leaves the page dimmed and unresponsive to touch — a stuck, undismissable
overlay. Intent: backdrop-tap (or an explicit close control) must fully tear
down the modal and restore touch/interaction to the rest of the page in every
case, not only the submit-then-close path. Check `buildOneOffModal` /
`decideOneOffTerminalAction` in `public/js/common.js` specifically.

### FR-7 — Download icon glyph

The one-off download button/nav entry (`.icon-download`) shows the wrong
glyph. **Current state, confirmed by reading the code:** `.icon-download` is
a fixed Unicode character (`content: "\2B07"`, DOWNWARDS BLACK ARROW),
explicitly documented in `style.css` as "independent of the icon-set system"
— unlike `.icon-home`/`.icon-folder`/`.icon-delete`/etc., which are real
Material-Symbols SVG masks with per-icon-set (Outlined/Rounded/Filled) and
per-era variants. **No `download`/`file_download` SVG asset exists today** in
any of `public/assets/icons/{,rounded/,filled/}`. Give it the proper download
glyph — flagged for Design to choose the approach (see the open decision
below); either way this should read as a correct, recognizable "download"
icon.

### FR-8 — Subscription default download count → 3

Dean: *"default download for a subscription should be 3, not 25. 25 is just
aggro."* Change the DEFAULT number of videos a NEW subscription pulls from 25
to 3. The knob: `DEFAULT_MAX_VIDEOS` in `lib/ytdlp/config.js` (currently
`25`), which backs `FILETUBE_YTDLP_MAX_VIDEOS`'s fallback AND is the
effective default whenever a new subscription's "Max videos" field
(`sub-add-maxvideos` in `lib/ytdlp/views/subscriptions.html`, placeholder
`"Max videos (blank=default, 0=unlimited)"`) is left blank. A fresh
subscription should grab the last 3 by default; the user can still raise it.
Preserve the existing bounds/validation (non-negative integer) and the `0` =
unlimited semantics. This is a DEFAULT for NEW subscriptions only — do not
retroactively change any already-persisted subscription's effective
`maxVideos`.

## Out of scope

- Any new architecture, SPA/router changes, or player-controller redesign
  beyond what FR-1/FR-2's root-cause fix and FR-4's watch-page toggle
  genuinely require. This round builds on top of v1.16.0's shell/router/
  player-controller architecture; it does not change it.
- Re-enabling general mobile autoplay for the initial load (FR-4's changes
  are scoped to the `'ended'`-triggered autoplay-next path and the
  resume-overlay/ended-reset refinements around it).
- Weakening the `DELETE /api/videos/:id` endpoint's existing behavior/
  contract (FR-3), the one-off download endpoint's validation (FR-6), or the
  yt-dlp config's existing bounds/validation (FR-8). Only the FR-8 *default
  constant* changes; no guard is loosened anywhere.
- A general client-side `removeAnyway` UI overhaul (FR-3) — closing that
  pre-existing gap is optional/nice-to-have, not mandated by this round.
- Retroactively changing any existing subscription's `maxVideos` (FR-8).
- Checked against `docs/CONTRIBUTING.md`: nothing above defers a mandatory
  standard. In particular, every FR still ships with tests where the
  bug/feature is code-verifiable (FR-3, FR-4, FR-6, FR-7, FR-8); FR-1/FR-2/
  FR-5 are inherently CSS-layout/iOS-lifecycle bugs with **no headless-
  browser test infrastructure in this repo** (`docs/RELIABILITY.md`
  explicitly states there are no automated E2E tests and FFmpeg/browser-
  dependent paths are verified manually) — for those three, Dean's on-device
  pass is the documented, non-negotiable arbiter, not a deferred-testing
  loophole. Design should still extract and unit-test any pure helper it can
  (e.g., an overflow-source check, a scroll-position helper, a resume-
  overlay-suppression flag) wherever the fix produces one.

## Constraints

- **No regressions.** Every item is additive/corrective on top of existing
  v1.16.0 behavior — the persistent shell, docked mini-player, prev/next,
  existing autoplay-OFF default, existing delete-confirmation step, existing
  resume-overlay logic for non-autoplay navigation, and the one-off download
  flow's successful-submit path must all keep working unchanged except where
  an FR explicitly changes them.
- **Security-sensitive touchpoints get extra scrutiny at review time**, even
  though none of them changes a security *guard*:
  - FR-3: the delete endpoint (`DELETE /api/videos/:id`) and its `EROFS`/
    `EACCES` → `409` + `removeAnyway` contract must not change; only client
    UI/UX around it changes.
  - FR-6: the modal-teardown fix must not reintroduce an `innerHTML` XSS
    surface (`CONTRIBUTING.md`'s textContent-not-innerHTML standard) while
    fixing the stuck-overlay bug.
  - FR-8: the yt-dlp `maxVideos` bounds/validation (non-negative integer,
    `0` = unlimited) must not loosen; only the fallback constant's value
    changes.
- **`docs/CONTRIBUTING.md`:** vanilla DOM (no framework/bundler), CommonJS
  server, 2-space/semicolons/single-quotes, `textContent` over `innerHTML`
  for new dynamic strings, `node:test` for new/regression tests, lint 0
  warnings, no new runtime dependencies.
- **Era-theme system preserved.** Any new CSS (the trash-can affordance, the
  watch-page autoplay toggle, a toast if chosen for FR-3a, any FR-1/FR-2
  layout fix) uses existing theme tokens (CSS custom properties) — no
  hardcoded colors that break the 2005/2009/2014/2021 eras or light/dark
  modes.
- **Node 22 LTS** toolchain (`engines` ≥20); CI/build-verify runs tests on
  Node 22.
- **No headless-browser/E2E infra exists** (per `docs/RELIABILITY.md`) — this
  bounds what FR-1/FR-2/FR-5 can realistically claim as `[UNIT]`/
  `[INTEGRATION]`; Dean's on-device pass is the documented arbiter for those,
  not a gap in rigor.

## Design

### Approach

This is a polish/bug bundle on top of the shipped v1.16.0 shell/router/
persistent-player architecture — no new architecture, no layer moves. Every
item is a localized corrective or a small additive control that reuses an
existing server endpoint or CSS token. The two product decisions are locked:
FR-3(a) ships a minimal auto-dismissing toast helper; FR-7 adds a real
`download` Material-Symbols mask across the icon sets (not a patched Unicode
arrow). FR-1 and FR-2 are investigated jointly (below) but resolve to two
distinct fixes on different axes; the rest are independent and parallelizable.

Two shared helpers are introduced in `public/js/common.js` (already the home of

### FR-1 + FR-2 — joint mobile-layout investigation (anchor)

These were investigated together per the assignment. Conclusion up front,
recorded in the Decision log: **they are the same era/family but not a single
shared element — FR-1 is a horizontal (X-axis) `>100vw` overflow; FR-2 is a
vertical (Y-axis) iOS dynamic-viewport/fixed-position interaction. Treat as two
fixes, investigated jointly.** Static analysis ruled out a shared JS cause: the
dock/expand path (`player.js` `mountInSlot`/`dock`) does a plain `appendChild`
reparent and never writes inline `top/left/width/height` — the "measured
geometry" FLIP is explicitly *sketched but not implemented* (`player.js` lines
87-97), so there is no residual inline width driving overflow and no
scroll-tracked geometry driving the jump.

**What was ruled OUT (documented so the SDE does not churn them):** the
header/search box is already defensively patched (`min-width:0` on
`.header-search`/`.search-form`/`.search-input`, `flex-shrink:0` on the button;
`style.css` 303-344) and collapses cleanly to the mobile column layout
(1987-2022); `box-sizing:border-box` is global (227-230) so padding never
overflows; the viewport meta is identical and correct on all four shells
(`width=device-width, initial-scale=1.0, viewport-fit=cover`); `.watch-container`
already goes single-column with a 100%-width sidebar at `<=1024px` (1892-1899),
so the 380px `.watch-sidebar` is not the culprit; `#view-root` has no CSS box of
its own; `#player-dock` is `display:none` while `[hidden]` and, when shown, is
right-anchored (`right; width`) so it cannot extend past the right edge.

**FR-1 — required first step (mandatory, gates the fix).** Because prior rounds
already patched the obvious points, the offending element must be *measured*,
not guessed, before any rule changes — the AC demands a *named* element and
forbids another blanket `overflow-x`/`min-width` band-aid. Run this in Safari
(Web Inspector console, on-device or the responsive simulator) on the home,
watch, and setup pages at a 375-414px width; it lists every element whose right
edge crosses the layout viewport:

- `[...document.querySelectorAll('*')].filter(e => e.getBoundingClientRect().right > document.documentElement.clientWidth + 1).map(e => e.tagName + '.' + e.className + ' -> ' + Math.round(e.getBoundingClientRect().right))`

The named offender from that list is what gets fixed at the source.

**FR-1 — ranked suspects and the matching source fix** (apply the one the
diagnostic confirms; do not apply all speculatively):

- **(most likely) The home sort-controls `<select id="sort-select">`.** It has
  no `max-width`/`min-width:0` (`style.css` 517-526) and its longest option
  ("I'm Feeling Lucky (random)") gives it a large intrinsic width. Inside the
  wrapped `.section-actions` row it can refuse to shrink below its content width
  and push past the viewport — matching the roadmap "sort row overflows /
  Shuffle again cut off" report. **Fix:** `.section-actions .sort-select {
  max-width: 100%; min-width: 0; }` (and confirm `.section-actions { min-width:
  0 }` so the flex item can shrink). This is a source fix on the element that
  overflows, not a shell-level `overflow-x` mask.
- **The mobile header column can exceed width only if a child sets an intrinsic
  min-width** — re-verify the search `<input>` actually shrinks (it has
  `min-width:0`, so it should); no change expected unless the diagnostic points
  here.
- **A media/thumbnail or related-card element with an unconstrained `<img>` or
  fixed px width** (`.related-thumb` is 120px, fine; verify no card image lacks
  `max-width:100%`).

If the diagnostic points at an element not in this list, fix *that* element
(same discipline: constrain the offender with `max-width:100%`/`min-width:0`/
`flex-wrap`, never a new `overflow-x:hidden`).

**FR-1 — folded-in "certain videos wrong-size" + "cards read large".** These are
almost certainly *symptoms* of the same zoom-out, not separate bugs: once an
element forces the layout viewport wider than the device, iOS scales the whole
page down, so the fixed header/search read tiny and cards/player look
mis-scaled; pinch-zoom-out "fixes" it only because the user is manually
re-fitting. Re-verify after the overflow fix; the mobile player cap
(`.player-container { max-height:45vh }` in the portrait query, 2226-2230) is
correct as written (width stays 100%, aspect-ratio letterboxes; no horizontal
contribution) and needs no change unless on-device shows otherwise. Document
whether the residual "wrong-size" reproduces once overflow is gone.

**FR-2 — watch-page scroll-jump: hypothesis + suspects.** Symptom (scroll up →
bottom panel jumps to mid-screen; scroll down → snaps back; refresh settles it;
video-dependent) is the classic iOS Safari **dynamic-viewport (address-bar
collapse) recompute**. Two concrete suspects, both keyed off values that change
as the iOS URL bar shows/hides on tall (scrollable) pages — which is exactly why
it is "video-dependent" (only videos with enough title/description/comments to
scroll trigger it) and why a refresh (URL bar reset to expanded) settles it:

- **(primary) `env(safe-area-inset-top)` is baked into BOTH the fixed
  `header` height (`min-height: calc(96px + env(safe-area-inset-top))`, 1990)
  AND `.app-container { padding-top: calc(96px + env(safe-area-inset-top)) }`
  (1996).** When the address bar collapses on scroll, iOS re-resolves the safe
  area, both values jump together, and the content block below shifts — read as
  the "bottom panel jumping." **Fix direction:** stop letting the scroll-driven
  safe-area value move the content offset — e.g. give `.app-container` a stable
  pixel `padding-top` and absorb the notch inset with a dedicated fixed spacer
  (or move the inset onto `header` padding only, not the content offset), so the
  content top is invariant across the address-bar animation.
- **(secondary) fixed bottom-anchored `.bottom-nav` (`position:fixed;
  bottom:0`, 1941-1955)** and the docked `#player-dock` bottom offset repaint at
  stale positions during the same address-bar animation. Verify whether the
  "panel" Dean sees is actually the bottom nav; if so the same stable-offset
  approach applies.

FR-2 is iOS-lifecycle and this repo has no headless-browser harness, so Dean's
on-device pass is the arbiter; the design names the suspects and the fix
direction as required. Extract no pure helper here (there is no pure decision to
extract); FR-1's diagnostic filter is the one testable-in-console artifact.

### FR-5 — PWA force-close: pause + persist, clean resume

Add lifecycle listeners in `player.js` alongside the existing foreground
`visibilitychange` re-assert (280-285), which stays intact. On `pagehide`,

### FR-6 — Stuck one-off download modal teardown

**Root cause (confirmed):** `.oneoff-modal-backdrop` sets `display: flex`
(`style.css` 1629) with **no `[hidden]` override**. An author `display` rule
beats the UA `[hidden] { display:none }`, so `backdrop.hidden = true` (the entire
teardown in `closeModal`, common.js 1135-1141) does not actually hide it — a
full-viewport `position:fixed` overlay at `z-index:2100` stays painted and eats
every touch: the page reads dimmed and dead. (The sibling
`.playlists-sheet-backdrop` avoids this by using the `:not([hidden])` pattern,
2025.)

**Fix (two parts, one teardown path):**

- CSS source fix: add `.oneoff-modal-backdrop[hidden] { display: none; }` (or
  fold the `display:flex` into `.oneoff-modal-backdrop:not([hidden])`, matching
  the sheet precedent). The attribute selector out-specifies the base rule; no
  `!important`.
- Harden `closeModal` into a full teardown so both dismiss paths (backdrop tap,
  `[x]`, Esc, and the `done` auto-close) share one implementation: remove the
  backdrop node from the DOM (`backdrop.remove()`), `stopPolling()`, null
  `currentJobId` **and** `modalState` so `openModal` rebuilds fresh next time
  (the once-bound `keydown` Esc handler already guards on `modalState &&`, so
  nulling it makes that handler an inert no-op). Preserve the existing
  `createElement`/`textContent` DOM construction — no `innerHTML` introduced.

Unit-testable with the existing fake-`document` pattern (`buildOneOffModal` is
already DOM-injectable): assert the backdrop is detached and `modalState` reset
after a backdrop-dismiss-without-submit.
