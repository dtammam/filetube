# v1.13.0 -- v1.12.0 polish + mobile fixes

Target release: **v1.13.0**. Feature id: `v1.13-polish`. Branch:
`feature/v1.13-polish` (off `main`, currently v1.12.0 shipped; prior feature
record `docs/exec-plans/completed/2026-07-06-ytdlp-metube-parity.md`).

This is a **right-sized, light-process** polish round on top of the shipped
v1.12.0 yt-dlp MeTube-parity feature. It does **not** re-litigate v1.12.0's
locked decisions (D1-D5) or its security core (T3/T4), and does not re-open
the FR-G synthetic-folder / E1 mount-loss architecture beyond the narrow
order-persistence gap described below.

## Goal

Polish the v1.12.0 yt-dlp module surface, fix two mobile/UX regressions and
one always-on server DELETE UX gap, and clean up two further mobile/menu
polish + duplicate-metadata bugs surfaced since ship -- without touching the
v1.11.0-v1.12.0 architecture, security core, or disabled-no-op guarantees.

## Scope

Seven items:

1. Fix the janky `/subscriptions` "Your subscriptions" list (per-character
   vertical wrap of names/URLs) by giving the list its own row layout.
2. Cap the mobile-portrait player height (currently a 70vh cap that's too
   tall for a phone) to a sensible ~40-50vh target, without touching desktop,
   landscape, or audio-mode/audio-bg-art layout.
3. Make the synthetic "Downloads" folder's Setup-page **order** persist
   across reload, while keeping it out of `db.folders` (rename already
   works; only reorder is broken). Small PE design required.
4. Add a filetype/container dropdown (one-off download + subscription):
   video -> mp4/mkv/webm/default; audio -> mp3/m4a/opus/default. Wire to
   `lib/ytdlp/args.js` (`--merge-output-format` / `-x --audio-format`), add a
   validated `filetype` field to the subscription record + one-shot body,
   backfill existing subs. Small PE design required (merge vs recode,
   allowlist shape).
5. Graceful `DELETE /api/videos/:id` behavior on read-only/permission-denied
   mounts: a specific EROFS/EACCES message instead of a generic 500, plus an
   opt-in "remove from library anyway" that deletes the db entry even when
   the file can't be unlinked, with an honest re-add-on-rescan caveat.
6. Mobile (portrait) layout polish for the Setup page (Library Settings) and
   the `/subscriptions` page: comfortable spacing, wrapping, control sizing,
   and tap targets. Extends/overlaps item 1's row-layout fix for
   `/subscriptions` specifically.
7. Stop displaying the redundant "Synopsis" line on downloaded items' watch
   pages -- yt-dlp's `--embed-metadata` writes the same text into both
   `description` and `synopsis` (and the URL into `comment`); FileTube
   currently surfaces all three. Fix: drop `synopsis` from the displayed
   embedded-tags set. Keep Description and Comment.

## Out of scope

- Any re-architecture of the yt-dlp module's scan-root / disabled-no-op
  design (D1-D5, FR-G, T3/T4) -- items 3 and 4 extend it narrowly, they do
  not redesign it.
- The deep prune/mount-loss redesign tracked as tech-debt tracker item #10 --
  item 3's order-persistence fix must not depend on it or attempt to close it.
- A general "hide any embedded tag that duplicates Description" nicety for
  item 7 -- the mandate is narrowly to drop the Synopsis line; a broader
  dedup rule is a possible future nicety, not required here (see Decision
  log).
- A job-queue / progress-bar redesign, WebSocket/SSE transport, multiple
  cookies files, or any other item from v1.12.0's out-of-scope list -- all
  still out of scope, unchanged.
- **Not deferrable:** tests, `npm run lint` zero-warnings, and the
  disabled-no-op guarantee are in scope for every item that touches
  module-gated code (CONTRIBUTING.md's "every feature/bugfix ships with
  tests" applies uniformly; nothing below softens that).

  stays the sole authoritative scan/mount-loss root. The synthetic "Downloads"

  SEPARATELY (e.g. a `folderSettings.order` field or a dedicated pref) and

  FS cleanup succeeded" contract is the DEFAULT path; "remove anyway" is an
  explicit, separately-requested opt-in, never silent.
- **CONTRIBUTING.md, unconditionally:** vanilla DOM, `textContent` not
  `innerHTML` (KEEP the XSS-safe rendering already in `subscriptions.js`),
  `node:test`, lint 0 warnings, **no new dependencies**. 2-space indent,

- **Git ownership:** the coordinator (EM) owns all git; SDEs implement and
  report, never commit/stage/push.
- **Toolchain:** Node 22 LTS is the test/verify standard (per
  `docs/CONTRIBUTING.md`); build-verify must run the suite on Node 22, not
  only a newer local default.

## Functional Requirements

**FR-1 (subscriptions list layout).** The `/subscriptions` "Your
subscriptions" list must render each row with its own layout (not the Setup
page's `folder-item-row` class + collapsing `flex:1; min-width:0` info
column) so the name and URL wrap normally (word-wrap / break-word at
sensible boundaries), not one character per line. The add/edit/one-shot
controls in the same page should be tidied (adequate spacing, no obviously
cramped controls). The existing `textContent`-only rendering and the
disabled-page-absence behavior are preserved exactly.

**FR-2 (mobile player height).** `.player-container`'s mobile-portrait
`max-height` cap (currently 70vh) is reduced to a value in the ~40-50vh
range. Desktop, landscape, and the audio-mode/audio-bg-art layout are
unaffected. The implementer records the exact before/after CSS values and
the media-query breakpoint touched.

**FR-3 (synthetic folder order persistence).** The synthetic "Downloads"
folder's display order in Setup (and the sidebar) persists across reload
after a reorder, via an order value stored OUTSIDE `db.folders` (in
`folderSettings` or a dedicated pref, merged at read time), while the folder
itself remains absent from `db.folders`. Real (non-synthetic) folder
reordering continues to work exactly as before. (PE: design the storage
field and the merge-at-read logic; keep the change minimal and reuse the
existing real-folder order mechanism.)

**FR-4 (filetype/container dropdown).** A third dropdown -- filetype/
container -- is added to the add-subscription form, the one-shot form, and
the per-subscription edit panel. Options: video -> `mp4` (default) / `mkv` /
`webm` / `default` (yt-dlp's own choice); audio -> `mp3` (default) / `m4a` /
`opus` / `default`. The subscription record and the one-shot request body
gain a validated `filetype` field; `lib/ytdlp/args.js` wires a non-`default`
video selection to `--merge-output-format <ext>` and a non-`default` audio
selection to `-x --audio-format <fmt>`; an unset/invalid value behaves as
`default` (today's forced `mp3`/no-container-control behavior, respectively,
for audio; today's no-container-control behavior for video). Existing
subscriptions without a `filetype` field are treated as `default` without
requiring a destructive migration. (PE: decide merge-vs-recode strategy for
video -- default recommendation is `--merge-output-format`, not
`--recode-video` -- and the exact allowlist/validation shape.)

**FR-5 (graceful read-only-mount delete).** `DELETE /api/videos/:id`
catches an unlink failure and returns a message that distinguishes
EROFS/EACCES ("read-only or permission-denied location") from other errors,
instead of today's generic "Could not delete file: `<raw message>`". An
opt-in `removeAnyway` request flag (query param on the same DELETE endpoint,
surfaced via a follow-up confirm in the same delete flow once the client
sees the specific read-only/permission error) deletes the db entry
(`metadata`/`progress`) even though the underlying file could not be
unlinked, and its response/UI copy carries an explicit caveat: if the mount
is still scanned, the item will reappear on the next scan. The default (no
flag) path is UNCHANGED otherwise: db is left untouched on an unlink
failure, and the happy path (successful unlink -> thumbnail/transcode/db
cleanup) is unchanged.

**FR-6 (mobile settings/menu polish).** The Setup page (Library Settings,
`public/setup.html` / `public/js/main.js` / `public/css/style.css`) and the
`/subscriptions` page (`lib/ytdlp/views/subscriptions.html` /
`lib/ytdlp/client/subscriptions.js`) get a mobile-portrait layout pass:
adequate spacing between rows/controls, form fields that wrap instead of
overflowing/clipping, and tap targets sized comfortably for touch. This
extends FR-1's `/subscriptions` row-layout fix into a fuller mobile pass for
that page's forms/controls, and is the first mobile pass for the Setup
page's forms/controls. Desktop layout for both pages is unaffected.

**FR-7 (drop duplicate Synopsis).** The watch page's embedded-tags display
no longer shows a `synopsis` line. `server.js`'s `parseFfprobeTags`
whitelist (`EMBEDDED_TAG_WHITELIST`, ~line 762/789) drops `'synopsis'` (or
equivalently the watch-page render in `public/js/watch.js`'s
`renderEmbeddedTags`, ~line 241, filters it) so it is never surfaced.
Description and Comment continue to display as before. This is an
accepted, Dean-approved narrow limitation: a non-yt-dlp file with a
genuinely distinct, non-duplicate synopsis tag will also no longer display
it -- there is no yt-dlp-vs-other-source distinction made at this layer, and
none is required.

## Non-functional requirements

**NFR-Security (item 4, spawn-arg discipline -- same bar as T3/T4).** The
filetype/container value must be allowlist-validated server-side against a
fixed, hardcoded enum (`mp4`/`mkv`/`webm`/`default` for video;
`mp3`/`m4a`/`opus`/`default` for audio) before it ever reaches an argv
array; any other value (including one containing shell metacharacters,
flags, or path-like content) is rejected and treated as `default`, never
passed through. Spawn discipline (arg-array, no `shell: true`,
`--restrict-filenames` retained, cookies redaction unaffected) is unchanged
from v1.12.0.

**NFR-Reliability (item 5, graceful FS-failure handling).** Per
`docs/RELIABILITY.md`: the unlink is wrapped in try/catch; on failure the
server logs and returns an explicit JSON error with a specific HTTP status,
never crashes the process; a bad delete never corrupts `db.json` (the
existing `updateDatabase` mutator/atomic-save discipline is reused
unchanged for both the default and remove-anyway paths).

**NFR-Non-regression (items 1/3/4, disabled-no-op + E1).** With
`FILETUBE_YTDLP_ENABLED` off: no new route, no new DOM, no new poll
behavior, byte-identical to today. With it on: `extraScanRoots()` remains
the sole authoritative scan/mount-loss root regardless of the synthetic
folder's Setup-page order or rename state; no scan/prune decision anywhere
reads the synthetic folder's order or its `db.folders` membership (it has
none).

## Design

Only items 3 and 4 carry real design work; 1, 2, 5, 6, 7 are direct fixes
(their approach is captured in the task breakdown below). The two designed
items both EXTEND v1.12.0's locked surfaces narrowly -- item 3 rides the
existing `folderSettings` rename channel (never `db.folders`); item 4 rides
the existing `format`/`quality` validate-at-boundary + re-assert-before-argv
pipeline. No new dependencies, no schema migration, no new endpoints.

### Design for item 3 -- synthetic "Downloads" folder order persistence

#### Approach (item 3)

The reorder gap is purely server-side. The synthetic entry already
round-trips through the client as a normal folder: `GET /api/config` appends
it to `folders`, `setup.html` loads the whole array into `configuredFolders`
and renders it with the existing up/down `.reorder-btn` handlers
(`public/setup.html:256-289`), and the save POSTs the reordered
`configuredFolders` back as `folders` (alongside `folderSettings`). Rename
persists because `POST /api/config` keeps the synthetic root's
`folderSettings[root]` entry even while stripping the root from `db.folders`
(`server.js:1477`, `:1489`, `:1498`). The ONLY thing lost on the round trip
is the synthetic's POSITION -- `GET` always `push`es it LAST
(`server.js:1386`).

Fix: persist an integer `order` on the synthetic root's existing
`folderSettings[root]` entry (the SAME entry that already carries `name`),
and have `GET /api/config` splice the synthetic entry in at that index
instead of always appending. The order value is DERIVED SERVER-SIDE from the
submitted `folders` array position on POST -- the client needs no new logic,
it simply keeps sending the reordered array it already sends. The synthetic
root never enters `db.folders`; `extraScanRoots()` is never read or written
by this path.

#### Where order is stored (item 3)

`db.folderSettings[<resolvedDownloadDir>].order` -- an integer, alongside the
existing `name`/`hidden` on that same synthetic-root entry. Real
(`db.folders`) folders get NO `order` field: their order stays their array
position in `db.folders`, exactly as today. `order` is only ever written for
a key that is a synthetic root (`syntheticRoots.has(resolvedKey)`), so no
real-folder semantics change.

#### POST /api/config -- deriving and persisting order (item 3)

In the `validFolders` loop (`server.js:1457-1480`), the synthetic root is
currently skipped with a bare `continue` (`:1477`). Change that branch to
FIRST record its intended index, then continue:

- Add `const syntheticOrders = new Map();` before the loop.
- At the synthetic-root branch, `syntheticOrders.set(resolved, validFolders.length)`
  (the count of real folders that preceded it in submitted order == its
  display index among real folders) BEFORE `continue`. It is still never
  pushed into `validFolders`.

In the `cleanSettings` loop (`server.js:1485-1503`), when the stored key is a
synthetic root, add `order` to the persisted object:

- `order = syntheticOrders.has(resolvedKey) ? syntheticOrders.get(resolvedKey)
  : (Number.isInteger(s.order) ? s.order : undefined)`. The fallback to the
  client-submitted `s.order` preserves a previously-stored order when a
  client POSTs `folderSettings[root]` WITHOUT round-tripping the synthetic in
  the `folders` array (same resilience the existing `name` handling has).
  Only include `order` in the stored object when it is an integer.

Real-folder settings objects are unchanged (`{ name, hidden }`).

#### GET /api/config -- merge-at-read (item 3)

Replace the always-last `folders.push(root)` (`server.js:1386`) with an
order-aware splice:

- Compute `const stored = folderSettings[root] && folderSettings[root].order;`
- `const idx = Number.isInteger(stored) ? Math.max(0, Math.min(stored, folders.length)) : folders.length;`
- `folders.splice(idx, 0, root)` (only when the root is not already present).

`idx == folders.length` reproduces today's append-last behavior for a
synthetic root that has never been reordered (backward compatible: existing
installs with no stored `order` are unchanged). Clamping keeps a stale order
(e.g. real folders later removed) in range. The sidebar
(`renderSidebarFolders` in `public/js/main.js`) and the playlists sheet
(`renderPlaylistsSheet` in `public/js/common.js`) render `data.folders` in
the returned order, so both pick up the merged position with NO client
change.

#### Round-trip (rename + reorder together) (item 3)

Rename writes `folderSettings[root].name`; reorder writes
`folderSettings[root].order`. Both live on the one synthetic-root entry, both
survive the POST prune (the entry is kept because `syntheticRoots.has` is
true), and both are re-read by GET. Renaming and reordering in the same save
persist together (AC9). After any round trip the synthetic root is still
absent from `db.folders` (AC10) because the `continue` still skips the
`validFolders.push`.

#### Invariants preserved (item 3)

- `extraScanRoots()` is neither read nor written by the order logic -- it
  remains the sole authoritative scan/mount-loss root (E1 intact, AC12).
- No scan/prune path reads `folderSettings[...].order`; it is display-only.
- Disabled module -> `syntheticRoots` is empty -> the `validFolders`/GET
  branches never fire -> `GET`/`POST /api/config` are byte-identical for real
  folders (C3/C7 disable-no-op intact, AC11).
- Client (`setup.html`/`main.js`) needs no functional change; the SDE only
  verifies the save payload already carries the synthetic entry inside
  `folders` and its `folderSettings[root]` (it does -- that is what makes
  rename work today).

#### Alternative considered (item 3)

A dedicated top-level `db.ytdlpFolderOrder` pref keyed by root. Rejected: it
splits the synthetic entry's persisted state across two places (rename in
`folderSettings`, order in a new pref), adds a new prune/backfill surface,
and touches `loadDatabase` defaults -- more blast radius for no benefit.
Co-locating `order` on the existing `folderSettings[root]` entry reuses the
exact channel rename already proved out.

### Design for item 4 -- filetype/container dropdown

#### Approach (item 4)

Add a third, format-dependent dropdown (`filetype`) to the add-subscription
form, the one-shot form, and the per-row edit panel. The value flows through
the SAME pipeline `format`/`quality` already use: validated at the API
boundary in `lib/ytdlp/store.js`, persisted on the subscription record via
`updateDatabase`, and RE-ASSERTED defensively in `lib/ytdlp/args.js`
immediately before it can influence an argv element. The enum is a single
hardcoded, format-partitioned allowlist owned by `args.js`; `store.js`
imports it so there is one source of truth. video -> lossless remux via
`--merge-output-format <ext>` (never `--recode-video`, no CPU transcode);
audio -> the existing `-x --audio-format <fmt>`, generalized from today's
hardcoded `mp3`.

#### The allowlist + the meaning of `default` (item 4)

Canonical, in `lib/ytdlp/args.js`:

```js
const VALID_FILETYPES = {
  video: new Set(['mp4', 'mkv', 'webm', 'default']),
  audio: new Set(['mp3', 'm4a', 'opus', 'default']),
};
const DEFAULT_FILETYPE = 'default';
```

`'default'` means "today's behavior" -- for audio, force `mp3` (unchanged
from the current hardcoded arg); for video, emit NO container flag (yt-dlp's
own choice, unchanged from today). Note audio `'mp3'` and audio `'default'`
produce identical args by construction (both `-x --audio-format mp3`); that
is intentional and harmless. The DROPDOWN pre-selects `mp4` (video) / `mp3`
(audio) for a NEW record (best-compatibility default, per the Decision log),
while an UNSET field on an existing record resolves to `'default'`.

#### Validation -- single-sourced, format-aware (item 4)

- `args.normalizeFiletype(format, raw)` (new, in `args.js`, mirrors
  `normalizeQuality`): returns `raw` when `format` is `'audio'`/`'video'` AND
  `raw` is a string in `VALID_FILETYPES[format]`; otherwise `'default'`.
  Never throws. This is the LAST gate before argv and is what makes AC18
  hold: a hostile string (`;`, backticks, `--exec`, a path, a flag shape)
  can never reach the array -- it collapses to `'default'`.
- `store.validateFiletype(format, value)` (new, in `store.js`, mirrors
  `validateMaxVideos`; imports `VALID_FILETYPES`/`DEFAULT_FILETYPE` from
  `args.js` for single-sourcing): `undefined` -> `{ ok: true, value: undefined }`
  (unset == default at build time, non-destructive per AC16); a value in
  `VALID_FILETYPES[format]` -> `{ ok: true, value }`; anything else ->
  `{ ok: false, error }` for a clean `400` at the boundary.

#### args.js mapping (buildYtdlpDownloadArgs) (item 4)

Replace the current `format === 'audio'` branch (`args.js:361-365`):

```js
const filetype = normalizeFiletype(format, sub && sub.filetype);
if (format === 'audio') {
  const audioFmt = filetype === 'default' ? 'mp3' : filetype;
  args.push('-x', '--audio-format', audioFmt);
} else {
  args.push('-f', QUALITY_SELECTORS[quality]);
  if (filetype !== 'default') {
    args.push('--merge-output-format', filetype);
  }
}
```

`--merge-output-format` remuxes the already-downloaded streams into the
chosen container losslessly (AC14); `'default'` omits it (today's behavior).
Audio `default`/unset stays exactly `-x --audio-format mp3` (AC15). No
`--recode-video` anywhere.

#### Record shape + backfill (item 4)

The subscription record gains an OPTIONAL `filetype` field. Following the
`maxVideos` precedent (NOT the `paused` precedent), it is NOT backfilled to a
concrete value in `ensureYtdlp` -- an old record simply lacks the field, and
`normalizeFiletype(format, undefined)` resolves it to `'default'` at build
time (AC16, no destructive rewrite). `store.addSubscription` gains a
`filetype` param (fail-safe to `undefined` on an invalid value, matching its
`maxVideos` handling) and writes `filetype: resolvedFiletype` on the new
record.

#### Boundary wiring (item 4)

- `store.validateSubscriptionInput` (add path): after `format` is resolved,
  `validateFiletype(format, input.filetype)`; on `!ok` return the error
  (`400`); include `filetype` in the returned `value`.
- `store.validateSubscriptionPatch` (PATCH): `filetype` is optional. Because
  `format` may be absent from a patch, do a COARSE reject here (present +
  not a string / not a member of EITHER allowlist -> `400`, catches hostile
  garbage) and carry the value through; the authoritative format-aware
  coercion is `args.normalizeFiletype(effectiveFormat, ...)` at build time.
  A lingering audio-filetype-on-a-video-sub combo therefore degrades safely
  to `'default'` rather than erroring -- documented, acceptable.
- `store.updateSubscription`: apply `sub.filetype = body.filetype` when
  present (same defensive shape as the other fields).
- `POST /api/ytdlp/download` (`index.js:930`): after resolving `format`, add
  `const ft = store.validateFiletype(format, body.filetype);` -> `400` on
  `!ok`; pass `filetype: ft.value` into `runOneShot`.
- `runOneShot` (`index.js:694`): thread `filetype` through and put it on
  `syntheticSub` (`index.js:706`), so `buildYtdlpDownloadArgs` reads it the
  same way it reads `format`/`quality`.

#### Client wiring -- format-dependent select (item 4)

In `lib/ytdlp/client/subscriptions.js`, add a hardcoded mirror beside
`FORMAT_OPTIONS`/`QUALITY_OPTIONS` (`:34-39`):

```js
const FILETYPE_OPTIONS = {
  video: [ {value:'mp4',label:'MP4 (recommended)'}, {value:'mkv',label:'MKV'},
           {value:'webm',label:'WebM'}, {value:'default',label:'Default (yt-dlp)'} ],
  audio: [ {value:'mp3',label:'MP3 (recommended)'}, {value:'m4a',label:'M4A'},
           {value:'opus',label:'Opus'}, {value:'default',label:'Default (yt-dlp)'} ],
};
const DEFAULT_FILETYPE_OPTION = { video: 'mp4', audio: 'mp3' };
```

- `buildFiletypeSelect(doc, format, selected)` -> `buildSelect(doc,
  FILETYPE_OPTIONS[format || 'video'], selected || DEFAULT_FILETYPE_OPTION[format || 'video'], 'setup-select')`.
- A shared `repopulateFiletype(formatSelect, filetypeSelect, selected)`
  helper clears and rebuilds the filetype `<select>` from the current
  format's options (used by all three surfaces on `format`-`change` and at
  init). `textContent`-only option labels, no `innerHTML` (KEEP the XSS-safe
  posture).
- Add/one-shot forms: add `<select id="sub-add-filetype">` /
  `<select id="oneshot-filetype">` to `lib/ytdlp/views/subscriptions.html`
  next to the format/quality selects; a `change` listener on each format
  select calls `repopulateFiletype`; init to the recommended value on load.
  Include `filetype` in the add body (`:618`) and one-shot body (`:702`).
- Edit panel (`createSubscriptionRow`, `:265-300`): build
  `editFiletypeSelect = buildFiletypeSelect(d, sub.format, sub.filetype)`,
  wire `editFormatSelect` `change` -> `repopulateFiletype`, and add
  `filetype: editFiletypeSelect.value` to the `onSaveEdit` patch (`:293`).
- Optional nicety: append the filetype to `formatSubMeta` (`:62`). Not
  required by any AC.

#### Alternative considered (item 4)

`--recode-video <ext>` for the video path (guarantees the container even when
source streams do not natively fit). Rejected per the Decision log: it forces
a full CPU re-encode on a home server for the common case where a remux
suffices; `--merge-output-format` is lossless and cheap, and yt-dlp already
falls back gracefully when a stream cannot be placed in the chosen container.

#### Invariants preserved (item 4)

- Spawn discipline unchanged: arg-array via `spawn` (never `shell: true`),
  `--` before positional targets, `--restrict-filenames` retained, cookies
  redaction untouched (AC18).
- Disabled module -> the `/subscriptions` page, the new select, and the
  one-shot form stay absent (they are served from inside the `isEnabled`
  gate); no new always-on surface.

### Risks and mitigations (items 3 + 4)

- **Risk (item 3):** a client POSTs `folders` without the synthetic entry ->
  its `order` would be lost. **Mitigation:** the `cleanSettings` fallback
  preserves a client-submitted `s.order`; and the failure mode is merely
  "reverts to append-last," never a `db.folders` write.
- **Risk (item 4):** format/filetype drift in a PATCH that changes only one
  of them. **Mitigation:** `normalizeFiletype` is format-aware and coerces a
  mismatch to `'default'` at build time -- safe degradation, never a bad arg.
- **Risk (item 4):** client allowlist drifting from the server enum.
  **Mitigation:** the server re-validates on every request and re-asserts
  before argv, so client drift can only ever be neutralized, never trusted
  (the same posture the existing `FORMAT_OPTIONS`/`QUALITY_OPTIONS` mirror
  relies on).

### Performance impact (items 3 + 4)

No impact on any `docs/RELIABILITY.md` budget. Item 3 adds an O(1) map write
plus a single array splice per `GET/POST /api/config`. Item 4 adds a bounded
allowlist lookup per request and one extra argv token per download;
`--merge-output-format` is a remux (no transcode), so download CPU is
unchanged versus today for the common case.
