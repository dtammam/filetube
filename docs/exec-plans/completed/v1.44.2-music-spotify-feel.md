## 1. Goal (Dean's words, LOCKED)

Make picking and playing a song feel like Spotify. **Playback-side** (not
browse-side), **art-forward**, **phone-first**, and **a good mini-player is
enough** — NO full-screen now-playing takeover. Three pieces:

1. **Collapsing / sticky album (and artist) header** on the drill-in view:
   large cover art + title + artist + year·track-count + prominent **Play** +
   **Shuffle** at rest; as the tracklist scrolls the header collapses into a
   slim **sticky bar** (small thumb + title + play). **IntersectionObserver**
   on a sentinel — NEVER per-frame scroll math (the v1.37.2 / v1.43.1
   "measure, don't guess" scar). Only the album/artist drill-in gets this —
   NOT the flat Songs/Liked tabs (mixed albums, no single art to anchor).
2. **Playing-row highlight**: accent the row whose track id == the playing
   track + a small **equalizer glyph drawn in CSS** (never an emoji codepoint
   — iOS forces blue emoji). Driven from `playAt(i)` (the choke point every
   advance routes through: tap, on-page prev/next, lock-screen next).
3. **"Playing from &lt;Album&gt;"** context line.

## 2. Intake decisions (Dean answered 2026-07-18 — all "agree")

1. **Play target → DOCK.** Tapping a song plays in the **docked mini-player**
   (bottom-right corner: album art + play/pause + slim seek — already a usable
   music mini-player) and highlights the row, keeping the album header +
   tracklist visible (browse-while-playing). It no longer expands the FULL
   player at the top of the view (which pushed the header down). This changes
   `loadTrack`'s mount target.
2. **Version = v1.44.2**; RBAC stays v1.45.
3. **Fold in tech-debt #46** — a same-URL nav no-op guard in `navigate()`
   (the cleaner of the two tracked directions; benefits home/watch/read/music
   uniformly; also kills the identical `/read` strand).
4. **Desktop = same mechanism** as phone (one IntersectionObserver code path;
   the header just has more room on desktop — no divergent branch).

## 3. Architecture directives (repo lessons — violate NONE)

- **UI:** all view markup INSIDE `#view-root`; ALL styles in `style.css`
  (the v1.41.8 SPA-swap lesson — page-local `<head>` styles die on in-app
  nav). Typography via `--fs-*` tokens ONLY — a lock test forbids literal
  `font-size:` **even in comments** (the v1.44 gate tripped on this). Colors
  via era/mode tokens. Glyphs via CSS/SVG, never emoji codepoints.
  `[hidden]{display:none!important}` on any hide-via-`hidden` element.
- **Player:** EXTEND player.js, do not rebuild it (the iOS background-audio /
  reparent machinery is battle-won). The persistent single `<video>` host is
  reparented between `#player-slot` (FULL) and `#player-dock` (DOCKED); each
  reparent has a `wasPlaying`→`play()` recovery. Do NOT add a second reparent
  where one suffices.
- **`shouldDockOnTransition` is DUPLICATED** in common.js and player.js and
  MUST stay in lockstep (the v1.44.1 scar). This wave does not change it, but
  the same-URL guard (§4.5) sits just upstream of it — verify their
  interaction.
- **Pure builders stay pure + node:test-covered** (the `music-view.test.js`
  posture): any new markup helper (`buildDrillHeaderHtml`, the eq markup in
  `buildSongRowHtml`) is DOM-free and unit-tested for shape + escaping.
- **Do NOT remove** `#player-dock` / `#player-host-template` (a source-lock
  guards them). `#player-slot` stays too — it is the transient FULL-mount
  target and the dock-tap-expand target (§4.1).
- **IntersectionObserver lifecycle:** it is NOT covered by music.js's
  AbortController. It must be `disconnect()`-ed in `destroy()` AND before every
  re-render, or a stale observer leaks across the SPA `#view-root` swap.

## 4. Technical design

Scroll container = the **window** (`.main-content` has no overflow scroll; the
`header`/`.bottom-nav`/`#player-dock` are `position:fixed`). So the
IntersectionObserver uses the **viewport** (null root); the fixed header
overlaps the top, so the sentinel/rootMargin accounts for a **measured**
header height (offsetHeight once at wire-time — not a CSS-var guess).

### 4.1 Play → dock (player.js `opts.dock` + music.js loadTrack)

Today `loadTrack` does `player.load(id, data, { slot: playerSlot })` →
`mountInSlot` → `state = FULL`, host in `#player-slot` at the top of the view.

**player.js** — add a `dock` option to `load()` so a caller can mount straight
into the dock without the mount-FULL-then-reparent dance (avoids a double
reparent of the media element — an iOS-sensitive op):

```
function load(id, data, opts) {
  var options = opts || {};
  if (!ensureHost()) return false;
  var adopt = isAdoptLoad(currentId, id, state);
  if (adopt) {
    if (data && typeof data.browseCtx === 'string' && currentData) currentData.browseCtx = data.browseCtx;
    if (options.dock) dock(); else expand(options.slot);   // NEW: dock a re-tap
    return true;
  }
  ... capture autoplay flag, teardownMediaState, currentId=id, currentData=data ...
  if (options.dock) mountInDock(); else mountInSlot(options.slot);   // NEW
  setupForMedia(id, currentData);
  return true;
}
```

`mountInDock()` — the dock-targeted twin of `mountInSlot()` (mirrors dock()'s
reparent tail without the FULL→DOCKED-specific resume-overlay handling, which
never applies to a fresh music load — music has no resume prompt, §music
resumeMode):

```
function mountInDock() {
  var dockEl = document.getElementById('player-dock');
  if (!host || !dockEl) return;
  ensureDockChrome(dockEl);
  var wasPlaying = mediaPlayer && !mediaPlayer.paused;
  if (host.parentNode !== dockEl) dockEl.appendChild(host);
  dockEl.hidden = false;
  state = STATE_DOCKED;
  applyControlsMode();
  if (wasPlaying && mediaPlayer.paused) mediaPlayer.play().catch(function () {});
}
```

**music.js `loadTrack`** — dock instead of FULL; keep `#player-slot` hidden:

```
// (drop `playerSlot.hidden = false`)
window.FileTube.player.load(item.id, data, { dock: true });
```

`setTrackNav` + the resume-pointer POST are unchanged. `prewarmThenLoad` (ALAC)
is unchanged — it still calls `loadTrack` once ready.

**Dock-tap-to-expand — fix the latent music bug.** The dock click handler
today navigates to `readerHref` else `/watch.html?v=<currentId>`. A **music**
track has no `readerHref`, so tapping the docked music mini-player navigates to
`/watch.html?v=<trackId>` — a VIDEO route with a track id → 404/"not found".
Harmless-ish today (FULL was the default), but play→dock makes the dock the
primary surface, promoting this to a core interaction. Fix: music tracks carry
a dock-return href of `/music`. In `loadTrack`'s `data`, set
`readerHref: '/music'` (the field is already the generic "where tapping the
dock returns" — books use it for the reader, music for `/music`; a comment
records the generalization). Tapping the docked music bar then navigates to
`/music`; when already on `/music` the §4.5 same-URL guard makes it a benign
no-op (you are already viewing the highlighted row). We deliberately do NOT
build a FULL-on-/music now-playing expand for music (Dean: mini-player is
enough; keeps the header from being pushed down — the whole point of this wave).

### 4.2 Playing-row highlight + CSS equalizer glyph (music.js + style.css)

- Track the playing id in an `init`-closure var `playingId` (seeded from
  `window.FileTube.player.currentId` on `init`, so a nav back to `/music`
  while a track plays re-highlights the right row). `playAt(i)` sets
  `playingId = queue[i].id` and calls `applyPlayingHighlight()`.
- `applyPlayingHighlight()` — pure DOM toggle, NOT a re-render: clears
  `.playing` from all `.music-song-row`, adds it to the row whose
  `data-id === playingId` (if present in the current list). Called from
  `playAt`, after every `renderSongList()`, and once at `init`.
- **Equalizer markup lives in `buildSongRowHtml`** (keeps it pure/testable):
  add `<span class="music-eq" aria-hidden="true"><i></i><i></i><i></i></span>`
  to every row. CSS hides it (`display:none`) unless the row is `.playing`,
  then shows 3 bars with a staggered `@keyframes` height animation
  (`prefers-reduced-motion` → static bars). The `.playing` row also gets an
  accent (era-token left-border/background). No emoji, no new mask.
- Advance coverage: autoplay-at-end and lock-screen next both fire the
  `setTrackNav` `onNext` handler, which calls `playAt(i+1)` — so the highlight
  follows every advance for free. (Pause/play does not move the row, correct.)

### 4.3 Collapsing / sticky drill header (music.js + style.css)

Only the **drill** branch of `render()` (album OR artist). The header is built
PURELY from the already-loaded `queue` (the drill's tracks) + the `drill`
descriptor — NO new endpoint:

- `buildDrillHeaderHtml(drill, tracks)` → the large header:
  - Album drill: big art (`/albumart/<tracks[0].id>`), album title
    (`drill.label`), artist (`tracks[0].albumArtist || tracks[0].artist`),
    a meta line `year · N tracks` (year derived from the tracks — the most
    common non-empty `year`, else omitted), a **Play** button + a **Shuffle**
    button.
  - Artist drill: representative art (`tracks[0].id`), artist name, meta
    `M albums · N tracks` (album count = distinct `album` in tracks), Play +
    Shuffle. (Art-forward even for artists; a single representative cover.)
  - Escapes every field via `escapeMusicHtml`; pure + node:test-covered.
- `buildStickyBarHtml(drill, tracks)` → the slim collapsed bar: small thumb +
  title + a play button. Rendered up-front (inside the drill header block),
  `hidden`/opacity-0 until collapsed.
- **Layout:** the drill view renders, in `#music-content`:
  `[sticky-bar][big-header][sentinel][song-list]`. The sticky bar is
  `position: sticky; top: <measured header height>` and only becomes visible
  when `.collapsed` is on the drill container. Desktop: identical mechanism,
  the big header just has more vertical room (CSS `min-height`/art size scales
  up at the wide breakpoint) — no JS branch.
- **IntersectionObserver:** a zero-height `.music-drill-sentinel` sits right
  below the big header. Observe it with `root: null` and
  `rootMargin: '-<headerH>px 0px 0px 0px'` where `headerH` is the FIXED site
  header's `offsetHeight` measured ONCE at wire time (measure, don't guess).
  When the sentinel leaves the top (`!entry.isIntersecting` with a
  `boundingClientRect.top < headerH` disambiguation for the initial callback),
  toggle `.collapsed` on the drill container → CSS reveals the sticky bar and
  hides the big header. No scroll listener, no per-frame math.
- **Lifecycle:** store the observer in an `init`-closure var
  `stickyObserver`; `disconnect()` it at the TOP of `render()` (before
  rebuilding `#music-content`) and in `destroy()`. This is the SPA-swap safety
  the gate must verify: leaving `/music` (view-root torn down) must not leak an
  observer pointed at a detached sentinel.
- **Header Play/Shuffle** reuse existing paths: Play → `playAt(0)` on the
  drill's loaded `queue`; Shuffle → the existing `shuffleBtn` logic scoped to
  `drill` (`loadSongs({sort:'random', seed, scope: drill}).then(()=>playAt(0))`).
  The sticky bar's play button = the same Play.

### 4.4 "Playing from &lt;Album&gt;" line (music.js + style.css)

- A single `#music-nowplaying` element, INSIDE `#view-root`, above
  `#music-content` (a sibling, so it survives `#music-content` re-renders).
  Add it to `music.html`.
- `deriveNowPlayingLabel(currentData)` (pure, tested): returns
  `Playing from <album>` when a music track is loaded with a non-empty
  `album`, else `''` (hidden). music.js updates it from `playAt` (via
  `currentData`/the tapped item's album) and once at `init` (from
  `player.currentId` + the loaded item — re-deriving on a nav-back).
- Tapping it drills into that album (a small, additive affordance: sets
  `drill = {type:'album', key: <albumKey>, label:<album>}` and re-renders).
  Album-keyed off the playing item's `albumKey` (the client already receives
  grouping fields on song items; if `albumKey` isn't on the item, fall back to
  a plain non-interactive label — verify the `/api/music` item shape during
  T-impl and, if absent, either surface `albumKey` on the item or make the
  line non-interactive, DISCLOSED). Hidden entirely on a music-less/idle view.
- The corner dock is too small to hold this; the lock screen already gets the
  album via `setupMediaSession`. This line is the in-app context surface.

### 4.5 Tech-debt #46 — same-URL nav no-op guard (common.js)

In `navigate(url, options)`, AFTER `deriveRouteView` (so an unknown route still
falls back to a real navigation) and BEFORE `recordScrollForCurrentState()` /
the `navGeneration` bump, add:

```
// tech-debt #46: a same-URL in-app navigation (tapping the ALREADY-active nav
// item) otherwise tears down and rebuilds #view-root -- stranding a player
// that the incoming view's init() does not re-adopt (music/read mount into
// #view-root's #player-slot) and needlessly re-fetching every other view.
// Treat "navigate to where we already are" as a no-op. popstate is unaffected
// (this is navigate()-only); a genuine reload path can still force via a flag
// if ever needed.
var here = window.location.pathname + window.location.search;
if (parsed.pathname + parsed.search === here && !opts.reload) {
  return Promise.resolve();
}
```

Rationale + scope notes for the gate:
- With play→dock the docked player lives OUTSIDE `#view-root`, so the common
  music case no longer strands on `music→music`. The guard still matters for
  the FULL-on-/read case (`read→read`) and any future FULL-on-/music, AND it
  removes a needless full re-fetch+re-init on every "tap the active tab".
- **Behavior change to weigh:** re-tapping the active nav item no longer
  refreshes/reset-scrolls that view. There is no code path that RELIES on a
  same-URL click to refresh (home's dirty-reconcile runs via
  visibility/poll + `markHomeGridDirty`, not via a Home re-click; a search/
  filter/sort produces a DIFFERENT url and is unaffected; `?play=` on /music is
  a different search string and is unaffected). This matches standard SPA
  behavior (the active link is a no-op).
- `handlePopState` keeps its own path (no guard) — back/forward to the same
  URL still reattaches/re-renders as today.

Update `books-router-nav.test.js` (or the closest router test) with: a
same-URL `navigate()` performs no fetch and no view swap; a different-URL
`navigate()` still proceeds.

## 7. Known risks / questions for the (slim) adversarial gate

Brief the seat to assume the implementer AND a QA pass missed something, demand
a concrete failure scenario per finding, and specifically probe:

1. **Player lifecycle / reparent.** Does `mountInDock()` on a FRESH load
   (state CLOSED → DOCKED, playback not yet started) correctly start playback
   once `setupForMedia`/`handleResumePlayback` resolves the async progress
   read? Does the ALAC prewarm→`loadTrack` path still dock (not FULL)? Does a
   re-tap of the SAME docked track (adopt) stay docked and not double-reparent
   or restart? Does `close()` (dock ×) still fully tear down from this new
   entry path?
2. **IntersectionObserver across the SPA swap.** Leaving `/music` mid-drill
   (view-root replaced) must `disconnect()` the observer — verify no leaked
   observer fires on a detached sentinel; verify a drill→drill (album→artist)
   re-render disconnects the old one first. Verify the measured header offset
   is read AFTER the fragment is in the live DOM (not zero).
3. **Same-URL guard blast radius.** Enumerate every same-URL `navigate()`
   caller (active nav re-click, search-submit-to-same-query, dock-tap-return,
   programmatic). Confirm none RELIES on the old tear-down-and-rebuild to
   refresh state; confirm home dirty-reconcile is untouched; confirm popstate
   is unaffected.
4. **Highlight correctness.** After an autoplay-at-end advance and a
   lock-screen next, does the highlighted row match `player.currentId`? On a
   nav back to `/music` while playing, is the right row highlighted from
   `player.currentId` seeding? Is the eq glyph an emoji anywhere (it must not
   be)?
5. **`albumKey` on song items** (§4.4) — if the `/api/music` item shape omits
   it, the "Playing from" drill-in silently breaks; verify the shape and the
   fallback.
