# v1.20.0 "Subscribe button — real subscriptions from downloads"

## Goal

Turn the currently-cosmetic watch-page "Subscribe" button into a real,
one-tap subscription toggle wired to the existing yt-dlp subscription system,
driven by each file's own captured creator/channel identity — so after a
yt-dlp download (subscription-driven **or** one-shot) a user can subscribe to
that creator directly from the watch page, without visiting Settings/
Subscriptions, and see that creator's downloads surfaced as its own browsable
Playlist.

## Scope

- **FR-1**: The watch page's `#subscribe-btn-mock` becomes a real toggle.
  Tapping it while NOT subscribed opens a small, compact options modal
  (pre-filled defaults, read-only channel identity) that on confirm creates a
  real subscription; tapping it while subscribed is a direct one-tap
  unsubscribe. Button state always reflects the live subscription list.
- **FR-2**: Capture a file's channel URL + display name at download time
  (both the subscription-poll download path and the one-shot download path,
  going forward), persist it per downloaded item, use it to resolve the
  watch page's displayed creator name, and provide a pure, canonical
  channel-identity matcher used by FR-1/FR-3.
- **FR-3**: Hide the Subscribe button entirely (not disable/grey it) on any
  file with no resolvable channel identity, and whenever the yt-dlp module is
  disabled.
- **FR-4**: Surface each subscribed channel as its own browsable Playlist,
  reusing the confirmed-physical per-channel download subfolder and the
  existing recursive `root=` video filter — no synthetic-folder/db.folders
  changes.
- **FR-5**: Lower `DEFAULT_MAX_VIDEOS` (lib/ytdlp/config.js) from 3 to 2 for
  new subscriptions only, mirroring the v1.18.0 25→3 precedent exactly
  (`0` = unlimited, env override, and already-persisted subscriptions all
  unaffected).

## Out of scope

- Auto-subscribe on download or any post-download subscribe prompt (Dean:
  button-toggle-only trigger).
- Any change to `validateChannelUrl`'s host allowlist, shell-metacharacter
  rejection, or any other existing security guard (FR-1/FR-2 route
  server-derived channel URLs through the SAME, unmodified validator — no new
  bypass, no weakening).
- Card-level Subscribe affordance on the home grid (watch-page only, mirrors
  the FR-3 (download button) scope precedent from v1.19.0).
- Retroactively backfilling `channelUrl` for files downloaded before this
  ships (explicit, documented limitation — see FR-2's Decision log entry).
- Any change to the existing add/edit/pause subscription flows on the
  `/subscriptions` page itself, beyond reusing their option-control building
  blocks in the new watch-page modal.
- Any change to `db.folders`/`folderSettings` schema or the existing
  single top-level synthetic "Downloads" folder splice/rename/order logic
  (FR-4 is additive on top of it, not a replacement).

## Constraints

- NO REGRESSIONS to the existing yt-dlp CRUD routes (`GET/POST
  /api/subscriptions`, `PATCH`/`DELETE /api/subscriptions/:id`), the one-shot
  download route, or the disabled-module byte-identical guarantee.
- The channel URL that ever reaches a yt-dlp spawn — whether user-typed
  (existing `/subscriptions` add form) or file-derived (new, FR-1/FR-2) —
  MUST pass through `store.validateSubscriptionInput` → `url.validateChannelUrl`
  unchanged. No new code path may construct a subscription record or spawn
  argv from an unvalidated string.
- Channel metadata captured from yt-dlp's own output (FR-2) is untrusted
  input the same way user input is: it must be validated/bounded before
  persistence or use, never trusted blindly (SECURITY touchpoint for the
  two-reviewer gate — see below).
- The `db.folders`-vs-`folderSettings` synthetic-root invariant (a prior
  data-loss-class bug) is unchanged: FR-4 must never write a per-channel
  entry into `db.folders`.
- Follow `docs/CONTRIBUTING.md`: vanilla DOM (no framework/bundler), CommonJS
  server, 2-space/semicolons/single-quotes, `textContent` over `innerHTML`
  for new dynamic strings (the modal's creator name / channel URL display
  especially), `node:test` for new/regression tests, lint 0, no new runtime
  dependencies.
- Preserve the era-theme system (CSS custom properties). The new modal must
  use existing tokens and the corrected select-sizing pattern from the
  v1.19.0 FR-1 fix (never reintroduce the oversized-mobile-select bug), and
  must reuse the v1.17.0 FR-6 full-teardown backdrop-dismiss pattern (never
  reintroduce a stuck-overlay bug).
- Node.js 22 LTS; no new npm dependencies; no headless-browser/E2E
  infrastructure exists (`docs/RELIABILITY.md`) — visual/on-device claims are
  bounded accordingly, with Dean's on-device pass as the documented arbiter.

## Design

### Approach

The feature is built enabler-first. **FR-2** captures each downloaded file's
channel identity at yt-dlp download time, bridges it across the
download→scan gap onto `db.metadata[id]`, and provides a pure client-side
matcher; everything else (**FR-1** toggle+modal, **FR-3** hide, **FR-4**
per-channel playlist) consumes what FR-2 produces. No existing security guard
is modified: every channel URL that can reach a spawn — captured from yt-dlp
output or POSTed from the new modal — still passes through the unmodified
`url.validateChannelUrl` (and, for subscription creation,
`store.validateSubscriptionInput`).

Server-side work lands in the existing `lib/ytdlp/*` module plus a narrow,
layering-safe touch in `server.js`'s scan. Client-side work reuses the option
builders and modal primitives already present in `public/js/common.js` (the
one-off download modal shares `.oneoff-modal-*` CSS — which already carries
the v1.17.0 full-teardown and v1.19.0 select-sizing fixes — and
`buildOneOffSelect`/`ONEOFF_*`/`reduceOneOffFiletypeOptions`), so the new
watch-page modal takes on **no** dependency on the gated, lazy-loaded
`/js/subscriptions.js`. New pure helpers (the channel-identity matcher, the
show/hide predicate, the channel-meta parser/validator) are `node:test`-covered.

The capture mechanism is a single `--print after_move:` template added to the
DOWNLOAD spawn's argv — one line per downloaded video, parsed from the SAME
streamed stdout `makeLineSplitter` the progress path already uses (no second
spawn, no `--write-info-json`, no extra network round-trip). The
`after_move:` WHEN-prefix is load-bearing: a bare `--print` implies
`--simulate` (no download), whereas `after_move:` prints after the file is
moved to its final path and leaves the download intact.

### FR-2 — capture + store + match channel identity (the enabler)

**Capture template (both paths, `lib/ytdlp/args.js`).** In
`buildYtdlpDownloadArgs` add, unconditionally (audio and video, subscription
and one-shot), a fixed-literal print flag:

```text
--print after_move:FTCHMETA\t%(id)s\t%(channel_url)s\t%(channel_id)s\t%(uploader_url)s\t%(channel)s
```

`FTCHMETA` is a fixed sentinel (never interpolated with any per-sub/per-video
value, so it can never become an injection vector — same posture as
`SHORTS_MATCH_FILTER`/`VIDEO_FORMAT_SORT`). Missing fields render `NA`. The
flag is emitted before the `--`/positional targets. `channel_url` is yt-dlp's
canonical `/channel/UC…` form; `uploader_url` is the `/@handle` form —
capturing both maximizes match coverage against a subscription of either
shape (see the matcher below).

**Capture parse (`lib/ytdlp/run.js`).** Add a pure
`parseChannelMetaLine(line)` (exported for `node:test`): returns
`{ videoId, channelUrl, channelId, uploaderUrl, channelName } | null`,
matching only lines with the `FTCHMETA\t` sentinel and splitting on `\t`
(treating `NA`/empty fields as absent). In `spawnYtdlpDownload`, the existing
per-line handler first tries `parseChannelMetaLine`; a match is pushed to a
**bounded** `capturedMeta` array (hard cap, e.g. `MAX_CAPTURED_META = 1000`,
exported like `STDERR_TAIL_LIMIT`) and NOT forwarded to `onProgress`; a
non-match falls through to `parseProgressLine`/`onProgress` exactly as today.
`spawnYtdlpDownload`/`runDownload` return `channelMeta: capturedMeta` on the
result object. SF3 is preserved: `result.stdout` stays `''`; the only new
retained state is a small, capped, structured array (documented).

**Validate before persistence (`lib/ytdlp/store.js` + reuse `url.js`).**
Captured strings are UNTRUSTED. A new
`store.sanitizeCapturedChannelMeta(raw)` (pure, exported, `node:test`-covered):

- `channelUrl` ← first of `channel_url`, else `uploader_url`, that passes
  `url.validateChannelUrl` (returns the normalized value); if neither passes,
  the item gets **no** channel identity (dropped — falls to FR-3's hide rule,
  AC11).
- `channelHandleUrl` ← `uploader_url` if it independently passes
  `validateChannelUrl` and differs from `channelUrl` (broadens matching only).
- `channelId` ← `channel_id` only if it matches `^UC[A-Za-z0-9_-]{22}$`, else
  dropped.
- `channelName` ← `channel`, control-chars stripped and length-bounded
  (reuse the `MAX_STATUS_LENGTH`-style cap; e.g. 200 chars), else absent.
- `videoId` ← must pass `url.isSafeVideoId`, else the whole entry is dropped
  (no safe join key).

**Persist the capture map (`lib/ytdlp/store.js`).** A durable, keyed map
bridges download→scan: `db.ytdlp.downloadMeta = { [videoId]: { channelUrl,
channelHandleUrl?, channelId?, channelName?, capturedAt } }`.
`store.recordDownloadChannelMeta(deps, entry)` runs the sanitizer, drops
invalid entries, writes through the serialized `updateDatabase`, and enforces
a FIFO cap (e.g. `MAX_DOWNLOAD_META = 5000`, evicting oldest by `capturedAt`)
so the map can never grow unbounded even if scans somehow never consume it.
`ensureYtdlp` backfills `downloadMeta: {}` alongside the existing namespace
fields (disabled path stays byte-identical — nothing calls in).

**Wire capture into both orchestrators (`lib/ytdlp/index.js`).** After a
successful `run.runDownload` in `runSubscriptionCycle` and `runOneShot`, for
each entry in `downloadResult.channelMeta` call
`store.recordDownloadChannelMeta`. For the subscription path, if capture
produced nothing for a survivor, fall back to the sub's own already-validated
`sub.channelUrl` + `sub.name` (both already known and trusted) keyed by the
survivor id — so a subscription download always gets at least the sub's
identity. The one-shot path has no such fallback (no channel known up front);
a capture miss simply leaves the item without identity.

**Bridge onto `db.metadata` (`server.js` scan + a `lib/ytdlp` helper).** The
scan already computes `id = getMediaId(filePath)` and, for files under
`ytdlpDownloadRoots`, runs `cleanDisplayTitle`. Add a sibling pure helper
`extractYtdlpVideoId(baseName)` (mirrors `cleanDisplayTitle`'s regex: the
trailing space-plus-`[<11-char id>]` bracket → the id, else `null`). Inside the
existing Phase-2 `updateDatabase(fresh => …)` mutator (which already mutates
each item for `rootFolder`), for each new/updated item rooted under a
download root with an extractable videoId, call a new module helper
`ytdlp.consumeDownloadChannelMeta(fresh, videoId)` — it reads (and validates)
`fresh.ytdlp.downloadMeta[videoId]`, **deletes** that key (bounded growth:
entries live only until first index), and returns
`{ channelUrl, channelHandleUrl?, channelId?, channelName? } | null`. The
returned fields are assigned onto the item. Keeping all `db.ytdlp` structural
knowledge inside the module preserves layering (server.js already imports
`ytdlp` for `extraScanRoots`). Reuse-fast-path (unchanged) items keep their
already-persisted identity; pre-feature downloads have no map entry → no
identity (documented limitation AC12).

**Creator display precedence (`public/js/common.js`, `resolveChannelName`).**
Extend to: captured `item.channelName` (non-empty) → existing chain (mapped
folder friendly-name → `item.artist` → `item.folderName` → `'Library'`).
`channelName` ranks FIRST **only when present**, so non-yt-dlp files are
completely unchanged (AC13); a downloaded file no longer shows the generic
"Downloads" root name as its creator. Both `/api/videos` and
`/api/videos/:id` already spread `...item`, so the new fields reach the home
grid and watch page with no route mapping change.

**Matcher (`public/js/common.js`, pure, `node:test`-covered).** Button state
is a client concern, so the matcher lives client-side (the server never needs
to match a file to a sub). Two helpers:

- `canonicalizeChannelUrl(url)` → a canonical key or `null`. Parse; lowercase
  host; then: `/channel/<UC…>` → `channel:<UCid>` (case preserved — ids are
  case-sensitive); `/@handle` → `handle:<lowercased>`; `/user/<name>` →
  `user:<lowercased>`; `/c/<name>` → `c:<lowercased>`; a `youtu.be`/`/watch`
  video URL → `null` (a video URL is not a channel identity). Conservative:
  anything unrecognized → `null`.
- `channelIdentityMatches(fileIdentity, subUrl)` → boolean. Build the file's
  key-SET from `{ canonicalizeChannelUrl(channelUrl),
  'channel:'+channelId (when present), canonicalizeChannelUrl(channelHandleUrl)
  }` (dropping nulls); match iff `canonicalizeChannelUrl(subUrl)` is in that
  set. This is what lets a file whose `channel_url` is `/channel/UC…` match an
  existing subscription added as `/@handle` (via the shared handle key from
  `uploader_url`) or as `/channel/UC…` (via the channel-id key) — and, being
  set-membership on canonical keys, it never false-matches two forms that
  can't be proven equal (AC14).

A `resolveFileChannelIdentity(item)` helper returns
`{ channelUrl, channelId, channelHandleUrl } | null` (null when no
`channelUrl`), single-sourcing what FR-1/FR-3 consume.

### FR-1 — subscribe toggle + compact options modal

**State + wiring (`public/js/watch.js`).** `populateMetadata` computes
`identity = resolveFileChannelIdentity(mediaData)` and, when the button is
shown (FR-3), fetches `GET /api/subscriptions`, finds the matching sub via
`channelIdentityMatches(identity, sub.channelUrl)`, and renders "Subscribe"
vs. "Subscribed" (AC1/AC2). The matched `sub.id` (if any) is held for the
unsubscribe path. The button element is `#subscribe-btn-mock` (id kept).

**Subscribe (not subscribed) → compact modal (`public/js/common.js`).** A new
`buildSubscribeModal(doc, opts, handlers)` mirrors `buildOneOffModal`'s
structure and REUSES its primitives: `.oneoff-modal-backdrop`/`.oneoff-modal`
CSS (v1.17.0 full-teardown + v1.19.0 `flex:0 0 auto` select-sizing come for
free), `buildOneOffSelect`, `ONEOFF_FORMAT_OPTIONS`, `ONEOFF_QUALITY_OPTIONS`,
`ONEOFF_FILETYPE_OPTIONS`, `reduceOneOffFiletypeOptions`/
`repopulateOneOffFiletypeSelect`. Contents:

- READ-ONLY channel identity: `channelName` + `channelUrl`, both via
  `textContent` (AC3/AC30) — never an editable field.
- type (`format`) select, pre-filled from `mediaData.type`
  (audio/video); quality select pre-filled `'best'`; filetype select
  (format-coupled via the shared reducer, AC7); a "download last N" number
  input pre-filled the FR-5 default; a skip-Shorts checkbox (default off).
- Subscribe + Cancel buttons.

**maxVideos default source (AC26).** To avoid a second hardcoded literal, the
gated `GET /api/subscriptions/health` body gains `defaultMaxVideos:
config.DEFAULT_MAX_VIDEOS` (single server-side source, FR-5's constant). The
watch page already needs a health probe for FR-3; it reads `defaultMaxVideos`
from that same response (fallback `2` only if absent).

**Confirm → create.** POST `/api/subscriptions` with
`{ channelUrl: identity.channelUrl, format, quality, maxVideos, skipShorts,
filetype }` through the EXISTING `store.validateSubscriptionInput` →
`url.validateChannelUrl` (no bypass, AC4). The `channelUrl` sent is the file's
stored, already-validated canonical value; the server re-validates regardless.
On success: full-teardown the modal, flip the button to "Subscribed", record
the new sub id. Cancel / backdrop / `[x]` / Esc → the shared full-teardown
(`backdrop.remove()` + null state), no POST (AC5).

**Unsubscribe (subscribed) → one-tap.** DELETE
`/api/subscriptions/:id` for the matched sub id; flip back to "Subscribe"
(AC6). No modal (Dean's explicit direction; low blast radius — DELETE only
stops future polling, never deletes files).

Both surfaces read the same `GET /api/subscriptions` list, so the watch button
and the Subscriptions tab never diverge.

### FR-3 — hide when no channel / module disabled

Pure `shouldShowSubscribeButton({ moduleEnabled, channelIdentity })` in
`common.js` (`node:test`-covered): `true` iff `moduleEnabled === true` AND
`channelIdentity` is non-null (AC15). `moduleEnabled` comes from the
`GET /api/subscriptions/health` probe (200 → enabled; 404 → disabled, the
existing capability signal). `watch.js` removes `#subscribe-btn-mock` from the
DOM when the predicate is false (`.remove()` — absent, not greyed); the SPA
re-fetches watch.html on each mount, so a later load with a resolvable channel
re-provides and shows it. With the module disabled the probe 404s → button
absent regardless of metadata, preserving the disabled-module byte-identical
contract (AC16/AC17/AC27) — no new always-present DOM, no new reachable route.

### FR-4 — each subscribed channel = a playlist

Confirmed physical per-channel subfolders (`args.resolveChannelDir`) +
`GET /api/videos?root=` pure path-prefix filter (`underFolder`) mean **no new
backend filtering primitive** is needed. Two additive pieces:

- **Expose the resolved dir.** The gated `GET /api/subscriptions` handler
  (`lib/ytdlp/index.js`, which has `config` in scope) enriches each returned
  sub with `channelDir: args.resolveChannelDir(config, sub)` (wrapped in
  try/catch — a sub that fails confinement simply omits the field). This is
  read-only and computed per-request; `db.folders` is never written (AC21).
  `store.listSubscriptions` stays pure/unchanged; enrichment happens in the
  route so `config` layering is respected.
- **Surface the link.** Each subscription renders a playlist link to
  `/?root=<encodeURIComponent(channelDir)>` — reusing the exact
  `/?root=<path>` pattern the Playlists sheet and sidebar already use — so it
  confines to that channel's own subfolder (AC19/AC20). Surface it in BOTH the
  Subscriptions tab rows (`createSubscriptionRow`, `lib/ytdlp/client/
  subscriptions.js`, an anchor built via `textContent`/`escapeAttr`) and,
  optionally, the mobile Playlists sheet. Because the Playlists sheet
  (`common.js`) is driven by `GET /api/config` (`db.folders`/`folderSettings`)
  and must NOT gain synthetic per-channel entries there, the per-channel
  playlist links are sourced from `GET /api/subscriptions` (module-gated),
  rendered as a distinct "Subscriptions" group appended to the sheet only when
  the module is enabled — never merged into the `db.folders` list. This keeps
  the synthetic-root invariant provably intact (no per-channel entry ever
  reaches `db.folders`/`folderSettings`).

### FR-5 — default download count 3 → 2

`lib/ytdlp/config.js`: `DEFAULT_MAX_VIDEOS` `3` → `2`, and update its comment
(it currently also mis-cites the prior change as v1.17.0 — correct to the
v1.18.0 25→3 precedent while here). New subscriptions only; `0` = unlimited,
the `FILETUBE_YTDLP_MAX_VIDEOS` env override, and already-persisted
subscriptions are all unaffected (`parseMaxVideos`/`subMaxVideos ??
config.maxVideos` are untouched). The FR-1 modal's pre-fill reads this same
constant via `GET /api/subscriptions/health`'s `defaultMaxVideos` (AC26), so
there is no second literal to drift.

### Component changes

- **`lib/ytdlp/args.js`**: add the fixed `--print after_move:FTCHMETA…`
  template to `buildYtdlpDownloadArgs` (both paths). SPAWN-ARGS surface →

- **`lib/ytdlp/run.js`**: `parseChannelMetaLine` (pure); capture bounded
  `channelMeta` in `spawnYtdlpDownload`, return it on the result; `MAX_CAPTURED_META`.
- **`lib/ytdlp/store.js`**: `sanitizeCapturedChannelMeta`,
  `recordDownloadChannelMeta`, `consumeDownloadChannelMeta` (or a paired
  reader), `downloadMeta` backfill in `ensureYtdlp`, `MAX_DOWNLOAD_META`.
  UNTRUSTED-INPUT validation surface → two-reviewer gate.
- **`lib/ytdlp/index.js`**: persist capture in `runSubscriptionCycle`/
  `runOneShot`; enrich `GET /api/subscriptions` with `channelDir`; add
  `defaultMaxVideos` to `/api/subscriptions/health`.
- **`server.js`**: `extractYtdlpVideoId` helper; call
  `ytdlp.consumeDownloadChannelMeta` in the Phase-2 scan mutator to attach

- **`public/js/common.js`**: `canonicalizeChannelUrl`,
  `channelIdentityMatches`, `resolveFileChannelIdentity`,
  `shouldShowSubscribeButton`, `resolveChannelName` precedence,
  `buildSubscribeModal`.
- **`public/js/watch.js`**: subscribe button wiring (state, modal open/
  confirm/teardown, unsubscribe, FR-3 show/hide). WATCH → SUBSCRIPTION

- **`public/watch.html`**: keep `#subscribe-btn-mock` (default `hidden`).
- **`public/css/style.css`**: reuse `.oneoff-modal-*`; add only a small

  (era-theme tokens only, AC28).
- **`lib/ytdlp/client/subscriptions.js`**: render the per-sub playlist link
  from `sub.channelDir`.
- **`lib/ytdlp/config.js`**: `DEFAULT_MAX_VIDEOS` 3 → 2.

### Data model changes

- New `db.ytdlp.downloadMeta` map (`{ [videoId]: { channelUrl,
  channelHandleUrl?, channelId?, channelName?, capturedAt } }`), FIFO-capped,
  consumed-and-pruned at scan time. Backfilled by `ensureYtdlp`.
- New optional `db.metadata[id]` fields on yt-dlp downloads: `channelUrl`,
  `channelName`, `channelId`, `channelHandleUrl`. Absent on all non-yt-dlp and
  pre-feature items (documented limitation, AC12). No schema migration; no
  change to `db.folders`/`folderSettings`.

### API changes

- `GET /api/subscriptions`: each sub gains a computed `channelDir` (read-only;
  no request/DB shape otherwise changed).
- `GET /api/subscriptions/health`: body gains `defaultMaxVideos`.
- `GET /api/videos` / `GET /api/videos/:id`: no code change; new item fields
  flow through the existing `...item` spread.
- No change to `POST /api/subscriptions`, `DELETE /api/subscriptions/:id`,
  `POST /api/ytdlp/download`, or any validator.

### Alternatives considered

- **Recover channel URL from embedded ffprobe tags** instead of download-time
  capture. Rejected: the `comment` tag carries the video's `webpage_url`, not
  a channel URL (PM finding, confirmed) — no reliable channel identity exists
  in the whitelisted tags.
- **A JSON sidecar file per download dir** for the bridge map instead of
  `db.ytdlp.downloadMeta`. Rejected: it would introduce a second persistence
  mechanism outside the serialized `updateDatabase` writer, with its own
  concurrency/cleanup story; the db namespace is already the module's durable,
  lock-safe store.
- **A second yt-dlp spawn (`--dump-json`/`--print` simulate) to fetch channel
  metadata.** Rejected: an extra network round-trip per download for data the
  download pass can emit inline via `after_move:` on the existing spawn.
- **URL-shape-only matching** (no `channel_id`). Rejected: cannot match a
  `/@handle` sub against a `/channel/UC…` file; capturing `channel_id` +
  `uploader_url` closes that gap conservatively (AC14).
- **Reuse `/js/subscriptions.js`'s option builders directly.** Rejected: that
  module is gated/lazy-loaded only on `/subscriptions`; `common.js` already
  carries an equivalent set loaded on every page, avoiding a circular/load
  dependency.

### Risks and mitigations

- **Risk**: `--print` implicitly enabling `--simulate` (no download). →
  **Mitigation**: the `after_move:` WHEN-prefix; add a regression test
  asserting the download argv still downloads (arg-shape test + the existing
  download integration harness).
- **Risk**: hostile/malformed channel metadata from yt-dlp output reaching
  persistence or a spawn. → **Mitigation**: `sanitizeCapturedChannelMeta` runs
  every field through `validateChannelUrl`/`isSafeVideoId`/charset bounds
  before storage; the POST path re-validates; two-reviewer gate on FR-2.
- **Risk**: unbounded `downloadMeta` growth. → **Mitigation**: consumed +
  pruned at first index; FIFO hard cap as backstop.
- **Risk**: duplicate subscriptions when a channel was previously subscribed
  under a different URL shape. → **Mitigation**: the matcher shows
  "Subscribed" (blocking a duplicate create) whenever any canonical key
  matches; documented that the store's own id-dedup is unchanged.
- **Risk**: FR-4 links leaking another channel's or the whole Downloads root's
  videos. → **Mitigation**: `channelDir` is `resolveChannelDir`'s confined
  per-channel path; `root=` is an exact-prefix match; two-reviewer gate
  re-proves the `db.folders` invariant.
- **Risk**: `after_move:` not firing for an edge container/postprocess. →
  **Mitigation**: capture miss degrades gracefully (button hidden, or the
  subscription-path fallback to `sub.channelUrl`), never an error.

### Performance impact

No expected impact on `docs/RELIABILITY.md` budgets. Capture is one extra
stdout line per downloaded video parsed by the existing line-splitter; the
scan adds an O(1) map lookup + optional field assignment per new download-
rooted item; the watch page adds one `GET /api/subscriptions` fetch on load
for files that show the button (module-enabled, resolvable channel only). No
new spawns, no new always-on polling, no change to the disabled path.

### Security preservation

- Every captured channel URL is validated by the UNMODIFIED
  `url.validateChannelUrl` before it is persisted, matched, displayed as a
  subscribe target, or POSTed.
- The new modal creates subscriptions ONLY via the existing
  `POST /api/subscriptions` → `store.validateSubscriptionInput` →
  `url.validateChannelUrl` path; no new code constructs a subscription record
  or spawn argv from an unvalidated string.
- The `--print` template is a fixed literal (sentinel + `%(field)s` only),
  never interpolated with a per-sub/per-video value.
- FR-4 computes `channelDir` from the confined `resolveChannelDir`; no
  per-channel entry is ever written to `db.folders`/`folderSettings`.
- Disabled-module byte-identical contract preserved: the button/modal require
  the 200 health probe, and every new backing behavior lives behind the
  existing `isEnabled` route gate.

### Security touchpoints (explicit re-confirmation)

- **Capture-injection (HIGH, remediated):** `CHANNEL_META_PRINT_TEMPLATE` now
  emits a single JSON object (`%(.{id,channel_url,channel_id,uploader_url,channel})j`)
  instead of tab-delimited interpolation, so an embedded newline in a
  hostile `channel` display name can no longer forge a second capture line;
  `parseChannelMetaLine` additionally only ever runs against stdout (never
  stderr). Both fixes are locked by `test/unit/ytdlp-run.test.js`'s
  forged-newline regression tests.
- **No validator bypass:** every code path that can reach a yt-dlp spawn or
  persist a channel URL — captured metadata (`sanitizeCapturedChannelMeta`/
  `consumeDownloadChannelMeta`, both re-run `url.validateChannelUrl`) and the
  new watch-page modal's `POST /api/subscriptions` — routes through the
  SAME, unmodified `store.validateSubscriptionInput`/`url.validateChannelUrl`.
  Confirmed by direct code read; no new/parallel validator exists anywhere in
  the diff.
- **`db.folders`/`folderSettings` invariant:** `enrichSubscriptionWithChannelDir`
  is a per-request, read-only, in-memory computation that never calls
  `updateDatabase`; regression-locked by
  `ytdlp-channel-dir-playlist.test.js`'s AC21 test.
- **XSS-safety:** every new dynamic string (modal identity block, error text,
  per-sub playlist link, playlist link label) is rendered via
  `textContent`/`createElement`+`.href` only; confirmed by direct code read
  plus a static-source regression guard test for `buildSubscribeModal`.
