# v1.19.0 "UI polish"

## Goal

Polish four rough edges Dean has actually hit using FileTube on his iPhone and
desktop — an oversized/"brutal" one-off download modal, a cramped and (per the
roadmap) janky `/subscriptions` page, no way to pull an already-hosted file
down to a device, and a synthetic library folder that looks second-class in
Setup — closing out the "UI polish" section of `ROADMAP.md` with small,
low-risk, mostly-additive fixes (no shell/router/player rearchitecture).

## Scope

Four functional requirements, one branch (`feature/v1.19-ui-polish`, off
`main` at `v1.18.0` — confirmed via `package.json`'s `"version": "1.18.0"`).

### FR-1 — One-off download modal polish (mobile "brutal"/oversized selects)

**Root cause, confirmed in code.** `public/css/style.css`'s base rule for the
modal's three `<select>`s:

```css
.oneoff-modal-row select {
  flex: 1 1 150px;
  min-width: 150px;
  ...
}
```

(style.css ~1910-1920) sets `flex-basis: 150px` — a sensible *width* hint when
`.oneoff-modal-row` is `flex-direction: row` (the desktop/default layout,
`flex-wrap: wrap`). The `max-width: 768px` mobile block then does:

```css
.oneoff-modal-row { flex-direction: column; }
.oneoff-modal-row select { width: 100%; }
```

(style.css ~2429-2436) — switching the row to a **column** flex direction so
the three selects stack, but it never resets the inherited `flex: 1 1 150px`
shorthand. Under `flex-direction: column`, the main axis is now **vertical**,
so that same `flex-basis: 150px` is reinterpreted as a **minimum height**, and
the still-active `flex-grow: 1` lets each select **stretch to fill whatever
vertical space `.oneoff-modal-row`'s box has to give** — three flex items each
independently growing tall. On a ~650-700px-tall phone viewport this produces
exactly Dean's screenshot: format/quality/filetype rendering as ~150px+ tall
boxes, each roughly a quarter of the screen, instead of a normal single-line
native control. This is a **width-intent-reinterpreted-as-height-intent** bug
under the direction flip, not a missing style altogether — `buildOneOffModal`
(`public/js/common.js` ~933-1022) itself is unaffected; this is CSS-only.

**Resolved fork — target select height (no open question).** The Setup page's
own `<select>` styling is the right visual bar:

```css
.setup-select {
  padding: 10px 12px;
  border: 1px solid var(--border-dark);
  border-radius: var(--radius);
  font-size: 13px;
  ...
}
```

(style.css ~1528-1536), which renders as a normal, single-line, ~36-40px-tall
native control. **Default: the one-off modal's three selects must render at
that same normal, content-driven, single-line height on mobile — not stretched
via `flex-grow` to fill the modal's vertical space** — while keeping the
existing `.oneoff-modal-row select`'s 44px comfortable-tap-target minimum
(style.css ~2437-2441, part of the same mobile block) as the tap-target floor,
not a growth target. The exact CSS mechanism (resetting `flex`/`flex-basis`,
or a more targeted mobile override) is Design's call — the requirement is the
visible outcome, not the rule.

Everything else about the modal must be preserved: the URL input, the three
selects' existing options/behavior (including `repopulateOneOffFiletypeSelect`
on format change), the Download button, the v1.15.1 max-height/scroll modal
container, and the v1.17.0 FR-6 backdrop-dismiss teardown fix (full DOM
removal on every dismiss path). This is a CSS-scoped fix.

### FR-3 — Download-to-device button (new)

A "Download" affordance on the watch page that triggers a real browser
download of the current item's **original, already-served file** — reusing
the existing id-based `GET /video/:id` route (server.js ~2286-2361), never a
new filesystem-path-taking endpoint.

**Confirmed current behavior, grounded in code:**
- `GET /video/:id` looks up `db.metadata[req.params.id]` and serves
  `item.filePath` directly UNLESS `item.needsTranscode` is true, in which case
  it serves the **transcoded MP4** (or a live-transcode stream via `?live=1`)
  instead of the original — so a naive reuse of this route for "download the
  file" would sometimes hand back a transcode, not the source Dean explicitly
  wants ("download the ORIGINAL source file, not the transcode").
- No route in `server.js` sets `Content-Disposition` today — this is
  genuinely new server-side work, not a one-line addition.
- `GET /api/videos/:id` (server.js ~2037) already spreads the full metadata
  item (including `title`/`ext`/`filePath`) to the client, and
  `public/js/watch.js` (~233) already renders `mediaData.filePath` (the raw
  absolute path) directly into the existing watch page (file-type text +
  delete-confirmation modal). **The raw path is therefore already visible in
  the existing UI** — FR-3 introduces no new path-disclosure surface; the
  requirement is that the actual *file bytes* download stays keyed by `id`,
  never a client-supplied path, exactly like playback already is.

**Resolved fork — mechanism (non-blocking, PE confirms exact shape).**
Default: extend the **existing** `GET /video/:id` route with an explicit
download intent (e.g. a `?download=1` query param or equivalent), which (a)
**always serves `item.filePath`** (the original), bypassing the
`needsTranscode` branch entirely, and (b) sets `Content-Disposition:
attachment; filename="<clean title>.<ext>"` (RFC 5987 / `filename*=UTF-8''`
encoding for non-ASCII titles), so the browser saves rather than plays. Same
id→metadata lookup as playback; no new confinement logic needed since
`item.filePath` is the same server-resolved path playback already streams
from. A sibling, equally-acceptable shape (a distinct
`GET /video/:id/download`-style route sharing the same lookup) is Design's
call — the constraint is "same id-based lookup, one shared code path with
playback's confinement, never a second path-resolution mechanism."

**Resolved fork — scope (non-blocking).** Default: **watch-page only** for
this round (a clear "Download" button near the existing Delete/star-rating
row in `.watch-actions`, `public/watch.html` ~148-163, reusing the
already-shipped `.icon-download` glyph from v1.17.0 FR-7). A card-level
download affordance (`public/js/main.js`) is deferred as a fast-follow, not
in scope here, per the brief's own stated default.

**Filename + audio/video/transcoded distinction:**
- Filename: the item's display `title` (already yt-dlp-cleaned via
  `cleanDisplayTitle` where applicable) + its **original** `ext` — never the
  raw on-disk `name` (which may carry the yt-dlp `--restrict-filenames`/`[id]`
  suffix or ffmpeg-transcode naming).
- Must download the **original** file (`item.filePath`) even when
  `item.needsTranscode` is true and a cached transcode exists at
  `transcodedPath(item.id)` — the transcode is a browser-playability sidecar,
  not the canonical file.
- Works for both audio and video items (the same route, same `item.filePath`,
  regardless of `item.type`).
- iOS Safari note: `Content-Disposition: attachment` plus the anchor
  `download` attribute is the standard same-origin download path on modern
  iOS Safari (13+); Dean's on-device iPhone pass is the arbiter for the exact
  save/share-sheet UX, not fully proxyable by `node:test`.

**Security-sensitive touchpoint — flagged for the two-reviewer gate.** This
touches the file-serving route. Non-negotiable: the download path must stay
strictly `id → db.metadata[id].filePath` (never accept or interpolate a
client-supplied path/filename into the filesystem lookup), must not weaken
any existing `/video/:id` behavior (Range support, 404s, the transcode
branch's existing behavior for ordinary playback), and the new
`Content-Disposition` filename must be safely encoded (no header injection
from a title containing CR/LF or quote characters).

### FR-4 — Synthetic "Downloads" folder: renamable + reorderable in Setup

**Grounded finding: this is ALREADY FULLY IMPLEMENTED AND TESTED — confirmed
by reading both the server and client code, not assumed from the roadmap
description.** `ROADMAP.md`'s "UI polish" section still lists this as an open
item ("can't be renamed or reordered... like real folders"), but:

- `GET /api/config` (server.js ~1596-1625, "FR-G part 2" / "Item 3 (v1.13.0,
  order persistence)") already merges the yt-dlp module's download directory
  into its **response** as a synthetic folder entry — spliced in at a
  **persisted display index** (`folderSettings[root].order`, defaulting to
  append-last) — and defaults its name to `'Downloads'` when no rename has
  ever been persisted.
- `POST /api/config` (server.js ~1628-1788, `FIX-2`) already accepts a rename
  (`folderSettings[resolvedRoot].name`) and a reorder (derived `order` from
  the synthetic root's position in the submitted `folders` array) for the
  synthetic root, persisting **both into `folderSettings` only** — the
  synthetic root is explicitly, unconditionally excluded from ever entering
  `db.folders`, even when a normal settings-page save round-trips `GET`'s
  synthetic-merged response straight back into `POST` (the exact "client
  doesn't know it's special" scenario this feature needs to survive).
- `public/js/setup.js`'s `renderFolders()`/`loadConfig()` (lines 38-96) treat
  `configuredFolders` (== `data.folders`, which already includes the spliced
  synthetic root) generically — **every** row, synthetic or real, gets the
  same rename `<input>`, "Hide from home"/"Hide from sidebar" checkboxes,
  up/down reorder buttons, and native drag-and-drop
  (`common.js`'s shared `moveArrayItem`/`computeDropIndex` helpers) — "zero
  client change" was the explicit v1.13.0 design note, and reading the
  current code confirms it is still true.
- `test/integration/ytdlp-synthetic-folder.test.js` already exercises exactly
  this contract end-to-end: rename-only persistence (AC44), reorder
  persistence across a fresh GET (AC8), rename+reorder together (AC9),
  never-in-`db.folders` after either (AC10), an out-of-range stored order
  clamping safely (AC10b), the disabled-module no-op (AC11/AC46), and the
  scan/mount-loss guard's independence from any of this (AC12/AC47).

**Conclusion for this round:** FR-4's core ask — a persisted friendly name and
a persisted position, surfaced through Setup's normal folder-management UI,
without ever entering `db.folders` — is **already shipped and tested**. This
plan does not need to (and must not) rebuild it. `ROADMAP.md` should be
corrected to move this line to "Shipped" as a housekeeping follow-up (flagged
for the EM/Dean, same as FR-2b).

**One genuine rough edge found, worth closing this round (resolved
default).** The synthetic row's "×" remove button
(`public/js/setup.js` ~93/183-192) behaves identically to a real folder's:
clicking it removes the row from the client's in-memory `configuredFolders`
and deletes its `folderSettings` entry, then a Save would submit a `folders`
array without it. But because the synthetic root is **never** actually
persisted into `db.folders` (by design — the CRITICAL invariant this feature
must never violate), removing it accomplishes **nothing durable**: the very
next `GET /api/config` re-derives and re-splices it right back in (self-heal,
AC45). Today the UI gives no signal that this button is a no-op for that one
row, which could read as "I removed the download folder from tracking" when
nothing changed. **Resolved default:** disable (not hide — the row itself
must still display) the remove button specifically on the synthetic row, with
a tooltip/label explaining it always reflects the module's live download
directory and cannot be removed here (only by disabling the yt-dlp module
itself). This is a small, additive client-only affordance change — it must
not touch the `db.folders`-exclusion invariant in any way, which stays
entirely server-side and untouched.

**Resolved fork — default friendly-name.** Already resolved by the shipped
code: `'Downloads'` (server.js ~1621) — no change needed, restated here for
completeness per the assignment's explicit ask to resolve this fork.

**Security/data-safety-sensitive touchpoint — flagged for the two-reviewer
gate.** Even though this FR's core behavior is a confirm-not-build, any touch
to `server.js`'s folder/config handling or `public/js/setup.js`'s folder
rendering carries the CRITICAL invariant from the v1.12.0 data-loss class of
bug: **the synthetic download folder must never be written into
`db.folders`** — only into `folderSettings[dir]`. The two-reviewer gate is
warranted specifically to re-prove this invariant holds after the remove-
button change, not because new risk is being introduced.

### FR-5 — Thumbnail-preserve fix (v1.18.0 icon regression) — ALREADY IMPLEMENTED, folded in

Shipped into this branch ahead of the rest (commit "Fix (v1.18 regression):
preserve thumbnails on upgrade scan"). Root cause: v1.18.0's FR-1b reuse-guard
required `videoCodec`/`audioCodec` fields, so every pre-v1.18 video fell into the
full re-extract path that regenerated (clobbered) its thumbnail on the first
post-upgrade scan — users lost their file "icons". Fix: `probeCodecsOnly()`
(codec-only ffprobe, no frame-grab, never touches the thumbnail) + a third scan
branch for an unchanged legacy video missing codec fields that reuses the existing
entry (thumbnail/title/duration/addedAt preserved), backfills codecs, recomputes
`needsTranscode`, and regenerates the thumbnail ONLY if genuinely missing. Tests
reproduce the regression (fail pre-fix) and lock the fix. **Two-reviewer gate**
(scan/data path) — reviewed as part of the v1.19 diff; weight it heavily, since the
original finding was under-scored in the v1.18 gate.

- [ ] [MANUAL] After upgrade, an unchanged legacy video keeps its existing thumbnail (no frame-grab), and codec fields are backfilled so FR-1b transcode detection works.
- [ ] [MANUAL] A legacy video whose thumbnail was already lost by the v1.18 scan gets it regenerated.
- [ ] [INTEGRATION] Genuinely new/changed files still full-extract; audio bypasses; already-migrated videos take the plain reuse fast-path (no spawn). (Covered by test/integration/scan-thumbnail-preserve.test.js.)

## Design

### Approach

This round is six small, additive, mostly CSS-scoped changes on top of the
existing v1.13.0–v1.18.0 architecture — no shell/router/player/scan/transcode
rearchitecture. Three items are pure CSS in `public/css/style.css` (FR-1, FR-2,
FR-6), each in a distinct rule block; one adds a server-side download intent to
the existing `GET /video/:id` route plus a watch-page button (FR-3); one is a
client-only Setup affordance backed by one small additive read-only field on
`GET /api/config` (FR-4); and one (FR-5, the thumbnail-preserve fix) is
**already implemented and merged into this branch** — no work here, it only
rides the v1.19 review gate (see "FR-5" below).

The root causes for FR-1, FR-2, and FR-6 are all the **same CSS class of bug**
this codebase has hit before: a flex item whose intrinsic/`flex-basis` sizing is
misapplied on the mobile breakpoint. FR-1 is a `flex-basis` reinterpreted as a
min-height under a `flex-direction` flip; FR-6 is a flex child's default
`min-width: auto` (intrinsic content width) forcing horizontal overflow — the
exact same family as the v1.17.0 `.sort-select`/`.section-actions` overflow fix.
Naming the offending element/rule (not masking with `overflow-x`) is a hard
requirement of FR-6's `[PROCESS]` AC and is satisfied below for all three.

Integration points: FR-3 reuses the existing `id → db.metadata[id].filePath`
resolution and the existing `fs.createReadStream` send in `GET /video/:id`
(server.js ~2381–2456) — no new path-resolution mechanism, no new endpoint that
takes a filesystem path. FR-4 reuses `renderFolders()`'s existing generic row
construction (setup.js ~53–191) and adds a pure, unit-testable
"is this folder synthetic?" decision fed by a new read-only `syntheticFolders`
array on the `GET /api/config` response — leaving the `db.folders`-exclusion
invariant (entirely in `POST /api/config`) completely untouched.

### Component changes

- **`public/css/style.css` — FR-1 (one-off modal selects, mobile).**
  Root cause **confirmed against the live CSS**: the base rule
  `.oneoff-modal-row select` (~1910) is `flex: 1 1 150px; min-width: 150px`.
  The `max-width: 768px` block (~2429) flips `.oneoff-modal-row` to
  `flex-direction: column` and sets `select { width: 100% }` but never resets
  the inherited `flex` shorthand. Under a column main axis the `150px`
  flex-basis becomes a **minimum height** and `flex-grow: 1` stretches each
  select to fill the modal's vertical space — the ~1/4-screen-tall selects in
  Dean's screenshot. **Fix (mobile block only):** add `flex: 0 0 auto;` to the
  mobile `.oneoff-modal-row select` rule so flex-basis reverts to `auto`
  (content height) and flex-grow is 0 (no vertical stretch):

  ```css
  .oneoff-modal-row select {
    flex: 0 0 auto;
    width: 100%;
  }
  ```

  The existing grouped `min-height: 44px` rule (~2437) stays as the tap-target
  floor, giving a normal single-line control at ~44px — matching
  `.setup-select`'s visual weight (padding `10px 12px`, ~40px). Desktop is
  untouched (the base row layout is unchanged). This is CSS-only;
  `buildOneOffModal` (common.js) is not touched. **The existing
  `test/unit/oneoff-modal-mobile-polish.test.js` FIX-5 assertions still pass
  unmodified** (they match `width: 100%` and `min-height >= 44px` with
  permissive `[^}]*` patterns); add one assertion that the mobile
  `.oneoff-modal-row select` rule now carries the `flex` reset (no unreset
  `1 1 150px`/`flex-grow` on the column axis) to close FR-1's `[UNIT]/[PROCESS]`
  root-cause AC.

- **`public/css/style.css` — FR-2a (subscriptions list box).**
  `#sub-list-container` (subscriptions.html line 217) shares the
  `.folder-list-builder` class (~1545: `padding: 12px; max-height: 240px`) with
  `#folders-builder-list` and `#oneshot-list-container`. **Fix:** a
  **scoped `#sub-list-container` rule** (an ID selector — higher specificity, no
  new markup hook needed) that raises the box height and padding for the
  subscriptions list only:

  ```css
  #sub-list-container {
    max-height: 420px;
    padding: 16px;
  }
  ```

  The Setup folder builder and the one-shot job list keep their existing
  240px/12px sizing (per resolved fork #2). Values are Design's call; 420px /
  16px is the proposed default (~75% taller, roomier padding). Place the rule
  immediately after `.folder-list-builder`.

- **FR-2b (regression-lock only, no CSS change).** The FR-2a change touches only
  the **container** (`#sub-list-container` padding/max-height) — it does **not**
  touch `.sub-row` / `.sub-row-info` / `.sub-row-name` / `.sub-row-url`
  (~1651–1703), so the v1.13.0 AC1 fix (`.sub-row-info { flex: 1 1 240px;
  min-width: 240px }` + `overflow-wrap/word-break: break-word`, never
  `break-all`) is structurally unaffected. The regression bar is the existing
  subscriptions row tests passing unmodified plus Dean's on-device confirmation
  that names/URLs still wrap on horizontal lines.

- **`public/css/style.css` — FR-6 (mobile search sizing). Root cause named.**
  Search reuses the **home view** and its `#video-grid.video-grid` element
  (index.html line 121; main.js sets only the header text on `?search=`), so the
  grid is sized identically to home (already fixed to `minmax(140px, 1fr)` on
  mobile at ~2398) — the grid is **not** the culprit. The offending element is
  **`#videos-section-header`** (index.html line 105), the heading `<span>` inside
  `.section-title`. As a flex child it has the UA default `min-width: auto`
  (= intrinsic content width). On home it reads "Recently Added" (short); on
  search it reads `Search Results for "<query>"`, and a long/unbroken query
  token establishes an intrinsic min-width **wider than the phone viewport**,
  forcing horizontal overflow — the exact mechanism the v1.17.0 FR-1 comment
  documents for `.sort-select` (mobile Safari then renders the whole page zoomed
  to fit, so the search results "read too big"). **Fix (mobile block only,
  scoped to the heading so desktop is unchanged):**

  ```css
  #videos-section-header {
    min-width: 0;
    overflow-wrap: break-word;
    word-break: break-word;
  }
  ```

  `min-width: 0` lets the heading shrink to its row (`.section-title` is already
  `flex-wrap: wrap` and `.section-actions` is `width: 100%` on mobile, so the
  heading takes its own line); `overflow-wrap`/`word-break` wrap a long query
  token instead of forcing overflow. This is a **source fix at the named
  element**, not an `overflow-x` band-aid, and reconciles with the existing
  mobile grid (unchanged) and the v1.17.1 overflow fixes (same pattern, adjacent
  element).

- **`server.js` — FR-3 (download intent on `GET /video/:id`).** Extend the
  existing route (~2381) with an explicit `?download=1` intent (resolved fork
  #4, query-param shape). When present:
  1. **Bypass the transcode branch entirely** — do not enter the
     `if (item.needsTranscode)` block (~2393); `filePath` stays `item.filePath`
     (the **original**), even when `needsTranscode` is true and a cached
     `transcodedPath(item.id)` exists. The `?live=1` path is also never taken.
  2. **Set `Content-Disposition: attachment`** with a safely-encoded filename
     (see below) before the existing Range/`writeHead` send. The existing 404s
     (missing item ~2384; missing file on disk ~2410), Range handling (~2427),
     `Content-Type`/`Content-Length`, and all non-download behavior are
     **unchanged for any request without `download=1`**.
  The download intent keys strictly off `id → db.metadata[req.params.id]` — no
  client-supplied path or filename ever reaches a filesystem call.

- **`server.js` — FR-3 filename encoding (new pure helper).** Add a small pure,
  exported (per CONTRIBUTING's "export pure helpers for tests") helper, e.g.
  `contentDispositionAttachment(title, ext)`, that builds a header-injection-safe
  value from the item's **display `title`** (not the raw on-disk `name`) + its
  **original `ext`**:
  - ASCII fallback: `filename="<ascii>"` where `<ascii>` =
    `String(title).replace(/[^\x20-\x7E]/g, '_').replace(/["\\]/g, '_')` — the
    first replace strips **all control chars incl. CR/LF (they are outside
    0x20–0x7E)** and non-ASCII; the second strips `"`/`\`. Ext sanitized to
    `[A-Za-z0-9.]` only.
  - RFC 5987 form: `filename*=UTF-8''<encodeURIComponent(title+ext)>` for the
    real (non-ASCII) name; `encodeURIComponent` percent-encodes CR/LF as
    `%0D%0A`, so no injection is possible via `filename*` either. (It
    over-encodes a few chars RFC 5987 leaves bare; browsers accept this.)
  - Result: `attachment; filename="<ascii>"; filename*=UTF-8''<encoded>`.
  A missing/empty title falls back to `'download'`.

- **`public/watch.html` + `public/js/watch.js` — FR-3 button.** Add a
  **Download** control to `.watch-actions` (~148–163), next to Delete, reusing
  the shipped `.icon-download` glyph. Implement as an anchor styled with the
  existing themed `.btn` class (no new CSS needed), e.g.
  `<a class="btn" id="download-media-btn" download><i class="icon-download"></i>
  Download</a>`. In `watch.js`'s `populateMetadata` (~221–233, where `mediaData`
  is in scope), set `href = '/video/' + mediaData.id + '?download=1'` per media
  load (the SPA reuses the node across navigations, so set it each load; keep it
  inert/`href`-less until a media item is loaded). The browser saves via the
  server's `Content-Disposition: attachment` (authoritative, and the standard
  same-origin path on iOS Safari 13+); the anchor `download` attribute is a
  belt-and-suspenders hint for desktop. Works identically for audio and video
  (same route, same `item.filePath`).

- **`server.js` (`GET /api/config`) + `public/js/setup.js` + `public/js/common.js`
  — FR-4 (disable the synthetic row's remove button).** See the New-fork note
  below for the `syntheticFolders` server field. In `setup.js`: capture
  `syntheticFolders` from the config response; in `renderFolders()`, for each
  row compute `isSynthetic` via a pure helper, and for the synthetic row render
  the remove button **disabled** (not hidden — the row still shows) with a
  static explanatory `title`, e.g. "This is the auto-managed downloads folder —
  rename or reorder it here, but it can't be removed (disable the yt-dlp module
  to remove it)." Set `disabled`/`title` as DOM properties on the button node
  after row creation (a disabled button dispatches no click, so the existing
  no-op remove handler never runs for it; add a defensive `if (isSynthetic)
  return;` guard too). The tooltip is a static literal (no dynamic/user data),
  so no `innerHTML` interpolation of dynamic strings is introduced. **No change
  to `POST /api/config`, the splice/rename/order logic, or the
  `db.folders`-exclusion invariant.**

- **FR-4 pure decision helper (for the `[UNIT]` AC).** Add a tiny pure function,
  e.g. `isSyntheticFolder(folder, syntheticFolders)` (return
  `Array.isArray(syntheticFolders) && syntheticFolders.includes(folder)`), in
  `public/js/common.js` next to `moveArrayItem`/`computeDropIndex` (which already
  ride the `module.exports` guard at ~2120) so it is unit-testable without a
  headless browser.

### API changes

- **`GET /video/:id`** gains an optional `?download=1` query param (additive;
  absent = byte-identical existing behavior). When present, always serves the
  original file with `Content-Disposition: attachment`.
- **`GET /api/config`** response gains an additive, read-only `syntheticFolders`
  array (the resolved synthetic root paths). Existing consumers/tests that read
  `folders`/`folderSettings` are unaffected (they do not assert response-key
  exclusivity — confirmed against `ytdlp-synthetic-folder.test.js`'s AC42/AC44
  assertions).
- No changes to any subscriptions API, `POST /api/config`, `POST /api/settings`,
  or any other route.

### Alternatives considered

- **FR-3 mechanism — a dedicated `GET /video/:id/download` route** (resolved
  fork #4's sibling option). Pros: cleaner separation, no branching inside the
  hot playback handler. Cons: duplicates the id-lookup / `fs.existsSync` /
  Range-send logic (or requires extracting a shared helper), i.e. **two send
  paths to keep confined** instead of one. **Rejected** in favor of the
  `?download=1` param on the existing route — a single shared code path with
  playback's confinement is exactly the constraint's "never a second
  path-resolution mechanism," and the download branch is a 2-line divergence
  (skip transcode branch + set one header).
- **FR-4 identification — strictly client-only, no server field** (the AC's
  literal "client-only" wording). The client would have to learn the yt-dlp
  download dir from the subscriptions module surface and re-resolve/normalize it
  to match the server's `path.resolve(downloadDir)` — **fragile** (client-side
  path resolution can't reliably match the server's) and couples `setup.js` to
  the ytdlp module. **Rejected** in favor of the additive read-only
  `syntheticFolders` field (New fork below), which is authoritative, trivial,
  and leaves the invariant-bearing `POST` logic untouched.
- **FR-4 marker via `folderSettings[root].synthetic = true`.** Rejected: the
  client round-trips `folderSettings` back into `POST`, so the marker would
  leak into persisted settings. A separate top-level array the client only
  reads (never sends back) is cleaner.
- **FR-6 — a shell-level `overflow-x: hidden`.** Explicitly rejected by FR-6's
  `[PROCESS]` AC (no band-aid); it would hide the symptom while the heading
  still forces the wrong layout width and the zoom-out persists.

### New fork surfaced during Design (with recommended default)

**Fork FR-4b — how does the client identify the synthetic folder row?** The
current `GET /api/config` response (`{ folders, folderSettings }`) does **not**
flag which entry is synthetic; the synthetic root is just one path in `folders`.
The plan's FR-4 AC says the change is "client-only," but there is **no robust
client-only signal** to distinguish the synthetic row (see the rejected
alternatives above). **Recommended default (Option A):** add a small, additive,
**read-only** `syntheticFolders` array (the resolved synthetic roots, i.e. the
`synthRoots` already computed at server.js ~1696) to the `GET /api/config`
response. This does **not** touch the synthetic-root *handling* the invariant
depends on — the splice, the rename/order persistence, and the
`db.folders`-exclusion all live in code paths that are unchanged; the
`ytdlp-synthetic-folder.test.js` suite passes unmodified because it never
asserts response-key exclusivity. The only deviation is that FR-4's `[PROCESS]`
AC "no change to `server.js`'s `GET`/`POST /api/config`" must be relaxed to "no
change to the synthetic-root *handling* / `db.folders`-exclusion invariant; one
additive read-only field on the `GET` response is permitted." **This is
non-blocking — Option A is recommended and implementation can proceed on it;**
EM/Dean should ratify the minor AC-wording relaxation at task-breakdown time.

### Risks and mitigations

- **Risk:** FR-3 accidentally serves the transcode (or takes the `?live=1`
  path) for a download. → **Mitigation:** the `?download=1` branch must be
  evaluated **before** and short-circuit the `if (item.needsTranscode)` block;
  cover with a `[UNIT]/[INTEGRATION]` test using a `needsTranscode: true`
  fixture with an existing cached transcode, asserting the original bytes/size.
- **Risk:** header injection via a title containing CR/LF or quotes. →
  **Mitigation:** the pure `contentDispositionAttachment` helper strips all
  chars outside `0x20–0x7E` (removes CR/LF) plus `"`/`\` from the ASCII
  fallback, and percent-encodes the `filename*` form; unit-test with a
  CR/LF/quote/non-ASCII title.
- **Risk:** FR-3 weakens existing playback (Range, 404s) for normal requests. →
  **Mitigation:** the download path is gated entirely behind `download=1`; add
  an integration test asserting a request *without* the param is byte-identical
  to today, and that a bad/missing id or missing-on-disk file returns the same
  404.
- **Risk:** FR-4's server field or the `renderFolders` change disturbs the
  `db.folders`-exclusion invariant. → **Mitigation:** the server change is a
  read-only additive field on `GET`; `POST` is untouched; the full
  `ytdlp-synthetic-folder.test.js` suite (AC42/44/45/46/47/8/9/10/10b/11/12) is
  the regression bar and must pass **unmodified**.
- **Risk:** FR-1/FR-2/FR-6 CSS changes regress an adjacent mobile fix. →
  **Mitigation:** each is scoped to a single rule/ID in its own block; the
  existing `oneoff-modal-mobile-polish.test.js` and subscriptions row tests
  gate FR-1/FR-2; FR-6 touches only `#videos-section-header`. Dean's on-device
  pass is the documented visual arbiter (no E2E infra per RELIABILITY.md).

### Performance impact

No expected impact on any budget in `docs/RELIABILITY.md`. FR-3 streams the
original file via the same `fs.createReadStream` the route already uses (no
extra copy, no transcode — in fact it *avoids* a transcode); the transcode
cache LRU/age budget is unaffected (a download never touches the cache). FR-4's
`syntheticFolders` is the already-computed `extraScanRoots()` result. The CSS
changes are static. No new runtime dependencies.
