# v1.18.0 "iOS playability + player polish"

## Goal

Make FileTube's media actually **play on iOS** — the whole point of a
self-hosted, mobile-first YouTube-style server — by transcoding based on the
real codec inside a file (not just its file extension) for both yt-dlp
downloads and manually-added library files, and polish the v1.16.0 persistent
player so switching videos never flashes a stale poster/frame, plus restore
the pre-existing "here's what will transcode" rescan feedback that regressed.

## Scope

Three functional requirements, one focused branch (`feature/v1.18-ios-playability`,
off `main` at v1.17.1):

### FR-1 — iOS mp4 playability (ANCHOR)

**Root problem, confirmed in code:** `needsTranscode(ext)` in `server.js`
(line 229-231) decides whether to transcode purely by file **extension**:

```js
const TRANSCODE_EXTENSIONS = ['.avi', '.flv', '.wmv', '.mpg', '.mpeg'];
function needsTranscode(ext) {
  return TRANSCODE_EXTENSIONS.includes(ext);
}
```

Every `.mp4`/`.mkv`/`.mov`/`.webm`/`.m4v` is assumed browser-playable and
served as-is (`reconcileTranscode`, line 825-844, and the scan-time seed at
line 1193, both key off this same ext-only function). But those containers
can carry a codec iOS/browsers can't decode — H.265/HEVC video, AC-3/DTS/
E-AC-3 audio, 10-bit/exotic profiles, VP9/AV1 — so the file appears in the
library but silently fails to play. Two complementary sub-fixes:

#### FR-1a — yt-dlp downloads (what we control)

`lib/ytdlp/args.js`'s `buildYtdlpDownloadArgs` currently selects video streams
via a plain `QUALITY_SELECTORS` map (`bestvideo+bestaudio/best`, height-capped
per quality tier) with no codec preference, and only sets
`--merge-output-format <ext>` when `sub.filetype` (normalized via
`normalizeFiletype`) resolves to an explicit non-`'default'` container —
`'default'` emits no container flag at all, so yt-dlp's own choice (often
VP9/AV1 in a `.webm`) stands. Make video downloads intended to be mp4 land as
**H.264 (avc1) video + AAC (mp4a) audio**, instead of yt-dlp's default,
via format-sorting/selection (`-S "vcodec:h264,acodec:aac"` and/or an explicit
`-f` selector preferring `bestvideo[vcodec^=avc1]+bestaudio[acodec^=mp4a]/best[ext=mp4]`),
plus the existing `--merge-output-format mp4` behavior, with an optional
`--recode-video mp4` fallback only when no compatible stream exists (CPU-cost
tradeoff — see Decision log). This must preserve every existing yt-dlp
security constraint unchanged: arg-array/no-shell construction (`args.js`
never builds a shell string), the `--` separator immediately before
positional targets, the channel-URL host allowlist (`lib/ytdlp/url.js`,
untouched), `resolveChannelDir`/SF4 path confinement, `--download-archive`
dedup (subscriptions) vs. `--no-download-archive --force-overwrites`
(one-off), `--windows-filenames`, the Shorts `--match-filter`, and the
one-off-vs-subscription branch in `buildYtdlpDownloadArgs` — this task only
adds format-selection args to the existing builder.

#### FR-1b — manually-added / non-yt-dlp files

`extractMetadataAndThumbnail` (server.js line 888-934) already runs `ffprobe`
at scan time for duration + embedded tags (`-show_entries
format=duration:format_tags`, line 899) — extend that same probe to also read
each stream's `codec_name`/`codec_type` (`-show_streams` or
`-show_entries stream=codec_name,codec_type`), and generalize `needsTranscode`
from extension-only to **extension-or-codec**: a file whose container is
nominally web-ok but whose codec is NOT on the browser-compatible allowlist
(HEVC/H.265, MPEG-4 Part 2, AC-3/DTS/E-AC-3 audio, VP9/AV1 where not
allowlisted) gets queued for the SAME transcode-to-H.264/AAC-mp4 path the
AVI flow already uses (`processTranscodeQueue`, server.js line 724-819, the
existing `-c:v libx264 ... -c:a aac` FFmpeg invocation at line 744-750 —
unchanged, just triggered by more inputs). The browser-compatible allowlist
target is **H.264 (AVC) video + AAC audio only** — conservative and correct
per Dean's own framing: iOS Safari's HEVC support is real but
version/hardware-dependent, so it is NOT treated as safe-to-skip-transcode in
this round.

**Performance/disk consideration (explicit, non-negotiable):** codec-based
detection can flag many more files than the old extension-only set on a large
library. Probing the codec at scan time is effectively free (the ffprobe call
already runs for every file; this only adds fields to its existing output
parse) — but actually **transcoding** must stay **lazy/on-demand**, exactly
like the existing AVI-on-mobile flow (`processTranscodeQueue`'s single-worker
queue, the transcode cache's size-capped LRU eviction, and the mobile
"preparing…" overlay + poll). This round must not eagerly transcode a whole
library on scan.

### FR-2 — Player polish: stale poster + FOUC on Next

Side effect of the v1.16.0 persistent player (`public/js/player.js`'s single
reused `<video>`/`#audio-bg-art` element, so playback survives navigation):
`teardownMediaState()` (line 970-987) clears the audio-bg-art image and hides
skip controls, etc., on every load, but never resets the `<video>` element's
visible poster/last-rendered frame — `setupForMedia()` (line 989-1040) only
ever sets `mediaPlayer.poster` for the **audio** branch (line 995,
`'/thumbnail/' + id`); the video branch never sets or clears a poster at all.
Consequence: (a) loading a new item, the PREVIOUS video's last decoded frame
(or, for audio, the previous item's bg-art) visibly lingers until the new
media decodes its own first frame or playback starts; (b) Dean also reports a
FOUC specifically on **Next** (the prev/next-advance path, `player-lifecycle`/
`handleAutoplayNext` territory). Fix: on every genuine new `load()` (not the
`adopt` same-media dock→full path), immediately reset the visible
poster/art to a neutral, static placeholder BEFORE the new source is assigned
— clearing both the video frame/poster and the audio-bg-art — so there is no
flash of the outgoing item's image during the transition, and the new media
then paints cleanly once ready. This must not reintroduce a reload, must not
change `load()`'s existing adopt/non-adopt branching, and must preserve every
other persistent-player behavior (playback continuity across navigation,
dock/full transitions, iOS inline playback, Media Session, resume overlay,
transcode "preparing…" overlay).

### FR-3 — Rescan should list what it'll transcode (regression restore)

Per `ROADMAP.md`'s "Bugs & regressions" section: the Setup "Save & Rescan"
flow used to surface the videos it would convert/transcode; it no longer
does. Today, `GET /api/scan-status` (server.js line 1675-1688) only returns a
**count** (`transcoding`, filtered on `i.needsTranscode && i.transcodeStatus`
in `{pending, processing}`-equivalent, excluding `ready`/`failed`) and
`public/js/setup.js`'s `pollScanStatus` (line 294-335) renders only
`` `Converting ${s.transcoding} file(s) in the background…` `` — no names, no
list. Restore the actual pending-transcode **set** (count + names/titles) so
the user can see what's queued before/while a rescan runs, surfaced in the
Setup page's `#scan-status` line (`public/setup.html` line 123) via the same
polling path. This pairs directly with FR-1b: codec-based detection will flag
more files than extension-only detection did, so the restored list must
reflect the **same, generalized** `needsTranscode`/`transcodeStatus` logic —
not a second, divergent detection path.

## Out of scope

- **Any rearchitecture of the transcode queue, cache, or persistent-player
  shell.** This is a focused playability + polish round on top of the
  v1.16.0/v1.17.0 architecture, not a redesign of either.
- **Eager, whole-library transcoding on scan.** FR-1b's codec detection at
  scan time is a probe only; actual transcoding stays lazy/on-demand (see
  FR-1's performance/disk note above). Checked against `docs/CONTRIBUTING.md`
  and `docs/ARCHITECTURE.md`'s documented "lazy transcoding" key decision —
  nothing here overrides it.
- **Re-litigating any yt-dlp security guard.** FR-1a only adds
  format-selection args to the existing arg-array builder; the URL allowlist,
  path confinement, dedup/archive semantics, and one-off-vs-subscription
  behavior are unchanged.
- **A new native Picture-in-Picture / mini-player redesign.** Out of scope;
  v1.16.0's docked persistent player is the shell FR-2 polishes, not replaces.
- **Restoring iOS-native HEVC playback as an allowlisted "skip transcode"
  path.** Rejected in this round per Dean's own framing (be conservative —
  H.264+AAC is the safe target); revisit only as a future, explicit,
  device-capability-aware enhancement if ever desired.
- Checked against `docs/CONTRIBUTING.md`: nothing above defers a mandatory
  standard (tests, lint, `textContent`-not-`innerHTML`, no new deps) — every
  file touched by every FR in this round still ships tests per the
  "every new feature or bugfix ships with tests" standard, including FR-2's
  MANUAL-heavy player polish (any extractable pure helper gets unit coverage)
  and FR-1b's codec detection (unit-tested with mocked ffprobe output, since
  ffmpeg isn't installed in CI per `docs/RELIABILITY.md`).

## Constraints

- **No regressions.** Every existing extension-based transcode case
  (`.avi`/`.flv`/`.wmv`/`.mpg`/`.mpeg`) must keep working identically through
  the generalized `needsTranscode` (FR-1b) — the existing
  `test/unit/pure-helpers.test.js` cases are the regression bar. The existing
  desktop live-transcode (`?live=1`) path, the mobile lazy-transcode +
  "preparing…" overlay + poll, the transcode cache's size-capped LRU eviction,
  and the single-worker transcode queue are all reused unchanged — only the
  set of files that can enter that pipeline grows.
- **Security-sensitive touchpoints — flagged for focused review, not a green
  light to loosen anything:**
  - **FR-1a — the yt-dlp arg-array spawn core (`lib/ytdlp/args.js`
    `buildYtdlpDownloadArgs`).** Every existing guard (arg-array/no-shell,
    `--` separator, `ALLOWED_HOSTS` allowlist in `lib/ytdlp/url.js`, SF4 path
    confinement via `resolveChannelDir`/`isPathUnder`, download-archive dedup
    vs. one-off bypass, `--windows-filenames`, the Shorts match-filter) must
    remain byte-identical; only new format-selection/preference args are
    added, never interpolated from unsanitized input.
  - **FR-1b — the ffprobe extension on user-supplied files.** The scan
    already runs `ffprobe` via `exec()` with the file path interpolated into
    a shell command string (server.js line 899:
    `` `ffprobe -v error -show_entries format=duration:format_tags -of json "${filePath}"` ``)
    — a pre-existing pattern, not newly introduced by this round, but this
    task extends that exact command. The Design stage should confirm this
    extension does not widen the injection surface (e.g., a media filename
    containing a shell metacharacter or embedded quote) beyond what already
    exists today, and flag it as a hardening opportunity if warranted (not
    mandatory to fix in this round, but must not be made worse).
  - **The transcode spawn (`processTranscodeQueue`, server.js line 744-750)**
    now fires for a larger, codec-triggered set of files, not just the fixed
    extension list — same spawn code (`ffmpeg -i <src> -c:v libx264 ...`),
    higher potential volume/frequency. The existing single-worker queue +
    cache LRU eviction are the mitigation; no change to the spawn itself is
    in scope.
- **Performance/disk (FR-1b).** Codec probing at scan time is effectively
  free (piggybacks on the existing per-file `ffprobe` call). Actual
  transcoding must remain lazy/on-demand (triggered on watch, mirroring the
  existing AVI-on-mobile flow) — never eagerly transcoding a whole library on
  scan. This is a hard constraint, not a suggestion.
- **`docs/CONTRIBUTING.md`:** vanilla DOM (no framework/bundler), CommonJS
  server, 2-space/semicolons/single-quotes, `textContent` over `innerHTML`
  for any new dynamic strings (FR-3's restored transcode-list rendering),
  `node:test` for new/regression tests, lint 0 warnings, no new runtime
  dependencies.
- **Era-theme system preserved.** Any new CSS (FR-2's neutral placeholder
  styling, FR-3's restored transcode-list styling in Setup) uses existing
  theme tokens (CSS custom properties) — no hardcoded colors that break the
  2005/2009/2014/2021 eras or light/dark modes.
- **No headless-browser/E2E test infrastructure exists** (per
  `docs/RELIABILITY.md`) — this bounds what FR-1 (real-device playability)
  and FR-2 (perceived smoothness/no-flash) can claim as `[UNIT]`/
  `[INTEGRATION]`; Dean's on-device iPhone pass is the documented arbiter for
  both, not a rigor gap. Design should still extract and unit-test every pure
  helper each fix produces (codec-allowlist check, ffprobe-stream-parsing,
  any player visual-state-reset helper, the pending-transcode-list
  derivation).
- **Node 22 LTS** toolchain (`engines` ≥20); CI/build-verify runs tests on
  Node 22.

## Product forks — resolved with recommendation, plus one new fork

Dean's brief already resolves most of the tradeoffs explicitly; restated here
for traceability, plus one genuinely new fork this Discovery surfaced:

1. **FR-1a: format-selection-only vs. add a `--recode-video mp4` fallback
   (CPU cost).** RESOLVED per Dean's own framing: prefer format-SELECTION
   (pick an already-H.264/AAC stream via `-S`/`-f`) as the primary
   mechanism; add `--recode-video mp4` only as a fallback when no compatible
   stream exists at all, since recoding re-encodes (real CPU/time cost on a
   home server). PE should confirm the exact `-S`/`-f` syntax and whether the
   fallback is worth the added complexity vs. simply accepting that a small
   minority of sources stay non-ideal without it.
2. **FR-1a: scope of the format-selection preference.** NEW fork, not
   resolved by the brief. `buildYtdlpDownloadArgs`'s filetype/container logic
   only forces `--merge-output-format mp4` when `sub.filetype` resolves to
   the explicit `'mp4'` value — the `'default'` filetype (today: no container
   flag, yt-dlp's natural choice) is a separate, more common case and is
   arguably the bigger real-world source of VP9/AV1-in-webm downloads.
   **Recommend:** apply the H.264/AAC format-sort preference (`-S`) to
   **every** video download regardless of filetype selection (it is a soft
   stream *preference*, not a container-forcing flag, so it's low-risk even
   for `'default'`/`'mkv'`/`'webm'` selections — it simply makes yt-dlp
   prefer an avc1/mp4a-carrying stream when one exists) — while leaving
   `--merge-output-format`'s explicit-`'mp4'`-only trigger unchanged. This
   maximizes the playability win without silently changing anyone's chosen
   container. Flag for PE/Dean to confirm at Design.
3. **FR-1b: lazy-on-watch vs. eager-on-scan transcoding.** RESOLVED per
   Dean's own framing and `docs/ARCHITECTURE.md`'s existing "lazy
   transcoding" key decision: reuse the exact same lazy/on-demand approach as
   the AVI flow. Not an open question — restated as a hard constraint above.
4. **FR-1b: exact browser-compatible codec allowlist.** RESOLVED per Dean's
   own framing: H.264 (AVC) video + AAC audio, full stop — deliberately
   conservative (does not allowlist HEVC despite partial iOS support, does
   not allowlist VP9/AV1). PE should name this as an explicit, documented
   constant (mirroring `TRANSCODE_EXTENSIONS`'s existing pattern) so it's
   easy to audit/extend later.
5. **FR-2: exact neutral-placeholder mechanism.** NEW fork. No placeholder
   image asset exists anywhere in `public/assets/` today (confirmed by
   inspection — same "not a trivial one-line asset fix" flag as v1.17's FR-7
   download-icon precedent). Options: (a) a blank/transparent static image
   set as `mediaPlayer.poster` before the new `src` is assigned, (b) a new
   FileTube-branded placeholder graphic, (c) a CSS-only treatment (e.g., a
   background color/fade on the host while no frame is painted) requiring no
   new asset at all. **Recommend (c)** as the lowest-risk, no-new-asset,
   era-theme-token-friendly option, with (a)/(b) as viable alternatives if
   Design finds a CSS-only treatment insufficient on-device. Non-blocking —
   Design/Dean's call.
6. **FR-3: payload size for the restored pending-transcode list.** NEW fork.
   FR-1b's codec-based detection can flag substantially more files than the
   old extension-only set on a large library — an uncapped names list in
   `GET /api/scan-status`'s JSON response could grow large. **Recommend** a
   cap (e.g., first N names + a "+K more" remainder count) rather than an
   unbounded list, matching the spirit of the existing `--playlist-end`-style
   "bound it, don't buffer it all" pattern used elsewhere in this codebase.
   Non-blocking — Design's call on the exact N.

If a genuine new fork surfaces during Design beyond these, PE should state it
with a recommendation rather than block.

## Design

### Approach

Four independent, small edits on top of the v1.16.0/v1.17.0 architecture — no
new modules, no dependencies, no queue/cache/player rearchitecture. FR-1a adds a
single fixed soft-sort flag to the existing yt-dlp video arg branch. FR-1b adds
two fields to the already-running scan-time `ffprobe` output, generalizes the
existing `needsTranscode` predicate to be codec-aware behind a backward-compatible
signature, and lets those newly-flagged files ride the *existing* lazy transcode
pipeline unchanged. FR-2 is a visual reset inside the existing
`teardownMediaState()` sequence in `public/js/player.js`. FR-3 extends one API
response and one polling renderer to surface the (bounded) pending-transcode set
that FR-1b's generalized predicate now produces.

The one cross-FR dependency is that FR-3's list must be derived from FR-1b's
generalized `needsTranscode`/`transcodeStatus` logic — so FR-3 lands after
FR-1b. FR-1a and FR-2 are fully independent of everything else.

Note on code drift vs. the Scope section: `lib/ytdlp/args.js` has advanced past
what the Scope narrative describes. The DOWNLOAD pass no longer targets the
channel URL — it builds explicit per-survivor `watch?v=<id>` URLs from
`targetIds` (C1 fix, already shipped), and it *already* emits
`-f QUALITY_SELECTORS[quality]` plus `--merge-output-format <ext>` (the latter
only when `normalizeFiletype` resolves to a non-`'default'` container). FR-1a is
therefore purely *additive*: one new soft-sort flag next to the existing `-f`.

### Component changes

- **`lib/ytdlp/args.js` — `buildYtdlpDownloadArgs` (FR-1a).** Add a module-level
  fixed literal `const VIDEO_FORMAT_SORT = 'vcodec:h264,acodec:aac';` (exported,
  mirroring the `SHORTS_MATCH_FILTER` pattern) and, in the existing **video**
  branch only, push `'-S', VIDEO_FORMAT_SORT` immediately after the existing
  `args.push('-f', QUALITY_SELECTORS[quality]);`. This is a *soft* format-sort:
  yt-dlp keeps the user-specified sort fields first and appends its own default
  ordering (res/fps/bitrate) as tie-breakers, so within each quality tier's
  height ceiling an H.264 video + AAC audio stream is preferred when one exists,
  without *excluding* any format (so the `2160p`/`1440p` tiers, where YouTube
  only serves VP9/AV1, still resolve to the best available rather than failing).
  Placement: after `-f`, before `--merge-output-format`, and well before the
  `--`/positional targets — all inside the `else` (video) branch, so the
  `format: 'audio'` path (`-x --audio-format`) is untouched. Because it lives in
  the video branch unconditionally (not gated on `filetype`), the preference
  applies to **every** video download and **every** quality tier, including
  `'default'`/`'mkv'`/`'webm'` selections (fork #2, recommended scope) — while
  the `--merge-output-format`'s explicit-`'mp4'`-only trigger stays byte-identical.
  `VIDEO_FORMAT_SORT` is a fixed literal, never interpolated from any input, so
  it adds no injection surface.

- **`server.js` — `extractMetadataAndThumbnail` ffprobe call (FR-1b, part 1).**
  Extend the `-show_entries` list from
  `format=duration:format_tags` to
  `format=duration:format_tags:stream=codec_name,codec_type`, which adds a
  `streams[]` array to the same single JSON probe (no second spawn, no doubled
  scan cost). **Hardening (recommended, in-scope for this task):** convert this
  one call from the shell-string `exec()` to `execFile('ffprobe', [args...],
  ...)` with the `filePath` as a distinct arg-array element, removing the
  path-interpolation on the exact line we are already editing. This is strictly
  better than "don't make it worse" and is low-risk/contained. The two *other*
  `exec()` calls in this function (the ffmpeg art/frame extractors, lines ~920 /
  ~927) are left as-is this round and logged to the tech-debt tracker — extending
  scope to them is not required and would widen the diff. Parsing stays in a new
  pure helper (see below), so ffmpeg/ffprobe are not needed in CI.

- **`server.js` — pure helpers (FR-1b, part 2).** Add, next to `parseFfprobeTags`:
  - `parseFfprobeStreams(input)` — accepts the parsed object OR raw stdout
    string (same robustness contract as `parseFfprobeTags`: `JSON.parse` in a
    try/catch, returns `{}` on anything malformed, never throws). Returns
    `{ videoCodec, audioCodec }` — the lowercased `codec_name` of the first
    `codec_type === 'video'` stream and first `codec_type === 'audio'` stream
    respectively, or `undefined` for a stream type that is absent.
  - `PLAYABLE_VIDEO_CODECS = new Set(['h264', 'avc1'])` and
    `PLAYABLE_AUDIO_CODECS = new Set(['aac'])` — named, exported, documented
    allowlist constants mirroring `TRANSCODE_EXTENSIONS`. (ffprobe reports
    `h264`; `avc1` included defensively.)
  - `codecNeedsTranscode(videoCodec, audioCodec)` — returns `true` only on a
    *positive* identification of a non-allowlisted codec:
    `if (videoCodec && !PLAYABLE_VIDEO_CODECS.has(videoCodec)) return true;`
    `if (audioCodec && !PLAYABLE_AUDIO_CODECS.has(audioCodec)) return true;`
    `return false;`. Crucially, `undefined`/missing codecs → `false`, so a
    failed/ambiguous probe never *falsely* flags a file (satisfies the
    "degrade safely" AC).

- **`server.js` — `needsTranscode` generalization (FR-1b, part 3).** Extend the
  signature to `needsTranscode(ext, videoCodec, audioCodec)` with the codec args
  **optional**: `if (TRANSCODE_EXTENSIONS.includes(ext)) return true;` (existing
  extension flow preserved exactly — `.avi`/`.flv`/… always transcode);
  `return codecNeedsTranscode(videoCodec, audioCodec);`. Called with one arg
  (codecs `undefined`) it is byte-identical to today, so every existing
  `test/unit/pure-helpers.test.js` case passes unchanged. Update the two call
  sites: `reconcileTranscode` →
  `needsTranscode(item.ext, item.videoCodec, item.audioCodec)` (this is the
  authoritative recompute; it runs inside the scan's `updateDatabase` mutator
  after the probe has attached codecs to the item). The scan-time seed at
  line ~1193 stays ext-only (codecs aren't known yet at that point — it is an
  initial value the `reconcileTranscode` pass overwrites).

- **`server.js` — item model + one-time backfill (FR-1b, part 4).** Store the
  probed codecs on the item alongside `duration`/`tags`: `item.videoCodec` /
  `item.audioCodec`, set to the probed lowercased string **or `null`** when the
  probe ran but the stream type was absent (so a probed-but-no-video item is
  distinguishable from an un-probed one after a JSON round-trip). Extend the
  "reuse unchanged metadata" guard at server.js:1156 so that a **video** item is
  only reused when it already carries the codec fields — a pre-v1.18 library
  entry (or one that predates the probe) is re-extracted **once** on the next
  scan to backfill its codecs. This is probe-only (never an eager transcode), so
  it stays within the "probing is free" budget while making existing iOS-broken
  libraries actually benefit without a forced full rescan. Audio items skip this
  (reconcile short-circuits `type === 'audio'`).

- **`server.js` — lazy transcode wiring (FR-1b, part 5): NO new code.** Because
  `reconcileTranscode`, `queueTranscode`, and the `/video/:id` handler all key
  off `item.needsTranscode`, a codec-flagged `.mp4`/`.mkv`/etc. flows into the
  *existing* lazy pipeline automatically: `reconcileTranscode` only seeds/clears
  status (no queue kick), and the real `queueTranscode` fires on first mobile
  watch in `/video/:id` (line ~2159) exactly as for AVI today. `player.js`'s
  `setupForMedia` already branches on `data.needsTranscode`/`data.transcodeStatus`,
  which now become `true`/in-flight for these files — so the "preparing…"
  overlay + poll and the desktop `?live=1` path work with **no player change**.

- **`public/js/player.js` — `teardownMediaState()` (FR-2).** Add a visual reset
  that runs on every genuine (non-`adopt`) load, *before* `setupForMedia`
  assigns the new source. The `adopt` (dock↔full) path returns early in `load()`
  before `teardownMediaState()` is ever called, so continuity is preserved
  untouched. Concretely, in `teardownMediaState()` after `mediaPlayer.pause()`:
  `mediaPlayer.removeAttribute('poster');` (clears the audio branch's
  `/thumbnail/<prevId>` poster) and then drop the last-decoded video frame with
  `mediaPlayer.removeAttribute('src'); mediaPlayer.load();`. `removeAttribute` +
  `load()` resets the element to the media-empty state, which paints *nothing* —
  revealing the existing `#000` `.player-container` background (the CSS-only
  neutral placeholder; no image asset). The `audio-bg-art` clear already present
  at line 983 stays. `setupForMedia` then assigns the fresh `src` a few lines
  later as today. This is a media-element `load()`, **not** a page reload, and
  does not touch `load()`'s adopt/non-adopt branching. `removeAttribute('src')`
  (not `src = ''`) avoids the "empty string resolves to the page URL" reload
  quirk.

- **`server.js` — `GET /api/scan-status` (FR-3).** Derive the pending set from
  the SAME filter that already produces `transcoding`:
  `pending = items.filter(i => i.needsTranscode && i.transcodeStatus &&
  i.transcodeStatus !== 'ready' && i.transcodeStatus !== 'failed')`. Add to the
  JSON response: `transcodeNames` (the first `TRANSCODE_LIST_CAP` items' `title
  || name`, a bounded `string[]`) and `transcodeOverflow`
  (`max(0, transcoding - transcodeNames.length)`). Keep the existing
  `transcoding` count for backward compatibility. `TRANSCODE_LIST_CAP = 10`
  (fork #6 — bounded, useful, small payload).

- **`public/js/setup.js` — `pollScanStatus` (FR-3).** Extend the existing
  `if (s.transcoding > 0)` block (in both the in-progress and "Scan complete"
  branches) to append the names:
  `` `: ${s.transcodeNames.join(', ')}` `` plus, when `s.transcodeOverflow > 0`,
  `` ` +${s.transcodeOverflow} more` ``. The whole message continues to be
  assigned via `statusText.textContent` (never `innerHTML`), so user-controlled
  filenames/titles are safe by construction. The empty-set case renders nothing
  new (the `> 0` guard already suppresses "Converting 0 file(s)"). No new HTML
  element and no new CSS is required (the message stays on the existing
  `#scan-status` line), keeping era-theme tokens untouched.

### Data model changes

- Item metadata gains two optional fields: `videoCodec` and `audioCodec`
  (lowercased `codec_name` string, or `null` when probed-but-absent, or the
  property absent for un-probed/legacy entries). No migration script; the
  backfill (FR-1b part 4) fills them lazily on the next scan of each video item.
- `GET /api/scan-status` response gains `transcodeNames: string[]` (capped at 10)
  and `transcodeOverflow: number`. Additive; existing `transcoding` count kept.

### API changes

- `GET /api/scan-status`: additive fields only (`transcodeNames`,
  `transcodeOverflow`) — no breaking change, existing consumers unaffected.
- `needsTranscode(ext)` → `needsTranscode(ext, videoCodec, audioCodec)` with the
  new args optional and defaulting to today's behavior. Internal + exported for
  tests; backward compatible.
- No new HTTP endpoints, no yt-dlp arg-array contract changes beyond one added
  fixed `-S` flag.

### Alternatives considered

- **FR-1a: hard `-f` codec filter (`bestvideo[vcodec^=avc1]+bestaudio[acodec^=mp4a]`)
  instead of a soft `-S` sort.** Rejected: a hard filter *excludes* non-avc1
  formats, so the `1440p`/`2160p` tiers (YouTube serves those only as VP9/AV1)
  would silently downgrade to ≤1080p or fail selection. The soft `-S` prefers
  H.264/AAC when present and falls through to best-available otherwise — the
  broad, low-risk behavior Dean asked for.
- **FR-1a: emit `--recode-video mp4` as an unconditional fallback.** Rejected as
  a default this round (see Decision log / Dean-decision below): yt-dlp's
  `--recode-video` is *container*-triggered, not codec-conditional, so it cannot
  be made to fire "only when no H.264 stream exists" without a pre-download
  probe — it would either over-fire (wasteful re-encode of already-H.264 content
  muxed to mp4) or under-fire (VP9-in-mp4 passes through). The soft `-S` already
  captures essentially all real-world YouTube content (H.264 up to 1080p exists
  for nearly everything). Left as a documented, deferred enhancement.
- **FR-1b: add `.webm`/`.mkv` to `TRANSCODE_EXTENSIONS` (stay extension-based).**
  Rejected: over-transcodes legitimately-playable H.264/AAC `.mkv`/`.mp4` files
  and still misses HEVC-in-`.mp4`. Codec detection is the correct axis and is
  free (piggybacks the existing probe).
- **FR-2: a new blank/branded placeholder image set as `poster`.** Rejected per
  fork #5: no asset exists today and the `#000` container background is already
  the correct neutral. `removeAttribute('poster')` + element `load()` reveals it
  with zero new assets and zero new CSS.
- **FR-3: unbounded names array.** Rejected per fork #6: codec detection can flag
  a large fraction of a big library; capped at 10 + overflow count.

### Risks and mitigations

- **Risk**: the `-S` soft sort downgrades resolution on a source where the only
  H.264 stream is low-res. → **Mitigation**: the per-tier height ceiling in `-f`
  still bounds the pool; yt-dlp's default res/fps/bitrate tie-breakers are
  appended after our codec fields, so within the ceiling it still picks the best
  H.264 rendition. Dean's on-device pass is the arbiter for the codec/quality
  tradeoff.
- **Risk**: the codec backfill re-probes a whole existing library on the first
  post-upgrade scan. → **Mitigation**: probing is a metadata-only ffprobe call
  (no transcode), bounded by the existing single-threaded scan loop; it happens
  once per item, then the codec fields are cached. Transcoding stays strictly
  lazy/on-watch.
- **Risk**: a media filename with a shell metacharacter in the ffprobe call. →
  **Mitigation**: the recommended `execFile` conversion removes the interpolation
  entirely on that line (strictly narrows the pre-existing surface).
- **Risk**: `mediaPlayer.load()` in teardown interrupts playback continuity. →
  **Mitigation**: `teardownMediaState()` only runs on genuine new loads; the
  `adopt` (dock↔full) path returns before it. `teardownMediaState()` already
  calls `mediaPlayer.pause()` — the `load()` is on media that is being replaced
  anyway.
- **Risk**: FR-1b flags many files → higher transcode volume → disk pressure. →
  **Mitigation**: reuses the existing size-capped LRU cache eviction + single-
  worker queue unchanged; only the *trigger set* grows, not the mechanics.

### Performance impact

No change to the transcode queue, cache, or eviction mechanics (RELIABILITY.md
budgets unchanged). Codec probing adds two fields to an ffprobe call that already
runs per file — effectively free, per the plan's hard constraint. Actual
transcoding remains lazy/on-watch — no eager scan-time transcode path is
introduced. The one new cost is the one-time codec backfill re-probe of existing
video items on the first post-upgrade scan (metadata-only, no transcode).
