## Verified bottlenecks (file:line evidence from the sweep)

- **B1 (freeze, O(NÂ²)):** every scan re-runs subtitle-sidecar detection per file via
  `readdirSync` of the file's own directory (`lib/subtitles.js:134`, invoked
  unconditionally for unchanged files at `server.js:1602-1611`) â€” ~1300 readdirs of the
  same ~1300-entry dir per rescan (~1.7M entry-regex ops), synchronous.
- **B2 (unresponsive):** `POST /api/scan` awaits the entire scan before responding
  (`server.js:2625-2643`); scan walk is `readdirSync`/`statSync` per file
  (`server.js:2290`, `:2323`) on the event loop; client awaits then hard-reloads
  (`main.js:590-597`).
- **B3 (slow open/scroll):** every hot route does a full `loadDatabase()`
  (readFileSync + JSON.parse of the whole db) per request â€” including
  `/thumbnail/:id` (`server.js:3948`), `/video/:id`, `/audio/:id`, `/api/videos`
  (`:2832`), progress, config. Scrolling = hundreds of ~800KB parses.
- **B4 (payload/render):** `/api/videos` returns everything unpaginated, full copy +
  sort per request (`server.js:2831-2882`); client renders all cards in one
  synchronous `innerHTML` join (`main.js:479-551`). Thumbnails already lazy.
- **B5 (write amplification):** `saveDatabase` stringifies + fsyncs the whole db
  synchronously on every write (`server.js:202-223`) including the 4s playback
  progress ping (`player.js:2595-2606` â†’ `server.js:2930-2962`).
- **B6 (boot):** full synchronous scan at startup before the server is usefully
  responsive (`server.js:4444`).
- **B7 (one-shot visibility gaps):** (a) done-edge consumed into
  `seenDoneOneShotJobIds` even when `refreshLibraryInPlace` had no home target
  (`common.js:4799-4801`), and `restoreHomeFromCache` deliberately does not re-render
  the grid (`common.js:2633-2689`) â†’ stale grid until manual rescan; (b) chip poll
  early-returns while `document.hidden` (`common.js:4782-4785`) so a backgrounded PWA
  can miss the edge if the server snapshot drops the done entry before resume.
- Already-good (do NOT redo): ffprobe/thumbnail extraction is correctly incremental
  (`server.js:1864-1897`); scan overlap guard coalesces concurrent triggers
  (`server.js:1681-1704`); thumbnails `loading="lazy"`.

## Scope

### Lane A â€” Performance at scale (the daily pain)

- **A1: Kill the O(NÂ²)** â€” one directory listing per directory per scan (cache the
  readdir; or skip sidecar re-detection for unchanged files, consistent with the
  incremental posture). (B1)
- **A2: Non-blocking scan** â€” `POST /api/scan` returns 202 immediately; scan runs
  cooperatively (async fs / yielding batches) so other requests keep flowing; client
  polls `/api/scan-status` and refreshes the grid in place on completion (no
  `window.location.reload`). Boot scan must not gate first responses. (B2, B6)
- **A3: In-memory DB cache** â€” parse db.json once, serve all reads from memory,
  invalidate/update on write; hot routes (esp. `/thumbnail/:id`) stop re-parsing per
  request. (B3)
- **A4: Cheap writes** â€” progress pings and other frequent writers stop rewriting the
  whole file synchronously every 4s (debounce/batch progress; keep crash-safety
  posture for real mutations; atomic-rename discipline preserved). (B5)
- **A5: Paginated `/api/videos` + lazy/infinite scroll grid** â€” server takes
  limit/offset (or cursor), keeps total count for headers; client renders the first
  window fast and appends on scroll (IntersectionObserver sentinel), preserving
  search/sort/filter semantics across the full library. (B4; Dean's UX pick)
- **A6: SQLite decision point** â€” design stage evaluates whether A3+A4 suffice or the
  migration is warranted (better-sqlite3-class, synchronous-API-friendly, safe
  one-way migration from db.json with backup + rollback path). Dean authorized it;
  evidence decides. If deferred, record as explicit tech debt with trigger criteria.

### Functional requirements (FR, one per deliverable)

- **FR1 (rescan-at-scale responsiveness):** A rescan that finds zero changed
  files must not block the Node event loop for a duration that makes the
  server feel frozen to a concurrent client. This must be true at Dean's real
  scale (~1300 items) and is asserted by an automated proxy for "the event
  loop is still responsive," not by subjective feel.
- **FR2 (non-blocking scan):** `POST /api/scan` must acknowledge the request
  and return before the scan finishes; the scan must run cooperatively so it
  does not starve other HTTP handlers; the client must reflect progress and
  completion via polling + in-place DOM update, never a full-page reload.
- **FR3 (fast open + smooth scroll at scale):** With 1300+ items, the first
  render must be bounded to a "page," not the whole library, and repeat hot
  requests (thumbnails in particular) must not re-parse the whole on-disk
  database file per request.
- **FR4 (cheap progress writes):** The playback-progress heartbeat (client
  cadence: every 4s, `POST /api/progress`) must not force a whole-file
  stringify+fsync on every ping. Real mutations (deletes, config/settings
  changes, scan merges) are NOT part of this relaxation and keep the existing
  atomic-write+fsync guarantee unchanged.
- **FR5 (one-shot download visibility):** A completed one-shot download must
  become visible in the home grid without the user taking any action, across
  three surfaces: currently on home, navigated away and back (cache restore),
  and PWA resumed from background. The done-edge must never be silently
  dropped when no refresh target was live.
- **FR6 (active-downloads chip):** The corner chip must reflect only
  currently-downloading jobs; it must be absent when nothing is actively
  downloading (queued-only is NOT "active"); it must be tappable to a detail
  view with progress/attribution; queued items must never expose a
  stop/dequeue affordance.
- **FR7 (visual polish cluster):** Typography must be tokenized (no stray
  hardcoded font sizes reintroduced) while preserving the existing 16px input

### A6 â€” SQLite decision: DEFER (verdict, with evidence)

**Verdict: DEFER SQLite. Ship the incremental path (A1 + A2 + A3 + A4 + A5).
Filed as tech-debt tracker item #28 with measurable trigger criteria (below and
in `docs/exec-plans/tech-debt-tracker.md`).**

Evidence-based justification (the O(NÂ²)/parse-storm findings in hand):

- **The two headline bottlenecks are NOT storage-engine problems.** B1 (the
  freeze) is an *algorithmic* O(NÂ²): one `readdirSync` of a file's own directory
  per file (`lib/subtitles.js:134` via `server.js:1602-1611`). A SQLite
  migration does nothing for it â€” the fix is to stop listing the same directory
  1300 times (A1). B2 (unresponsive) is a *synchronous walk on the event loop*
  (`readdirSync`/`statSync` per file at `server.js:2290/2323`) plus `POST
  /api/scan` awaiting the whole scan â€” again storage-agnostic; the fix is
  cooperative async fs + a 202 ack (A2). Migrating the store would leave both
  intact.
- **The read storm (B3) is better solved by an in-memory cache than by SQLite.**
  The whole DB fits in memory (measured 179 KB at 284 items â†’ ~800 KB at ~1300,
  ~1.2 MB projected at ~2000). Serving hot reads (`/thumbnail/:id`, `/video/:id`,
  `/api/videos`) from an already-parsed in-memory object is O(1) with *zero*
  query/serialization cost â€” strictly faster than any per-request SQLite query.
  SQLite would help only if the working set no longer fit in memory, which is
  orders of magnitude beyond this app's scale.
- **The write storm (B5) is fully bounded by batching (A4).** The amplifier is
  the 4 s progress ping rewriting the whole file. Coalescing progress to â‰¤1
  write per â‰¤5 s window meets the â‰¥5:1 target. A ~1.2 MB `JSON.stringify` +
  `fsync` costs single-digit-to-low-double-digit ms and happens for *real*
  mutations only (infrequent) â€” comfortably inside budget for a single-user LAN
  box. SQLite's per-row write advantage is real but solves a problem this app
  does not have at this scale.
- **Cost/risk of migrating is high and lands on the exact surface this wave must
  not destabilize.** `better-sqlite3` is a *native* addon: it must have a
  prebuilt binary (or compile) matching Docker Alpine/musl **and** both Node 22
  **and** Node 24 (AC8.2 explicitly tests Node 24) â€” adding ABI-matching
  fragility to the release gate. Migration also demands rewriting every
  `db.json` reader/writer, the `updateDatabase` re-read-merge mutex, the scan
  final-merge, and re-validating *every* crash-safety and preserved-guard AC
  (prune/mount-loss, incremental reuse, overlap coalescing) against a new store
  â€” plus a hard one-way-migration-with-backup-and-rollback requirement. That is
  an enormous blast radius on a crash-safety-sensitive codebase, for a box the
  incremental path already makes fast.

**Conclusion:** A3 (in-memory read cache) + A4 (batched progress writes) + A2
(cooperative scan) + A1 (kill the O(NÂ²)) clear the FR1/FR3/FR4 bounds at ~1300
with clear headroom to ~2000+. SQLite is authorized but not warranted by the
evidence. Deferral is tracked, not silent (item #28).

**Trigger criteria to revisit SQLite (tracker #28):** revisit if, after A1â€“A5
ship, ANY of these is observed â€”

1. The FR1 heartbeat proxy still records event-loop stalls > 50 ms/stretch
   during an unchanged rescan at N â‰¥ 2000 items; or
2. `db.json` exceeds ~5 MB (â‰ˆ several thousand items) such that a single
   real-mutation `saveDatabase` (`JSON.stringify` + `fsync`) exceeds ~50 ms; or
3. Boot parse of `db.json` (first `loadDatabase`) exceeds ~100 ms; or
4. The single-writer assumption is broken (multi-user / a second writer process
   is introduced) â€” the in-memory-cache coherency argument (below) no longer
   holds and a real transactional store becomes warranted.

### A1 â€” sidecar-detection cache shape (B1 / AC1.3, AC1.7)

**Mechanism: a per-scan directory-listing cache, passed into
`findSubtitleSidecar`.** One `readdir` per unique directory per scan, memoized
in a `Map<dir, string[]>` that lives for the duration of a single
`runScanDirectories` pass and is discarded when it ends.

- `lib/subtitles.js`: extend `findSubtitleSidecar(filePath, fsImpl, dirCache)`
  with an optional `dirCache`. When present: (a) resolve the directory listing
  from the cache (populate on miss via one `readdirSync`); (b) answer the
  `<base>.vtt` and `<base>.srt` exact-match probes from *listing membership*
  instead of two per-file `existsSync` calls; (c) run the identical anchored
  `langVttRe` scan over the cached entries. The regex, precedence, and
  fail-closed semantics are byte-for-byte unchanged â€” only the *source* of the
  entry list changes. With no `dirCache` (every existing direct caller / unit
  test), behavior is byte-identical to today.
- `server.js`: `runScanDirectories` constructs one `perScanReaddirCache` and
  threads it through `applyHasSubtitlesDetection(existing, filePath, dirCache)`
  â†’ `findSubtitleSidecar`. Recomputation-every-scan is preserved (a freshly
  dropped/removed sidecar is still picked up on the next scan â€” the cache is
  per-scan, not persistent), honoring the existing `applyHasSubtitlesDetection`
  contract at `server.js:1602-1617`.
- **AC1.7 (change-detection untouched):** the cache memoizes *only* directory
  listings. The `unchanged = existing.filePath === filePath && existing.size
  === info.size` decision (`server.js:1868`) is not read or written by this
  path â€” a changed file is still detected exactly as today.
- **Instrumentation (AC1.3):** the cache exposes a hit/miss counter (or the
  developer injects an `fsImpl` whose `readdirSync` is a counting spy). Test:
  seed N unchanged files in one directory, run a rescan, assert the readdir
  count is 1 (constant) for that directory, not N.

**Alternative considered:** *skip* sidecar re-detection for unchanged files
entirely. Rejected â€” it changes an observable behavior (a `.srt` dropped next to
an existing, unchanged video would not be detected until the video itself
changed), which the current code deliberately avoids. The per-scan cache gets
the same O(1)/dir win without that regression.

### A2 â€” cooperative / non-blocking scan (B2, B6 / AC2.1â€“AC2.5)

**`POST /api/scan` acks immediately; the walk is async + batch-yielding; the
boot scan runs after `listen`; the client polls and refreshes in place.**

- **202 ack (AC2.1):** `POST /api/scan` calls `scanDirectories()` **without
  `await`** (fire-and-forget, mirroring `POST /api/config` at `server.js:2621`)
  and responds `202 { scanning: true, alreadyInProgress: <bool> }` synchronously.
  If a scan is already running it still flags `rescanRequested` (coalescing) and
  returns 202 with `alreadyInProgress: true`. Ack is O(1) â†’ â‰¤100 ms trivially.
- **Cooperative walk (AC1.1, AC1.2):** convert `scanDirRecursive` to async
  `fs.promises.readdir(dir, { withFileTypes: true })` per directory, processing
  file entries in batches of `SCAN_YIELD_BATCH` (default 64) with an
  `await new Promise(setImmediate)` between batches so no synchronous stretch
  exceeds the 50 ms bound. `fs.promises.stat` per file yields naturally. The
  `unreadable` set is populated exactly as today (a rejected `readdir`/`stat`
  â†’ `unreadable.add(dir)` / retain), so the mount-loss and per-file guards
  (`server.js:2287-2350`) are preserved bit-for-bit.
- **Final merge stays atomic (AC1.6/1.7, crash-safety):** the single
  `updateDatabase` final-merge mutator (root backfill, `transcodeStatus` seed,
  `mergeScannedMetadata`, progress prune â€” the one documented in
  ARCHITECTURE.md) is unchanged. It is the one intentionally-synchronous stretch;
  it operates on in-memory objects and one atomic write (~10 ms at 1.2 MB),
  inside budget. Incremental ffprobe/thumbnail reuse (`server.js:1864-1897`) is
  untouched.
- **Boot scan not gating (AC2.4):** move the boot `scanDirectories()` call to
  **inside/after the `app.listen` callback** via `setImmediate`, so route
  serving is never sequenced behind the scan's first synchronous stretch. `GET
  /api/config` issued immediately post-boot returns while the scan runs.
- **Progress + status (AC2.2):** extend `scanState` to
  `{ scanning, lastScan, rescanRequested, processed, total, phase }`. The
  cooperative walk increments `processed`/`total` monotonically; `GET
  /api/scan-status` surfaces them. Progress never regresses within a pass.
- **Overlap coalescing preserved (AC2.5):** `scanDirectories`'s guard
  (`scanState.scanning` â†’ set `rescanRequested`), `MAX_RESCAN_FOLLOWUPS`, and
  `scheduleDeferredRescan` (`server.js:1681-1704`) are wholly unchanged â€” we
  make `runScanDirectories` cooperative *inside* the guard, not around it. The
  `currentDeferredRescanTimer`/`currentScanTimer` test accessors still prove at
  most one scan runs and a fresh trigger still starts promptly.
- **Client (AC2.3, main.js `rescanBtn` at `main.js:590-597`):** replace the
  `await fetch('/api/scan')` â†’ `window.location.reload()` with: POST â†’ 202 â†’
  poll `GET /api/scan-status` (existing chip-style backoff) until `scanning`
  flips false â†’ call `window.__filetubeRefreshLibrary()` (the in-place home
  refresh). **`window.location.reload` is removed from this path** â€” this
  generalizes the BUG-2 contract to scan completion. DOM-level test asserts the
  scan-completion handler calls the in-place refresh and never `location.reload`.

**Alternative considered:** keep the walk synchronous but move it to a
`worker_thread`. Rejected â€” a worker needs its own fs view and a serialization
boundary for results, far more surface than batch-yielding, and the extraction
loop already yields; cooperative async fs is the minimal change that satisfies
the heartbeat bound.

### A3 â€” in-memory DB cache + invalidation contract (B3 / AC3.3)

**A read-through cache in front of the existing `updateDatabase` mutex; the
mutex's disk re-read-merge invariant is preserved unchanged.**

- **Reads:** introduce `getCachedDatabase()`. If the cache is valid, return the
  cached parsed object; else call `loadDatabase()` (one disk read+parse), store
  it, mark valid, return it. All hot GET readers switch from `loadDatabase()` to
  `getCachedDatabase()`: `/thumbnail/:id` (`server.js:3948`), `/video/:id`,
  `/audio/:id`, `/api/videos` (`:2832`), `/api/videos/:id`, `/api/progress/:id`,
  `/api/config`, `/api/settings`, `/api/scan-status`, the subtitle route.
- **Writes (invariant preserved):** `updateDatabase` still loads a **fresh copy
  from disk** inside the lock, applies the mutator, and `saveDatabase`s
  atomically â€” the re-read-merge-on-save concurrency invariant in
  ARCHITECTURE.md is unchanged. The single added step: on a **successful** save,
  set `dbCache = <the just-saved object>; dbCacheValid = true` â€” inside the same
  synchronous critical section (no `await` between save and cache-set).
- **Coherency argument (no torn/stale reads):**
  1. Every writer is serialized through the one `updateDatabase` promise chain;
     each write's loadâ†’mutateâ†’saveâ†’cache-set is one synchronous tick.
  2. Node is single-threaded; a synchronous reader runs entirely before or
     entirely after any given write's critical section â€” never interleaved â€” so
     it always sees a fully-consistent snapshot.
  3. This process is the *only* writer of `db.json` (single-node, single-process
     â€” ARCHITECTURE.md constraint), so the cache cannot silently drift from disk.
  4. The cached object is **replaced by reference, never mutated in place** by
     readers or by real mutations (mutators operate on the freshly-loaded disk
     copy, then that copy *becomes* the cache) â€” so a reader holding a reference
     to a prior snapshot is always internally consistent.
- **AC3.3 instrumentation:** a module-level `loadDatabase` call counter with a
  test accessor `__getLoadDatabaseCallCount()`. Integration test: fire N
  sequential `/thumbnail/:id` against an unchanged DB, assert the counter is
  O(1) (1 population + 0 thereafter), not N.

**Alternative considered:** have `updateDatabase` also read from the cache
(skipping the write-path disk read, since cache == disk). Rejected for this
wave â€” it alters the documented "loads a FRESH db from disk" invariant and buys
little (real mutations are infrequent; A4 removes the progress-write frequency).
Kept as a possible future optimization, not this wave's risk.

### A4 â€” progress-durability relaxation bounds (B5 / AC4.1â€“AC4.3)

**A last-write-wins progress coalescer with a â‰¤5 s flush window; every real
mutation keeps 1:1 atomic-write+fsync.**

- **Coalescer:** a module-level `pendingProgress` `Map<id, {timestamp, duration,
  updatedAt}>`. `POST /api/progress` (`server.js:2930`): validate `id` exists
  (against `getCachedDatabase().metadata`), `pendingProgress.set(id, value)`,
  arm a single debounce timer (`PROGRESS_FLUSH_MS`, default 5000) if not already
  armed, respond `200 { success: true }`. No per-ping `updateDatabase`.
- **Flush** (timer fire): snapshot-and-clear the pending entries, then one
  `updateDatabase(db => { for (const [id, v] of snapshot) if (db.metadata[id])
  db.progress[id] = v; return true })` â€” a **single** whole-file atomic
  write+fsync covering the whole window. N pings â†’ ceil(window-span / 5 s)
  writes â‡’ â‰¥5:1 for any realistic burst (4 s ping cadence, 5 s window).
- **Read-your-writes (overlay, keeps A3's no-in-place-mutation rule):** progress
  readers overlay `pendingProgress` over the cache:
  `effectiveProgress(id) = pendingProgress.get(id) ?? cache.progress[id]`.
  Applied in `GET /api/progress/:id`, `GET /api/videos` (the per-item progress
  map at `server.js:2869`), and `GET /api/videos/:id`. So a just-posted position
  is visible immediately without mutating the shared cache object.
- **Crash-safety carve-out (AC4.2 â€” real mutations unaffected):** ONLY
  `POST /api/progress` uses the coalescer. `DELETE /api/videos/:id`,
  `POST /api/config`, `POST /api/settings`, the scan final-merge, and the new
  C2 like-membership routes all keep calling `updateDatabase` directly â‡’ exactly
  1 atomic write+fsync per invocation. Write-count instrumentation (a
  `saveDatabase` call counter, shared with AC4.1) asserts a 1:1 ratio for these
  routes and â‰¤ N/5 for the progress burst.
- **Shutdown / crash (AC4.3):** a `process.on('SIGTERM')` / `SIGINT` /
  `beforeExit` handler flushes `pendingProgress` synchronously (one final atomic
  `saveDatabase`). On a hard `SIGKILL`, the unflushed â‰¤5 s window of *watch
  position only* is lost; the on-disk file is never torn (atomic
  write-temp-then-rename is unchanged) and always parses (loadDatabase's
  corrupt-JSON recovery is unchanged). Nothing but watch position is ever at
  risk, because metadata/folders/settings/downloads never route through the
  coalescer. Test: simulate a crash at an arbitrary point relative to a flush,
  assert the file still parses and only bounded progress is lost.

### A5 â€” pagination contract (B4 / AC3.1â€“AC3.4)

**limit/offset with a server-authoritative sort+filter, returning a bounded page
plus a total count. Sort/filter/search move server-side so paging is a slice of
the fully-ordered full library.**

- **`GET /api/videos` request:** existing `search`, `folder`, `root` **plus** new

- **`GET /api/videos` response (API CHANGE):** from a bare array to

- **Server pipeline:** `getCachedDatabase()` â†’ hidden-folder filter (home only,
  unchanged) â†’ search â†’ `root`/`folder` filter â†’ `format` filter â†’ **sort the
  full filtered list** â†’ slice `[offset, offset+limit)` â†’ overlay pending
  progress â†’ return with `total = <full filtered length>`.
- **AC3.2 (cross-window correctness):** because sort+filter run over the FULL
  filtered library *before* slicing, `page(sort, filter)` concatenated equals
  `sort(filter(full))` then sliced â€” this is the property the test asserts
  (paged == unpaged-then-sliced; a match on "page 3" is found when that

  keeps `random` stable across sequential page fetches.
- **Shared, testable ordering:** extract the sort comparators, format predicate,

- **Client (main.js):** `loadLibrary` sends `sort` (`currentSort`), `format`
  (`getStoredFormatFilter()`), `limit`, `offset=0`, and a fresh `seed`; renders
  page 0 only. The home grid **stops re-sorting/re-filtering locally** for the
  paginated view (the server is now authoritative) â€” `renderSorted` becomes a

  `IntersectionObserver` sentinel at the grid tail fetches exactly one next page

  `renderMediaGridPage(items, { append: true })` path (createElement/append, not
  a full `innerHTML` rebuild). Sort change / format change / search / shuffle-
  again reset to a fresh page 0 (new `seed` for random).
- **AC3.1 latency:** sorting â‰¤2000 items in memory (no parse, from A3 cache) is
  sub-millisecond; first page â‰¤150 ms trivially.
- **AC3.4 DOM test (jsdom):** initial card count == page size (or total if

**Alternative considered:** cursor pagination. Rejected â€” the library reorders

### B1 â€” done-edge / dirty-flag (B7 / AC5.1â€“AC5.4)

**Consume the done-edge only when a live refresh actually happened; otherwise
mark the home grid dirty and reconcile on cache-restore and on
visibility/pageshow resume. Never `window.location.reload`.**

- **`refreshLibraryInPlace()` returns a boolean** (`common.js:4410`): `true`
  iff `window.__filetubeRefreshLibrary` was a function and was invoked.
- **Edge consumption (AC5.1, AC5.2 at `common.js:4799-4801`):** in `pollOnce`,
  for each newly-done one-shot: call `refreshLibraryInPlace()`; if it returned
  `true` (a live home target refreshed) add the jobId to `seenDoneOneShotJobIds`
  (today's behavior â€” AC5.2a). If it returned `false` (no live home target),
  **do not silently drop it**: add to the seen-set to stop re-processing AND
  call `markHomeGridDirty()` (a `localStorage` flag `filetube_home_dirty=1`,
  survives navigation and backgrounding). AC5.2b.
- **Cache-restore reconcile (AC5.2b, `restoreHomeFromCache` at
  `common.js:2633`):** at the top, `if (isHomeGridDirty()) { clearHomeGridDirty();
  homeViewCache = null; /* fall through to a fresh home load instead of
  reattaching the stale cached node */ }`. When dirty, a fresh
  `loadLibrary`-driven render replaces the stale cache so the completed item
  appears â€” no manual rescan, no reload.
- **Backgrounded-resume (AC5.3):** maintain a persistent `pendingOneShotJobIds`
  (`localStorage`) â€” on every active poll tick (and on `visibilitychange` â†’
  hidden), record the jobIds currently `queued`/`downloading`. Add a
  `visibilitychange` + `pageshow` listener that, on becoming visible: (a)
  triggers an immediate `pollOnce()` (overriding today's `document.hidden`
  early-return at `common.js:4782-4785` for this one catch-up call); (b) for any
  previously-pending jobId now absent-or-`done` in the fresh snapshot, treats it
  as completed â†’ `refreshLibraryInPlace()` or `markHomeGridDirty()`, then clears
  it from `pendingOneShotJobIds`. This surfaces a job that completed *and was
  dropped from the server snapshot* while hidden, because we remembered it was
  in flight.
- **AC5.4 (reload-never):** all four paths use in-place refresh /
  `loadLibrary` / dirty-reconcile; none call `window.location.reload`. Tests spy
  `location.reload` across all three surfaces and assert zero calls.

**Alternative considered:** keep the done-edge *unconsumed* (never add to the
seen-set) when off-home, and re-fire on every subsequent poll. Rejected â€” it
re-triggers a no-op refresh every tick and risks a duplicate refresh storm on
return; the dirty-flag is a single, cheap, idempotent reconcile signal.
