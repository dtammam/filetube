# v1.22.2 — Three-FR Polish Round (Audio Fullscreen, Favicon, Rating Wrap)

## Goal

Ship three small, independent fixes requested after Dean's on-device iPhone
pass of v1.22.1: (1) give audio playback a genuinely-working "fullscreen"
(CSS-expanded now-playing) view, (2) make the favicon display consistently
across browsers and in bookmarks, and (3) stop the watch-page "N / 5" rating
text from wrapping to two lines.

## Scope

### FR-1 — Working audio "fullscreen" (CSS-expanded now-playing view)

Add a full-viewport "expanded now-playing" view for AUDIO playback: a
CSS `position:fixed; inset:0` class-toggle overlay (high z-index) that
enlarges the cover art (`#audio-bg-art`) and the custom control bar to fill
the screen. This is explicitly **not** the browser Fullscreen API — iPhone
Safari refuses `webkitEnterFullscreen()`/`requestFullscreen()` on a
non-video element, which is exactly why v1.22.1 (FR-2) hid the mobile-audio
`#fs-btn` in the first place. Video fullscreen keeps using the real
Fullscreen API and is untouched by this FR.

### FR-2 — Favicon displays consistently (tab vs. bookmark, across browsers)

Add a raster favicon fallback (PNG and/or `.ico`, with explicit `sizes`/
`type`) alongside the existing SVG `rel="icon"` link, applied
byte-identically across all four HTML shells, so the icon renders
consistently in tabs, bookmarks, history, and browsers that don't consume
SVG favicons.

### FR-3 — Star-rating "N / 5" stays on one line

Fix the watch-page rating count text (`#rating-text` / `.rating-count`,
inside `.star-rating`) so "N / 5" never wraps under a width squeeze.

## Design

### Approach

Three independent, surgical fixes across three disjoint file sets, all
shippable in a single wave (no two concurrent tasks share a file):

- **FR-1 (audio expanded now-playing view)** is a CSS full-viewport
  class-toggle on the persistent host `#player-wrapper` (`.audio-expanded`),
  driven from `public/js/player.js` by **reusing the existing `#fs-btn`** with
  an `.audio-mode` branch — never the Fullscreen API. The JS (T1) toggles the
  class and force-clears it on every lifecycle exit; the CSS (T2) makes
  `#player-wrapper.audio-mode.audio-expanded` a `position: fixed; inset: 0`
  overlay and re-shows the mobile-audio `#fs-btn`. The video Fullscreen path
  (`enterFullscreen()`, `inNativeFullscreen()`, `webkitEnterFullscreen`/
  `requestFullscreen`, the `f`/`F` shortcut, rotate-to-fullscreen) is left
  byte-identical.
- **FR-2 (favicon)** adds PNG `rel="icon"` fallback `<link>`s (reusing the
  existing `/icons/icon-192.png` + `/icons/icon-512.png`) alongside the SVG
  link, byte-identically across all four shells (T3). No new asset, no `.ico`
  generation step, no dependency.
- **FR-3 (rating one-line)** adds `white-space: nowrap` (+ `flex-shrink: 0`)
  to `.rating-count` (T2, same `style.css` task as FR-1's CSS).

The cross-file contract between T1 and T2 (frozen): the toggled class is
`audio-expanded` on `#player-wrapper` (the host), and in `.audio-mode` the
`#fs-btn` toggles it instead of entering native fullscreen. Both tasks are
written against that exact class name.

### FR-1 mechanism (per-FR mechanics)

**Control: reuse `#fs-btn` (chosen).** The `#fs-btn` click handler
(`player.js` ~1707-1721) currently routes unconditionally through
`enterFullscreen()`. Add a single decision at the top of the handler and
branch:

- New module-level **pure helper** `resolveFsButtonAction({ audioMode })`
  returning `'audio-expand'` when `audioMode` is truthy, else
  `'native-fullscreen'`. Exported alongside the other pure helpers
  (`player.js` ~400-415) and unit-tested (repo's no-DOM `node:test`
  convention). This is the testable seam that documents and locks the
  audio-vs-video branch point (guards AC4's byte-identical video path).
- Live handler: read `audioMode` from `host.classList.contains('audio-mode')`,
  pass it to the helper. On `'audio-expand'` call a new `toggleAudioExpand()`;
  on `'native-fullscreen'` run the **existing** enter/exit-native block
  unchanged (byte-identical inner code).

Chosen over a **new control id** because a new control would require
byte-identical markup added to all four `#player-host-template` shells (the
`#pip-btn`/`#speed-btn` parity discipline). Reusing `#fs-btn` avoids that
four-shell churn entirely — the button already exists in every shell and is
already CSS-hidden only for mobile audio (the one rule we remove).

**Enter/exit + restore (AC3, AC5).**

- `toggleAudioExpand()`: FULL-only guard (`state === STATE_FULL`), then
  `host.classList.toggle('audio-expanded')`. FULL-only matches every other
  fullscreen/gesture guard in this file and the existing video precedent.
- `exitAudioExpand()`: `if (host) host.classList.remove('audio-expanded')`.
  Called from **`teardownMediaState()`** (next to the existing
  `host.classList.remove('audio-mode')` at ~1937 — covers every genuine
  non-adopt load / Next / navigate-away), **`dock()`** (~2069, so docking
  while expanded never docks a fixed-overlay wrapper), and **`close()`**
  (~2083). These three cover every transition out of FULL, so the expanded
  view can never get "stuck" (AC5).
- **Exit affordances:** primary is re-tapping the same `#fs-btn` (works on
  touch and pointer, inherently discoverable — it is the control the user just
  pressed). Secondary desktop convenience: a small dedicated `keydown`
  listener (added in `wireHostListeners`) that, only when
  `state === STATE_FULL && host.classList.contains('audio-expanded')`, calls
  `exitAudioExpand()` on `Escape`. Kept as a separate listener rather than
  folded into the existing FULL-only shortcut switch (~1869) so the `f`/`F`
  video path stays byte-identical and the existing `activeElement`/`BUTTON`
  early-return does not interfere with the exit.

**Scope: FULL-only** (confirmed). The toggle no-ops unless `state ===
STATE_FULL`; DOCKED audio keeps its tap-to-expand-to-watch behavior untouched.

**Coexistence (AC6).** Because `.audio-expanded` makes `#player-wrapper` a
`position: fixed; inset: 0` viewport overlay, it escapes the flow of its FULL
slot — so v1.21 **THEATRE mode** (`.watch-container.theater-mode`, an ancestor
layout class ~3939) has no effect on the fixed wrapper: no conflict. Media
Session, progress saving, and loop are all wired on `mediaPlayer` and are
independent of any wrapper class — untouched while expanded or after exit. The
dock FULL/DOCKED/CLOSED model is preserved because the class only ever exists
in FULL and is force-cleared on the way to DOCKED/CLOSED. Nothing is persisted
(AC7) — it is an in-memory class toggle only. The enlarged `#audio-bg-art`
(`background-size: cover`, already `inset: 0`) and `.player-controls`
(`position: absolute; bottom: 0`) fill the fixed wrapper with no new geometry.

### FR-2 format choice (favicon)

**Add PNG `rel="icon"` fallbacks; do not add a `.ico`.** Root cause: the only
tab icon is `rel="icon" type="image/svg+xml"`; browser bookmark bars, history,
and non-SVG-favicon browsers fall back to a default when no raster
`rel="icon"` exists. A PNG `rel="icon"` with explicit `type`/`sizes` is
consumed by every modern browser for tabs and bookmarks, reuses the two PNGs
already committed under `/icons/`, and needs **no** asset-generation step or
dependency. A real multi-resolution `favicon.ico` would add robustness only
for legacy browsers but requires committing a new binary or a generation tool
— rejected as not worth the friction (AC allows "PNG and/or .ico").

The icon `<link>` block (lines 7-9 today) becomes, byte-identically across all
four shells:

```html
  <link rel="icon" type="image/svg+xml" href="/favicon.svg">
  <link rel="icon" type="image/png" sizes="192x192" href="/icons/icon-192.png">
  <link rel="icon" type="image/png" sizes="512x512" href="/icons/icon-512.png">
  <link rel="apple-touch-icon" href="/icons/icon-192.png">
  <link rel="manifest" href="/manifest.webmanifest">
```

The existing SVG, `apple-touch-icon`, and `manifest` lines are unchanged
(AC13). The static route (`server.js` ~1789, `express.static(public,
{Cache-Control: no-cache})`) already serves `/icons/*.png`, so both PNGs
resolve with no 404 and no route/header change needed (AC12, AC14). SVG-first
ordering lets SVG-capable browsers keep the crisp vector while others take the
nearest PNG.

### FR-3 mechanics (rating one-line)

`.rating-count` (`style.css` ~1365-1369) is a flex child of `.star-rating`
with no wrap control, so `${rating} / 5` (`watch.js:779`) wraps under a width
squeeze in the `.watch-actions` row. Add `white-space: nowrap` (keeps
"N / 5" atomic) and `flex-shrink: 0` (keeps the count from being compressed)
to that rule only. Scoped to `.rating-count`/`.star-rating`; the home-card
`.card-rating` glyph variant (`main.js:468`, a different class, glyphs not
"N / 5" text) shares no selector and is unaffected (AC18). Existing tokens
(`--text-secondary`) preserved (AC19).

### Alternatives considered

- **New dedicated audio-expand control** (separate button/close chrome)
  instead of reusing `#fs-btn`. Pro: unambiguous glyph, no overloading. Con:
  requires byte-identical markup across all four `#player-host-template`
  shells (parity test + churn) for a button that duplicates the fullscreen
  affordance users already expect. Rejected — reusing `#fs-btn` is smaller and
  matches the existing mental model; the same-button-toggles pattern gives a
  clean exit for free.
- **Generate a real `favicon.ico`** (multi-res) for FR-2. Pro: maximal legacy
  coverage. Con: commits a new binary asset or adds a generation step; no
  measurable benefit over PNG `rel="icon"` for the browsers Dean actually
  uses. Rejected in favor of the zero-asset PNG fallback.
- **Fold Escape into the existing FULL-only keydown switch** (~1869) for
  FR-1's desktop exit. Con: that handler early-returns on a focused `BUTTON`
  (the `#fs-btn` the user just clicked) and shares the `f`/`F` video block we
  must keep byte-identical. Rejected — a small dedicated listener is cleaner
  and lower-risk.

### Risks and mitigations

- **Risk:** expanded view gets "stuck" across dock/close/navigate (AC5). →
  **Mitigation:** `exitAudioExpand()` is called from all three FULL-exit paths
  (`teardownMediaState`, `dock`, `close`); the toggle itself is FULL-only.
- **Risk:** re-showing the mobile-audio `#fs-btn` resurrects the v1.22.1
  dead-tap bug. → **Mitigation:** the audio-mode branch routes the button to
  `toggleAudioExpand()`, never `enterFullscreen()`; the pure helper + its test
  lock the branch.
- **Risk:** the fixed overlay lands under the site header/dock/modal. →
  **Mitigation:** T2 sets `z-index` above the known chrome tiers (header 1000,
  dock 950, nav 900); Dean's iPhone on-device pass is the arbiter (AC9).
- **Risk:** touching the video fullscreen path. → **Mitigation:** the
  native enter/exit block is copied unchanged inside the new branch; no edit to
  `enterFullscreen()`, `inNativeFullscreen()`, the `f`/`F` switch, or
  rotate-to-fullscreen.
- **Risk:** FR-2 icon-link ordering changes which icon a browser picks. →
  **Mitigation:** SVG stays first; PNGs are additive fallbacks with explicit
  `sizes`, and `apple-touch-icon`/manifest are untouched (AC13).

### Performance impact

No expected impact on any `docs/RELIABILITY.md` budget. FR-1 is a CSS class
toggle on an existing element (no new element, no rAF, no timers beyond one
idle `keydown` listener); FR-2 reuses already-committed assets served by the
existing static route; FR-3 is a two-property CSS change.
