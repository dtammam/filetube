# Execution Plan: v1.29 Downloads Reliability Wave

- **Date:** 2026-07-11
- **Status:** APPROVED 2026-07-11 (Dean) — full autonomous pipeline run authorized
- **Target release:** v1.29.0
- **Branch:** from `main` (v1.28.0 merge, clean)
- **Prep source:** code exploration 2026-07-11 (two read-only sweeps: backend yt-dlp pipeline, downloads UI); root-cause findings verified with file:line anchors.

## Goal

Make downloads trustworthy: failures explain themselves, retry actually retries,
one-shot downloads don't hold the UI hostage, status reflects reality (including
partial success), and there is a durable history of what happened.

## Root causes this plan addresses (verified)

1. **Exit-1 amplification** — one yt-dlp invocation per channel with N positional
   URLs; any single video failure makes yt-dlp exit 1, and the orchestrator treats
   any non-zero exit as total channel failure (`lib/ytdlp/index.js:1380`), even when
   most videos downloaded.
2. **Real error discarded** — the captured stderr tail (the actual reason: bot-check,
   429, age-gate, members-only) is dropped and replaced with generic
   `yt-dlp exited with code 1` (`lib/ytdlp/run.js:901`). Console log is redacted.
   Nothing durable records why a run failed.
3. **Retry replays the doomed request** — "retry" is re-pull with identical argv;
   `--download-archive` skips prior successes so retry re-targets exactly the subset
   that failed, with nothing changed. Busy coalescing (`{started:false, reason:'busy'}`)
   is a silent no-op in the UI. `/subscriptions` failed rows have no Retry button at all.
4. **Zero anti-bot affordances** — no sleep/rate/retry/extractor-args/user-agent flags
   (`lib/ytdlp/args.js:614-737`); cookies silently skipped if the configured file is
   missing (`args.js:252-265`). Likely root cause of the consistent exit-1 pattern.
5. **"Preparing → failed" with no progress** — the progress parser
   (`lib/ytdlp/progress.js`) exists and works; failing runs die in the listing pass or
   before the first percent line. (Do NOT rebuild the parser.)

## Scope

### In

- **T0 (prerequisite): Diagnostics foundation.**
  Promote the captured stderr tail into the persisted `lastStatus` and the live
  activity entry (bounded/sanitized). Add a durable per-run JSONL log under `data/`
  (run start/end, per-channel outcome incl. counts, per-item failures with reasons,
  timestamps; capped/rotated). Feeds T4.
- **T1: Retry that retries.**
  Retry buttons on `/subscriptions` errored subscription rows and one-shot rows, and
  on the one-off modal's error state — port the corner chip's working logic
  (`retryOneShot` `public/js/common.js:4611`, `retrySubscription` `:4632`).
  Surface busy-coalescing visibly ("queued behind current run") instead of silent no-op.
- **T2: Non-blocking one-shot download + auto-refresh.**
  On submit, minimize the one-off modal into the existing corner download chip
  (`injectDownloadStatusChip` `common.js:4560`); progress continues there; user keeps
  browsing/watching. On `done`, refresh the library grid IN PLACE (`loadLibrary()`),
  never `window.location.reload()` — client auto-rescan was deliberately disconnected
  as the "BUG 2" fix (`common.js:1942`) because reload-under-load hung. The server
  already rescans after `runOneShot`; the client only re-fetches.
- **T3: Meaningful, honest status.**
  (a) Partial-success reporting: per-video outcome threading so one failed video no
  longer marks the whole channel failed — "downloaded N, M failed: [real reasons]"
  using existing `failures[]` plumbing (`lib/ytdlp/failures.js`,
  `mapItemFailuresForActivity` `index.js:1428`). A channel with some successes is a
  partial success, not an error.
  (b) Resilience argv flags: `--sleep-requests`/`--sleep-interval`, `--retries` with
  backoff, consider `--extractor-args youtube:player_client` — all config-overridable
  via `FILETUBE_YTDLP_*` env vars consistent with `lib/ytdlp/config.js`, with safe
  defaults.
  (c) Loud warning (status + log) when `FILETUBE_YTDLP_COOKIES_FILE` is configured
  but the file is missing.
- **T4: Download history view.**
  Durable per-run history (backed by the T0 run log, capped) rendered on
  `/subscriptions`: states, timestamps, per-item failures. Today `lastStatus` keeps
  only the latest outcome and the live activity map dies on restart.

- yt-dlp argv changes must keep `shell:false` arg-array spawning and the byte-identical

- The one-shot `text/plain` / URL-normalization behavior shipped in v1.28.0 must not

- Status strings render via `textContent` — keep server-composed strings bounded and

- JSONL run log must be capped/rotated (single-user LAN box, but unbounded logs are a
  regression).
- Two-reviewer gate before acceptance; honest scoring (see thumbnail-backfill lesson);
  on-device pass is the final arbiter.

## Deliverables / "done" looks like

1. A failed channel run shows the REAL reason (bounded stderr tail / per-item reasons),
   not "exit code 1", both live and persisted across restart.
2. `data/` contains a capped JSONL run history; `/subscriptions` renders it.
3. Retry buttons exist on failed rows/modal, do something visible immediately
   ("queued behind current run" when busy), and re-run the target.
4. One-shot: submit → modal minimizes to corner chip → user keeps browsing → on done
   the library grid refreshes in place and the new video appears without manual rescan.
5. A channel where 9/10 videos downloaded reports partial success with the 1 failure
   attributed — not "channel failed."
6. Resilience flags in the argv with env overrides documented in README's ENV table.
7. `npm test` + `npm run lint` green; two-reviewer gate passed; version bumped to
   v1.29.0 with release notes.

## Complexity signal (why plan-gated)

Multi-domain: child-process spawn layer + orchestrator state machine (`lib/ytdlp/`),
new durable storage (run log), HTTP API surface, and three UI surfaces
(`/subscriptions` page, one-off modal, corner chip) — plus behavior changes to
external-service interaction (yt-dlp pacing flags) that affect download reliability
for every subscription. A wrong move in the exit-code/partial-success accounting
could mask real failures (the inverse of today's problem), so acceptance criteria
must test both directions.

## Suggested task ordering

T0 → T3(a) (shared per-video outcome threading) → T1 → T3(b,c) → T2 → T4 → gate →
release. EM may re-split during the Tasks stage.

## Requirements

Numbered, testable requirements grouped by task. Each is one verifiable
behavior. Anchors verified against the current tree on 2026-07-11.

### T0 — Diagnostics foundation (prerequisite)

- **R0.1** [UNIT] When a channel download spawn exits non-zero, the bounded,
  redacted stderr tail already captured by `run.js` (`safeStderr`, the value
  currently discarded at `run.js:901` in favor of the literal string
  `"yt-dlp exited with code <n>"`) is surfaced as (or appended into) the
  persisted `lastStatus` string instead of being replaced by the generic
  exit-code message.
- **R0.2** [UNIT/INTEGRATION] The surfaced stderr-tail-derived reason is
  bounded (reuses the existing `STDERR_TAIL_LIMIT`/redaction path, not a new
  unbounded field) and sanitized (control characters stripped) before it is
  ever stored or rendered — consistent with `failures.js`'s
  `sanitizeReason`/`MAX_REASON_LENGTH` posture.
- **R0.3** [INTEGRATION] The live activity entry for an in-progress/just-failed
  subscription (the ephemeral snapshot `/subscriptions` polls) carries the same
  real-reason text as R0.1, not just the persisted `lastStatus`.
- **R0.4** [INTEGRATION] After a server restart (fresh `loadDatabase()`, no live
  activity map), a subscription's persisted `lastStatus` still shows the real
  failure reason from its last run — i.e., the reason lives in `db.json`, not
  only in the in-memory activity map that is lost on restart.
- **R0.5** [UNIT] A new JSONL run-log writer appends one line per run event
  (at minimum: run start, run end/outcome) under a `data/`-relative path
  (e.g. `data/ytdlp-runs.jsonl` or equivalent — exact name is a Design-stage
  decision), with fields covering: timestamp, subscription/one-shot id,
  outcome (success/partial/failure/cancelled), succeeded/failed counts, and
  per-item failure reasons.
- **R0.6** [UNIT] The JSONL run log is capped/rotated (e.g. max line count or
  max file size with rotation) — appending indefinitely without bound is a
  FAIL. The cap value and rotation strategy must be stated in code
  comments/Design and be a concrete, testable number.
- **R0.7** [INTEGRATION] With `FILETUBE_YTDLP_ENABLED` off, no run-log file is
  ever created and no run-log route is reachable (disabled-module no-op
  guarantee, per constraints).
- **R0.8** [UNIT] A cancelled download (`cancelledSubscriptionIds` latch) still
  persists `lastStatus: 'cancelled'`, not a synthesized SIGKILL error — this
  existing behavior (`index.js:1406-1408`) must not regress when R0.1's
  stderr-tail promotion is added.

### T3(a) — Partial-success reporting (shared prerequisite; ordered before T1 per plan)

- **R3a.1** [UNIT] Given a `downloadResult` where some target ids succeeded and
  some failed (per-item `itemFailures` attributed via `failures.js`'s
  `parseItemFailureLine` / `mapItemFailuresForActivity`), the computed
  subscription outcome is `partial` (or equivalent explicit state), not
  `error`/`failed`.
- **R3a.2** [UNIT] The partial-success status string names both counts and
  reasons, matching the shape "downloaded N, M failed: [reason1, reason2,
  ...]" (exact wording is a Design-stage decision; the count/reason content is
  the testable contract).
- **R3a.3** [UNIT] Given a `downloadResult` where every target id failed (or a
  channel-level failure with zero attributable per-item successes — e.g.
  bot-check/429/age-gate/members-only before any item started), the outcome is
  `error`/`failed` with the real reason — NOT `partial` and NOT silently
  `success`. (This is failure-masking Direction A — see the dedicated section
  below.)
- **R3a.4** [UNIT] Given a `downloadResult` where every target id succeeded
  (today's already-correct case), the outcome remains `success` — R3a.1's new
  `partial` branch must not reclassify a full success.
- **R3a.5** [UNIT] `downloadMeta` (or equivalent per-item persisted record) is
  recorded ONLY for successfully downloaded ids; a failed survivor id gets no
  stale `downloadMeta` entry (tech-debt #17, addressed as a natural byproduct
  of per-video outcome threading).
- **R3a.6** [INTEGRATION] The `/subscriptions` UI (or its status JSON) renders
  a `partial` outcome distinctly from `error` (different status string/class),
  so a human glancing at the row can tell "9/10 downloaded" from "channel
  failed" without opening logs.

### T1 — Retry that retries

- **R1.1** [INTEGRATION] `/subscriptions` renders a visible Retry action on any
  subscription row whose `lastStatus` is an error/failed/partial-with-failures
  state — the plan's "buried Re-pull only in the settings sheet"
  (`lib/ytdlp/client/subscriptions.js:1314`) is no longer the only path.
- **R1.2** [INTEGRATION] Clicking Retry on a failed/partial subscription row
  calls the existing `POST /api/subscriptions/:id/repull` route
  (`lib/ytdlp/index.js:2923`) and the row visibly changes state (e.g. to
  "retrying"/"queued") within one poll tick — not a no-op with no visible
  change.
- **R1.3** [UNIT] A one-shot row in the error state exposes a Retry action that
  re-submits the same URL/format/quality/folder via `POST
  /api/ytdlp/download`, reusing the existing `retryOneShot`
  (`common.js:4611`) request-body-building logic (`buildOneShotRetryBody`) —
  ported to the modal's error state, not reimplemented.
- **R1.4** [UNIT] The one-off modal's error state (distinct surface from the
  corner-chip one-shot row) exposes an equivalent Retry action that re-submits
  the same request.
- **R1.5** [INTEGRATION] When a retry request lands while `runPoll` is already
  in flight for that subscription (or globally, per the existing
  single-flight poll guard), the busy outcome (`{started:false,
  reason:'busy'}`, currently computed at `index.js:1710` but discarded
  unread by the `:id/repull` route at `index.js:2923-2935`) is propagated to
  the HTTP response and the client renders an explicit "queued behind current
  run" state — replacing today's silent no-op where the client only ever
  sees `202 {accepted:true}` regardless of whether the poll actually started.
- **R1.6** [UNIT] Tech-debt #8 (busy-coalescing re-reads the full subscription
  set via `loadDatabase` on every trigger): if addressed, a test demonstrates
  the in-flight poll's subscription id set is cached/reused rather than
  re-read per retry click during the same in-flight run. (Optional — only
  required if the task's natural shape covers it, per the plan.)

### T2 — Non-blocking one-shot download

- **R2.1** [UNIT] On one-shot submit, the modal minimizes into the existing
  corner chip (`injectDownloadStatusChip`, `common.js:4560`) rather than
  staying open/blocking; the user can navigate/interact with the rest of the
  page immediately after submit.
- **R2.2** [UNIT] Progress for the minimized one-shot continues to update
  inside the chip (reusing the chip's existing poll loop), not a separate
  polling mechanism.
- **R2.3** [UNIT] On the one-shot reaching `done`, the library grid is
  refreshed via `loadLibrary()` (a targeted re-fetch/re-render), and
  `window.location.reload()` is never called anywhere in this path — a
  regression test asserts `decideOneOffTerminalAction`'s (`common.js:1956`,
  the BUG 2 fix) `rescan: false` / no-reload contract still holds after T2's
  changes.
- **R2.4** [INTEGRATION] After a one-shot reaches `done`, the newly downloaded
  video appears in the library grid without the user manually triggering a
  rescan (the server already rescans post-`runOneShot`; the client's
  `loadLibrary()` call is the only new client-side behavior needed).
- **R2.5** [MANUAL/ON-DEVICE] On a real device, submitting a one-shot and then
  immediately browsing/playing another video confirms the modal does not
  block navigation and the chip continues showing accurate progress.
- **R2.6** [INTEGRATION] The v1.28.0 one-shot `text/plain` body handling and
  URL-normalization (`lib/ytdlp/url.js` extraction, the iOS Shortcut path)
  are unchanged: existing tests for these behaviors still pass unmodified

### T3(b) — Resilience argv flags

- **R3b.1** [UNIT] `buildYtdlpDownloadArgs` (and/or the list-args builder)
  emits `--sleep-requests`/`--sleep-interval` and `--retries` (with backoff)
  flags with safe, non-zero, finite default values when no override is
  configured.
- **R3b.2** [UNIT] Each new flag's value is independently overridable via a
  new `FILETUBE_YTDLP_*` env var, parsed through `lib/ytdlp/config.js`
  following the existing `parse*` bounds-checking pattern (e.g.
  `parseDownloadTimeoutMinutes`'s posture: invalid/out-of-range input falls
  back to the default, never produces a zero/negative/non-finite value passed
  to yt-dlp).
- **R3b.3** [UNIT] If `--extractor-args youtube:player_client` (or equivalent)
  is added, it is emitted as fixed hardcoded literal tokens — never built
  from interpolated user/subscription data — matching the existing
  non-injection posture of `VIDEO_FORMAT_SORT`/`SHORTS_MATCH_FILTER`.
- **R3b.4** [UNIT] The full argv produced by `buildYtdlpDownloadArgs` after
  adding these flags still uses `shell:false` array spawning (asserted via
  the existing spawn call sites/tests), still emits the mandatory `--`
  separator immediately before positional targets, and every existing
  injection-guard test (host allowlist, `FORBIDDEN_CHARS`, SF4 path
  confinement, decoded-id charset) passes byte-identically — no existing
  injection-guard test's expected argv shape changes except for the
  appended new flags.
- **R3b.5** [DOC] Each new `FILETUBE_YTDLP_*` var appears as a new row in the
  README's ENV table (`README.md` ~line 181-188), with default value and a
  one-line description, matching the existing table's format.

### T3(c) — Cookie-missing warning

- **R3c.1** [UNIT] When `FILETUBE_YTDLP_COOKIES_FILE` is set in config but
  `fs.existsSync(config.cookiesFile)` is false (the exact condition
  `cookiesUsable` at `args.js:252-253` already detects and currently causes a
  silent `cookiesArgs` no-op), a loud warning is produced: surfaced in the
  live status/activity for the affected run AND written to the JSONL run log
  (R0.5) — not merely a `console.error` line.
- **R3c.2** [UNIT] When `FILETUBE_YTDLP_COOKIES_FILE` is unset entirely (the
  common/default case), no cookie warning is produced — the warning is scoped
  to "configured but missing," not "cookies not in use."
- **R3c.3** [UNIT] The cookie-missing warning does not itself fail the run —
  a members-only video is still cleanly skipped (via `shouldSkip`) rather than
  attempted and erroring; the warning is additive, not a new failure mode.

### T4 — Download history view

- **R4.1** [INTEGRATION] `/subscriptions` renders a history view (list, per
  the plan's UI-research §5 corner-chip/list conventions) backed by the T0
  JSONL run log, showing at minimum: timestamp, target (subscription/one-shot
  name), outcome, and per-item failure reasons for partial/failed runs.
- **R4.2** [INTEGRATION] The history view reflects entries written before the
  current server process started (i.e., it reads the persisted JSONL file, not
  only in-memory state) — proving durability across restart.
- **R4.3** [UNIT] The history view/endpoint respects the same cap as R0.6 — it
  never attempts to render/return more than the capped number of entries even
  if the log file is manually inflated beyond the cap.
- **R4.4** [INTEGRATION] With `FILETUBE_YTDLP_ENABLED` off, no history

## Acceptance criteria

Explicit pass/fail criteria for each of the plan's 7 "done" deliverables, plus
the required both-directions failure-masking tests. "PASS" requires the named
observable signal, not a subjective impression.

### Deliverable 1 — real reason surfaced live AND persisted across restart

- [ ] **AC1.1** A channel run whose spawn exits non-zero with distinctive
  stderr (e.g. a synthetic "Sign in to confirm your age" fixture) produces a
  live activity entry whose status text contains that reason text (or a
  bounded/sanitized derivative of it) — NOT the literal string "yt-dlp exited
  with code 1" alone.
- [ ] **AC1.2** After the run settles, `db.json`'s `lastStatus` for that
  subscription contains the same real-reason text.
- [ ] **AC1.3** After a simulated server restart (re-`loadDatabase()` in a
  fresh test process/fresh `require`, per this repo's isolated-`DATA_DIR` test
  pattern), the subscription's `lastStatus` still shows the real reason — the
  live activity map (which does die on restart) is not the only place the
  reason lived.
- [ ] **AC1.4** The persisted/rendered reason text is bounded in length
  (matches `STDERR_TAIL_LIMIT`/`MAX_REASON_LENGTH`-class bounding) and free of
  control characters, verified against a pathological (very long, control-char
  laden) synthetic stderr fixture.

### Deliverable 2 — capped JSONL run history rendered on /subscriptions

- [ ] **AC2.1** After a run completes, a new line is appended to the JSONL run
  log file under `data/`, valid JSON, containing at minimum timestamp, target

- [ ] **AC2.2** Feeding the log more entries than its stated cap (via direct
  writes in a test) results in the file staying at-or-under the cap after the
  next write — old entries rotated out, not unbounded growth.
- [ ] **AC2.3** `GET /subscriptions` (or its backing JSON endpoint) returns/
  renders entries from this log; a DOM query (or JSON response) for the

  test.
- [ ] **AC2.4** With `FILETUBE_YTDLP_ENABLED` unset/false, no file is created
  under `data/` for the run log and the history endpoint/DOM is absent
  (404 or not rendered).

### Deliverable 3 — retry buttons do something visible immediately

## Design

Verified against the current tree on 2026-07-11. Every anchor below was read
before designing on top of it. The one architectural addition (the JSONL run
log, `lib/ytdlp/runlog.js`) is also reflected in `docs/ARCHITECTURE.md`.

### Guiding shape of the change

The spine of this wave is ONE new pure decision function plus ONE new durable
writer; every UI and status change hangs off those two:

1. **`computeDownloadOutcome({ ok, itemFailures, targetIds })`** — a new pure
   function in `lib/ytdlp/failures.js` (the existing pure attribution module,
   already the fixture-test seam) that classifies a download spawn's result
   into `success` / `partial` / `error` with `succeeded`/`failed` counts. This
   is the single place the exit-1-amplification bug (root cause 1) and the
   both-directions failure-masking hazard are resolved, and it is directly
   unit-testable from stderr fixtures (feed lines through `parseItemFailureLine`
   → `itemFailures[]` → `computeDownloadOutcome`), no live network.
2. **`lib/ytdlp/runlog.js`** — the durable, capped JSONL history writer/reader
   (see ARCHITECTURE.md). Feeds T4 and records the real failure reason (root
   cause 2) durably.

Everything else is wiring: promoting the already-captured stderr tail (T0),
threading the outcome (T3a), porting existing retry logic to new surfaces (T1),
closing the modal into the existing chip (T2), appending pacing flags in the
one spot in the arg-builder that is safe (T3b), a cookie-missing warning (T3c),
and a read-only history list (T4).

### T0 — Diagnostics foundation

#### Promoting the real reason (R0.1–R0.4, R0.8)

The stderr tail is already captured, bounded (`STDERR_TAIL_LIMIT = 4096`,
`run.js:168`) and redacted (`safeStderr`, `run.js:884`). It is discarded at
`run.js:901`, where the `error` field is overwritten with the literal
`yt-dlp exited with code <n>`. The fix is local to `run.js`'s `close` handler:

- Add a tiny pure helper `pickStderrReason(safeStderr)` (in `run.js`, or lifted
  into `failures.js` beside `sanitizeReason`) that returns the most meaningful
  bounded line from the already-bounded/redacted `safeStderr`: the **last line
  matching `/^ERROR:/i`** if present, else the **last non-empty line**, else
  `''`. It never introduces a new unbounded field — it only selects from the
  already-`STDERR_TAIL_LIMIT`-bounded, already-redacted tail (R0.2).
- At `run.js:896-904`, compose the resolved `error` as
  `pickStderrReason(safeStderr) || 'yt-dlp exited with code ${resultCode}'`
  (still wrapped in the existing `redactString`). The generic string survives
  ONLY as the fallback when stderr yielded nothing usable.
- No consumer change is required for R0.1/R0.3/R0.4: `safeErrorStatus`
  (`index.js:764`) already reads `downloadResult.error`, bounds it again to
  `MAX_STATUS_LENGTH = 300`, and prefixes `error:`. That value is what the
  error branch persists as `lastStatus` (`index.js:1393` → returned →
  `processSubscription` persists at `index.js:1628`, surviving restart in
  `db.json`, R0.4) AND writes onto the live activity entry (`index.js:1430`,
  R0.3). One `run.js` change lights up all four.
- **R0.8 (cancelled not regressed):** the `cancelledSubscriptionIds` override
  at `index.js:1406-1408` runs AFTER `safeErrorStatus` and unconditionally
  forces `status = 'cancelled'`. Since T0 only changes how the *error reason
  string* is composed upstream, that override still wins — a cancelled run's
  SIGKILL is still persisted as `'cancelled'`, never a synthesized error. The
  one-shot mirror is `cancelledOneShotJobs` at `index.js:2138-2142`. This
  ordering is a HARD INVARIANT the implementation must preserve (regression
  test: cancel mid-run, assert persisted `lastStatus === 'cancelled'` even
  though `safeStderr` was non-empty).

#### The JSONL run log (R0.5–R0.7)

New module **`lib/ytdlp/runlog.js`**, required directly in `index.js` like
`args`/`run` (import is side-effect-free — no fs, no route, consistent with
`config.js`'s posture):

- **File:** `ytdlp-runs.jsonl` in the app `DATA_DIR` (same directory as
  `db.json`; resolved the same way `store.js` locates the db, NOT
  `config.downloadDir`). Rationale in ARCHITECTURE.md: it is app state, must
  stay on stable local disk, and must not contend on the `updateDatabase`
  mutex.
- **Line schema (one JSON object per line, one line per COMPLETED run):**

  ```json
  {
    "ts": "2026-07-11T12:34:56.789Z",
    "kind": "subscription",
    "id": "<sub.id | one-shot jobId>",
    "name": "<bounded, sanitized channel/target name>",
    "outcome": "success | partial | error | cancelled",
    "succeeded": 9,
    "failed": 1,
    "reason": "<bounded status/reason string, empty for clean success>",
    "cookieWarning": false,
    "failures": [ { "videoId": "abc123 | null", "title": "…?", "reason": "…" } ]
  }
  ```

  All strings pass through the existing `sanitizeReason`/`MAX_REASON_LENGTH`
  (500) and `MAX_STATUS_LENGTH` (300) bounding; `failures[]` is already bounded
  (`MAX_CAPTURED_META = 1000` upstream) and re-capped on render.
- **Writer home / one call site each:** the log entry is emitted from
  `processSubscription` (which owns the final `lastStatus` and, via the
  structured return described in T3a, the outcome/counts/failures) and from
  `runOneShot`'s terminal branches — exactly the two places a run completes.
  Centralizing to those two points (rather than scattering across the 4-5
  terminal branches inside `runSubscriptionCycle`) keeps the writer calls
  auditable and covers the `processSubscription` catch-all throw path too
  (synthesized `{ outcome: 'error', succeeded: 0, failed: 0 }`).
- **Cap / rotation — concrete testable number:** `YTDLP_RUNLOG_MAX_ENTRIES =
  500`. Each append does read → parse-defensively → push → `slice(-500)` →
  atomic rewrite (unique temp in the same dir, then `rename`, mirroring
  `saveDatabase`). Runs are infrequent on a single-user box, so an O(n) rewrite
  per completed run is negligible and gives an EXACT bound (R0.6). AC2.2 test:
  seed the file with >500 lines via direct writes, trigger one more write,
  assert the file is ≤500 lines afterward.
- **Disabled no-op (R0.7):** the writer is only ever called from the
  enabled-only `processSubscription`/`runOneShot` paths, and the reader route
  is registered only inside `registerRoutes`, which early-returns at
  `index.js:2599` when `!isEnabled(config)`. Import creates nothing. So with
  the flag off: no file, no route. AC2.4 test: boot disabled, assert no
  `ytdlp-runs.jsonl` under `DATA_DIR` and a 404 on the history route.

### T3(a) — Partial-success reporting (ordered before T1)

#### The outcome function (R3a.1, R3a.3, R3a.4 + AC-FM-A/B/D)

New pure function in `failures.js`:

```js
function computeDownloadOutcome({ ok, itemFailures, targetIds }) {
  const target = Array.isArray(targetIds) ? targetIds.length : 0;
  const failures = Array.isArray(itemFailures) ? itemFailures : [];
  if (ok) return { outcome: 'success', succeeded: target, failed: 0 };
  // !ok with ZERO per-item attribution === a channel-level failure
  // (bot-check / 429 / age-gate / members-only before any item started):
  // credit NOTHING, surface the real reason. (Direction A guard.)
  if (failures.length === 0) return { outcome: 'error', succeeded: 0, failed: target };
  const attributed = new Set(
    failures.map((f) => (f && typeof f.videoId === 'string' && f.videoId ? f.videoId : null))
            .filter(Boolean),
  );
  const unattributed = failures.filter(
    (f) => !f || typeof f.videoId !== 'string' || !f.videoId,
  ).length;
  const failed = attributed.size + unattributed;
  const succeeded = Math.max(0, target - failed);
  if (succeeded > 0) return { outcome: 'partial', succeeded, failed };
  return { outcome: 'error', succeeded: 0, failed }; // all attributable ids failed
}
```

**Mutual exclusivity (the AC-FM crux), stated explicitly and testable:**

- `success` ⇔ `ok === true`.
- `partial` ⇔ `!ok` AND `failures.length > 0` AND `succeeded > 0`.
- `error` ⇔ `!ok` AND (`failures.length === 0` OR `succeeded === 0`).

These three are exhaustive and disjoint. **Direction A** (AC-FM-A) is held by
the `failures.length === 0` early return (a channel-level failure has no
per-item lines, so it can NEVER be miscounted as partial/success — nothing is
credited) and by the final `succeeded === 0` branch (every targeted id
attributed as failed ⇒ `error`, not `partial`). **Direction B** (AC-FM-B) is
the `succeeded > 0` branch. **AC-FM-D** (unattributed failure not dropped) is
already guaranteed upstream by `parseItemFailureLine` returning
`{ videoId: null, reason }`; `computeDownloadOutcome` counts those in `failed`
(so they cannot silently inflate `succeeded`) and `mapItemFailuresForActivity`
already passes them through to the surfaced failures list.

#### Threading it through `runSubscriptionCycle` / `processSubscription`

The current post-download block (`index.js:1353-1483`) has only two arms:
`if (!downloadResult.ok)` → total error, else → success. Restructure the
post-quarantine tail into three arms driven by `computeDownloadOutcome`:

- Keep the unconditional confinement quarantine (`index.js:1369-1378`) as-is on
  every path.
- **`error`:** the existing error branch (`index.js:1380-1435`) — including
  `cleanupFailedDownloadIntermediates`, the `cancelledSubscriptionIds`
  override, and `mapItemFailuresForActivity` → activity write. Status via
  `safeErrorStatus` now carries the real reason (T0).
- **`partial` (new):** do NOT run the failed-download intermediate cleanup
  (some items succeeded). Record channel meta ONLY for the succeeded ids
  (`survivorIds` minus the attributed failed ids — see R3a.5 below). Write the
  activity entry `{ state: 'done', percent: 100, phase: null, outcome:
  'partial', failures }` (reusing `mapItemFailuresForActivity`), and return a
  status string of the shape **`partial: downloaded N new video(s), M failed:
  <reason1; reason2; …>`** (R3a.2; bounded to `MAX_STATUS_LENGTH`). The exact
  punctuation is an implementation detail — the testable contract is that it
  names both counts and includes the real reasons.
- **`success`:** unchanged (`index.js:1458-1482`), plus the run-log emission.

`runSubscriptionCycle` changes its return type from a bare status string to a
small structured result `{ status, outcome, succeeded, failed, failures,
cookieWarning }` (its ONLY caller is `processSubscription`, so blast radius is
contained; the listing-failure branch at `index.js:1221-1234` and any throw
return `{ outcome: 'error', succeeded: 0, failed: 0, failures: [] }`).
`processSubscription` persists `result.status` as `lastStatus` and emits the
single run-log line.

#### `downloadMeta` for succeeded ids only (R3a.5, tech-debt #17)

The confirmed seam is the survivor loop at `index.js:1458-1471`, which today
records channel meta for EVERY `survivorId`. In the `partial` arm, compute
`succeededIds = survivorIds.filter((id) => !attributedFailedIds.has(id))` and
iterate that instead. Attributed failed ids get no `downloadMeta` entry
(closing #17); unattributed failures (`videoId: null`) remove no specific id,
which is correct (we cannot prove which id they belong to). AC5.3 test: 9/10
scenario, assert meta recorded for the 9, not the 1.

#### Cutoff-advance interaction

`processSubscription`'s `advanceCutoffDate` gate (`index.js:1621`) currently
tests `lastStatus.startsWith('ok')`. Broaden to advance on `success` OR
`partial` (progress was made), i.e. `result.outcome === 'success' ||
result.outcome === 'partial'`. See Risks for the deliberate tradeoff (a
partial run's failed items are not auto-re-listed by a later poll once the
cutoff advances; the primary retry value is the channel-level case where
nothing succeeded and the cutoff did NOT advance).

#### Rendering partial distinctly (R3a.6, AC of Deliverable 5)

`formatRowStatusLine` (subscriptions.js) already renders `lastStatus`/live
`entry` via `textContent` into `.sub-row-status`, and `formatFailuresLine`
already renders the per-item failure detail into `.sub-row-failures`
(`subscriptions.js:1047-1058`). Add a `partial` discriminator: the client sets
a `sub-row-status-partial` class when `entry.outcome === 'partial'` or the
status string starts with `partial:` (distinct from the existing error class).
No new CSS tokens/typography — reuse the existing status-class mechanism
(off-limits guard honored).

#### Visible busy-coalescing + the repull-route timing (R1.5, PM FLAG 2)

Today `POST /api/subscriptions/:id/repull` (`index.js:2923-2935`) responds
`202 {accepted:true}` and only THEN fires `runPoll(...).catch(...)`, never
inspecting `runPoll`'s `{started:false, reason:'busy'}` return (computed at
`index.js:1710`). The client thus always sees an accepted 202 regardless of
whether a poll actually started.

**Decision (PM FLAG 2): keep the 202 status, add a discriminator in the body,
via an early-decision callback — do not block the response on the long poll.**

`runPoll` cannot be awaited for the `started:true` case (it resolves only after
the whole channel download finishes). But its busy-vs-started decision is made
synchronously, before its first `await` (`pollBusy` check at `index.js:1698`,
set at `index.js:1713`). So:

- Add an optional `onDecision` callback to `runPoll(deps, config, subId, {
  onDecision })`. `runPoll` invokes it synchronously with
  `{ started: false, reason: 'busy' }` at the coalesce point (`index.js:1710`)
  and `{ started: true }` immediately after `pollBusy = true`. Because this
  fires before the first `await`, it is race-free (single-threaded; the route's
  synchronous `runPoll(...)` call runs `runPoll`'s prefix to completion before
  yielding).
- The route uses `onDecision` to send the response ONCE:
  `res.status(202).json({ accepted: true, started: <bool>, reason: <'busy'|
  undefined> })`. The status stays 202 in both cases (no caller that assumed an
  always-202 *status* breaks — only the *body* gains fields). The
  fire-and-forget `.catch` still guards the long-running poll for the started
  case.
- Client: `retrySubscription` (and the new row Retry) read the body; when
  `started === false && reason === 'busy'`, render an explicit **"queued behind
  current run"** state (a defined class `sub-row-status-queued` / text) — AC3.5
  asserts this specific string/class is present rather than a silent no-op. The
  `requestPollRerun(subId)` coalescing at `index.js:1709` is unchanged, so the
  queued retry still runs once the in-flight poll finishes.

Apply the same `onDecision` discriminator to the repull-**all** route
(`index.js:2916`) for consistency (cheap; keeps the two routes symmetric).
**Caller audit:** the existing chip `retrySubscription` and settings-sheet
Re-pull only `.catch()` the fetch and ignore the body today, so adding body
fields is backward-compatible; no caller asserts an exact body shape.

#### Tech-debt #8 (R1.6, optional)

Each trigger does `store.listSubscriptions(deps)` (a `loadDatabase` read) at
`index.js:1682`, and the route does a second `loadDatabase` for its existence
check (`index.js:2927`) — so a rapid retry-click during a long poll costs two
disk reads per click. **Assessment: cheap but low-value on this single-user LAN
box.** A clean, natural cache: when `pollBusy` is set, snapshot the in-flight
poll's target sub-id set into a module-level `inFlightPollSubIds` and clear it
in the `finally`; the coalesce path can consult it instead of re-reading. I
recommend implementing it ONLY if it falls out cleanly during T1; otherwise
leave it (R1.6 is explicitly optional and the plan says "if cheap"). Not worth
adding indirection that complicates the busy path for a sub-millisecond disk
read.

### T3(b) — Resilience argv flags (PM FLAG 4: finalized)

  deprecates client identifiers across releases, and pinning the wrong one can
  itself CAUSE failures, directly contradicting the reliability goal.
  Committing a fixed literal would age badly against the pinned-but-upgraded
  yt-dlp binary. Instead add `FILETUBE_YTDLP_PLAYER_CLIENT` (default **unset**
  → flag NOT emitted → byte-identical to today for that flag). When set, emit
  `--extractor-args`, `youtube:player_client=<value>` as two argv elements,
  where `<value>` is validated against a strict allowlist charset
  `/^[a-z0-9_,\-]+$/` and length-bounded (rejecting → treated as unset, no
  flag). This honors R3b.3's "fixed-literal, never interpolated hostile data"
  intent: the `youtube:player_client=` prefix is a literal and the value is
  charset-confined the same way decoded video ids already are — it is operator
  config, never per-video/per-subscription data. This gives Dean an immediate
  on-device lever for the bot-check case without baking in a fragile default;
  if on-device testing shows a specific client is reliably needed, promote it
  to a hardcoded default in a fast-follow.
- **Injection-guard preservation (R3b.4, AC6.3):** the new flags slot in as
  fixed literal tokens (plus bounds-checked numeric values from config)
  immediately after `args.push('--no-warnings')` (`args.js:723`) and before
  `cookiesArgs`/`--print`/`-o`/`--`. They are NOWHERE NEAR the `--` separator
  and positional-target section (`args.js:735-736`), so the host allowlist,
  `--` separator, `FORBIDDEN_CHARS`, SF4 path confinement, and decoded-id
  charset are all untouched. `shell:false` array spawning is unchanged. Every
  existing injection-guard test passes byte-identically except for the appended
  new flags (AC6.3 asserts a byte-identical diff modulo the additions).
- **README ENV table (R3b.5, deliverable 6):** four new rows in the table at
  `README.md:181-188` (five if `FILETUBE_YTDLP_PLAYER_CLIENT` is documented,
  which it should be, marked "unset by default; advanced"). New vars:
  `FILETUBE_YTDLP_SLEEP_REQUESTS` (1), `FILETUBE_YTDLP_SLEEP_INTERVAL` (2),

### T4 — Download history view (R4.1–R4.4)

### Alternatives considered

- **Per-video isolation via one-spawn-per-video** (instead of
  `computeDownloadOutcome` attribution). Rejected: explicitly OUT of scope in
  the plan; it would serialize N spawns per channel, multiply the rate-limit
  surface (worsening the very bot-check problem), and rebuild the orchestrator.
  Attribution from the existing single-spawn stderr achieves honest reporting
  with far less risk.
- **Await `runPoll` fully in the repull route** (so the HTTP response reflects
  the real outcome). Rejected: a poll can run for the length of a whole
  channel's downloads; awaiting it would tie up the request for minutes/hours
  and change the route from fire-and-forget to blocking. The `onDecision`
  early-callback gives the busy/started discriminator without the block.
- **Storing run history inside `db.json`** (under `db.ytdlp.runs`). Rejected:
  an append-mostly, capped history would bloat the re-read-merge-on-save
  snapshot and contend on the `updateDatabase` mutex that already serializes
  every writer; a separate JSONL file is append-cheap and isolated.
- **Emitting a run-log `start` line as well as an `end` line** (R0.5's "at
  minimum run start, run end"). Rejected in favor of a SINGLE terminal line per
  completed run: a start-without-end (crash mid-run) would leave dangling
  entries and complicate the capped history and its rendering. Every AC
  (AC2.1/2.3) checks only the completion line, so the terminal-only design
  satisfies them while keeping "one line == one completed run" unambiguous.
  Flagged for EM/PM: this resolves R0.5's "at minimum" toward the outcome line.
- **Hardcoding `--extractor-args youtube:player_client=<client>`** as an
  unconditional default. Rejected as a default (version-fragility, can itself
  break downloads) but exposed as an unset-by-default env lever — see T3b.

### Risks and mitigations

- **Failure-masking, both directions (the headline hazard).** → Contained in
  ONE pure, fixture-tested function (`computeDownloadOutcome`) with three
  exhaustive, disjoint arms; the `failures.length === 0` early return is the
  Direction-A guard (channel-level failure credits nothing), the
  `succeeded === 0` branch prevents all-failed being called partial, and the
  `succeeded > 0` branch is the only path to partial. AC-FM-A/B/D drive it from
  stderr fixtures with no network.
- **Repull-route timing change.** → Kept at 202; only the body gains a
  discriminator. Existing callers ignore the body (audited). The `onDecision`
  callback fires synchronously before any await, so there is no race between the
  route's response and `pollBusy` being set.
- **Run-log unbounded growth.** → Hard cap of 500 entries enforced by
  read-trim-atomic-rewrite on every append (AC2.2). File lives on stable local
  `DATA_DIR`, not the (possibly network) download dir.
- **argv change interacting with the injection guard.** → New flags are fixed
  literals + bounds-checked numerics, inserted far from the `--`/positional
  section; `player_client` value is charset-allowlisted. AC6.3 requires a
  byte-identical injection-guard-test diff modulo the appended flags.
- **Cutoff advances on partial ⇒ a partial run's failed items are not
  auto-re-listed next poll.** → Deliberate: advancing prevents the unbounded
  re-list regression FIX-1 solved. The high-value retry case is channel-level
  failure (nothing succeeded ⇒ cutoff did NOT advance ⇒ re-pull re-lists the
  same window). A persistently-failing single item within an otherwise-good
  channel is a known, documented limitation (a per-item retry queue is out of
  scope) — surfaced honestly in the history view rather than silently retried
  with identical argv (root cause 3).
- **`loadLibrary` exposed on `window`.** → Namespaced
  (`__filetubeRefreshLibrary`), function-typed-guarded, home-page-only; a no-op
  everywhere else. The chip's done-edge tracking prevents repeated refreshes.

### Performance impact

No expected impact on any budget in `docs/RELIABILITY.md`. The serial
one-worker `runExclusive` model, the transcode/audio queues, and `progress.js`
are untouched. New per-run costs: one bounded stderr-tail select, one pure
outcome computation, and one atomic ≤500-line JSONL rewrite — all off the
`updateDatabase` lock and negligible at real-world run cadence. The new pacing
flags intentionally SLOW each yt-dlp invocation slightly (1s between requests,
2-5s pre-download sleep) — a deliberate reliability-for-latency trade to reduce
bot-checks/429s, fully env-tunable (set to 0 to restore prior pacing).
