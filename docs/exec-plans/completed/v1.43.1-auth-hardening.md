## A1 — restore rejects real bundles with 413 (CONFIRMED root cause)

The global `app.use(express.json())` (server.js ~4421, default 100 kb) runs
BEFORE the restore route; an oversized body throws `entity.too.large` there,
the global body-parser error middleware maps it to 413, and the route-scoped
`express.json({limit:'16mb'})` (server.js ~6215) has been DEAD since v1.42.
The test fixture (a few KB) masked it. Dean's prod bundle (~2943 items +
users + logo) is well past 100 kb.

Fix (agreed):
1. Exclude `POST /api/admin/restore` from the global JSON parser (wrap the
   global parser; skip when `req.path === '/api/admin/restore'` for POST).
   Consequence: the AUTH GATE now authenticates BEFORE any large parse —
   the big parse is confined to authenticated admins (a pre-auth 16/32mb
   parse would be a DoS amplifier).
2. Route-scoped `express.json({ limit: '32mb' })` on the restore route
   (Dean-approved cap: ~12 MB realistic worst case × ~2.5 headroom;
   math: ~2943 items × ~1.5–2 KB ≈ 4–6 MB + per-user state + 2 logo
   variants ≈ 2.7 MB b64 each).
3. Route-scoped body-parser error middleware in the SAME route chain,
   reusing `formatBodyParserError` — Express's error stack only walks
   FORWARD, so the global mapping middleware (registered earlier) can never
   catch this route's parser errors (the documented lib/ytdlp precedent,
   lib/bodyParserErrors.js top comment).
4. Census check: the ONLY dead route-scoped limit is restore. ytdlp's
   `express.text` (text/plain) and the logo's `express.raw` (image/*) never
   match the global json parser's content-type, so their route-scoped
   limits are alive. No other route-scoped express.json exists.

Test gap to close: a restore integration fixture comfortably OVER 100 kb
(proves the global-parser bypass), one near/over 32mb boundary behavior
(413 with clean JSON body), and an assertion that an UNAUTHENTICATED
oversized POST is refused by the gate WITHOUT parsing (401, not 413).

## A2 — scripts/reset-admin.js (recovery, script-only; reset OR create)

Operator-run recovery for a lost sole-admin password. No env lever (Dean:
no standing backdoor).

Contract:
- `node scripts/reset-admin.js <username>` — prompts for the new password
  (no password on argv: shell history/`ps` leak), or reads
  `FILETUBE_NEW_PASSWORD` from env for non-interactive use (documented as
  transient; never persisted).
- Resolves DATA_DIR exactly like server.js (env DATA_DIR → /app/data →
  cwd fallback semantics matched to server.js's rule).
- Opens the SAME adapter (`lib/db/sqlite.js openAdapter`) + user store
  (`lib/auth/store.js`) + hash via `lib/auth/crypto.js` — no parallel SQL,
  no auth-model bypass.
- Existing user → `updatePassword` (bumps token_version → every live
  session for that user is dead on the server's next per-request tv
  re-check; WAL makes the cross-process write visible immediately —
  works with the server RUNNING or stopped).
- Zero users → create as admin via the count-guarded insert (same
  statement class as /welcome setup — cannot race a concurrent /welcome).
- Existing non-admin username → resets the password but does NOT
  change role (a recovery tool must not be a privilege-escalation lever);
  says so in output. Missing username with users present → error listing
  nothing sensitive (no username enumeration beyond what the operator
  already owns — they have the DB file anyway; still keep output clean).
- If the user row is disabled → say so and require an explicit
  `--enable` flag to also re-enable (recovery ≠ silently un-banning).
- Destructive-work rigor: password strength floor same as the API's,
  refuses empty/short; clear post-run message that all sessions for the
  user were invalidated.

## B2 — chapters list clips its top entries on mobile

`.chapters-menu` is `position:absolute; bottom:calc(100%+6px)` anchored to
the control bar with `max-height:50vh` at the mobile breakpoint. Hypothesis
to verify: when the anchor sits low and 50vh exceeds the space above it
(within the clipping/positioning context), the menu's TOP overflows the
visible region — inner scrolling can't reach content whose box-top is
off-screen. Reproduce at mobile viewport geometry first; fix so the first
chapter is reachable without rubber-banding; static lock/test if one fits.

## B3 — space (and shortcuts) firing "while typing"

The FULL-only shortcut switch HAS guarded inputs since v1.16
(player.js ~4496). Real mechanism to prove with a repro: a clicked player
control `<button>` (e.g. play/pause) RETAINS FOCUS; the shortcut handler
early-returns on BUTTON without preventDefault, so a later Space keystroke
triggers the browser's default click-on-focused-button → pause. Fix
candidates: blur pointer-initiated clicks on player control buttons
(preserves Tab-key a11y), audit EVERY single-key shortcut surface. Also
fix the genuinely unguarded handler: read.js ~833 arrow keys flip book
pages while typing in any input. Verify desktop + mobile symptom.
