# v1.39.0 — Book narration: in-reader now-playing bar + chapter nav + reliable iOS start

Status: DESIGN (active, Dean-approved). One branch. Author: 2026-07-13.
Supersedes the v1.38.2/.3/.4 micro-patches (symptom-patching — the working-style
anti-pattern). Grounded in the v1.38.4 investigation report.

## Problem (from the investigation)

The TTS "Listen from Here" playback reused the shared player but **docked** it —
so on mobile the only surface was the 160px corner chip (play/pause hidden
behind an expand-bubble bug), i.e. "no usable controls". And the iOS start was
fragile (silent-clip unlock that never blessed; then a `begin()` gated behind a
`loadedmetadata` event iOS may never fire pre-gesture). The controls we want
ALREADY EXIST in the player's FULL audio bar — docking just hid them.

## Locked requirements (Dean)

1. A **full in-reader now-playing bar** at the bottom of the reader:
   `⏮  ▶/⏸  Chapter title · 0:42 ─●──── 8:15  ⏭` + book cover, plus working iOS
   **lock-screen** controls (play/pause/seek + prev/next).
2. **⏭/⏮ = next/previous CHAPTER** (synthesize + jump the reader page), not
   intra-chapter skip.
3. **Reuse** the battle-won player machinery (transport controls, MediaSession,
   background-audio lifecycle) — do not rebuild.
4. Process: investigate → design (this) → gate → ship. Full two-reviewer gate.

## Architecture — Option 1: mount FULL as a reader bar, never dock

The player's FULL audio state already renders `#player-controls` (play/pause +
scrub slider + elapsed/total time), uses the custom bar on mobile for audio
(`applyControlsMode` never sets native for audio), and wires seek/drag/fill +
MediaSession over the active element. We surface that as the reader bar:

- The reader owns a fixed bottom bar `#reader-nowplaying` containing (L→R): book
  cover, ⏮, the **player host** (mounted FULL via `#player-slot`, rendering
  `#player-controls`), ⏭, chapter label.
- Reader-scoped CSS reshapes the FULL audio host to a COMPACT strip: hide
  `#audio-bg-art` and the video-only buttons (fs/pip/cc/chapters), keep
  play/pause + seek + time (+ speed — useful for narration); fixed ~60px height.
- We **stop calling `player.dock()`** for book audio; the host stays FULL in the
  reader bar. (Docking on real navigation away still works via the router.)

Rejected: a reader-owned bar driving `#media-player` directly — it would
re-implement the already-wired play/pause/seek/drag/time + MediaSession and
break the `window.FileTube.player` boundary. Less reuse, more divergence.

## Design detail

### 1. The reader bar (read.html + read.js + style.css)
- read.html: wrap `#player-slot` in `#reader-nowplaying` (fixed bottom, hidden
  until the first Listen), with reader-owned `#reader-np-prev`, `#reader-np-next`
  (⏮/⏭), `#reader-np-cover` (img), `#reader-np-chapter` (label).
- read.js `loadReaderAudio`: mount FULL into `#player-slot`, **do not dock**;
  reveal `#reader-nowplaying`; set the cover + chapter label.
- style.css (durable, survives SPA swaps — lesson #4): `#reader-nowplaying` bar
  layout; reader-scoped overrides to compact the host (`#reader-nowplaying
  .audio-bg-art{display:none}`, hide fs/pip/cc/chapters, fixed height, controls
  static). Sits above `#bottom-nav`.

### 2. Reliable iOS start (read.js)
- Attempt `play()` IMMEDIATELY after mount (not gated on `loadedmetadata`, which
  iOS may never fire pre-gesture). Observe the promise:
  - resolves → playing, clear status.
  - rejects (NotAllowedError, iOS) → the reading pane becomes "▶ Tap to start
    listening" (`offerTapToPlay`). The tap is a fresh gesture with audio loaded →
    always plays, and unlocks the element for the session (later chapters
    auto-play — the player's own documented behavior).
- Seek to `blocks[blockIndex].startSec` on `loadedmetadata` (fires once the
  gesture-driven load has metadata), tolerant of it already being ready.

### 3. Prev/next chapter (read.js + a small additive player seam)
- `advanceChapter(delta)`: clamp `spineIndex + delta` to `[0, spine.length)`
  (spine length from `GET /api/books/:id`); if valid, run the SAME
  ensure→pollTtsReady→startTtsPlayback for the new index, and
  `adapter`/`rendition.display(spine[idx].href)` to sync the visible page +
  update the reader locator. Prefetch is already server-side (one ahead).
- Wire ⏮/⏭ buttons AND MediaSession `previoustrack`/`nexttrack`. The shared
  player deliberately leaves prev/next unwired (watch.js owns them), so add a
  minimal ADDITIVE seam: `player.setTrackNav({ onPrev, onNext })` that registers
  MediaSession `previoustrack`/`nexttrack` handlers over the active element;
  `close()`/a non-audio load clears them. The reader registers chapter nav; a
  null clears it. (Mirror how watch.js scopes its own prev/next.)

### 4. Lock screen (player.js — mostly already there)
- `setupMediaSession` already uses `data.artUrl` (book cover) + wires
  play/pause/seek/position over the active element; audio is lifecycle-pause
  exempt (keeps reading on lock). Add the prev/next handlers via §3's seam.
  Verify positionState updates for the TTS m4a.

### 5. Honesty + diagnostics (read.js)
- Distinct states: `Preparing audio…` / `▶ Tap to start listening` / (playing →
  clear) / `Audio isn’t available for this chapter` (synth failed) / a real
  error path that LOGS `err` (so a residual on-device failure is diagnosable,
  not guessed). Suppress the player's `/api/progress` READ for book audio too
  (avoid a stray resume overlay for a synthetic id) — extend the existing
  `suppressProgress` flag to gate the read in `handleResumePlayback`.

## Task decomposition (each testable where a DOM-free seam exists)
- **T1** player seam: `setTrackNav({onPrev,onNext})` + clear-on-close + suppress
  the progress READ under `suppressProgress`. Unit: the pure gate + a fake
  MediaSession asserting handlers set/cleared.
- **T2** read.js: mount-FULL-no-dock `loadReaderAudio`; reveal `#reader-nowplaying`;
  cover + chapter label. (DOM — validated on-device + a light fake-DOM unit where
  practical.)
- **T3** read.js: reliable start (immediate play + observe + tap-to-play,
  ungated), seek-on-metadata.
- **T4** read.js: `advanceChapter(±1)` clamping + page sync + status; wire ⏮/⏭ +
  MediaSession prev/next via T1. Unit: pure clamp helper against `spine.length`.
- **T5** read.html + style.css: `#reader-nowplaying` bar + compact host CSS.
- **T6** diagnostics/honest states + progress-read suppression.

## Risks
1. FULL-audio-as-compact-bar CSS: the audio-mode pointer-events/z-index swap
   (`.audio-mode`) — ensure `#player-controls` stays tappable above the (hidden)
   art in the bar. On-device tap test.
2. iOS first-play: the tap-to-play fallback is the guarantee; the immediate
   play() is best-effort. Do NOT gate the fallback behind any media event.
3. prev/next re-entrancy: guard against overlapping advances (disable while a
   new chapter is preparing).
4. Reuse boundary: the only player.js changes are the additive `setTrackNav`
   seam + the `suppressProgress` read-gate — no state-machine change.

## On-device definition of done
Open an EPUB on iPhone → tap Listen → (Tap to start listening →) it reads aloud
from the current paragraph; a bottom bar shows play/pause + scrub + time + ⏮/⏭;
lock the screen → keeps reading, lock-screen shows the cover + play/pause/seek +
chapter skip; ⏭ jumps to and reads the next chapter (page follows); desktop
unaffected.
