# v1.24 "UX Round"

## Goal

Ship ~25 real-usage UX improvements Dean has been collecting (downloads/yt-dlp,
subscriptions, library/discovery, player-adjacent mobile UX, mobile polish,
visual polish, mock comments) as one coherent branch/release, without
weakening the yt-dlp spawn-security guarantees or the disabled-module no-op
guarantee, and without touching video player CONTROLS except the one
deliberately-approved CC toggle.

## Scope

Organized into 7 domain clusters (A–G). Every FR below is grounded against the
live code (file/line pointers included); every FR ships with tests as normal
per `docs/CONTRIBUTING.md` (no deferred testing anywhere in this round).

### Cluster A — Downloads / yt-dlp

**A1. FR-1 — Multi-site URL support (HEAVIEST security gate)**
Per Dean's decision: **one-off downloads** (`POST /api/ytdlp/download`) accept
ANY yt-dlp-supported site, broadly. **Subscriptions** (`POST
/api/subscriptions`) stay **YouTube-only** — `lib/ytdlp/url.js`'s
`validateChannelUrl`/`ALLOWED_HOSTS` is UNCHANGED for the subscription path.
The one-off path gets a NEW, separate, broader-but-still-injection-safe
validator (design left to principal-engineer — see "Flag for the
principal-engineer" below). Folds in the two related ROADMAP items:
  - Accept share-sheet URL params (`?si=`, `?is=`) on the one-off path —
    normalize/strip tracking params before validation, without weakening the
    guard.
  - Recognize `/shorts/<id>` as a valid explicit one-off download target,
    without touching the per-subscription `SHORTS_MATCH_FILTER` skip-Shorts
    behavior (`lib/ytdlp/args.js` ~L310).

Acceptance criteria:
- [ ] [UNIT][A] A new one-off URL validator accepts a representative sample of
      non-YouTube yt-dlp-supported URLs (e.g. Vimeo, SoundCloud — PE to pick
      the concrete set) while still rejecting: shell metacharacters/whitespace/
      control chars, embedded userinfo, non-http(s) schemes, a leading `-`
      (option-injection), and oversized input — mirroring `validateChannelUrl`'s
      existing defense-in-depth checks in `lib/ytdlp/url.js`.
- [ ] [UNIT][A] `POST /api/subscriptions`'s validation path is provably
      UNCHANGED (existing `validateChannelUrl`/`ALLOWED_HOSTS` tests still pass
      byte-for-byte; a non-YouTube host is still rejected for subscriptions).
- [ ] [UNIT][A] Every one-off spawn still uses an arg array (never a shell
      string), still places the URL after a `--` separator, and never sets
      `shell: true` — asserted the same way existing `lib/ytdlp/args.js` spawn
      tests assert it today.
- [ ] [UNIT][A] `?si=`/`?is=` (and other known tracking params) are stripped
      before validation on the one-off path; the resulting URL still passes
      through the SAME injection-safety checks (no bypass via query-string
      smuggling).
- [ ] [UNIT][A] `/shorts/<id>` is accepted by the one-off classifier
      (`classifySingleVideo`-equivalent) as a single-video target; a
      subscription's poll cycle still excludes `/shorts/` via
      `SHORTS_MATCH_FILTER` and the JS `isShort` binding filter, unchanged.
- [ ] [PROCESS][A] `FILETUBE_YTDLP_ENABLED=false` still exposes zero new
      routes/UI/DOM for the one-off multi-site path (disabled-module no-op
      guarantee holds).
- [ ] [MANUAL][A] Two-reviewer adversarial gate: at least one reviewer
      attempts to construct a payload that bypasses the new validator's
      char/shape checks (option injection, embedded newline, encoded
      metacharacters per the existing SF5 decoded-charset lesson) before this
      wave is considered done.

**A2. FR-3 + FR-7 (merged) — Surface real per-item download failure reasons,
then fix common causes**
Dean currently only sees an opaque "yt-dlp exited with code 1" at 0% during
subscription downloads. Root cause (grounded): `runSubscriptionCycle`
(`lib/ytdlp/index.js` ~L377–551) calls `run.runDownload` ONCE per
subscription cycle with potentially many survivor video URLs; the result is a
single aggregate `{ code, stderr, error }` for the WHOLE batch — there is no
per-video attribution today (`lib/ytdlp/run.js`'s `spawnYtdlpDownload`
returns one bounded stderr tail for the entire spawn). Phase (a): capture and
surface the actual per-item reason (channel + video + why — bad URL,
members-only, geo, format, rate-limit). Phase (b): once real reasons are
visible, fix the most common root causes found (likely overlaps A1's Shorts/
share-param fixes and members-only/rate-limit handling).

Acceptance criteria:
- [ ] [UNIT][A] A pure parser attributes a yt-dlp per-video failure line
      (yt-dlp's own `ERROR: [extractor] <id>: <reason>`-shaped output) to the
      specific survivor video id it belongs to, given a batch's target id list
      — never silently drops or misattributes an error to the wrong id.
- [ ] [UNIT][A] The download status snapshot (`activity.setSubscription`
      shape, surfaced via `GET /api/subscriptions/status`) can carry a
      per-item `{ videoId, reason }` list, not just one aggregate `error`
      string, backward-compatible with existing consumers when nothing failed
      per-item.
- [ ] [MANUAL][A] The download status UI (home status chip / subscriptions
      view) shows, for a partially-failed batch, which channel + which video +
      a human-readable reason — verified against at least one real
      reproduced failure case (a members-only or geo-blocked video, or a
      deliberately malformed URL).
- [ ] [PROCESS][A] Phase (b) root-cause fixes (whatever A2's investigation
      surfaces as the top 1–2 recurring causes) are captured as their own
      sub-tasks once Phase (a) data is in hand — not guessed at up front.
- [ ] [UNIT][A] `lib/ytdlp/run.js`'s SF1 (cookies-path redaction) and SF3
      (bounded stderr, no unbounded buffer) guarantees are unaffected by the
      new per-item parsing.

## Constraints

- **Spawn-guard discipline (A1, A6, B2 Phase 2):** no `shell: true`, ever.
  Preserve arg-array spawns, the `--` separator, the host allowlist for
  SUBSCRIPTIONS (unchanged), `--match-filter`/`SHORTS_MATCH_FILTER`, SF4 path
  confinement, and the decoded-id charset (SF5). Widening one-off acceptance
  must not weaken subscription-path injection safety.
- **Disabled-yt-dlp-module no-op:** every yt-dlp FR (all of Cluster A, B2/B3/
  B4, C6) must preserve: `FILETUBE_YTDLP_ENABLED` off → no new route/UI/DOM
  reachable.

## Design

Depth-weighted: the security-heavy and data-architecture clusters (A1, A2/A3,
A6, B2/C1) get full treatment; the quick wins are sketched. Every yt-dlp FR
below is registered INSIDE the existing `registerRoutes` `isEnabled` gate and
touches only module-owned state, so `FILETUBE_YTDLP_ENABLED=false` keeps the
disabled-module no-op guarantee (no new route/UI/DOM reachable) — this is
stated once here and assumed for A1–A6, B1–B4, C6.

### Load-bearing grounding fact (drives C1, B2, C5)

`getMediaId(filePath)` (`server.js` ~L578) is `md5(filePath)` — the media id is
a hash of the PATH, not of content. Watch progress (`db.progress[id]`),
thumbnails (`THUMBNAIL_DIR/<id>.jpg`) and transcode sidecars
(`transcodedPath(id)`) are all keyed by that id. Therefore **any file move
changes the item's id**, and a naive `fs.rename`-then-rescan makes the moved
file look like a delete (old id pruned, progress lost) + a brand-new add (new
id, no history). The move machinery (C1) and the physical reconcile (B2 Phase 2)
must **re-key** `db.metadata`/`db.progress` and rename the thumbnail/transcode
sidecars from `getMediaId(oldPath)` to `getMediaId(newPath)` atomically inside
one `updateDatabase` mutator, so the next scan sees the file already indexed
under its new-path id and takes the reuse fast-path with history intact. This is
the single most important invariant in the round.

### A1 — FR-1 dual-validator (HEAVIEST security gate, Wave 6)

**Shape: two validators, one file, zero change to the subscription path.**
`lib/ytdlp/url.js` keeps `validateChannelUrl`/`ALLOWED_HOSTS`/`classifySingleVideo`
**byte-for-byte unchanged** — subscriptions (`POST /api/subscriptions`) and the
existing YouTube one-off classify path stay YouTube-only. Add two new exported
functions in the same file:

- `validateOneOffUrl(raw)` → `{ ok, url }` / `{ ok:false, error }`.
- `classifyOneOffTarget(raw)` → the one-off route's single entry point.

**How the one-off path selects between them.** `classifyOneOffTarget(raw)` runs
`validateChannelUrl(raw)` FIRST. If it passes (a YouTube URL), it delegates to
the EXISTING `classifySingleVideo` verbatim — YouTube one-offs are completely
unchanged (still reject a bare channel/playlist, still build the hardcoded
`www.youtube.com/watch?v=<id>` canonical URL). Only if `validateChannelUrl`
fails do we fall through to `validateOneOffUrl(raw)`; on success the result is a
`{ ok:true, kind:'video', targetUrl:<normalized-url> }` where `targetUrl` is the
user's own normalized URL passed through **verbatim** (no id extraction, no URL
reconstruction). YouTube behavior is thus provably untouched; the broadened
surface only ever engages for non-YouTube hosts.

**What `validateOneOffUrl` allows / rejects.** It reuses every
injection-relevant check from `validateChannelUrl` and drops ONLY the two
YouTube-SCOPE checks (`ALLOWED_HOSTS.has(hostname)` and the
`CHANNEL_PATH_PATTERNS`/`isPlausiblePath` shape check). In order:

1. Non-string / empty → reject. Length `> MAX_URL_LENGTH` (2048) → reject
   (reused constant).
2. Embedded-URL extraction pre-step (reused verbatim from `validateChannelUrl`
   — a share-sheet prose paste yields its first `https?://\S+` run).
3. Leading `-` → reject (option-injection).
4. **Raw-string metacharacter check** against a one-off constant
   `FORBIDDEN_CHARS_ONEOFF`. This is the ONLY deliberate widening: it rejects
   the full shell-operator + whitespace + control set
   (`` \s \x00-\x1f \x7f ; | ` $ < > ' " ( ) { } \ ``) **but permits the
   URI-structural characters `& = ? #`** that `validateChannelUrl`'s
   `FORBIDDEN_CHARS` rejects. `&`/`?`/`#` are legitimate, ubiquitous query/
   fragment syntax on non-YouTube sites; they are only "shell operators" inside
   a shell, and this module never uses one (see the security argument below).
5. `new URL(candidate)` parse → reject on throw.
6. Protocol `http:`/`https:` only (reused).
7. Embedded userinfo (`username`/`password`) → reject (reused SF6).
8. **Positive hostname-shape allowlist** (net-new, a HARDENING not a widening):
   the lowercased hostname must match a DNS-name shape
   `^[a-z0-9]([a-z0-9-]*[a-z0-9])?(\.[a-z0-9]([a-z0-9-]*[a-z0-9])?)+$` — i.e. at
   least one dot, no userinfo, no `:port`-only tricks, ASCII label charset only.
   This rejects bare hostnames / internal names (`localhost`, `router`) and
   keeps the broadened surface pointed at real public sites (a modest
   SSRF-lite hardening the YouTube path never needed because its allowlist
   already bounded the host).
9. **Tracking-param strip** (`?si=`, `?is=`, `?pp=`, `?feature=`, `utm_*`):
   `searchParams.delete(...)` for that fixed key set, then `parsed.toString()`.
   Because the strip happens AFTER `new URL` and re-serializes through the WHATWG
   URL serializer, the output is percent-encoded and cannot smuggle a
   metacharacter — the raw-string check in step 4 already ran on the pre-parse
   candidate, so normalization can only ever remove characters, never weaken a
   guard.

**The explicit security argument (for the adversarial gate).**

- **Injection safety is orthogonal to host scope.** `ALLOWED_HOSTS` in
  `validateChannelUrl` is a SCOPE control ("only YouTube"), not an injection
  control. The injection defenses are: the raw-string metacharacter reject, the
  leading-`-` reject, `http(s)`-only, userinfo reject, the length cap, and —
  structurally, downstream — the arg-array `spawn` with `--` before the
  positional and never `shell:true` (`lib/ytdlp/run.js`, unchanged). None of
  those depend on the host being youtube.com. Dropping the host allowlist for
  the one-off path removes a scope bound, not a safety bound.
- **`&`/`?`/`#` are inert in an arg array.** The normalized one-off URL becomes
  a single positional argv element after `--`, handed to `cp.spawn('yt-dlp',
  args)` with no shell. yt-dlp receives it as one opaque string; it is never
  re-tokenized by a shell, so `&` cannot background a command and `$(...)`
  cannot substitute. We still reject `` ; | ` $ < > ' " ( ) { } \ `` and all
  whitespace/control chars on the raw string as belt-and-suspenders, so even a
  hypothetical future call site that DID build a shell string would still be
  handed a metacharacter-free token.
- **SF5 (decoded-charset) does not apply, because there is no reconstruction.**
  SF5 exists in `validateChannelUrl` specifically because that path EXTRACTS a
  decoded `?v=`/`?list=` value and INTERPOLATES it into a hardcoded
  `https://.../watch?v=${id}` string via `buildWatchUrl`. The one-off path does
  the opposite: it passes the user's own URL through verbatim with no
  extraction and no interpolation, so there is no decoded value that ever
  becomes part of a string WE build. The one place we still reconstruct
  (YouTube one-offs via `classifySingleVideo`) keeps SF5 unchanged.
- **Subscriptions are provably unchanged.** `validateChannelUrl` and
  `classifySingleVideo` are not edited; a unit test asserts a non-YouTube host
  is still rejected for `POST /api/subscriptions` byte-for-byte.

**Shorts one-off + the download-target plumbing.** `/shorts/<id>` on a YouTube
host already passes `validateChannelUrl` (it matches neither `/watch` nor a
channel-path pattern → today `classifySingleVideo` classifies it `channel` and
rejects it). Fix inside `classifySingleVideo` ONLY for the one-off classify
path: recognize `pathname` matching `^/shorts/<id>$` and treat it as a
single-video target, extracting `<id>` through the SAME `isSafeVideoId` gate and
building the canonical watch URL. This does **not** touch `SHORTS_MATCH_FILTER`
or the poll-loop `rules.isShort` binding filter (subscription Shorts-skip is
unchanged — the two live in different code paths). For non-YouTube one-offs the
target is the verbatim URL, so `buildYtdlpDownloadArgs` gains an
`opts.targetUrl` (a single pre-validated absolute URL): when present it is
re-validated via `validateOneOffUrl` (defense-in-depth, mirroring
`requireValidUrl`) and becomes the positional after `--` instead of mapping
`targetIds` through `buildWatchUrl`; `--no-playlist` is added on this branch so a
pasted playlist/channel URL grabs only the referenced item. Subscriptions never
set `opts.targetUrl`, so their arg array is byte-identical.

### A2 — FR-3 + FR-7 per-item failure attribution (Wave 5)

**Root cause (grounded, `lib/ytdlp/index.js` L479–483, `args.js` L638–643):**
`runSubscriptionCycle` calls `run.runDownload` ONCE with N positional
`watch?v=<id>` URLs and no `--abort-on-error`; the result is a single aggregate
`{ code, stderr, error }`. There is no per-video attribution to capture today.

**Mechanism chosen: parse-and-correlate, NOT per-URL spawns.** Splitting into
one spawn per video would give unambiguous exit codes but N× process + network +
archive read/write overhead and loses the batch efficiency the whole C1
survivor design was built around; rejected for the subscription path. Instead we
correlate yt-dlp's OWN per-item error output against the known target-id set —
run.js already line-splits the download streams for progress (FR-E), so this is
a second consumer of the same lines, not a new spawn:

- New pure parser `lib/ytdlp/failures.js` → `parseItemFailureLine(line,
  knownIds)`. It recognizes yt-dlp's `ERROR: [<extractor>] <id>: <reason>` line
  shape, extracts the id, and returns `{ videoId, reason }` **only when the
  extracted id is a member of `knownIds`** — an id it can't confidently
  attribute returns `{ videoId: null, reason }` (an "unattributed" entry:
  surfaced, never silently dropped, never misattributed to the wrong id).
  `reason` is control-char-stripped and length-capped; cookies-path redaction is
  applied downstream via the existing `redactString`.
- `run.spawnYtdlpDownload` gains `opts.knownIds` (the target-id set) and
  accumulates matched failures into a bounded `itemFailures[]` array on the
  result — same bounded-capture posture and cap constant as the existing
  `channelMeta[]` capture (SF3 unaffected: bounded array, no unbounded buffer;
  SF1 unaffected: reasons pass through `redactString`).
- `runSubscriptionCycle` maps `downloadResult.itemFailures` into an additive
  `items` (or `failures`) list on the activity LiveEntry via
  `activity.setSubscription(sub.id, { failures: [{ videoId, title?, reason }] })`
  — `title` filled from captured `channelMeta`/list metadata when available.
  `activity.mergeEntry` already shallow-merges unknown fields, so no
  activity.js change is needed. Backward-compatible: absent when nothing failed.

**Surfacing.** `GET /api/subscriptions/status` returns the snapshot verbatim
(no route change). The home status chip / `subscriptions.html` render the
per-item `{channel, video, reason}` list — this is what makes "10–15 failing,
can't tell which/why" diagnosable. Phase (b) root-cause fixes (Shorts one-off,
members-only-without-cookies, geo/format) are captured as follow-on sub-tasks
once Phase (a) reasons are visible, not guessed now.

### A3 — Cancel an in-progress download (Wave 5)

The child handle is created inside `run.js`'s spawn wrappers and not currently
exposed. Add a minimal, opt-in registration hook rather than a global registry:
`spawnYtdlpDownload` accepts `opts.onChild(child)`, invoked synchronously right
after a successful `cp.spawn`. `runOneShot` passes an `onChild` that stores the
handle in a module-level `Map<jobId, ChildProcess>` in `index.js`, deleted when
the promise settles. New route `POST /api/ytdlp/download/:jobId/cancel` (inside
the `isEnabled` gate): if the job has a live handle, `child.kill('SIGKILL')`
(same posture as the existing timeout-kill path — a killed download is already a
handled outcome, leaving a partial file that `cleanupFailedDownloadIntermediates`
reaps), and `activity.setOneShot(jobId, { state: 'cancelled' })` — a NEW terminal
state distinct from `error`. An id with no live handle → clean `404`, never a
crash. `cancelled` is added to `activity.TERMINAL_STATES` so it TTL-prunes like
`done`/`error`. (Scope note: cancel targets the one-shot path — the home status
chip's cancellable job — which is where Dean initiates ad-hoc downloads;

### B2 — FR-4 reconcile (HEAVIEST data gate, Wave 7)

**Phase 1 — virtual grouping (always-on, primary, ships first).** Server-side
query-time union in `GET /api/videos`: when the requested `?root=<dir>` equals a
subscription's resolved `channelDir`, the result additionally includes items
whose captured `db.metadata[id].channelUrl` equals that subscription's
`channelUrl`, regardless of physical folder, deduplicated by id. A pure helper
`unionChannelItems(items, channelDir, channelUrl)` (union of "under
`channelDir`" and "matching `channelUrl`", dedup by id) is unit-tested; an item
already under `channelDir` is never double-counted. Read-only, no file movement.

**Phase 2 — physical move on subscribe (ships second).** Trigger = **the
subscribe action itself** (`POST /api/subscriptions`), fire-and-forget AFTER the
subscription persists — NOT on every poll (would re-scan/fight repeatedly) and
not a manual button (Dean's decision: automatic on subscribe). It reuses the C1
move machinery (id re-key + sidecar rename + progress preservation) for each
item whose `channelUrl` matches the new subscription AND whose physical folder
`!==` the subscription's `resolveChannelDir`. Safety:

- **Confinement:** only items whose current path is already UNDER the download
  root are eligible (one-offs live in the synthetic `One-Off` folder under the
  download root) — a user's own local-library file is never moved into a channel
  folder. Target path is `resolveChannelDir(config, sub)` (its own traversal
  guard) + the file's basename, re-checked via `isPathUnder`.
- **Idempotent:** an item already under `channelDir` is skipped, so
  re-subscribing / a duplicate subscribe never re-moves.
- **Dedup archive not disturbed:** the archive keys by extractor+id, not path;
  moving a file within the download root does not change its archive entry, so no
  re-download on the next poll.
- **Scan-safe / progress-preserving:** via the C1 id re-key (see the load-bearing
  fact) — the moved file is already in `db.metadata` under its new-path id when
  the next scan runs, so it's a reuse, not delete+add.

Both phases are two-reviewer-gated; Phase 2 gets the same path-confinement +
scan-interaction rigor as delete and yt-dlp downloads.

### C1 — Move files between folders (Wave 3)

New `POST /api/videos/:id/move` mirroring the `DELETE /api/videos/:id` pattern
(`server.js` L2319). Body `{ targetFolder }`. Steps, all inside one
`updateDatabase` mutator after the FS work:

1. Resolve target under an ALLOWED mount (same confinement discipline as delete +
   yt-dlp paths); reject before any FS op if it escapes.
2. Same-device → `fs.rename`; cross-device (`EXDEV`) → copy+`unlink` fallback.
3. Compute `newId = getMediaId(newPath)`; **re-key** `db.metadata[oldId]` →
   `newId` (with `filePath` updated), move `db.progress[oldId]` → `newId`, and
   `fs.rename` the thumbnail (`<oldId>.jpg`→`<newId>.jpg`) and transcode sidecar
   (`transcodedPath(oldId)`→`transcodedPath(newId)`), each best-effort/idempotent.
4. Any subtitle sidecar (A6) moves alongside the media file.

Result: watch progress survives the next scan (unit-locked), no delete+new-add.
Pure helper `computeMoveTarget(filePath, targetFolder, allowedRoots)` →
`{ ok, newPath }` for the confinement/rename decision is unit-tested. UI: a
per-item "Move to…" picker over known folders. Focused two-reviewer gate.

### Alternatives considered

- **FR-1 curated multi-site host allowlist** (vs the structurally-open
  shape-checked validator chosen): safer-sounding but needs perpetual
  maintenance as yt-dlp adds extractors, and contradicts Dean's "any
  yt-dlp-supported site, broad" decision. Rejected — the injection safety comes
  from arg-array/`--`/no-shell + char/scheme/userinfo guards, not from the host
  set, so the open validator is equally injection-safe with a positive
  hostname-shape check as the hardening.
- **A2 one-spawn-per-video** (vs parse-and-correlate): perfectly attributable
  but N× overhead and loses batch efficiency + archive-write serialization
  simplicity. Rejected for the subscription path.
- **C4 cached stats aggregate** (vs live compute): rejected — invalidation
  complexity for no benefit at home-server scale.
- **B2 physical move on every poll / manual button** (vs on-subscribe):
  rejected — poll-triggered fights the scan repeatedly; a manual button
  contradicts Dean's "automatic on subscribe" decision.

### Risks and mitigations

- **Risk:** the FR-1 char-guard widening (`&`/`?`/`#`) is misread as a shell
  exposure. → **Mitigation:** the explicit no-shell/arg-array argument above +
  the adversarial two-reviewer gate + a unit test proving the arg array places
  the URL after `--` and never sets `shell:true`.
- **Risk:** a move that partially completes (file renamed, sidecar rename fails)
  leaves inconsistent state. → **Mitigation:** re-key inside one `updateDatabase`
  mutator; sidecar renames are best-effort/idempotent and self-heal on the next
  scan (thumbnail/transcode regenerate; progress already re-keyed).
- **Risk:** B2 physical move races the periodic scan. → **Mitigation:** the
  re-key makes the moved file already-indexed under its new id, so a concurrent
  scan sees a reuse, not delete+add; the move runs inside the serialized
  `updateDatabase` lock.
- **Risk:** cumulative regression across a 25-item single branch. →
  **Mitigation:** build+QA at the END of EACH wave against a small diff (per the
  one-branch-health section); `lib/ytdlp/index.js` re-read fresh at each wave.

No expected impact on the RELIABILITY.md budgets. `GET /api/stats` and the B2
