# v1.21.0 "The Polish Release"

## Goal

Ship Dean's full post-v1.20 feedback backlog as one cohesive release: fix the
subscription count-edit bug, overhaul the audio player with custom theme-aware
blocky controls (the single highest-leverage UI change), rearchitect
`/subscriptions` to be subscriptions-first, add channel metadata/pinning
polish, tighten the mobile library grid, add an extra-deliberate delete step
for irreplaceable local files, add download retry + a clearer status chip,
add watch-page theatre mode, and refresh the README — without regressing any
existing player, subscription-CRUD, delete, or folder/config behavior.

## Scope

Ten functional requirements, one cohesive v1.21.0 round. Grounded directly in
`docs/ui-research-2026-07.md`, `ROADMAP.md`'s Planned section (every item
below already appears there, confirming this is not new scope creep), and the
live code (see Decision log for every grounding finding).

### FR-1 — [BUG] Subscription count-edit reverts

**Root cause, grounded by reading the code (not assumed):**
`lib/ytdlp/client/subscriptions.js`'s live-status poll (`pollStatusOnce`,
`STATUS_POLL_BASE_MS = 2500`) calls `renderSubscriptions()` on **every** tick
(as often as every 2.5s), which does `clearChildren(listContainer)` and
rebuilds **every** row from scratch via `createSubscriptionRow` — including a
brand-new, closed `editPanel` with a fresh `editMaxVideosInput` re-seeded from
the cached (pre-edit) `sub.maxVideos`. If a user opens the inline Edit panel
and changes "download last N" but hasn't clicked Save within the poll
cadence, the very next poll tick silently destroys and replaces the open
panel — the input visibly reverts to the original value, exactly matching
Dean's report ("switches back to the original number even after pressing
Edit"). Server-side persistence (`PATCH /api/subscriptions/:id` →
`store.validateSubscriptionPatch`/`updateSubscription`) was read in full and
is **correct** — `maxVideos` (including `0` = unlimited) round-trips
properly. This is a client-side render-timing bug, not a validation/storage
bug.

Fix the poll-clobbers-open-edit race so an in-progress/open edit (any field,
not just maxVideos) survives a live poll tick, regardless of how this
manifests mechanically (skip the row's full rebuild while its edit surface is
open; a targeted partial DOM update for live status only; or another
approach — PE's call). Must be fixed regardless of FR-3, per Dean's explicit
instruction, even though FR-3's settings-detail-page redesign is likely to
structurally sidestep this exact bug class as a side effect (see FR-3).

Security-adjacent (subscription config mutation surface) — light gate; the
server-side validator/mutator path is unchanged, so the two-reviewer gate is
not required here (contrast FR-3/FR-5's heavier gates).

### FR-2 — [BIG, high-leverage] Audio player overhaul (custom controls)

Replace the native `<audio>`/`<video>` browser control bar (`controls`
attribute on the single shared `<video id="media-player">` element every
media type — audio and video — plays through, per `player.js`/`watch.html`)
with custom, app-owned blocky controls. Per the research this one change
delivers four of Dean's asks simultaneously:

- **(a) Theme-aware "now playing" surface.** The audio background-art layer
  (`#audio-bg-art`, `.audio-mode`) and the new control bar follow the active
  era + light/dark mode — no more pure-white native chrome regardless of
  theme.
- **(b) Blocky, non-derpy controls.** Square/beveled play-pause, seek,
  volume, and time controls replacing the rounded native Chromium bar —
  Winamp/early-Flash-retro per the research.
- **(c) Click-the-cover-art to play/pause.** `#audio-bg-art` becomes a click
  target toggling play/pause (with a fading glyph overlay), isolated from any
  other click handler on the player host (notably the v1.16 dock's
  tap-to-expand, which lives on `#player-dock`, a different element).
- **(d) Persist volume across tracks.** `volumechange` → localStorage; applied
  on load, before playback, clamped 0–1.

**Scope is audio-first; video must not regress.** Because both media types
share one `<video id="media-player">` element (confirmed by reading
`player.js`/`watch.html`/`lib/ytdlp/views/subscriptions.html`'s
`#player-host-template`), removing the native `controls` attribute
necessarily replaces video's control bar too — there is no way to give audio
custom controls while leaving video on native ones without forking the
player element. This is a resolved (non-blocking) fork: build one shared
control-bar component, verify video is not regressed (fullscreen,
rotate-to-fullscreen, the existing skip-controls overlay, live-transcode),
and apply the theme-aware/blocky treatment uniformly.

**Must preserve everything the current player already does** (all grounded
by reading `public/js/player.js` in full): inline iOS playback
(`playsinline`/`webkit-playsinline`/`x5-playsinline`), ±15s skip (buttons,
double-tap, press-hold-to-2x, keyboard `←`/`→`), the transcode "Preparing
this video…" overlay + `pollTranscodeUntilReady`, the resume overlay, the
v1.16.0 persistent player's full FULL↔DOCKED↔CLOSED state machine (reparent
between `#player-slot`/`#player-dock`, tap-to-expand, `[x]` close, iOS
reparent-resume-guard), Media Session (metadata/playback-state/position —
"near-free once controls are custom" per the research; keep and consider
extending action handlers), and progress saving (interval + pause +
PWA-lifecycle pause/persist).

**Known platform constraint (grounded, not assumed) to build around, not
"fix":** iOS Safari does not allow script to control `HTMLMediaElement.volume`
— it is fixed at `1.0` and controlled only by the hardware volume buttons.
FR-2(d)'s volume-persistence is therefore a desktop/Android-effective
capability; on iOS a custom volume slider must either be hidden or clearly
documented as inert, never presented as broken. Flagged explicitly so Dean's
iOS on-device pass doesn't mistake a platform limitation for a regression.

Biggest/riskiest FR — flagged for the **heaviest two-reviewer/adversarial
gate** (shared-player regression risk spans both audio and video and the
entire v1.16 persistent-shell/dock machinery) plus Dean's on-device pass
(especially iOS Safari) as the arbiter for feel and for any regression this
plan's tests cannot reach (no headless-browser/E2E infra exists, per
`docs/RELIABILITY.md`).

### FR-3 — [BIG, headline] Subscriptions page rearchitect (subscriptions-FIRST)

**Grounded by reading `lib/ytdlp/views/subscriptions.html` and
`lib/ytdlp/client/subscriptions.js` in full.** Today's page order is: (1) the
add-subscription form (URL, format, quality, filetype, maxVideos,
members-only, skip-Shorts), (2) the one-off download form + its job list,
(3) "Your subscriptions" — a `.folder-list-builder` box hard-capped at
`max-height: 240px` (`public/css/style.css`), each row a cramped inline
cluster of Pause/Edit/Re-pull/Delete buttons plus (once toggled) the FR-1
inline edit panel. This exactly matches the research's "cramped inline
edit"/"management surface wants a list" findings and Dean's own "not
intuitive or smooth to find a sub" complaint.

Rearchitect so the subscriptions LIST is the primary, first-encountered
content:

- A dense vertical **list** (not grid): one row = avatar/letter + channel
  name + one muted metadata line (format/quality/count) + a single trailing
  kebab/gear — not today's four-button inline cluster.
- **Row tap** (outside the kebab) opens that channel's videos (the existing
  `/?root=<channelDir>` per-channel playlist link from v1.20 FR-4, promoted
  to the row's primary target).
- **Kebab tap** opens a dedicated per-subscription **settings surface**
  (detail page or bottom-sheet — PE's call) with quality, count (maxVideos),
  type (format), name, skip-Shorts, filetype — replacing the cramped inline
  edit panel. This is also the FR-1 fix's natural home: a dedicated
  page/sheet is not subject to the list's live poll re-render at all, so the
  edit-clobber race class cannot recur there by construction.
- Add-subscription and one-off-download become **secondary/collapsible**
  (e.g. a collapsed section or a distinct "+" affordance) — not the
  top-of-page focus, but still fully reachable/functional.
- The list gets a bigger, better-proportioned layout, no longer bound by the
  `.folder-list-builder` 240px scroll cap.
- Preserve the disabled-`FILETUBE_YTDLP_ENABLED`-module no-op: `/subscriptions`
  still 404s server-side when disabled; nothing new is reachable.

Heavy gate (touches subscription CRUD UI — same tier as v1.20's FR-1/FR-4).

### FR-4 — ["Subscribed to" date + clickable channel link]

**Grounded finding:** `store.js`'s `addSubscription` already stamps
`addedAt: new Date().toISOString()` at creation time and has done so since
the module's original v1.11.0 design — `updateSubscription`/`ensureYtdlp`'s
backfill loop deliberately never touches it, confirming it is a stable,
already-persisted field for every subscription ever created through the
normal add path. Practically all existing subscriptions already have it; the
only theoretical gap is a hand-edited/corrupted `db.json` missing the field,
which must degrade to "date unknown" rather than crash or fabricate a date.

Also grounded: `createSubscriptionRow`'s `urlEl.textContent = sub.channelUrl`
renders the channel URL as **plain text today**, not a link.

Scope:
- Render "Subscribed on `<date>`" per row from the existing `addedAt` field
  (no new stored field for new subscriptions).
- Render the channel URL as a real `<a href>` opening the actual channel
  (`target="_blank" rel="noopener noreferrer"`), built via `.href`/
  `textContent` only — `channelUrl` already passed `validateChannelUrl`
  before persistence (v1.11.0+), so this is a safe value to link, not a new
  trust boundary.

textContent/href-safe (no XSS via channel name/url) — light gate (display-only,
reuses already-validated/persisted fields).

### FR-5 — Pin a channel's "View as Playlist" to persistent nav

**Grounded finding, load-bearing for this FR's design:** `POST /api/config`
(`server.js`, ~1916–1957) prunes `db.folderSettings` to keys that are either a
real `db.folders` entry or the single synthetic top-level "Downloads" root
(`syntheticRoots`) — **any other key, including a per-channel `channelDir`, is
silently dropped on the very next Settings save.** `folderSettings` is
therefore **not usable** as FR-5's pin store; it must be a new, separate,
persisted structure that a Settings save never touches or prunes.

Per the research, Subscribe (follow) and Pin (quick-access shortcut) are
different intents — keep them separate. Investigate (done): the current
Playlists surface is `public/js/common.js`'s `renderPlaylistsSheet`/
`openPlaylistsSheet`, driven purely by `GET /api/config`'s
`folders`/`folderSettings` — v1.20 FR-4 already deliberately keeps
per-channel playlist links OUT of that data source (surfaced instead via
`GET /api/subscriptions`'s `channelDir` field) specifically to protect this
invariant; FR-5 must uphold the same discipline.

Scope:
- A discoverable star/pin **toggle** (not drag-only) on a channel's "View as
  Playlist" surface (the existing `channelDir` link, or FR-3's new row).
- A pinned channel persists as an entry under Playlists/Folders (sidebar
  and/or the mobile Playlists sheet) — a one-tap return shortcut — surfaced
  **alongside**, never merged into, the `db.folders`-driven list.
- New, dedicated persisted store (server-side, e.g. a new `db.ytdlp`-adjacent
  namespace — PE's exact shape) that a `POST /api/config` save can never
  read, write, or prune.
- **Resolved, non-blocking fork:** store the pin as a **snapshot**
  (`channelDir` + a display label), not a live join keyed only by
  subscription id — so an unpinned-but-still-downloaded channel (unsubscribe
  without deleting files) doesn't silently break the pin, matching the
  research's "pin and subscribe are different intents" guidance. PE may
  revise if a stronger reason emerges.

Data-safety gate (folders/config invariant) — two-reviewer; must re-prove
`db.folders`/`folderSettings` are never written to or read from by the pin
feature.

### FR-6 — Library grid mobile density

**Grounded finding:** the requested `aspect-ratio:16/9; object-fit:cover`
thumbnail treatment is **already implemented**
(`.thumbnail-container`/`.thumbnail-img`, `public/css/style.css` ~565–578) —
no work needed there. A v1.15.0 mobile-specific rule already exists
(`@media (max-width:768px)`, `.video-grid { grid-template-columns:
repeat(auto-fill, minmax(140px, 1fr)); gap: 12px; }`, on top of
`.main-content`'s 16px padding) that numerically should already fit 2 columns
on most phone widths. Per Dean's continued real-device report (echoed in
`ROADMAP.md` and the research pass) it is evidently still not comfortable in
practice. Scope is narrowed accordingly: further tighten the mobile
`minmax()` floor (and/or add a narrower breakpoint) until Dean's on-device
pass confirms ≥2 comfortable columns, while leaving the already-correct
aspect-ratio/object-fit treatment and the untouched desktop rule
(`minmax(210px, 1fr)`, outside the mobile media query) alone.

Scoped CSS-only fix (`public/css/style.css`); desktop unaffected; on-device
Dean pass is the arbiter for "comfortable" density; light gate.

### FR-7 — Extra-deliberate delete for local (non-yt-dlp) files

A yt-dlp-downloaded file is re-downloadable — the existing v1.17.0 two-tap
card arm and the watch-page confirm modal (`showConfirmModal`, grounded by
reading `public/js/watch.js`'s `deleteBtn` handler) are fine for it. A LOCAL
(non-yt-dlp) file is irreplaceable and today gets the exact same,
single-tier confirm.

**Detection signal (reuse, not new):** `db.metadata[id]`'s already-present
`channelUrl`/`channelId`/`channelName` fields (v1.20 FR-2 — populated only
for yt-dlp downloads, present on every response since `GET /api/videos`/
`GET /api/videos/:id` already spread `...item`) are the existing signal for
"this file is yt-dlp-managed." A file with none of these is local. This must
fail SAFE: any ambiguity (including every pre-v1.20 local file, which has
none of these fields) is treated as local/irreplaceable, never the reverse.

Scope: for a local file only, add an ADDITIONAL, more deliberate delete step
— a distinct hard-warning confirm, visually/interactionally different from
the existing confirm modal / two-tap arm — so deleting an irreplaceable file
is a conscious 3rd action. The existing `DELETE /api/videos/:id` endpoint and
its graceful read-only-mount (`removeAnyway`) path are unchanged — this is a
client-side confirm-flow escalation only. yt-dlp files keep their existing,
lighter flow completely unchanged.

Gate (destructive-action class) — two-reviewer; adversarial review must
confirm the detection signal can never fail toward LESS friction than today.

### FR-8 — Download retry + clearer active-download status

**(a) Retry.** Grounded: a subscription-driven failure already has an
existing (always-available, not failure-contextual) "Re-pull all" button
(`POST /api/subscriptions/:id/repull`, wired via `repullOne`) — usable as the
retry mechanism for a failed subscription cycle, surfaced CONTEXTUALLY on the
failed row (not just the generic always-on button). A one-shot download job
(`createOneShotRow`) has **no retry today** — only Dismiss (`onDismiss`);
this is the real gap. On a failed one-shot job, surface a Retry that
re-invokes the same download (PE decides the exact mechanism: re-POST
`/api/ytdlp/download` with the job's original URL/options, since no
persisted one-shot "retry" endpoint exists today).

**(b) Status surface.** Replace/augment the current in-page-only status
(`sub-repull-status`, `oneshot-status`, the `oneShotListContainer` rows —
all confined to `/subscriptions`) with a persistent, app-wide corner chip
(candidate bottom-left per the research) visible from any page while a
download is active, reusing the EXISTING `GET /api/subscriptions/status`
polling data (`lib/ytdlp/activity.js`'s `getSnapshot` — no new backend
polling primitive). Collapsed = "N downloading · X%" summary; expands to a
small per-item progress panel. Completed items auto-dismiss; errored items
stay sticky until acknowledged. Coordinates with the FR-2 audio dock so the
two never overlap on mobile. Retro-styled: beveled box, segmented/striped
progress, 1px borders, era tokens.

Light-to-medium gate.

## Out of scope

- Any change to `validateChannelUrl`, `validateSubscriptionInput`/
  `validateSubscriptionPatch`, or any other existing yt-dlp security guard —
  FR-1/FR-3/FR-4/FR-5/FR-8 all reuse existing, unmodified validators/routes.
- Retroactive `channelUrl` backfill for pre-v1.20 downloads (unchanged,
  documented v1.20 limitation) — FR-7's local-file detection deliberately
  fails safe (treats an unmarked file as local) precisely because of this.
- Any change to `db.folders`/`folderSettings` schema, the single top-level
  synthetic "Downloads" folder splice/rename/order logic, or the
  `POST /api/config` prune behavior — FR-5 is strictly additive via a NEW,
  separate store.
- A client-side router library, build tooling, or any new npm runtime
  dependency for the custom audio/video controls (FR-2) or the status chip
  (FR-8) — vanilla DOM/CSS only, per `CONTRIBUTING.md`.
- New screenshots for the README (FR-10) — Dean's to provide; text/copy only
  in this release.
- Native OS-level Picture-in-Picture (superseded by the existing v1.16 docked
  mini-player; not reopened here).
- Checked against `docs/CONTRIBUTING.md`: nothing above defers a mandatory
  standard (tests, lint, `textContent`-not-`innerHTML`, no new deps) — every
  file touched by every FR still ships tests per CONTRIBUTING's "every new
  feature or bugfix ships with tests" rule; no FR's scope narrowing above
  trades away a mandatory standard.

## Constraints

- **Additive / no regressions.** Every FR layers on top of existing
  behavior. FR-2 is explicitly required to preserve every existing player
  feature (inline iOS playback, ±15s skip, transcode overlay, resume
  overlay, the full v1.16 FULL/DOCKED/CLOSED state machine, Media Session,
  progress saving) across BOTH audio and video.
- **Disabled-yt-dlp-module byte-identical guarantee preserved** for
  FR-1/FR-3/FR-4/FR-5/FR-8 — with `FILETUBE_YTDLP_ENABLED` off, no new
  route/UI/DOM is reachable; `/subscriptions` still 404s server-side exactly
  as today.
- **`db.folders`/`folderSettings` synthetic-root invariant unchanged**
  (FR-5): a pin is NEVER written into `db.folders` or `folderSettings`; the
  pin store is never scanned or pruned as a folder.
- **No existing security guard weakened.** FR-1/FR-3/FR-4/FR-5/FR-8 all
  route through the SAME, unmodified `validateChannelUrl`/
  `validateSubscriptionInput`/`validateSubscriptionPatch`/
  `DELETE /api/videos/:id` paths — no new parallel validator, no bypass.
- **FR-7 fails safe.** The yt-dlp-vs-local detection signal must never
  misclassify toward LESS delete friction than today; an ambiguous/missing
  signal is always treated as local/irreplaceable.
- **`docs/CONTRIBUTING.md`:** vanilla DOM (no framework/bundler), CommonJS
  server, 2-space/semicolons/single-quotes, `textContent` over `innerHTML`
  for new dynamic strings (FR-4's channel link/date, FR-3's detail surface,
  FR-5's pin labels, FR-8's chip), `node:test` for new/regression tests,
  lint 0 warnings, no new runtime dependencies.
- **Era-theme system preserved.** All new CSS (FR-2's control bar, FR-3's
  list/detail surface, FR-6's grid tightening, FR-8's chip, FR-9's theatre
  layout) uses existing CSS custom-property tokens — no hardcoded colors
  breaking the 2005/2009/2014/2021 eras or light/dark modes.
- **Node.js 22 LTS** toolchain (`engines` ≥20); CI/build-verify runs tests on
  Node 22.
- **No headless-browser/E2E infrastructure exists** (`docs/RELIABILITY.md`).
  Dean's on-device pass is the documented arbiter for FR-2 (audio, especially
  iOS Safari's fixed-volume constraint), FR-6 (grid density feel), FR-9
  (theatre feel), and FR-3 (list/detail-page feel) — not a testing gap to be
  closed, consistent with every prior round's posture.

## Design

### Approach

This release is ten mostly-independent surfaces bolted onto an unchanged
core. The design keeps every server security boundary (`validateChannelUrl`/
`validateSubscriptionInput`/`validateSubscriptionPatch`/`DELETE
/api/videos/:id`/`POST /api/config` prune) byte-identical and adds only two
small server surfaces: a NEW gated pin store + routes (FR-5) and two
additive fields on the one-shot activity entry for retry (FR-8). Everything
else is client/CSS. Every new persisted structure lives in `db.ytdlp` (the
module-owned namespace `POST /api/config` never touches) — never in
`db.folders`/`folderSettings`.

The single biggest design decision is FR-2's shared control bar. Because
audio and video share one `<video id="media-player">` cloned once into the
persistent host (`player.js`), the custom bar is wired ONCE inside
`ensureHost()`/`wireHostListeners()` exactly like the existing skip controls,
so it travels with the host across FULL↔DOCKED↔CLOSED "by construction" — no
per-view re-wiring, no new listener churn. The design deliberately keeps the
bar ALWAYS-VISIBLE (no auto-hide) to shrink the regression surface, and moves
only ONE existing behavior: desktop fullscreen retargets from the `<video>`
element to the `.player-container` host so the custom bar is visible in
fullscreen (iOS keeps `webkitEnterFullscreen`'s native controls untouched).

FR-1 lands first and independently as a tiny poll-guard on the current file;
FR-3 then rebases on it and structurally retires the race (settings move to a
sheet the poll never re-renders). File-overlap sequencing is in the proposed
wave plan at the end of this section.

### FR-1 — poll-clobber fix (client render-timing)

**Mechanism.** The `~2.5s` poll's success branch calls `renderSubscriptions()`
(→ `clearChildren(listContainer)` + full rebuild), destroying any open edit
panel. Fix: track which subscription ids currently have an OPEN edit surface
in a controller-scoped `openEditIds` Set, and SKIP the poll-driven list
rebuild while that set is non-empty. `renderOneShots()` and the status-poll
cadence continue unaffected; explicit `loadSubscriptions()` (add/save/delete)
still rebuilds because those actions close the edit first.

- **Pure helper (AC1/AC4, unit-tested):** `shouldSuppressListRerender(openEditIds)`
  → `openEditIds.size > 0`. Because the guard leaves the entire open row's
  DOM intact, it is field-agnostic — every edit field (format/quality/
  filetype/maxVideos/skipShorts) survives, not just `maxVideos`.
- **Wiring:** `createSubscriptionRow` gains an `onEditToggle(subId, isOpen)`
  handler, called from the Edit/Cancel/Save button handlers; the controller
  adds/removes the id in `openEditIds`. `pollStatusOnce`'s success branch
  becomes `if (!shouldSuppressListRerender(openEditIds)) renderSubscriptions();`.
- **Integration (AC2):** regression-lock the already-correct `PATCH → GET`
  `maxVideos` round-trip (including `0` = unlimited).

This is a small, low-risk change on the CURRENT file. When FR-3 lands, the
inline edit panel is gone entirely, so the race cannot recur (AC22) and the
guard becomes a harmless no-op that FR-3 removes.

### FR-2 — custom blocky audio/video controls (HEAVIEST)

**Markup (all four shells' `#player-host-template`: `index.html`,
`watch.html`, `setup.html`, `subscriptions.html`).** Remove the `controls`
attribute from `<video id="media-player">`. Add, inside `#player-wrapper`, a
single control bar plus an art play/pause glyph:

- `<div id="player-controls" class="player-controls">` containing: a
  play/pause `<button id="pp-btn">`, `<span id="time-cur">`, `<input
  type="range" id="seek-bar">`, `<span id="time-dur">`, a mute `<button
  id="mute-btn">`, `<input type="range" id="vol-bar" min="0" max="1"
  step="0.01">`, and a fullscreen `<button id="fs-btn">`.
- `<div id="art-play-glyph" class="art-play-glyph"></div>` — the fading
  play/pause overlay glyph for the click-the-art surface.

The template is replicated across four shells; keep them byte-identical (the
existing FOUC/dock comments already note this replication).

**Controller (`player.js`, primary surface).** All listeners wired once in
`wireHostListeners()`:

- **Play/pause:** `pp-btn` toggles `mediaPlayer.play()/pause()`; existing
  `play`/`pause` listeners drive a new `updatePlayPauseUI()` (glyph + button
  state) alongside the existing `setPlaybackState`.
- **Click-the-art (AC9):** `#audio-bg-art` click toggles play/pause and
  flashes `#art-play-glyph`, but ONLY when `state === STATE_FULL`
  (`e.stopPropagation()` there). When DOCKED the click is left to bubble to
  `#player-dock`'s existing tap-to-expand handler — so the art surface never
  fights the dock and never navigates unexpectedly. This mirrors the existing
  `state !== STATE_FULL` guards on hold-to-2x/keyboard.
- **Seek scrub vs. commit (AC15):** an `isScrubbing` flag set on `seek-bar`
  `input` (visual only — updates a `--seek-fill` CSS var + `#time-cur`, never
  touches `currentTime`); cleared on `change`, which is the ONLY place
  `currentTime` (or `startLiveStream`) is committed. A pure reducer
  `seekCommitTarget({ duration, ratio, liveMode, liveTotal })` (unit-tested)
  computes the absolute target from the slider ratio for both normal and
  live-transcode sources.
- **Visual fill via rAF (not `timeupdate`):** a `requestAnimationFrame` loop
  runs while playing (and not scrubbing) updating `--seek-fill` and
  `#time-cur` from `currentAbsTime()`; cancelled on pause/close. The existing
  `updatePositionState` `timeupdate` wiring for Media Session is untouched.
- **Live-transcode sync (AC16):** the bar reflects
  `currentAbsTime() / (currentData.duration || mediaPlayer.duration)`; a
  committed seek in `liveMode` calls `startLiveStream(target)` exactly as the
  existing `skip()` does. Duration display prefers `currentData.duration` for
  live sources.
- **Volume + persistence (AC10/AC11):** pure helper `clampVolume(raw)` →
  a number in `[0,1]` or `null` for garbage (unit-tested). On init, BEFORE
  playback, read `localStorage['ft-volume']`, `clampVolume`, and apply to
  `mediaPlayer.volume`. On `volumechange`, persist. **iOS treatment:** a
  browser-only feature-detect `volumeIsSettable(el)` (save current, set a
  distinct probe value, read back, restore) runs once at init; when volume is
  NOT settable (iOS Safari), hide `#vol-bar`/`#mute-btn` and skip the apply —
  degrade silently, never an error (AC11). The clamp/parse helper is what
  `node:test` covers; the feature-detect is browser-only.
- **Fullscreen retarget (AC14, load-bearing):** introduce
  `enterFullscreen()` preferring `mediaPlayer.webkitEnterFullscreen()` (iOS —
  keeps native fullscreen controls) else `host.requestFullscreen()` (desktop
  — so the custom bar is visible in fullscreen, since it's a sibling of the
  video, not a child). Update the `f`-key handler, the rotate-to-fullscreen
  path, and `#fs-btn` to call it. This is the one existing behavior that
  changes, and it exists specifically so removing native `controls` doesn't
  strip desktop-fullscreen controls.

**Everything preserved (AC12):** skip controls, ripples, hold-to-2x,
double-tap, resume/transcode overlays, the FULL/DOCKED/CLOSED machine,
Media Session, progress saving, iOS reparent-resume — all untouched; the bar
is purely additive markup + listeners on the same host. Media Session stays
as-is; adding play/pause/seek action handlers is the optional near-free
extension (AC13), not required.

**Styling (`style.css`, new `.player-controls` section):** fixed-height flex
bar pinned to the bottom of `.player-container`, always visible; `border-
radius: 0`, beveled inset/outset borders from era tokens; square range thumbs
via `::-webkit-slider-thumb`/`::-moz-range-thumb` (`border-radius:0`) and
segmented fill via `::-webkit-slider-runnable-track`/`::-moz-range-progress`
plus the `--seek-fill` var; `color-scheme: dark light` on the wrapper. All
colors from era CSS vars so dark mode "just works" (AC7/AC8). In `.audio-mode`
the bar sits over the art; note `#audio-bg-art` is currently
`pointer-events:none` — the click-to-play handler attaches there and must
re-enable pointer events for that layer in audio mode only.

### FR-3 — subscriptions-first rearchitect (HEAVY)

**New IA (`subscriptions.html`).** Reorder `#view-root` so "Your
subscriptions" is FIRST and primary. Wrap the add-subscription and one-off
forms in native `<details>`/`<summary>` collapsible sections ("+ Add a
subscription", "One-off download") placed BELOW the list — fully reachable,
not the top-of-page focus (AC18/AC23), no JS needed. Replace the
`.folder-list-builder` (240px cap) list container with a new `.sub-list`
container with no scroll cap (AC24).

**New row (`subscriptions.js` `createSubscriptionRow` rewrite).** One dense
row = avatar/letter (`name[0]`, beveled) + channel name + ONE muted metadata
line (`formatSubMeta` + FR-4's subscribed date) + the FR-4 channel link + a
single trailing kebab/gear `<button>` (AC19). The inline Pause/Edit/Re-pull/
Delete cluster and the inline `editPanel` are removed.

- **Row tap (AC20):** the row body (outside the kebab) navigates to
  `/?root=<channelDir>` via `window.FileTube.navigate` (the v1.20 FR-4
  per-channel link, promoted to primary). Rows without a resolved
  `channelDir` are non-navigating (unchanged fail-safe).
- **Kebab → settings sheet (AC21):** PE's call is a **bottom-sheet detail
  surface** (not a separate route) — keeps the SPA-lite shape and the
  disabled-module no-op (`/subscriptions` still 404s when disabled; nothing
  new is server-reachable, AC25). The sheet contains
  quality/count(maxVideos)/type(format)/filetype/skipShorts plus
  Pause/Resume, Re-pull, Delete, and Save. Save PATCHes via the existing,
  unmodified `PATCH /api/subscriptions/:id`. Because the sheet is NOT part of
  the poll-rerendered list, FR-1's race class cannot recur here by
  construction (AC22).
- **Poll rerender:** with edit out of the list, the poll may still rebuild
  rows safely; to avoid 2.5s flicker and losing kebab state, the poll updates

### FR-5 — pin a channel playlist to persistent nav (HEAVY, data-safety)

**Store (NEW, never `db.folders`/`folderSettings`).** A new `db.ytdlp.pins`
array of snapshots `{ id, channelDir, label, pinnedAt }`, where `id =
getMediaId(channelDir)` (stable key; avoids slashes in route params) and
`label` is a display SNAPSHOT (channel name at pin time) so unsubscribing
without deleting files never breaks the pin (resolved fork). `ensureYtdlp`
backfills `pins: []` alongside `subscriptions`/`downloadMeta`; because
`db.ytdlp` is only ever materialized by an enabled-module write, the
disabled path stays byte-identical. `POST /api/config` writes only
`db.folders`/`db.folderSettings` (confirmed lines ~1916–1964) — it never
reads, writes, or prunes `db.ytdlp.pins` (AC33/AC34/AC38).

New store functions (in `store.js` or a new `lib/ytdlp/pins.js`, tested):
`listPins(deps)`, `addPin(deps, {channelDir, label})` (idempotent by id),
`removePin(deps, id)`.

**Routes (gated inside `registerRoutes`'s `isEnabled`, so absent/404 when
disabled — AC69):**

- `GET /api/subscriptions/pins` → `listPins`.
- `POST /api/subscriptions/pins` → validate + `addPin`.
- `DELETE /api/subscriptions/pins/:id` → `removePin`.

**Add-pin validation (security):** the client submits `channelDir` (untrusted
string). Before storing, confine it under `config.downloadDir`
(`path.resolve` prefix check, same posture as `resolveChannelDir`) so a
hostile pin can never point `/?root=` at an arbitrary path; strip control
chars and length-bound `label`. (`/?root=` is already a pure path-prefix
filter over the INDEXED library, so worst case is "matches nothing," but this
keeps the store clean and the label safe.)

### FR-8 — download retry + status chip (LIGHT-TO-MEDIUM)

**(a) Retry (AC52/AC53).**

- **Subscription failure:** contextual "Retry" on an errored row (or its
  settings sheet) calling the existing `POST /api/subscriptions/:id/repull` —
  no new endpoint.
- **One-shot failure (the real gap):** re-invoke by re-POSTing the SAME
  `POST /api/ytdlp/download` with the failed job's original
  `url/format/quality/filetype/folder`. The one-shot `LiveEntry` today stores
  only `label`/`url`, so add `format`/`quality`/`filetype` to the entry
  written by the download route and `runOneShot` (small `index.js` change) so
  the client can reconstruct the request. The retry is a normal new one-shot
  through the SAME `classifySingleVideo`/validation path — no bypass, no new
  persisted retry endpoint.

**(b) App-wide status chip (`common.js` component, AC54–AC58).** A fixed
bottom-left corner chip mounted only when the module is enabled (reuse the
existing subscriptions-nav capability probe). It polls the existing
`GET /api/subscriptions/status` (no new backend primitive) at a slow app-wide
cadence, backing off via the existing `nextPollDelay` pattern and hiding
itself when the snapshot is empty. Collapsed = "N downloading · X%"; tap
expands to a per-item panel (name/%/state) with Retry on errored one-shots
and Dismiss. Completed items auto-dismiss; errored stay sticky until
acknowledged. To avoid redundant polling, the chip suppresses itself on
`/subscriptions` (that page owns its inline status). All chip text via
`textContent`. Retro styling: beveled box, segmented/striped progress, 1px
borders, era tokens (`style.css`).

**Coordination with FR-2 dock (AC57):** the chip is bottom-LEFT and sits
above the mobile bottom-nav; `#player-dock` occupies the opposite/lower-right
region — the CSS positions the chip to clear both the dock and the bottom-nav
so they never overlap on mobile.
