## In scope ÔÇö three isolated task commits

### T1 ÔÇö Sticky home filter/action bar
**Problem:** the filter/action row (`.section-title`: sort, All/Videos/Audio,
Shuffle, Rescan) scrolls away with the grid, so re-shuffling mid-scroll means
scrolling back to the top. **Acceptance:** scroll the home grid on mobile; the
filter bar stays pinned directly below the fixed header, and Shuffle/Rescan
stay tappable at any scroll depth. Grid content scrolls cleanly *under* it (no
see-through, no overlap of the sort-menu popover or the player dock/modals).

**Approach (CSS only):** `#library-content .section-title` (home-scoped ÔÇö
`.section-title` and `#library-content` exist only on the home view; other
views use `books-section-title`/`sidebar-section-title`) becomes
`position: sticky` with:
- `top: var(--sticky-bar-top)` ÔÇö new var: 56px default (header height),
  overridden to `var(--mobile-header-h)` in the `max-width:768px` block.
- solid `background: var(--bg-color)` so grid rows don't show through.
- `z-index: 20` ÔÇö above grid cards (z-index 2), below the sort-menu (30, a
  descendant so it still overlays) and below the dock/modals.

### T2 ÔÇö Incremental-pop Home (restore-your-place) ÔÇö LOAD-BEARING
**Problem:** tapping Home always jumps to the top of a fresh feed, discarding
where you were. Dean uses Home as his omni-back on mobile. **Decided model
(Dean, override of the convention default):** the Home control walks *up one
level per tap*, restoring the scroll/view you left at each level; "home" is
simply the top of that walk. NOT a two-state jump (that breaks the deep
creator-dive case); NOT a double-tap overload.

**Key recon finding:** the router (`common.js:2604-3345`) ALREADY has a full
pushState/popstate stack with per-entry `scrollY` **save**
(`recordScrollForCurrentState`) and **restore** (`swapToView`/
`restoreHomeFromCache`/`handlePopState`). This is NOT a history-stack rework.
Incremental-pop reuses `history.back()` Ôćĺ the existing popstate restore path.

**Net-new:**
1. `depth` field in the history state (`buildHistoryState` 4th arg; preserved
   by `parseHistoryState` and `recordScrollForCurrentState`) ÔÇö how many in-app
   entries sit behind the current one. Seeded 0 at `bootRouter`; `pushState`
   sites set `current.depth + 1`; `replaceState` sites keep `current.depth`.
   `depth > 0` Ôçĺ there IS an in-app level to pop to (guarantees `history.back()`
   stays inside the app ÔÇö depth only rises on our own pushes, so it can never
   walk back past the session's first stamped entry to an external referrer).
2. Pure `nextHistoryDepth(currentState, isReplace)` and
   `resolveHomeButtonAction(depth, currentPathAndSearch)` Ôćĺ
   `'back' | 'go-home' | 'noop'`. Exported for node:test.
3. Runtime `goHomeControl()`: reads live `history.state.depth` + current path,
   applies the decision. `'back'` Ôćĺ guarded `history.back()`; `'go-home'` Ôćĺ
   `navigate('/')` (depth-0 deep-link entry that isn't home); `'noop'` Ôćĺ
   nothing (already at session-root home).
4. `handleDocumentClick`: an intercepted click whose target resolves to the
   home root (`/` or `/index.html`, no query) routes to `goHomeControl()`
   instead of `navigate()`. A `/?root=ÔÇŽ` folder drill is a normal forward nav.
5. **Race guard:** `homeBackPending` flag set when `goHomeControl` issues
   `history.back()`, cleared in `handlePopState`. A second Home tap while
   pending is ignored (coalesced), so a fast double-tap can never pop two
   levels / past the app before `window.history.state` reflects the new depth.

**Acceptance:** drill home Ôćĺ folder Ôćĺ creator Ôćĺ video; tap Home Ôćĺ creator
(scroll restored); tap Home Ôćĺ folder; tap Home Ôćĺ home top. From a fresh
deep-link into a video (no in-app history), tap Home Ôćĺ `/`. At home already Ôćĺ
no-op. Browser back button unchanged. No exit-to-referrer on rapid taps.

### T3 ÔÇö Music drill sticky-header rotate re-measure
**Problem (provable instance of Dean's "styling breaks on rotate"):**
`music.js` `wireStickyObserver` measures the fixed-header height ONCE
(`siteHeader.offsetHeight`) and bakes it into `--music-sticky-top` + the
observer's `rootMargin`, with no re-measure on rotate. Mobile header is 96px
(`--mobile-header-h`) vs 56px, so portraitÔćĺlandscapeÔćĺportrait leaves the
sticky offset / collapse threshold stale until the next `render()`.
**Approach:** a debounced (`~150ms`) `resize`+`orientationchange` listener,
registered inside `init` with the existing `{ signal }` AbortController
(auto-removed on the SPA `destroy()`), re-invokes `wireStickyObserver()` (which
disconnects + re-measures + recreates the observer, and no-ops when not on a
drill). `content.isConnected` guard covers a trailing debounce timer firing
after `destroy()`. **Acceptance:** on a music album drill, rotate
portraitÔćĺlandscapeÔćĺportrait; the sticky header stays parked directly under the
header and the collapse threshold is correct in both orientations.
