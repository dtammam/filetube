# v1.22.0 "Player Parity + Roadmap"

## Goal

Fix the v1.21.0 mobile-video player regression by restoring native controls
on mobile video (while keeping the loved v1.21 custom bar everywhere else),
then work through the rest of Dean's post-v1.21 roadmap backlog: retroactive
Subscribe-button backfill, a subscriptions-page deep link from the watch
page, an era-theme light-mode contrast audit, desktop-sidebar channel pins,
a configurable max-duration download gate, a loop/repeat toggle, desktop
cross-tab playback + native Picture-in-Picture, card-level download, and a
scroll-jump re-verification — without regressing any existing player,
subscription, security, or folder/config behavior.

## Scope

Ten functional requirements, one cohesive v1.22.0 round. FR-1 is the
top-priority regression fix and leads the round; FR-2 through FR-10 work
through the rest of the `ROADMAP.md` v1.22 candidate list. Grounded directly
in the live code (see Decision log for every grounding finding), the
completed `2026-07-06-v1.16-watch-experience.md` (persistent-player/dock
architecture) and `2026-07-08-v1.21-polish-release.md` (custom control-bar
introduction) exec plans, and `docs/ui-research-2026-07.md`.

### FR-1 — [TOP PRIORITY, heaviest] Diverging-but-cohesive controls: native on mobile VIDEO, custom on desktop + mobile audio

**Grounded regression, confirmed by reading `public/js/player.js` and
`public/watch.html`'s `#player-host-template` in full:** v1.21.0 FR-2 removed
the native `controls` attribute from the single shared
`<video id="media-player" preload="metadata" playsinline webkit-playsinline
x5-playsinline style="display: none;">` app-wide (confirmed byte-identical
across all four shells: `public/index.html`, `public/setup.html`,
`public/watch.html`, `lib/ytdlp/views/subscriptions.html`) and replaced it
with the custom `#player-controls` blocky bar for **every** viewport and
**every** media type. On desktop this is a clean win (Dean: "desktop is
solid"). On mobile VIDEO it is a regression: the bar's `.pc-btn`/`.pc-range`
controls are small/tap-unfriendly at mobile widths (only the existing
`@media (max-width:768px)` rule at style.css:3074 lightly restyles them, no
size floor), the fullscreen button (`#fs-btn` → `enterFullscreen()`) is
unreliable on some mobile browsers, mobile lost the native menu's playback
speed and download affordances, there's no AirPlay/PiP entry point, and
**the press-and-hold-2x gesture now visibly triggers the browser's
text-selection highlight** — confirmed by reading `public/css/style.css`:
`.player-container`, `#media-player`, and `.skip-controls` carry **no**
`user-select`/`-webkit-touch-callout` rule anywhere (the only
`user-select: none` rules in the file, at lines 1189/1671/2842, are on
unrelated UI, not the player surface).

**Locked decision (Dean + coordinator), responsive by media-type AND
form-factor:**
- **Mobile VIDEO → restore NATIVE `controls`** on `#media-player` (fullscreen,
  playback speed, download, AirPlay, PiP, and correct native sizing all
  return "for free," exactly matching Dean's pre-v1.21 experience). **KEEP**
  FileTube's own gesture layer on top, exactly as it worked pre-v1.21: single
  tap → play/pause, press-and-hold → 2x speed, double-tap → ±15s skip. Fix
  the text-selection glitch with `user-select: none` /
  `-webkit-user-select: none` / `-webkit-touch-callout: none` on the gesture
  surface.
- **Mobile AUDIO → KEEP the v1.21 custom themed bar** (dark-mode,
  click-the-cover-art-to-play) — audio has no native fullscreen/speed/
  download menu to miss, so custom stays strictly better — but **drop the
  volume slider on mobile** (hardware volume buttons make `#vol-bar`
  redundant there).
- **DESKTOP → unchanged.** The v1.21 custom bar drives BOTH audio and video,
  byte-identical to today (no regression to what Dean likes).

**Mechanism must be clean within the existing v1.16 persistent-player/dock
architecture** — grounded, load-bearing findings from reading `player.js` in
full:
- The single shared `<video>` is cloned exactly ONCE from
  `#player-host-template` (`ensureHost()`) and every listener is wired
  exactly ONCE (`wireHostListeners()`), so it survives FULL↔DOCKED↔CLOSED
  reparenting "by construction." Whatever toggles `controls` must fit this
  same one-time-wiring model — not something re-decided per reparent.
- The only existing mobile/desktop signal in the codebase today is
  `isMobileViewport()` (`player.js`, `window.matchMedia('(max-width: 768px)')`
  — used for the desktop-only initial-autoplay gate and the live-transcode
  branch). FR-1 explicitly calls for a **more robust signal than bare
  width** (the same shell serves a desktop browser window narrowed below
  768px, and a large-screen tablet in portrait) — e.g. combining viewport
  width with a coarse-pointer/no-hover media query or touch-capability
  check. The exact combined heuristic is a PE Design-stage decision (flagged
  below); this plan requires that whatever it is, it is a **named, testable,
  pure helper**, not an inline `matchMedia` call duplicated at each site.
- `.audio-mode` (video vs. audio branch) is orthogonal to the mobile/desktop
  signal — the two combine into 4 cases (desktop video, desktop audio,
  mobile video, mobile audio), only one of which (mobile video) changes
  behavior from today.
- `wireSkipHoldGestures(mediaPlayer)` (no `onSingleTap`) is **already** wired
  unconditionally to `#media-player` for every video today — restoring
  `controls` on mobile video requires **no change** to this wiring; the
  gesture layer already coexists with a control strip the way it did
  pre-v1.21 (gestures fire on taps outside the native bottom control strip,
  exactly as native `<video controls>` has always allowed). This materially
  shrinks the mechanism's risk surface.
- `enterFullscreen()` currently retargets desktop fullscreen to `host`
  (`.player-container`, a sibling of `<video>`) so the custom bar stays
  visible in fullscreen; iOS already branches through
  `mediaPlayer.webkitEnterFullscreen()` (native iOS chrome, untouched by the
  custom bar entirely) — this iOS branch requires no change. A newly-native
  mobile `<video controls>` on non-iOS mobile (Android Chrome, etc.) gets
  fullscreen for free via the native bar's own button, so `#fs-btn`
  presence/absence must track the same native-vs-custom decision.
- Docked player state (`dock()`/`mountInSlot()`/`close()`), prev/next
  (`common.js`'s `deriveOrderedIds`/`computeNeighbors`), autoplay-next
  (`handleAutoplayNext`), the transcode/resume overlays, Media Session, and
  progress saving are **all** driven off `mediaPlayer`/`host` state that is
  completely independent of the `controls` attribute — none of that logic
  reads or branches on `controls` anywhere today, so toggling it must not
  require touching any of those systems, only the visibility of
  `#player-controls` and the presence of the `controls` attribute.

Flagged as the **heaviest item this round** — two-reviewer/adversarial gate
plus Dean's iOS on-device pass as the arbiter for feel; independently
shippable early as fast regression relief if desired (self-contained to
`player.js` + one CSS section, no dependency on FR-2..FR-10).

### FR-2 — [high] Retroactive Subscribe-button backfill

**Grounded root cause, confirmed by reading `server.js`'s scan merge (~lines
1578–1620) and `lib/ytdlp/index.js`/`store.js` in full:** channel identity
(`channelUrl`/`channelId`/`channelName`) is bridged onto `db.metadata[id]`
**only** when `freshlyScannedIds.has(item.id)` — i.e., only on a file's very
first scan after it lands as new/changed. An already-indexed item that hits
the `reusable`/`legacyVideoCodecBackfillOnly` fast paths is **never**
revisited by this bridge, "keep[ing] whatever identity it already has"
(comment, server.js ~1359). This explains BOTH gaps Dean reported:
- **Pre-v1.20 downloads**: scanned long before `channelUrl` capture existed
  — permanently un-bridged.
- **Subscription-path downloads** (Dean's report that even NEW
  subscription downloads sometimes lack the button): a plausible root cause,
  grounded by reading `runSubscriptionCycle` (`lib/ytdlp/index.js` ~475–545)
  — `persistCapturedChannelMeta`/the survivor fallback write
  `db.ytdlp.downloadMeta[videoId]` **before** `processSubscription`'s
  post-download `scanDirectories()` trigger, so the happy path is correct —
  but if the periodic background auto-scan (`armScanTimer`, independent of
  the download-triggered scan) races ahead and indexes the same new file
  FIRST (before `downloadMeta` is written, or in a scan pass that doesn't
  overlap the download's own trigger), that file is marked
  `freshlyScannedIds` **without** a matching `downloadMeta` entry yet — the
  bridge finds nothing to consume, and because bridging only ever happens on
  a file's FIRST scan, the identity is lost forever on every subsequent scan
  (the file now takes the `reusable` fast path). This must be verified (not
  assumed) and, if confirmed, is an inherent race in the "bridge only on
  first scan" design that the retroactive fix below also incidentally heals.

**Fix (retroactive, no re-download):** yt-dlp downloads land in per-channel
subfolders (`resolveChannelDir(config, sub) = downloadDir/sanitizeChannelName(sub.name
|| sub.channelUrl)`, `lib/ytdlp/args.js`). Infer a missing channel identity
from the item's download FOLDER: for any `db.metadata[id]` item lacking
`channelUrl` whose `filePath` resolves under a subscription's `channelDir`,
backfill `channelUrl`/`channelName` (and `channelId` if available) from that
subscription — **at scan time**, so the Subscribe button appears on
existing videos too and reliably shows "Subscribed" for subscription
content. Grounded building blocks already exist for this: `lib/ytdlp/index.js`
already exports `enrichSubscriptionWithChannelDir` (computes
`{...sub, channelDir: args.resolveChannelDir(config, sub)}` — the exact
per-subscription channelDir a matcher needs), `config.downloadDir` is
available wherever `extraScanRoots()`/`consumeDownloadChannelMeta` are
already called from server.js's scan, and `item.filePath` is already read
in the same merge loop that currently does the `freshlyScannedIds`-scoped
bridge (server.js ~1607) — this backfill is a natural sibling to that
existing block, but must NOT be scoped to `freshlyScannedIds` (the whole
point is to also catch already-indexed items).

**Security (must not weaken anything):** an inferred/backfilled
`channelUrl` must still pass `lib/ytdlp/url.js`'s `validateChannelUrl`
before being persisted/used — it is already a subscription's own
`channelUrl` (which passed this same validator at subscribe-time), but this
plan requires the backfill path to defensively re-validate rather than
trust it blindly a second hop removed from where it was first validated
(defense-in-depth, matching this codebase's existing "re-validate
downstream" posture, e.g. `args.js`'s builders re-validating `sub.channelUrl`
before every spawn). Flagged for the **two-reviewer gate** (data mutation +
spawn-guard-adjacent surface).

### FR-4 — Grey-on-white contrast in light mode on certain era themes

Some elements (the Subscribe button explicitly called out, possibly others)
render grey-on-white / low-contrast in LIGHT mode under certain era skins.
**Grounded:** every relevant color in `public/css/style.css` already flows
through CSS custom properties (`--text-primary`, `--text-secondary`,
`--btn-bg`, `--card-bg`, `--yt-red`, etc.) rather than hardcoded hex on the
Subscribe-adjacent surfaces read (`.btn`/`.btn-primary`,
`.subscribe-modal-identity*`) — so this is a **token-VALUE** problem in one
or more era/light combinations (e.g. a `--text-secondary`/background pairing
that clears contrast in dark mode but not light, or vice versa), not a
missing-token problem. No automated contrast tooling exists in this repo.
Audit contrast across all era themes (`:root`/`[data-theme]` blocks,
`[data-mode]` light/dark variants) in LIGHT mode specifically; fix the
offending token/element combos. On-device Dean arbiter for the final verdict
(no CI-enforceable contrast check exists or is being added this round).

### FR-5 — Pins in the desktop sidebar (v1.21 FR-5 follow-up)

**Grounded finding:** v1.21.0 FR-5 shipped channel pins rendering ONLY in
the mobile Playlists sheet (`common.js`'s `renderPinnedPlaylists`/
`derivePinnedPlaylistEntries`, targeting `#playlists-sheet-list`) — the
desktop left-nav sidebar (`#sidebar-folders-list`) was explicitly out of
that task's lane because, unlike the Playlists sheet (one owner,
`common.js`), **`renderSidebarFolders` is independently duplicated in THREE
files** (confirmed by reading all three): `public/js/main.js:208`,
`public/js/watch.js:982`, and `public/js/setup.js:241` — each view module
renders `#sidebar-folders-list` itself on its own `init()`, because that
element lives in the persistent shell OUTSIDE `#view-root` (same reason
`common.js`'s `restoreHomeFromCache` needs the `restoreSidebar()` hook —
see its C3 remediation comment). Add the desktop-sidebar counterpart so
pinned channels show alongside (never merged into) the `db.folders`-driven
list, in ALL THREE sidebar-rendering call sites, consistently. Reuse the
SAME gated pin store and the SAME pure `derivePinnedPlaylistEntries` helper
(no new store, no new pin data model) — mirroring `renderPinnedPlaylists`'s
stricter `createElement`/`textContent`/`createTextNode`-only DOM
construction (never `innerHTML`) for the same reason: a pin's `label` is a
creator-controlled snapshot. **Hard invariant, unchanged from v1.21:** a pin
is NEVER written into `db.folders`/`folderSettings`; `POST /api/config`
never reads/writes/prunes the pin store.

### FR-6 — Max-duration download gate (parameter, default 2h, changeable like max-videos)

A configurable maximum item LENGTH so very long items (11–20h live streams)
are skipped, bounding download size + transcode time, wired to yt-dlp
`--match-filter "duration < <seconds>"`. **Grounded pattern to mirror
exactly**, read in full: `lib/ytdlp/config.js`'s `DEFAULT_MAX_VIDEOS`/
`parseMaxVideos` (env-configurable global default, `0`/unset = a documented
sentinel), `lib/ytdlp/store.js`'s `validateMaxVideos`/`MAX_SUB_MAX_VIDEOS`
(per-subscription override validation, reused identically across
`validateSubscriptionInput`/`validateSubscriptionPatch`/`addSubscription`),
and `lib/ytdlp/args.js`'s `effectiveMaxVideos = subMaxVideos ?? config.maxVideos`
resolution feeding `playlistEndArgs`/`--playlist-end`. Make max-duration a
**real parameter, same shape**: a new `FILETUBE_YTDLP_MAX_DURATION_SECONDS`
env default (**sensible default of 2 hours = 7200 seconds**), a
per-subscription `maxDurationSeconds` override validated the same way
`maxVideos` is, and `0`/blank meaning **unbounded** (mirroring
`DEFAULT_MAX_VIDEOS`'s `0` semantics exactly — do not invent new semantics).
Wire the resolved value into `buildYtdlpListArgs` (`args.js`) as an
additional `--match-filter "duration < <n>"`, combined correctly alongside
the existing skip-Shorts `--match-filter` (`SHORTS_MATCH_FILTER`) — both
are FIXED-shape strings with only a validated integer interpolated (never
raw user text), matching this file's existing "no argv-injection possible,
because the value is a bounded, re-validated integer, and the array is
passed to `spawn` directly, never a shell" security posture. Surface the new
field in the v1.21 FR-3 settings sheet (subscription-level) alongside
`maxVideos`, and in the add-subscription form. Flagged for the
**two-reviewer gate** (yt-dlp args/spawn surface).

### FR-7 — Loop / repeat toggle

A watch-page toggle to loop/repeat the current media when it ends (replay
instead of stop/advance). Audio + video. **Grounded reconciliation point:**
`player.js`'s `mediaPlayer.addEventListener('ended', ...)` already has THREE
separate listeners (progress-reset-to-0, Media Session `'none'`, and
`handleAutoplayNext` for the v1.16/v1.21 `autoplayNext` setting) — loop
needs to be a FOURTH, coordinating one: when loop is ON, replay THIS item
(`currentTime = 0` + `.play()`, or `startLiveStream(0, true)` for a
live-transcode source, mirroring the existing `resumeNoBtn` handler's own
restart logic) instead of letting the existing reset-to-0/autoplay-next
behavior stand — **loop takes precedence over autoplay-next when both are
on** (reconciling with the v1.17.0 `autoplayNext` setting per Dean's
instruction: loop = repeat THIS; autoplay = next). Persist the toggle.
**Recommended fork resolution:** mirror the v1.21 FR-9 theatre-mode pattern
exactly (`localStorage['ft-theater']` + pure
`isTheaterModeActive`/`theaterModeStorageValue` helpers) rather than a
server `db.settings` setting — loop is a watch-page-local preference like
theatre mode, not a cross-device Settings-page preference like
`autoplayNext`. PE may revise if a stronger reason emerges.

### FR-8 — Desktop cross-tab video + native Picture-in-Picture

**(a) Scope pause-on-hidden to mobile/PWA.** Grounded: `player.js`'s
`shouldPauseForLifecycleEvent` (pure helper) + its three listeners
(`pagehide`, `freeze`, `visibilitychange`-hidden) currently pause+persist
ANY playing video unconditionally, regardless of viewport — this is
FileTube's OWN v1.17.0 FR-5 rule, correct for the mobile PWA (don't keep an
invisible video playing) but wrong for desktop (switching tabs souldn't
pause a video the user is intentionally background-playing, e.g. in a
second monitor or picked back up later). Scope this to mobile/PWA only (or
a setting) — reuse the SAME robust mobile/desktop signal FR-1 introduces
(do not invent a second one) — so desktop video keeps playing across tab
switches. Audio must be unaffected either way (already exempted by
`ctx.isAudio` — untouched). **Preserve the existing mobile "smart" behavior
exactly** (regression-tested).

**(b) Native Picture-in-Picture.** Add a PiP button (feature-detected via
`document.pictureInPictureEnabled`/`video.requestPictureInPicture`) so a
video can pop out into a native OS PiP window and persist across tabs/apps.
**Grounded:** no PiP wiring or `disablePictureInPicture` blocker exists
anywhere in the codebase today — this is a clean net-new addition to the
custom control bar (`#player-controls`), shown only where both the browser
supports it AND the custom bar is the active control surface (i.e., desktop
— and mobile audio, where it's harmless/hidden by feature-detection on
platforms lacking support). Native OS-level PiP was explicitly deferred by
v1.16.0 in favor of the docked in-app mini-player; this is the first time
it's being added, additively, alongside (not replacing) the existing dock.

### FR-10 — Watch-page scroll-jump — verify + fix if still repro

**Grounded: this bug was already root-caused and fixed in v1.17.0.** Reading

presumed-still-broken work: Dean confirms on-device whether it still
reproduces after the v1.17.x/v1.21 layout work; if it does, root-cause and
fix (new finding, scoped then); if not, close as resolved with no code
change required.

## Constraints

- **Additive / no regressions.** FR-1 is explicitly required to preserve
  every other v1.16/v1.21 player feature (dock/prev-next/autoplay-next,
  transcode + resume overlays, Media Session, progress saving, the
  audio-bg-art layer) across the new native/custom split — the persistent
  `<video id="media-player">` element and its one-time listener wiring are
  unchanged in kind, only the `controls` attribute + `#player-controls`
  visibility become conditional.
- **Desktop and mobile-audio custom-bar behavior must be byte-identical to
  v1.21** — no regression to what Dean already likes. This is the load-bearing
  acceptance bar for FR-1: three of the four (mobile-form-factor ×
  media-type) combinations must be provably unchanged; only mobile video
  changes.
- **Disabled-yt-dlp-module byte-identical guarantee preserved** for
  FR-2/FR-3/FR-5/FR-6 — with `FILETUBE_YTDLP_ENABLED` off, no new
  route/UI/DOM is reachable; `/subscriptions` still 404s server-side exactly
  as today.
- **`db.folders`/`folderSettings` synthetic-root invariant unchanged**
  (FR-5): a pin is NEVER written into `db.folders`/`folderSettings`; the pin
  store is never scanned/pruned as a folder.
- **No existing security guard weakened.** FR-2's inferred `channelUrl`
  passes the SAME, unmodified `validateChannelUrl` before use; FR-6's
  `--match-filter` value is a re-validated bounded integer, never raw user
  text, passed via the existing `spawn` argv-array (never a shell).
- **`docs/CONTRIBUTING.md`:** vanilla DOM (no framework/bundler), CommonJS
  server, 2-space/semicolons/single-quotes, `textContent` over `innerHTML`
  for new dynamic strings (FR-3's channel link, FR-5's sidebar pin labels),
  `node:test` for new/regression tests, lint 0 warnings, no new runtime
  dependencies.
- **Era-theme system preserved.** All new/changed CSS (FR-1's mobile-video
  native-controls area + gesture-surface `user-select` fix, FR-4's contrast
  fixes, FR-5's sidebar pin styling, FR-8's PiP button) uses existing CSS
  custom-property tokens — no hardcoded colors breaking the 2005/2009/2014/
  2021 eras or light/dark modes. FR-4 specifically may CHANGE token
  *values* in one or more era/light combinations (that is the fix), never
  introduce a new hardcoded color bypassing the token system.
- **Node.js 22 LTS** toolchain (`engines` ≥20); CI/build-verify runs tests on
  Node 22.
- **No headless-browser/E2E infrastructure exists** (`docs/RELIABILITY.md`).
  Dean's on-device pass — especially iOS Safari — is the documented arbiter
  for FR-1 (mobile feel, the native-vs-custom split, the gesture layer),
  FR-4 (contrast), and FR-8(b) (PiP feel) — not a testing gap to be closed,
  consistent with every prior round's posture.

   signal (viewport width + a coarse-pointer/no-hover media-query or touch-
   capability check), exposed as one named, pure, unit-testable helper — not
   a bare `matchMedia('(max-width: 768px)')` reused as-is. PE owns the exact
   heuristic; it must be reused (not re-implemented) by FR-8(a)'s
   pause-on-hidden scoping, since both features need the same "is this
   mobile" answer.
2. **FR-2 backfill scope** → infer purely from the download FOLDER path
   against a subscription's `channelDir` (no yt-dlp re-invocation, no
   metadata re-probe); never overwrite an item that already has a captured
   `channelUrl` (only fill a genuine gap).
3. **FR-3 deep-link target** → deep-link to the specific channel/subscription
   when the identity is resolvable (reusing `decideSubscribeButtonState`'s
   existing resolution already read by `watch.js`); otherwise fall back to
   the general `/subscriptions` page. Not a blocking fork — either outcome
   is a correct, safe navigation.

## Design (FR-1)

### Approach

FR-1 is a **responsive control-surface split** applied to the SINGLE
persistent `<video id="media-player">` and its custom `#player-controls` bar.
The four `form-factor × media-type` combinations collapse to one behavioural
change: **mobile video** swaps its control STRIP from the v1.21 custom bar to
the browser's native `controls`, keeps FileTube's gesture layer on top, and
fixes the press-hold text-selection glitch. The other three combinations
(desktop video, desktop audio, mobile audio) keep the v1.21 custom bar — with
mobile audio additionally dropping its (redundant) volume controls. Only two
files change: `public/js/player.js` and `public/css/style.css`.

The mechanism fits the existing one-time-wiring model exactly. Native vs.
custom is **STATE, not a listener** — the `controls` attribute and a
`.ff-mobile` marker class are toggled per media-load / per FULL↔DOCKED
transition inside the reparent functions that already run synchronously at
those points (`mountInSlot`/`dock`), so nothing in `wireHostListeners()`
changes and every existing listener still fires exactly once and travels with
the host "by construction." Three new pieces are added to `player.js`: a pure,
unit-tested form-factor helper (`resolveMobileFormFactor` + its browser
wrapper `isMobileFormFactor`, reused later by FR-8(a)); a pure, unit-tested
control-surface helper (`resolveControlsMode(mediaType, isMobile)` →
`'native' | 'custom'`); and a thin impure applier (`applyControlsMode()`) that
toggles the `controls` attribute and the `.ff-mobile` class. CSS consumes
`.ff-mobile` to hide the custom bar on mobile video and drop volume on mobile
audio, and adds the `user-select`/`-webkit-touch-callout` guards to the
gesture surfaces. **No HTML shell changes are required** (see below).

### The four cases (the whole behavioural contract)

| form-factor × type | control strip | `controls` attr | custom bar | `#vol-bar`/`#mute-btn` | `#fs-btn` |
| --- | --- | --- | --- | --- | --- |
| Desktop video | custom bar | absent | shown | shown | shown |
| Desktop audio | custom bar | absent | shown | shown | shown |
| Mobile audio | custom bar | absent | shown | **hidden** | shown |
| Mobile video (FULL) | **native** | **present** | **hidden** | n/a (bar hidden) | n/a |
| Mobile video (DOCKED) | none (tap-to-expand) | **absent** | hidden | n/a | n/a |

Desktop (both types) and mobile audio (minus volume) are **byte-identical to
v1.21**: `resolveControlsMode` returns `'custom'` for all three, so the
`controls` attribute is never set and the custom bar's markup, listeners, and
CSS are untouched.

### Component changes

- **`public/js/player.js` — pure form-factor helper (new, exported).**
  `resolveMobileFormFactor(signals)` returns a boolean from explicit signals so
  it is `node:test`-able with no DOM:

  - Primary signal — a touch-first device with no precise hover pointer:
    `signals.coarsePointer && signals.noHover`. This is the robust combined
    heuristic Dean asked for (NOT bare width): a desktop window narrowed below
    768px still has a fine pointer + hover → **desktop** (keeps the custom bar,
    "desktop is solid"); a large tablet in portrait (>768px) has a coarse
    pointer + no hover → **mobile** (native video controls). A touchscreen
    laptop (touch **and** a trackpad) reports hover-capable → **desktop**,
    which is correct (it has a precise pointer).
  - Defensive fallback — only when the pointer/hover media queries are
    unsupported (very old browsers, `coarsePointer`/`noHover` passed as
    `undefined`): fall back to `signals.narrowViewport` (width). Modern
    browsers never hit this branch.

  The browser wrapper `isMobileFormFactor()` reads the three
  `window.matchMedia` results — `(pointer: coarse)`, `(hover: none)`,
  `(max-width: 768px)` — detecting unsupported queries via
  `mql.media === 'not all'` (→ `undefined`) and delegates to the pure helper.

- **`public/js/player.js` — pure control-surface helper (new, exported).**
  `resolveControlsMode(mediaType, isMobile)` → `'native'` iff
  `mediaType === 'video' && isMobile`, else `'custom'`. State-independent (the
  DOCKED-suppression is applied in the wrapper, below) so it stays a clean
  two-argument lookup for AC10.

- **`public/js/player.js` — `applyControlsMode()` (new, impure applier).**
  Reads `isMobileFormFactor()` and `currentData.type`, then:
  `host.classList.toggle('ff-mobile', mobile)`; and sets the `controls`
  attribute iff `resolveControlsMode(...) === 'native'` **and**
  `state === STATE_FULL` (the `&& FULL` clause is the AC9 dock-suppression —
  see below). Called from **`mountInSlot()`** and **`dock()`** — the two
  synchronous reparent/state-transition points every visible-player path
  already flows through: a genuine load (`load()` → `mountInSlot`), an adopt
  expand (`load()`/`expand()` → `mountInSlot`), and docking (`dock()`). This
  re-derives the mode at every FULL/DOCKED transition without touching
  `wireHostListeners()` and without any src/currentTime change (no restart).
  `teardownMediaState()` additionally calls
  `mediaPlayer.removeAttribute('controls')` (belt-and-suspenders: no native
  strip can flash on a mobile-video → mobile-audio load before `mountInSlot`
  re-runs microseconds later).

- **`public/js/player.js` — module exports.** Add `resolveMobileFormFactor`
  and `resolveControlsMode` to the existing `module.exports` block.

- **`public/css/style.css` — new `/* === v1.22 FR-1 === */` section.** (a) the
  gesture-surface `user-select`/`-webkit-touch-callout` guards; (b) hide the
  custom bar on mobile video; (c) drop the volume controls on mobile audio.
  Display/`user-select` only — no color tokens touched (AC12). The v1.21 FR-2
  section is **not modified**.

- **`wireSkipHoldGestures(mediaPlayer)` — NO change (confirmed).** It is
  already wired unconditionally to `#media-player` with no `onSingleTap`, and
  already coexisted with native controls pre-v1.21. Restoring `controls`
  requires no change: native controls own the bottom strip and single-tap
  play/pause; our layer adds press-hold-2x, double-tap ±15s skip, and the skip
  buttons on the surface ABOVE the strip, exactly as pre-v1.21. Deliberately
  **no** `onSingleTap` play/pause is added to the video surface — that would
  double-toggle against the native single-tap. (AC5's "single tap →
  play/pause" is delivered by the native control layer; our layer adds
  hold-2x + double-tap-skip.)

- **`isMobileViewport()` — left untouched.** Its two existing callers are the
  desktop live-transcode gate (`setupForMedia`, line ~1636) and the
  desktop-only initial-autoplay gate (`handleResumePlayback`, line ~873) —
  viewport heuristics unrelated to control-surface choice. FR-1 introduces the
  new `isMobileFormFactor()` for the control decision (and FR-8(a) will reuse
  it); conflating the two would change the desktop-AVI and mobile-no-autoplay
  behaviours, which is out of FR-1 scope. The two helpers answer different
  questions and are documented as such. AC78's "same helper" constraint binds
  FR-1 and FR-8 (both use `isMobileFormFactor`), which this satisfies.

### The `.ff-mobile` class is the single source of truth

CSS cannot set the `controls` attribute, so JS must drive the native-vs-custom
attribute regardless. To avoid a second, textually-duplicated "is this mobile"
condition in CSS (a divergence risk), CSS reacts to the `.ff-mobile` class that
`applyControlsMode()` sets from the SAME `isMobileFormFactor()` call that
toggles `controls`. One JS signal, reflected as one class plus one attribute;
CSS is purely reactive. New CSS (schematic):

```css
/* === v1.22 FR-1: responsive controls === */

/* AC6 — press-hold-2x must not select text / raise the iOS callout. Applied
   to the gesture surfaces wireSkipHoldGestures() binds to, NOT the native
   control strip (user-select/touch-callout never affect tap handling). */
#media-player,
#audio-bg-art,
.skip-controls,
.speed-badge {
  user-select: none;
  -webkit-user-select: none;
  -webkit-touch-callout: none;
}

/* AC4 — mobile VIDEO: native controls own the strip; hide the custom bar.
   :not(.audio-mode) keeps the bar for mobile AUDIO. */
#player-wrapper.ff-mobile:not(.audio-mode) .player-controls {
  display: none;
}

/* AC3 — mobile AUDIO: keep the custom bar, drop volume (hardware buttons;
   mirrors initVolume()'s existing iOS volumeIsSettable degrade, now
   deliberate for all mobile audio). */
#player-wrapper.ff-mobile.audio-mode #vol-bar,
#player-wrapper.ff-mobile.audio-mode #mute-btn {
  display: none;
}
```

The existing `@media (max-width: 768px)` block (style.css:3074, `.player-controls`
44px height / `.pc-btn` 36px) is left as-is: with the mobile-video bar now
hidden it governs the mobile-audio bar (and a narrowed desktop window's bar,
harmlessly), giving the mobile-audio bar comfortable touch targets.

### No HTML shell changes

The `controls` attribute is toggled entirely from JS; the
`#player-host-template` `<video>` markup stays exactly as today (no `controls`
in markup). Therefore FR-1 touches **only** `player.js` + `style.css` and
requires **zero** edits to the four shells
(`index.html`/`setup.html`/`watch.html`/`subscriptions.html`) — eliminating the

### Response to device change (orientation / resize) — AC7

The control mode is decided **once per media-load and per FULL/DOCKED
transition**, not via a live media-query listener. This is robust because the
primary signal — `(pointer: coarse)` / `(hover: none)` — reflects device
CAPABILITY, which does **not** change on orientation or resize (unlike width).
Rotating/resizing therefore never re-runs the decision, never reloads the
`<video>`, and never restarts playback. In the theoretical edge where a
capability genuinely changes mid-session (e.g. plugging a mouse into a tablet),
the `.ff-mobile`-driven CSS re-evaluates the bar visibility for free on the
next style pass, and the (possibly briefly stale) `controls` attribute is
harmless (worst case a hidden custom bar plus native controls, both
functional) until the next load/transition re-derives it.

### Native controls + gesture-layer coexistence — AC5

Native controls render as a shadow-DOM strip at the bottom of the video; taps
on those widgets are handled by the browser and do not reach our surface
gestures — proven by the pre-v1.21 layering (the custom skip buttons were
overlaid and native controls still worked). Our gesture listeners
(`touchstart`/`move`/`end`, `dblclick`) fire on taps on the video body above
the strip. `engageHold`/`revealSkipButtons` are already `state === STATE_FULL`-
guarded, so a DOCKED mini-player never engages 2x or reveals skip buttons. The
critical invariant: the FR-2-era `.audio-mode #media-player { pointer-events:
none }` rule stays **audio-only scoped** — mobile VIDEO keeps normal
`pointer-events` on `#media-player`, which both native controls and our
gestures require. No change to that rule. Dean's iOS on-device pass is the
arbiter for the feel (AC5/AC6/AC13).

### DOCKED mobile video — AC9

`applyControlsMode()`'s `&& state === STATE_FULL` clause means native
`controls` are present only while FULL; on `dock()` the attribute is removed,
so the mini-player shows a clean video frame with no overflowing native strip
and the dock's existing tap-to-expand handler works unobstructed. This mirrors
the existing `#player-dock .skip-controls, .speed-badge { display: none }`
"FULL-only affordance" precedent. Expanding back to FULL (`mountInSlot` via the
adopt path) re-adds `controls` with no src/currentTime touch (no restart).

### Proof that dock / prev-next / autoplay-next / overlays / Media Session / progress are unaffected

The toggle mutates exactly three things: the `controls` attribute on
`#media-player`, the `.ff-mobile` class on `#player-wrapper`, and (via CSS) the
`display` of `#player-controls`/`#vol-bar`/`#mute-btn`. **No** other player
system reads any of these:

- `dock()`/`mountInSlot()`/`close()`, prev/next (`deriveOrderedIds`/
  `computeNeighbors`), `handleAutoplayNext`, the transcode + resume overlays,
  `setupMediaSession`/`updatePositionState`, and every `saveProgressToServer`
  path branch on `mediaPlayer`/`host`/`currentData`/`state` — never on
  `controls`, `.ff-mobile`, or bar visibility.
- `wireHostListeners()` is unchanged; every listener still wires once and
  travels with the host.

This is grep-verifiable (`controls` appears in `player.js` only where
`applyControlsMode`/`teardownMediaState` set/remove it) and is a stated
reviewer check for the two-reviewer gate. A regression unit test can also lock
`resolveControlsMode`'s four-way table so a future edit can't silently flip a
desktop/mobile-audio case to native.

### Alternatives considered

- **Pure-CSS media query (no `.ff-mobile` class), hiding the bar via
  `@media (pointer: coarse) and (hover: none)`.** Rejected: the `controls`
  attribute cannot be set from CSS, so JS is required anyway; driving bar
  visibility from a separate CSS condition duplicates the "is mobile" logic and
  risks textual drift from the JS signal (and a width-fallback divergence). The
  `.ff-mobile` class keeps one source of truth.
- **Keep the custom bar on mobile video and re-implement fullscreen / speed /
  AirPlay / PiP / download in it.** Rejected: large, fragile, and precisely the
  native-for-free behaviour Dean wants back.
- **Re-derive the mode live via a `matchMedia` change-listener.** Rejected as
  unnecessary — pointer/hover capability is stable across orientation/resize;
  once-per-load/transition is simpler and robust (AC7).
- **Replace `isMobileViewport()` everywhere with the new helper.** Rejected for
  FR-1 scope — its two callers gate desktop live-transcode and desktop autoplay
  (viewport heuristics), and changing their definition of "mobile" risks the
  desktop-AVI and mobile-no-autoplay behaviours. Kept separate; FR-8 reuses the
  new helper.

### Risks and mitigations

- **Risk**: iOS pauses/resets on reparent when the `controls` attribute is
  toggled during a dock/expand. → **Mitigation**: `applyControlsMode()` runs
  synchronously inside the existing `mountInSlot`/`dock` reparent (no awaited
  work), and the attribute travels with the element; the existing
  `wasPlaying && paused → play()` re-assertion is unchanged. Dean's iOS
  on-device pass is the arbiter (AC13).
- **Risk**: native control taps conflict with the gesture layer. →
  **Mitigation**: proven pre-v1.21 coexistence; `.audio-mode` pointer-events

- **Risk**: `.ff-mobile` class / `controls` attribute drift. → **Mitigation**:
  both set from one `isMobileFormFactor()` call in `applyControlsMode()`;

- **Risk**: a future edit silently flips a byte-identical case to native. →
  **Mitigation**: unit test locks `resolveControlsMode`'s four-way table

## Design (FR-2..FR-10)

### Approach

FR-2..FR-10 are nine mostly-independent surfaces layered onto an unchanged
core, sequenced AFTER FR-1's fast-track lands (FR-7/FR-8 rebase on FR-1's
`isMobileFormFactor()` helper and its `#player-controls` model). The one
server-heavy item (FR-2, retroactive channel backfill) is a scan-time,
filesystem-path-only inference reusing the existing `resolveChannelDir`
confinement and the unmodified `validateChannelUrl` guard; the one args/spawn
item (FR-6, max-duration gate) mirrors the `maxVideos` plumbing end-to-end.
Every other FR is client/CSS. No new runtime dependency, no new validator, no
change to `db.folders`/`folderSettings`, and the disabled-`FILETUBE_YTDLP_ENABLED`
byte-identical contract holds for FR-2/FR-3/FR-5/FR-6 (no subscriptions →
nothing to match/pin/gate; `/subscriptions*` still 404s).

Two server files carry FR-2 and FR-6 respectively and are fully disjoint from
each other and from the client FRs, so they form a parallel server track. The
client FRs cluster on four hot shared files (`player.js`, `style.css`,
`watch.js`, `main.js`) and must serialize around FR-1 — see the task-breakdown
and file-overlap map at the end.

### FR-2 — retroactive Subscribe-button backfill / creator re-association

**Root cause (confirmed by reading `server.js` ~1356-1620 in full).** The
channel-identity bridge (server.js:1607) is gated on
`freshlyScannedIds.has(item.id)` — only a file's *very first* scan consumes
`db.ytdlp.downloadMeta[videoId]`. Every subsequent scan takes the `reusable`/
`legacyVideoCodecBackfillOnly` fast paths (server.js:1393/1408) and "keeps
whatever identity it already has." So a pre-v1.20 download (indexed before
capture existed) is never revisited, and — confirmed by reading
`runSubscriptionCycle` (index.js:530) + `processSubscription`'s
fire-and-forget `scanDirectories()` trigger (index.js:595) vs. the independent
periodic `armScanTimer` poll (index.js:1371) — a periodic auto-scan that
indexes a freshly-downloaded file *before* `persistCapturedChannelMeta` writes
its `downloadMeta` entry marks it `freshlyScannedIds` with nothing to consume;
because bridging only ever runs on first scan, that identity is then lost
forever. This is a real race, not assumed (AC20).

**Fix — a second, folder-based backfill pass in the SAME Phase-2 mutator, NOT
scoped to `freshlyScannedIds`.** yt-dlp downloads land *flat* directly in
`resolveChannelDir(config, sub) = downloadDir/sanitizeChannelName(sub.name ||
sub.channelUrl)` (args.js:170; `OUTPUT_TEMPLATE` has no path separator —
args.js:270), so an item's channel dir is exactly `path.dirname(item.filePath)`.
The new block runs for **every** item in `newMetadata` (reuse-fast-path items
included), immediately after the existing `freshlyScannedIds`-scoped bridge, in
the existing `updateDatabase(fresh => …)` mutator:

- **Guard (never overwrite):** skip any item that already has a truthy
  `item.channelUrl` (a genuine gap only — AC17). Skip any item not under
  `ytdlpDownloadRoots` (reuse `matchRootFolder`, exactly as the existing
  bridge does — a non-yt-dlp file is never fed the matcher, AC16-analogue).
- **Matcher (pure, exported, `node:test`-covered — AC15):** a new pure helper
  `matchChannelDirToSubscription(filePath, resolvedSubs)` in `lib/ytdlp/index.js`
  (or a small sibling module) that compares `path.dirname(path.resolve(filePath))`
  against each entry's already-resolved `channelDir` by **exact directory
  equality** (not a `startsWith` prefix — a file merely sharing a path prefix
  that is not its actual parent channel dir must NOT match, AC15), returning
  the matched subscription's `{ channelUrl, channelName, channelId }` or `null`.
  `resolvedSubs` is computed **once per scan** (memoized before the item loop)
  from `fresh.ytdlp.subscriptions` via the existing
  `enrichSubscriptionWithChannelDir(config, sub)` (index.js:1002) — each
  entry's `channelDir` wrapped in the same try/catch confinement it already
  uses, so a sub that fails confinement simply omits from the candidate set.
- **Security re-validation (AC18):** the matched `channelUrl` — although it is
  a subscription's own value that already passed `validateChannelUrl` at
  subscribe-time — is re-run through the **unmodified** `url.validateChannelUrl`
  before assignment (defense-in-depth, matching args.js's `requireValidUrl`
  "re-validate downstream" posture). Only a value that passes is written; a
  corrupted/hand-edited subscription record that fails is dropped (item stays
  identity-less → button hidden — fine).
- **Module wrapper (layering):** all `db.ytdlp` structural reads stay inside
  the module. Add `ytdlp.backfillChannelIdentityFromFolder(fresh, item, config)`
  (mirroring the existing `ytdlp.consumeDownloadChannelMeta` export,
  index.js:1634) that owns the `resolvedSubs` memoization + matcher + re-validate
  and returns the identity fields (or `null`); `server.js` assigns them and
  sets `dbChanged = true`, exactly like the existing bridge block.

**Race fix (AC20).** No re-sequencing of the download/scan triggers is needed:
the folder backfill *inherently heals* the race because it runs on **every**
scan for **every** identity-less yt-dlp-rooted item — so a file the periodic
scan indexed before its `downloadMeta` was written picks up its identity from
its own folder on the very next scan. Document this in the Decision log as the
confirmed-and-healed outcome (AC20 explicitly permits "as simple as this same
folder-based backfill catching it retroactively on the NEXT scan").

**Display re-association (Dean's clarification, folded into FR-2).**

1. **Real creator name shown.** `resolveChannelName` (common.js:256) *already*
   ranks a captured `item.channelName` FIRST (above the mapped folder
   friendly-name / `artist` / `folderName` / `'Library'` chain). The watch-page
   uploader area (`watch.js:290`, `uploaderChannelName.textContent =
   channelName`) and the home cards (`main.js:425`) both render that resolved
   value. Therefore, the *only* thing FR-2 must guarantee for the creator name
   to appear is that the backfill writes **`item.channelName`** (from the
   matched `sub.name`) — not just `channelUrl` — onto `db.metadata`. Once that
   lands, the generic "Downloads"/folder label is replaced by the real creator
   on rescan with **no client change** (the precedence already exists). This is
   a regression-lock + integration assertion, not new display code (new AC80).
2. **Save-to-device button confirmed visible.** The v1.19 save-the-original
   button is wired **unconditionally** at `watch.js:309` (`if (downloadBtn)
   { downloadBtn.href = /video/<id>?download=1; … }`) — there is no yt-dlp/
   re-associated gating on it anywhere. It renders on every watch page,
   including these re-associated items. FR-2 adds a regression-lock asserting
   it is not gated/hidden for yt-dlp/backfilled items (new AC81); no code
   change is expected (this is the "save the original file to device" button,
   matching Dean's interpretation).

**Tests.** Unit: the pure `matchChannelDirToSubscription` (exact-parent match;
rejects a non-parent prefix; multiple-subs disambiguation; `null` on no match)
plus the never-overwrite guard. Integration: an identity-less yt-dlp item under a
sub's `channelDir` gets `channelUrl`+`channelName` backfilled on scan (no
yt-dlp spawn) and shows the resolved creator; a non-yt-dlp item under an
ordinary library folder does not; an item that already has `channelUrl` is
untouched. Two-reviewer gate. Files: `server.js`, `lib/ytdlp/index.js` (+ tests).

### Definitive collision-free wave plan

No two CONCURRENT tasks may edit the SAME file (shared working tree). The SERVER
track (TA on `server.js`/`lib/ytdlp/index.js` ∥ TB on the rest of `lib/ytdlp/*`)
is fully disjoint from each other and from the CLIENT track, so both run in
parallel in Wave 1 alongside FR-1's two disjoint-file tasks.

- **Wave 1:** T1 (player.js) ∥ T2 (style.css) ∥ TA (server.js/index.js) ∥ TB (lib/ytdlp/*) — four disjoint file sets.
- **Wave 2:** TC (common.js/main.js/watch.js/setup.js/style.css) — plus TI (process, no files).
- **Wave 3:** TE (style.css) ∥ TD (watch.html/watch.js) — disjoint.
- **Wave 4:** TF (player.js/watch.html/watch.js/style.css).
- **Wave 5:** TG (player.js/four shells/style.css).
- **Wave 6:** TH (main.js/style.css).
