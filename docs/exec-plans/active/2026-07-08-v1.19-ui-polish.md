# v1.19.0 "UI polish"

## Goal

Polish four rough edges Dean has actually hit using FileTube on his iPhone and
desktop — an oversized/"brutal" one-off download modal, a cramped and (per the
roadmap) janky `/subscriptions` page, no way to pull an already-hosted file
down to a device, and a synthetic library folder that looks second-class in
Setup — closing out the "UI polish" section of `ROADMAP.md` with small,
low-risk, mostly-additive fixes (no shell/router/player rearchitecture).

## Scope

Four functional requirements, one branch (`feature/v1.19-ui-polish`, off
`main` at `v1.18.0` — confirmed via `package.json`'s `"version": "1.18.0"`).

### FR-1 — One-off download modal polish (mobile "brutal"/oversized selects)

**Root cause, confirmed in code.** `public/css/style.css`'s base rule for the
modal's three `<select>`s:

```css
.oneoff-modal-row select {
  flex: 1 1 150px;
  min-width: 150px;
  ...
}
```

(style.css ~1910-1920) sets `flex-basis: 150px` — a sensible *width* hint when
`.oneoff-modal-row` is `flex-direction: row` (the desktop/default layout,
`flex-wrap: wrap`). The `max-width: 768px` mobile block then does:

```css
.oneoff-modal-row { flex-direction: column; }
.oneoff-modal-row select { width: 100%; }
```

(style.css ~2429-2436) — switching the row to a **column** flex direction so
the three selects stack, but it never resets the inherited `flex: 1 1 150px`
shorthand. Under `flex-direction: column`, the main axis is now **vertical**,
so that same `flex-basis: 150px` is reinterpreted as a **minimum height**, and
the still-active `flex-grow: 1` lets each select **stretch to fill whatever
vertical space `.oneoff-modal-row`'s box has to give** — three flex items each
independently growing tall. On a ~650-700px-tall phone viewport this produces
exactly Dean's screenshot: format/quality/filetype rendering as ~150px+ tall
boxes, each roughly a quarter of the screen, instead of a normal single-line
native control. This is a **width-intent-reinterpreted-as-height-intent** bug
under the direction flip, not a missing style altogether — `buildOneOffModal`
(`public/js/common.js` ~933-1022) itself is unaffected; this is CSS-only.

**Resolved fork — target select height (no open question).** The Setup page's
own `<select>` styling is the right visual bar:

```css
.setup-select {
  padding: 10px 12px;
  border: 1px solid var(--border-dark);
  border-radius: var(--radius);
  font-size: 13px;
  ...
}
```

(style.css ~1528-1536), which renders as a normal, single-line, ~36-40px-tall
native control. **Default: the one-off modal's three selects must render at
that same normal, content-driven, single-line height on mobile — not stretched
via `flex-grow` to fill the modal's vertical space** — while keeping the
existing `.oneoff-modal-row select`'s 44px comfortable-tap-target minimum
(style.css ~2437-2441, part of the same mobile block) as the tap-target floor,
not a growth target. The exact CSS mechanism (resetting `flex`/`flex-basis`,
or a more targeted mobile override) is Design's call — the requirement is the
visible outcome, not the rule.

Everything else about the modal must be preserved: the URL input, the three
selects' existing options/behavior (including `repopulateOneOffFiletypeSelect`
on format change), the Download button, the v1.15.1 max-height/scroll modal
container, and the v1.17.0 FR-6 backdrop-dismiss teardown fix (full DOM
removal on every dismiss path). This is a CSS-scoped fix.

### FR-2 — Subscriptions page UI (small box + janky layout)

**FR-2a — the subscriptions list box is small/cramped (confirmed, genuine
gap).** `lib/ytdlp/views/subscriptions.html`'s "Your subscriptions" section
uses `<div id="sub-list-container" class="folder-list-builder">` (line 217).
`.folder-list-builder` (style.css ~1545-1553) is:

```css
.folder-list-builder {
  ...
  padding: 12px;
  max-height: 240px;
  overflow-y: auto;
}
```

— a fixed 240px scroll box with modest padding. This exact class is **shared
by three different lists**: the Setup page's folder-management builder
(`#folders-builder-list`), the subscriptions page's one-off job list
(`#oneshot-list-container`), and the subscriptions list itself
(`#sub-list-container`). Dean's ask ("the subscriptions box too small,"
`ROADMAP.md`) is specific to the subscriptions list. **Resolved default:**
give the subscriptions list more room (a taller box and/or more generous
padding) via a **scoped rule** (e.g. an `#sub-list-container`-targeted
override, or a dedicated modifier class), not by changing the shared
`.folder-list-builder` rule wholesale — so the Setup folder builder and the
one-shot job list are unaffected unless Design finds a shared bump is
actually desirable for all three (non-blocking, Design's call, but the
default is scoped-not-global).

**FR-2b — "janky /subscriptions UI... vertical text": ALREADY FIXED, confirmed
by code AND test inspection — this is a stale roadmap carry-over, not a live
bug.** `ROADMAP.md`'s "UI polish" section still lists "the channel name/URL
renders vertically (one character per line)... a flex width-collapse +
per-character `word-break`" as an open item, attributing it to v1.12.0. But:

  `.sub-row-name` / `.sub-row-url` / `.sub-row-actions` — a **dedicated** row
  layout (style.css ~1639-1703), explicitly NOT the generic

**Conclusion for this round:** FR-2b is a **regression-lock**, not a

### FR-3 — Download-to-device button (new)

A "Download" affordance on the watch page that triggers a real browser
download of the current item's **original, already-served file** — reusing
the existing id-based `GET /video/:id` route (server.js ~2286-2361), never a
new filesystem-path-taking endpoint.

**Confirmed current behavior, grounded in code:**
- `GET /video/:id` looks up `db.metadata[req.params.id]` and serves
  `item.filePath` directly UNLESS `item.needsTranscode` is true, in which case
  it serves the **transcoded MP4** (or a live-transcode stream via `?live=1`)
  instead of the original — so a naive reuse of this route for "download the
  file" would sometimes hand back a transcode, not the source Dean explicitly
  wants ("download the ORIGINAL source file, not the transcode").
- No route in `server.js` sets `Content-Disposition` today — this is
  genuinely new server-side work, not a one-line addition.
- `GET /api/videos/:id` (server.js ~2037) already spreads the full metadata
  item (including `title`/`ext`/`filePath`) to the client, and
  `public/js/watch.js` (~233) already renders `mediaData.filePath` (the raw
  absolute path) directly into the existing watch page (file-type text +
  delete-confirmation modal). **The raw path is therefore already visible in
  the existing UI** — FR-3 introduces no new path-disclosure surface; the
  requirement is that the actual *file bytes* download stays keyed by `id`,
  never a client-supplied path, exactly like playback already is.

**Resolved fork — mechanism (non-blocking, PE confirms exact shape).**
Default: extend the **existing** `GET /video/:id` route with an explicit
download intent (e.g. a `?download=1` query param or equivalent), which (a)
**always serves `item.filePath`** (the original), bypassing the
`needsTranscode` branch entirely, and (b) sets `Content-Disposition:
attachment; filename="<clean title>.<ext>"` (RFC 5987 / `filename*=UTF-8''`
encoding for non-ASCII titles), so the browser saves rather than plays. Same
id→metadata lookup as playback; no new confinement logic needed since
`item.filePath` is the same server-resolved path playback already streams
from. A sibling, equally-acceptable shape (a distinct
`GET /video/:id/download`-style route sharing the same lookup) is Design's
call — the constraint is "same id-based lookup, one shared code path with
playback's confinement, never a second path-resolution mechanism."

**Resolved fork — scope (non-blocking).** Default: **watch-page only** for
this round (a clear "Download" button near the existing Delete/star-rating
row in `.watch-actions`, `public/watch.html` ~148-163, reusing the
already-shipped `.icon-download` glyph from v1.17.0 FR-7). A card-level
download affordance (`public/js/main.js`) is deferred as a fast-follow, not
in scope here, per the brief's own stated default.

**Filename + audio/video/transcoded distinction:**
- Filename: the item's display `title` (already yt-dlp-cleaned via
  `cleanDisplayTitle` where applicable) + its **original** `ext` — never the
  raw on-disk `name` (which may carry the yt-dlp `--restrict-filenames`/`[id]`
  suffix or ffmpeg-transcode naming).
- Must download the **original** file (`item.filePath`) even when
  `item.needsTranscode` is true and a cached transcode exists at
  `transcodedPath(item.id)` — the transcode is a browser-playability sidecar,
  not the canonical file.
- Works for both audio and video items (the same route, same `item.filePath`,
  regardless of `item.type`).
- iOS Safari note: `Content-Disposition: attachment` plus the anchor
  `download` attribute is the standard same-origin download path on modern
  iOS Safari (13+); Dean's on-device iPhone pass is the arbiter for the exact
  save/share-sheet UX, not fully proxyable by `node:test`.

**Security-sensitive touchpoint — flagged for the two-reviewer gate.** This
touches the file-serving route. Non-negotiable: the download path must stay
strictly `id → db.metadata[id].filePath` (never accept or interpolate a
client-supplied path/filename into the filesystem lookup), must not weaken
any existing `/video/:id` behavior (Range support, 404s, the transcode
branch's existing behavior for ordinary playback), and the new
`Content-Disposition` filename must be safely encoded (no header injection
from a title containing CR/LF or quote characters).

### FR-4 — Synthetic "Downloads" folder: renamable + reorderable in Setup

**Grounded finding: this is ALREADY FULLY IMPLEMENTED AND TESTED — confirmed
by reading both the server and client code, not assumed from the roadmap
description.** `ROADMAP.md`'s "UI polish" section still lists this as an open
item ("can't be renamed or reordered... like real folders"), but:

- `GET /api/config` (server.js ~1596-1625, "FR-G part 2" / "Item 3 (v1.13.0,
  order persistence)") already merges the yt-dlp module's download directory
  into its **response** as a synthetic folder entry — spliced in at a
  **persisted display index** (`folderSettings[root].order`, defaulting to
  append-last) — and defaults its name to `'Downloads'` when no rename has
  ever been persisted.
- `POST /api/config` (server.js ~1628-1788, `FIX-2`) already accepts a rename
  (`folderSettings[resolvedRoot].name`) and a reorder (derived `order` from
  the synthetic root's position in the submitted `folders` array) for the
  synthetic root, persisting **both into `folderSettings` only** — the
  synthetic root is explicitly, unconditionally excluded from ever entering
  `db.folders`, even when a normal settings-page save round-trips `GET`'s
  synthetic-merged response straight back into `POST` (the exact "client
  doesn't know it's special" scenario this feature needs to survive).
- `public/js/setup.js`'s `renderFolders()`/`loadConfig()` (lines 38-96) treat
  `configuredFolders` (== `data.folders`, which already includes the spliced
  synthetic root) generically — **every** row, synthetic or real, gets the
  same rename `<input>`, "Hide from home"/"Hide from sidebar" checkboxes,
  up/down reorder buttons, and native drag-and-drop
  (`common.js`'s shared `moveArrayItem`/`computeDropIndex` helpers) — "zero
  client change" was the explicit v1.13.0 design note, and reading the
  current code confirms it is still true.
- `test/integration/ytdlp-synthetic-folder.test.js` already exercises exactly
  this contract end-to-end: rename-only persistence (AC44), reorder
  persistence across a fresh GET (AC8), rename+reorder together (AC9),
  never-in-`db.folders` after either (AC10), an out-of-range stored order
  clamping safely (AC10b), the disabled-module no-op (AC11/AC46), and the
  scan/mount-loss guard's independence from any of this (AC12/AC47).

**Conclusion for this round:** FR-4's core ask — a persisted friendly name and
a persisted position, surfaced through Setup's normal folder-management UI,
without ever entering `db.folders` — is **already shipped and tested**. This
plan does not need to (and must not) rebuild it. `ROADMAP.md` should be
corrected to move this line to "Shipped" as a housekeeping follow-up (flagged
for the EM/Dean, same as FR-2b).

**One genuine rough edge found, worth closing this round (resolved
default).** The synthetic row's "×" remove button
(`public/js/setup.js` ~93/183-192) behaves identically to a real folder's:
clicking it removes the row from the client's in-memory `configuredFolders`
and deletes its `folderSettings` entry, then a Save would submit a `folders`
array without it. But because the synthetic root is **never** actually
persisted into `db.folders` (by design — the CRITICAL invariant this feature
must never violate), removing it accomplishes **nothing durable**: the very
next `GET /api/config` re-derives and re-splices it right back in (self-heal,
AC45). Today the UI gives no signal that this button is a no-op for that one
row, which could read as "I removed the download folder from tracking" when
nothing changed. **Resolved default:** disable (not hide — the row itself
must still display) the remove button specifically on the synthetic row, with
a tooltip/label explaining it always reflects the module's live download
directory and cannot be removed here (only by disabling the yt-dlp module
itself). This is a small, additive client-only affordance change — it must
not touch the `db.folders`-exclusion invariant in any way, which stays
entirely server-side and untouched.

**Resolved fork — default friendly-name.** Already resolved by the shipped
code: `'Downloads'` (server.js ~1621) — no change needed, restated here for
completeness per the assignment's explicit ask to resolve this fork.

**Security/data-safety-sensitive touchpoint — flagged for the two-reviewer
gate.** Even though this FR's core behavior is a confirm-not-build, any touch
to `server.js`'s folder/config handling or `public/js/setup.js`'s folder
rendering carries the CRITICAL invariant from the v1.12.0 data-loss class of
bug: **the synthetic download folder must never be written into
`db.folders`** — only into `folderSettings[dir]`. The two-reviewer gate is
warranted specifically to re-prove this invariant holds after the remove-
button change, not because new risk is being introduced.

## Constraints

- **No regressions.** FR-1/FR-2's CSS changes must not alter the modal's
  function (URL input, three selects and their change-driven filetype
  repopulation, Download button, the v1.17.0 FR-6 backdrop-dismiss full
  teardown) or the subscriptions page's already-fixed row layout (FR-2b);
  FR-3 must not change `/video/:id`'s existing playback behavior (Range
  support, the `needsTranscode`/`?live=1`/lazy-transcode branches, 404s) for
  any request that isn't the new download intent; FR-4 must not touch the
  server-side `db.folders`-exclusion invariant at all, and must not change
  real (non-synthetic) folders' rename/reorder/hide behavior.
- **Security-sensitive touchpoints — flagged for the two-reviewer gate, not a
  green light to loosen anything:**
  - **FR-3 — the file-serving route (`GET /video/:id`).** The download intent
    must stay strictly `id → db.metadata[id].filePath`; never accept or
    interpolate a client-supplied path or filename into a filesystem
    operation; the `Content-Disposition` filename must be safely encoded
    (no header/CRLF injection from a title string).
  - **FR-4 — the folders/config model (`server.js`'s `GET`/`POST
    /api/config`, `public/js/setup.js`'s folder rendering).** The synthetic
    download folder must NEVER be written into `db.folders` — this is a
    repeat of the exact invariant a prior data-loss-class bug required
    fixing; any touch here (even the small remove-button change) re-proves
    the invariant via the existing `test/integration/ytdlp-synthetic-
    folder.test.js` suite passing unmodified, plus new coverage for the
    remove-button change itself.
  - FR-1/FR-2 are CSS/layout-scoped with no security-guard surface —
    **lighter single-QA no-regression** review is appropriate for those two.
- **`docs/CONTRIBUTING.md`:** vanilla DOM (no framework/bundler), CommonJS
  server, 2-space/semicolons/single-quotes, `textContent` over `innerHTML`
  for any new dynamic strings, `node:test` for new/regression tests, lint 0
  warnings, no new runtime dependencies.
- **Era-theme system preserved.** Any new/changed CSS (FR-1's select-height
  fix, FR-2a's list-box sizing, FR-3's Download button, FR-4's disabled-
  remove-button treatment) uses existing theme tokens (CSS custom
  properties) — no hardcoded colors breaking the 2005/2009/2014/2021 eras or
  light/dark modes.
- **Node 22 LTS** toolchain (`engines` ≥20); CI/build-verify runs tests on
  Node 22.
- **No headless-browser/E2E test infrastructure exists** (per
  `docs/RELIABILITY.md`) — this bounds what FR-1 (mobile select rendering)
  and FR-2 (row layout "feels clean"/list-box proportions) can claim as
  `[UNIT]`/`[INTEGRATION]`; Dean's on-device pass (mobile Safari + desktop)
  is the documented arbiter for both, not a rigor gap. FR-3 and FR-4 are
  substantially more code/test-verifiable (an HTTP route's response headers/
  body; a JSON config round-trip), with a lighter on-device/visual check
  layered on top.
