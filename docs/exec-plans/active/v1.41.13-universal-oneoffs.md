## Goal

Paste ANY URL that one of yt-dlp's ~1,800 **named** extractors supports into
the one-off download box and get a fully first-class library item: it
downloads into a sensible folder, indexes with correct metadata, plays,
deletes cleanly (tombstone/resurrect machinery intact), and never re-downloads
after deletion. YouTube behavior is byte-for-byte unchanged.

## Non-goals (explicitly out of this branch)

- **Non-YouTube subscriptions/channel polling.** The poller, backoff,
  breaker, cutoff-date model, and UU-feed logic stay YouTube-only. Follow-up
  branch.
- **The generic extractor.** Refused (Dean's call): yt-dlp's scrape-any-page
  fallback would let the UI make the server fetch arbitrary LAN/internal
  URLs (SSRF-shaped in a LAN-trusted app). Possible env opt-in later.
- **Playlist expansion.** Single-item semantics. **[REVIEW AMENDMENT — W1]**
  `--no-playlist` alone does NOT deliver this: yt-dlp's `_yes_playlist`
  (extractor/common.py:4036) still expands a pure-playlist URL (no companion
  video id) in full — `--no-playlist` only disambiguates video+playlist
  URLs. The one-off argv therefore carries BOTH `--no-playlist` (the
  ambiguous case) and `--playlist-items 1` (the hard single-item bound for
  pure playlists), and the job row notes when a playlist was reduced.
- Reheat/hydration for non-YouTube items beyond what falls out naturally
  (channel_url hydration remains YouTube/MeTube-scoped as shipped in v1.41.5).

## The design crux: the id model

Everything hard in this branch radiates from one fact: **FileTube's delete /
tombstone / archive / resurrect machinery is keyed to YouTube's 11-char
`[A-Za-z0-9_-]{11}` id shape**, via the filename bracket ` [<id>].ext`.
Non-YouTube ids are arbitrary: variable length, sometimes numeric, sometimes
containing `.`/`%`/unicode, and **only unique per-extractor** (Vimeo id
`76979871` and a Dailymotion id can collide as bare strings).

### Decision D1 — on-disk name keeps a bracket, extended shape

Output template for non-YouTube one-offs becomes:

    %(title)s [%(extractor_key)s=%(id)s].%(ext)s      (e.g. "Talk [Vimeo=76979871].mp4")

- The `extractor_key=` prefix makes the bracket **globally unique** and
  visually self-documenting, and — critically — keeps the EXISTING 11-char
  YouTube matcher's population disjoint: a YouTube file keeps its exact
  current name shape (` [dQw4w9WgXcQ]`), so **every shipped YouTube
  code path continues to match only YouTube files with zero behavior change**.
- New pure helper `extractMediaRef(baseName)` returns
  `{ source: 'youtube', id }` for the legacy 11-char shape OR
  `{ source: <extractorKey>, id }` for the new `key=id` shape, else null.
  Bracket grammar for the new shape: `` [<Key>=<id>]`` where `<Key>` is
  `[A-Za-z0-9_]{1,64}` (yt-dlp extractor_key charset) and `<id>` is
  `[^\[\]/]{1,128}` — no slashes (impossible in a filename component anyway),
  no nested brackets, length-capped. The ` [`/`_[` separator rule carries
  over unchanged.
- **Filename-hostile ids**: `%(id)s` can contain characters the filesystem or
  our parser can't round-trip. Resolution: the bracket is rendered as
  `[%(extractor_key)s=%(id)s]` and yt-dlp's own per-OS filename sanitization
  (already active via our `--windows-filenames`) is ACCEPTED — the bracket
  is **never required to round-trip**. The authoritative `{source, id}` pair
  lives in db metadata (D2) and the archive line (D3); the bracket is a
  same-folder recovery hint exactly as today for YouTube, and the v1.41.9
  SEAM-1/SEAM-2 confinement (same folder + same ext) carries over verbatim.
  The full raw-vs-sanitized contract — which consumer uses which string, and
  the forbidden comparisons — is Decision D5 below [REVIEW AMENDMENT — W4].
  Plugin extractor keys carry a `+plugin` suffix and are treated as
  metadata-only (no bracket key) [REVIEW S1].

### Decision D2 — metadata carries `{sourceExtractor, sourceId}`

New per-item fields `sourceExtractor` (yt-dlp `extractor_key`) and `sourceId`
(the RAW `%(id)s` — see D5 for the raw-vs-sanitized contract). `youtubeId`
stays exactly as-is for YouTube items (no migration, no rename).

**[REVIEW AMENDMENT — C1, the crux the first draft hand-waved]** The first
draft said "the downloadMeta bridge already carries structured fields per id
— extend it." The review proved that FALSE at every layer: the bridge's
capture key is gated by `isSafeVideoId` (`sanitizeCapturedChannelMeta`,
lib/ytdlp/store.js:899 — a non-conforming id DROPS THE WHOLE CAPTURE), its
consume key is the 11-char bracket (`extractYtdlpVideoId`, server.js:3740 —
returns null for the new bracket so the bridge block never runs), and the
print template (`CHANNEL_META_PRINT_TEMPLATE`, args.js:686) does not even
capture `extractor_key`. Real extractor ids from yt-dlp's own test corpus
(`'id=6'`, `'MTQ2NjMxOQ=='`, ids with spaces/colons/CJK) fail the gate — so
the two new fields would simply NEVER BE WRITTEN for the hard cases, and no
carry-forward checkpoint can protect a field that was never written. The
bridge redesign is therefore its own sub-design, a SIX-TOUCH chain:

1. `CHANNEL_META_PRINT_TEMPLATE` (args.js) gains `%(extractor_key)s` — the
   one-off variant only; the subscription template stays byte-identical.
2. `parseChannelMetaLine` (run.js:311) parses the new field.
3. `sanitizeCapturedChannelMeta` (store.js) becomes source-aware: for
   `source === 'Youtube'` (absent field = legacy = YouTube) the existing
   `isSafeVideoId` gate stays byte-identical. **[DELTA-2 AMENDMENT — the
   review's DELTA-1: the v2 draft's raw-id charset gate re-created the C1
   silent drop]** For any other source, the RAW id gets the MINIMAL gate
   only: non-empty string, length <= 256, control chars (incl. newline —
   archive-line forgery guard) stripped. NO charset restriction: real
   extractor ids contain slashes, quotes and spaces ('austrian/page=1',
   archive.org's 'lp_.../disc1/01.01. Bells Of Rostov.mp3' — yt-dlp's own
   corpus). D1's `[^\[\]/]` grammar is BRACKET-side only (the on-disk form,
   post-sanitizer, where `/` is impossible); the raw id never becomes a path
   or argv element, so there is nothing for a charset to protect. The join
   key becomes the composite `source + ' ' + id`, never the bare id
   (cross-extractor bare-id collisions are real: numeric ids). Note
   `rules.isArchived` already treats everything after the first space as the
   id, so spacey/slashy raw ids are safe for archive matching.
4. `recordDownloadChannelMeta` (store.js:980) keys `downloadMeta` by the
   composite key.
5. The consume site (server.js:3740) resolves its key via `extractMediaRef`
   (D1). **[DELTA-2 AMENDMENT — the review's DELTA-2: the v2 draft falsely
   claimed the rendered filepath was "already captured"]** It is NOT — the
   only `--print` in the codebase is CHANNEL_META_PRINT_TEMPLATE, which has
   no filepath field. The capture is a REAL chain change, owned here: touch
   #1 adds BOTH `extractor_key` AND `filepath` to the one-off print
   selector's `%(.{...})j` JSON set (same line, same parse); touch #2's
   `parseChannelMetaLine` carries both through. TRUST POSTURE (pinned): the
   captured filepath is yt-dlp-reported UNTRUSTED INPUT — the sanitize gate
   confines it to `path.basename(...)` -> `extractMediaRef` to derive the
   SANITIZED composite key; it is never resolved, stat'd, or otherwise
   treated as a filesystem path to act on. Key roles (D5, corrected
   wording): the record stores BOTH composites; the consume site looks up by
   the rendered-basename (sanitized) composite — the only form a dirent can
   give it — and the RAW pair is the authoritative payload it yields. Zero
   re-implementation of yt-dlp's sanitizer (endorsed by the review).
6. The bridge write onto the item sets `sourceExtractor`/`sourceId` (raw)
   alongside the existing channel-identity fields.

`backfillChannelIdentityFromFolder` (server.js:3803) is YouTube-root-scoped
and CANNOT act as a second chance for these — the bridge write is the ONLY
path, which is exactly why it must never silently drop (step 3's per-source
validation replaces drop-the-capture with record-what-was-captured).

THEN the five carry-forward checkpoints apply to the two new fields (scan
re-init carry-forward, Phase-2 merge guard, persist-gate OR-chain,
final-merge gap-fill, terminal writes — the six-strike class), each with its
own regression test per the stale-snapshot suite pattern.

### Decision D1a — the YouTube-vs-universal lane branch [REVIEW AMENDMENT — W2]

The review caught an identity-loss hole: yt-dlp's `YoutubeIE` matches URL
shapes `classifySingleVideo` (url.js:519) does not (e.g. `/clip/...`). Such
a URL would fall into the universal lane, download as `[Youtube=<id>]`, miss
the legacy 11-char matcher, and lose youtubeId/Share/reheat identity for a
genuinely-YouTube video. RULE (the output template is fixed at spawn, so
the branch is decided PRE-SPAWN): a URL that fails `classifySingleVideo` but
whose host passes the existing YouTube host allowlist is REFUSED with
today's error (byte-identical current behavior — no new YouTube shapes are
claimed by this wave; the design reviewer judged refusing /clip/ URLs
acceptable), and the universal lane only accepts NON-YouTube-host URLs.

**[DELTA-2 AMENDMENT — the review's DELTA-3: the host gate does NOT make
"[Youtube=id] can never exist" true]** yt-dlp's YoutubeIE matches dozens of
PROXY hosts outside our allowlist (the _INVIDIOUS_SITES list — yewtu.be et
al — plus hooktube.com, youtube-nocookie): `https://yewtu.be/watch?v=<id>`
sails through the non-YouTube-host gate into the universal lane and comes
back with `extractor_key='Youtube'`. Handling (the reviewer's recommended
option a): at the BRIDGE WRITE, when the captured `source === 'Youtube'`
AND the id passes `isSafeVideoId`, the item ALSO gets `item.youtubeId` set
— restoring Share/reheat/identity coherence for proxy-host YouTube items.
SEAM-2 is unaffected (the legacy secondary match still requires a legacy
bracket on the scanned file; a proxy-host item's on-disk bracket is
`[Youtube=<id>]` and matches only via the new-shape path, same-folder
confined). The archive line is `youtube <id>` either way (make_archive_id
lowercases the key), so dedup against a legacy-lane download of the same
video holds by construction. A yewtu.be case joins the D1a test matrix.

### Decision D5 — raw vs sanitized id: the three-strings contract [REVIEW AMENDMENT — W4]

yt-dlp sanitizes `%(id)s` in the rendered filename (`--windows-filenames` is
already on our argv → `/`→`⧸`, `?*:<>|"\`→fullwidth, controls stripped),
while `--print`/archive emit the RAW id. So three representations exist:
bracket (sanitized), metadata `sourceId` (raw), archive line (raw). CONTRACT
(each consumer named, raw-vs-bracket comparisons FORBIDDEN):
- Archive membership + delete-time append: RAW (`sourceExtractor.toLowerCase()
  + ' ' + sourceId`) — matches `make_archive_id` semantics.
- downloadMeta bridge [delta-review nit, corrected]: the record stores BOTH
  composite keys (raw and rendered-basename/sanitized); the CONSUME site
  looks up by the rendered-basename composite — the only form a dirent can
  ever give it — and the RAW `{source, id}` pair is the authoritative
  payload it yields. The sanitized key derives from the filepath yt-dlp
  itself reports in the (newly added — see D2 step 5) print capture; we
  never re-implement the sanitizer.
- Delete resolver / SEAM-2: bracket-vs-bracket only (both sides read
  dirents, so both are sanitized-form — internally consistent). The
  tombstone stores BOTH `sourceRef: {extractor, id}` (raw, from metadata)
  AND the bracket string observed at delete time; SEAM-2 matches on the
  bracket pair, confined to same-folder+same-ext exactly as v1.41.9.
- KNOWN RESIDUAL (disclosed): two distinct raw ids can sanitize to one
  bracket; within one folder+ext that could bind SEAM-2 to the wrong
  sibling, bounded by the existing `mtime <= deletedAt` gate and the
  fresh-db re-checks. Same order of likelihood as the disclosed v1.41.9
  same-id residual; the implementation gate's adversarial seat is briefed
  to attack precisely this with hostile-id fixtures (`'id=6'`,
  `'MTQ2NjMxOQ=='`, an id with a space — the review's corpus).

### Decision D3 — archive lines use yt-dlp's native format

`--download-archive` lines are natively `<extractor_lowercase> <id>` — yt-dlp
already writes the right thing for every site. Changes needed on OUR side:
- `recordOneShotInArchive(config, videoId)` hardcodes `youtube ${id}` →
  gains a source param: `recordInArchive(config, source, id)`; the delete
  route resolves `{source, id}` from metadata (D2) falling back to the
  bracket via `extractMediaRef`.
- The archive-append-on-delete scope check (`matchRootFolder` + id
  recoverable) generalizes through the same helper.

### Decision D4 — tombstones carry `{sourceExtractor, sourceId}`

`deleteTombstones[id].youtubeId` stays (legacy, YouTube items). New optional
`sourceRef: { extractor, id }` written for non-YouTube items. The scanner's
SEAM-2 secondary match generalizes: same folder + same ext + same
`extractMediaRef` result (bracket OR persisted ref) — the v1.41.9 gate's
confinement rules are LOAD-BEARING and copied, not re-derived.

## Extractor gating (named-only, no generic)

`--use-extractors default,-generic` (alias `--ies`) on every one-off spawn.
yt-dlp then refuses URLs only the generic extractor would match, with a
clear per-URL error we surface in the job row ("No supported site matched
this URL").

**[REVIEW-VERIFIED against yt-dlp master @ aefce1e]** — all four premises
hold in source: `default,-generic` composes (`orderedSet_from_options`,
utils/_utils.py:5336, regex-removal of `IE_NAME='generic'`; options.py:408
documents this exact invocation); the no-match error is a clean classifiable
stderr line (`No suitable extractor ... found for URL`, YoutubeDL.py:1724);
`extractor_key` is the Python class name minus `IE` → `[A-Za-z0-9_]`
(common.py:835) — EXCEPT plugin extractors gain a `+plugin` suffix
(common.py:4118), so D1's Key grammar adds `+` ONLY when followed by a
suffix, or (chosen) plugin-suffixed keys are treated as metadata-only, no
bracket key [REVIEW S1]; archive lines lowercase the key
(`make_archive_id`) and rules.js's isArchived already compares extractor
case-insensitively + id exactly [REVIEW S2 — the case contract: bracket
carries TitleCase, archive carries lowercase, EVERY source comparison in
FileTube is case-insensitive; pinned by a unit test].

## URL intake (server + client)

- `lib/ytdlp/url.js`'s YouTube allowlist REMAINS the rule for
  subscriptions/channels (unchanged). One-offs get a new, deliberately loose
  gate: `isPlausibleMediaUrl(url)` = http(s), no embedded credentials, no
  localhost / private-range / link-local literal IPs, length-capped.
  **[REVIEW AMENDMENT — C2, the first draft's rationale was FALSE]** The
  draft claimed named extractors match by hostname so a private-IP URL
  matches none — the review refuted this from source: several NAMED, ENABLED
  extractors match ARBITRARY hosts by path shape alone (`MediasiteIE`'s
  `_VALID_URL` is `https?://[^/]+/Mediasite/Play/...` — mediasite.py:24;
  also MediasiteNamedCatalog, Bandcamp-style `[^/]+` hosts, and others). So
  `http://<internal-host>/Mediasite/Play/<32hex>` reaches a named extractor
  and yt-dlp WILL fetch it despite `-generic`. Consequences adopted:
  (a) the literal-IP/localhost/private-range check is **load-bearing**, the
  only SSRF defense — not belt-and-suspenders; it gets its own unit matrix
  (v4/v6 literals, decimal/octal IP encodings, `[::1]`, 0.0.0.0/8).
  (b) HONEST RESIDUAL (disclosed, accepted for the LAN-trusted single-user
  threat model): a DNS hostname resolving to private space (incl. rebinding)
  and post-match redirects are NOT defended — doing so would require
  resolve-then-pin or a proxy, disproportionate here. The realistic exposure
  is "the sole trusted user pastes a hostile link"; noted in the ROADMAP
  entry and the tech-debt tracker with the v1.42 multiuser tranche as the
  revisit trigger (multi-user changes the threat model and MUST revisit
  this).
  yt-dlp (with the extractor gate) remains the real URL validator.
- Downloads page copy: "YouTube URL" → "Media URL (any yt-dlp-supported
  site)"; client-side validation relaxed to the same plausible-URL rule.
  Subscriptions intake copy unchanged.
- **Error copy (Dean's on-device report, 2026-07-16):** the current refusal
  leaks the internal parameter name ("channelUrl host is not an allowed
  YouTube host") on the one-off surface. The universal lane's refusals get
  user-facing copy per cause: gate-refused ("This doesn't look like a
  media URL"), yt-dlp no-match ("No supported site matched this URL —
  FileTube supports any site yt-dlp has a named extractor for"), private-IP
  refused (plain "not allowed" without echoing internals). No internal
  field names in any user-visible message.
- **Share-shortlink shapes (Dean's real case: facebook.com/share/r/<id>/) —
  Decision D6, the bounded redirect resolver:** VERIFIED against yt-dlp
  master source (byte-level grep, 2026-07-16): NO named Facebook extractor
  claims `/share/r/<base62>` — FacebookIE's _VALID_URL covers
  permalink/video/watch/posts/events/groups/watchparty (+FacebookReelIE's
  /reel/<digits>) with ids constrained to `pfbid…|\d+`. Stock yt-dlp handles
  share links ONLY via the GENERIC extractor's redirect-following — which
  this design blocks. Since the share sheet is THE mobile flow, the design
  adds a bounded server-side redirect resolver rather than reopening
  generic:
  - Runs ONLY when the pasted URL fails the named-extractor match (P1
    implements the match probe; cheap common case untouched).
  - Max 3 hops; requests carry no cookies/auth; response bodies are never
    consumed (redirect status + Location header only); http(s) Locations
    only (data:/file:/ftp: refused).
  - The LOAD-BEARING private-IP/localhost/link-local guard is re-applied to
    EVERY hop's target before that hop is fetched — strictly stronger than
    yt-dlp generic's own redirect-following, which has no IP guard at all.
    The DNS-rebinding residual is unchanged and stays disclosed.
  - The FINAL URL re-runs the named-extractor gate; still no match → the
    honest no-match error. The resolver's result is what gets queued (and
    what requeuePendingOneShots re-validates), so the durable queue never
    holds an unresolved shortlink.
  - Fixtures: Dean's facebook.com/share/r/ shape (redirects to /reel/<id>);
    a redirect chain ending at a private IP (refused at the hop); a >3-hop
    chain (refused honestly); a shortlink resolving to a still-unclaimed URL
    (no-match error); a shortlink resolving to a YOUTUBE URL (enters the
    legacy lane — see below); relative + protocol-relative Locations;
    userinfo-bearing Location (refused).
  - **[Delta-3 prescriptions — the focused design review APPROVEd D6 with
    these five additive constraints, all folded in as requirements:]**
    1. LANE: resolution runs BEFORE the lane branch; the resolved URL
       re-enters intake FROM THE TOP, byte-for-byte as if pasted (YouTube
       host → D1a/classifySingleVideo → legacy lane or today's refusal;
       else → named gate → universal lane). A shortener can therefore never
       launder a YouTube URL into the universal template, and a share link
       resolving to youtube.com/watch gets the legacy lane + youtubeId for
       free.
    2. LOCATION HANDLING: every Location header is resolved to an ABSOLUTE
       URL against the current hop (`new URL(location, currentUrl)`) BEFORE
       the scheme allowlist + IP guard run; hop targets carrying userinfo
       are refused outright (mirrors url.js's SF6 reject).
    3. GUARD COMPLETENESS + RESOLVE-THEN-CHECK: the per-hop guard is the
       same canonicalizing check as the intake gate (alternate IPv4
       encodings: decimal/octal/hex/short forms; IPv6 loopback, link-local
       incl. zone-id forms, ULA, unspecified, IPv4-mapped), AND each hop
       hostname is getaddrinfo-resolved with the fetch refused if ANY
       resolved address is private/loopback/link-local. RESIDUAL WORDING
       (corrected per review): D6's fetch primitive broadens the C2 class
       (FileTube now fetches arbitrary unmatched hosts); resolve-then-check
       narrows it back; the remaining residual is TOCTOU rebinding only —
       disclosed, revisit at v1.42 multiuser.
    4. BOUNDING (this repo's starvation scars): the resolver runs UNLOCKED
       (never inside the download gate — the probeChannel 5-minute-hold
       lesson), hard per-hop timeout (~3s) + total resolver budget (~8s);
       GET with `redirect: 'manual'` and the response destroyed immediately
       after status+Location are read — NEVER the HTTP client's auto-follow,
       which would bypass the per-hop guard entirely.
    5. P1 PRE-FLIGHT (efficacy) — DONE, CONFIRMED (2026-07-16): Dean's exact
       link `facebook.com/share/r/1Hk9jStL2C/` returns a single HTTP **302**
       to `facebook.com/reel/897928030021587/?rdid=…&share_url=…`
       (verified with `curl -o /dev/null`, no auth, no cookies, no body
       read). The target is claimed by FacebookReelIE (`/reel/<digits>`,
       facebook.py:959). So D6 resolves this shape in ONE hop with exactly
       the header-only, unauthenticated primitive specified — the efficacy
       risk the review raised is retired for the reel case. NOTE for the
       resolver: the `?rdid=&share_url=` query survives on the resolved URL;
       harmless to FacebookReelIE (id comes from the path), but the resolved
       URL queued is the full 302 Location verbatim.
- **iOS Shortcut / share-sheet path (Dean-confirmed requirement, 2026-07-16):**
  there is exactly ONE one-off intake — `POST /api/ytdlp/download` — which
  already accepts `text/plain` bodies specifically for Shortcuts (v1.28.0),
  so the D1a lane branch at that route's validation seam covers Shortcut
  engagements automatically. REQUIREMENT pinned: the universal lane runs the
  SAME v1.28.0 normalization pre-steps (trim → surrounding-punctuation peel
  → first-http(s)-run prose extraction — already URL-generic, verified in
  url.js) BEFORE `isPlausibleMediaUrl`, so a Shortcut sharing
  `"Title\nhttps://vimeo.com/<id>?si=x"` behaves identically to the web
  modal. Test matrix gains Shortcut-style fixtures: prose-wrapped +
  quote-glued + text/plain-body non-YouTube URLs.
- **[Design-review note 1, adopted]** The downloadMeta record's secondary
  entry is keyed by the FULL rendered basename string (already captured via
  filepath), and the consume site looks up by exact basename equality —
  strictly more robust than bracket-derived keys for bracket-unparseable
  ids (a `]`-bearing or >128-char id); the bracket stays load-bearing only
  in the delete resolver / SEAM-2. The gate seat probes a `]`-bearing and a
  200-char synthetic id.

## Folder placement

Non-YouTube one-offs land in per-uploader folders exactly like the v1.25.1
channel-folder model: folder name = `%(uploader)s` fallback
`%(extractor_key)s` (e.g. `Vimeo/`), via the same computed-relocation path
the YouTube one-offs use. ⚠ uploader can be absent/hostile — the existing
folder-name sanitizer (v1.25.1) is reused verbatim; design review should
check it against non-YouTube uploader shapes.
