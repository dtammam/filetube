# v1.16.0 "Watch Experience"

## Goal

Keep the video playing and keep the user's place across the FileTube watch/home
experience — navigating away from a playing video should not interrupt it, moving
between videos and back to browsing should feel continuous rather than a series of
hard page reloads — and accept real-world YouTube share-sheet URLs into the
optional subscription feature safely, without weakening any existing security guard.

## Scope

Five items, one cohesive v1.16.0 round:

### FR-1 — Docked persistent mini-player + SPA-lite navigation (REVISED, architectural centerpiece)

**This item was revised on 2026-07-06. It is NO LONGER native Picture-in-Picture.**
See the Decision log for the full rationale.

Dean's hard requirement, verbatim: *when on a video and you tap the home button /
navigate to other views, the player must NOT reload — the video keeps playing
smoothly.*

FileTube is currently a **multi-page app** (`watch.html`, `index.html`,
`setup.html` — and, when the optional yt-dlp module is enabled,
`lib/ytdlp/views/subscriptions.html`): every in-app navigation is a full browser
reload that tears down the `<video>` element. Delivering "smooth, no reload"
therefore requires the video element to **survive navigation**, which requires:

1. **A persistent app shell.** The header/nav chrome and the media player
   (`<video>`/audio element) live in a shell that is never torn down by in-app
   navigation; only the per-view content area is replaced.
2. **SPA-lite navigation.** In-app link clicks (nav links, folder/search links,
   related-video cards, the home/logo button, prev/next from FR-2) are
   intercepted; the destination view's content is fetched and swapped into the
   content container **without a full page reload**; `history.pushState` keeps
   the URL in sync so back/forward and deep links keep working. The persistent
   player keeps playing throughout every swap.
3. **A docked mini-player.** Leaving the watch view (going home / to another
   in-shell view) while media is playing shrinks/docks the player to a
   persistent corner (YouTube-web style); it keeps playing there. Tapping/
   clicking it expands back to the full watch view (same media, same position,
   no reload). An explicit close (`[x]`) stops playback and fully dismisses it.
   When you are *on* the watch view, it is the full-size player, exactly as
   today.
4. **Graceful degradation.** SPA-lite navigation is a **progressive
   enhancement** — a fresh page load or an externally-shared deep link to any
   main view (`/`, `/watch.html?v=<id>`, `/setup.html`, and `/subscriptions`
   when the yt-dlp module is enabled) still works as a complete, correct full
   page load. The smooth/no-reload behavior only applies to *in-app*
   navigation.
5. **Feature preservation.** Every existing player feature keeps working
   identically in *both* the full and docked states: inline iOS playback
   (`playsinline`/`webkit-playsinline`), ±15s skip controls, the audio
   background-art mode, the transcode "preparing…" overlay + polling, and
   Media Session lock-screen metadata/position.

Native OS-level Picture-in-Picture is **not** part of this round (superseded by
the docked-player decision below) — it may be revisited as a separate,
future item if Dean wants both.

This is the heavy, architectural item: FileTube moves from multi-page-reload to
a persistent-shell-plus-client-side-view-swapping model. It touches the whole
navigation/shell layer and the player's lifecycle, with real risk of breaking
deep links, the watch page, the back button, and scroll — flagged for the
**heavier two-reviewer/adversarial QA gate** (see Acceptance criteria). Dean's
on-device pass (especially the smooth-no-reload *feel* and iOS Safari) is the
arbiter.

### FR-2 — Prev/next video navigation (derived playlist order) — foundation for FR-3

Add prev/next controls on the watch page that step to the previous/next media
in a derived ordered "playlist," recommended to be **the current home sort
order** (`localStorage` `filetube_sort`, applied by `sortItems` in
`public/js/main.js`). The order and current position are **re-derived on the
watch page** from `GET /api/videos` + the persisted sort (not solely passed via
URL params), so a refresh/deep-link still computes the correct position. Prev
is disabled at the start of the order and Next is disabled at the end (no
wrap-around).

### FR-3 — Autoplay next (Settings toggle, OFF by default)

A persisted toggle (recommended: a server `db.settings` setting `autoplayNext`,
`defaultView`-style) that, when ON, auto-plays the next media (per FR-2's
derived order) when the current one ends (the `<video>` `'ended'` event). OFF
by default. Depends on FR-2's derived-next foundation. Does not re-enable
general mobile autoplay for the initial load (`watch.js`'s existing
desktop-only initial-autoplay posture is unchanged).

### FR-4 — Persist home-view state on return from watch (REVISED / reconciled)

**Reconciled on 2026-07-06.** This was originally a standalone item requiring
its own persistence mechanism (sessionStorage / history.state / bfcache). It is
now **largely subsumed by FR-1's persistent shell**: because in-app navigation
no longer tears down and reloads the home view, the home view's DOM (active
folder/search filter, sort, and scroll position) is naturally retained across
an in-app round trip to the watch page and back — there is no separate
persistence architecture to design. FR-4 is reduced to a thin acceptance
requirement that rides on FR-1: returning to home via in-app navigation
restores the exact prior view and scroll position, and a fresh/deep-link home
load always starts clean.

### FR-5 — Share-URL validator robustness (`lib/ytdlp/url.js`) — security-sensitive

Make `validateChannelUrl` robust to the real-world YouTube share-sheet payload
(e.g. `"Title\nhttps://youtu.be/<id>?si=<token>"`) **without weakening any
existing guard**: coerce to string and trim first; if internal whitespace
remains, extract the first `/https?:\/\/\S+/` substring as the candidate; run
that candidate through the **unchanged** strict validation. This is isolated to
`lib/ytdlp/url.js` (server-side, yt-dlp path only) and shares no surface with
the client watch/home work in FR-1–FR-4.

## Out of scope

- **Native OS-level Picture-in-Picture.** Superseded by the FR-1 pivot to a
  docked in-app mini-player; not built in this round.
- **Re-enabling general mobile autoplay for the initial load.** FR-3's
  autoplay-next only fires on `'ended'`, riding an existing user-gesture
  chain; the desktop-only initial-autoplay posture (`watch.js`) is unchanged.
- **Weakening any existing `validateChannelUrl` guard.** FR-5 only normalizes
  the input before the unchanged strict checks.
- **A client-side router library or build tooling.** SPA-lite navigation is
  built with vanilla DOM/`history` APIs per `CONTRIBUTING.md` (no framework,
  no bundler, no new dependency).
- **Converting every page in the app to the shell.** Only the four main
  in-shell views are in scope: home (`index.html`), watch (`watch.html`),
  setup (`setup.html`), and subscriptions (`lib/ytdlp/views/subscriptions.html`,
  only reachable when the yt-dlp module is enabled). No other routes exist
  today, and none are added.
- Checked against `CONTRIBUTING.md`: nothing above defers a mandatory standard
  (tests, lint, textContent-not-innerHTML, no new deps) — all of those remain
  in scope for every file touched by every item, including the new SPA-lite
  nav/shell code.

## Constraints

- **Additive / no regressions.** Every item layers on top of existing
  watch/home behavior. The existing resume overlay, skip controls, audio
  visualizer, related sidebar, transcode overlay, mobile-autoplay-disabled
  posture, and home sort/sidebar/search must keep working unchanged — this is
  *more* consequential than before, since FR-1 restructures the navigation/shell
  layer every one of those features lives inside.
- **The optional yt-dlp module's disabled-path no-op is sacred (FR-5).**
  `lib/ytdlp/url.js` only executes on the yt-dlp path; with
  `FILETUBE_YTDLP_ENABLED` off the module stays byte-identical no-op. FR-5 adds
  no routes/UI and changes no disabled-path behavior — it only normalizes the
  input to the existing validator. Correspondingly, when the module is
  disabled, the SPA-lite shell must not introduce a `/subscriptions`
  swappable-view entry, nav link, or route that doesn't already exist today.
- **FR-5 security is non-negotiable.** The existing strict guards in
  `validateChannelUrl` do not change: `FORBIDDEN_CHARS`, `MAX_URL_LENGTH`,
  leading-`-` reject, `new URL()` parse, http/https-only, userinfo reject,
  `ALLOWED_HOSTS` allowlist, `isPlausiblePath` shape check, and the SF5
  decoded `?v=`/`?list=` id charset+length check. Only the input is normalized
  *before* those checks (trim, then extract-first-URL if whitespace remains),
  and extraction must never smuggle a hostile URL past any check.
- **`docs/CONTRIBUTING.md`:** vanilla DOM (no framework/bundler — this applies
  directly to FR-1's SPA-lite nav/shell), CommonJS server, 2-space/semicolons/
  single-quotes, `textContent` over `innerHTML` for new dynamic strings,
  `node:test` for new/regression tests, lint 0 warnings, no new runtime
  dependencies.
- **Era-theme system preserved.** Any new CSS (the docked mini-player, prev/
  next controls, an autoplay-next toggle surface) uses existing theme tokens
  (CSS custom properties) — no hardcoded colors that break the 2005/2009/2014/
  2021 eras or light/dark modes.
- **Node 22 LTS** toolchain (`engines` ≥20); CI/build-verify runs tests on
  Node 22.

## Product forks — resolved with recommendation

1. **FR-2 order source** → the current home sort order (not `rankRelated`);
   PE owns how the order reaches the watch page (re-derive on the watch page
   from `GET /api/videos` + the persisted sort, not a URL-param id-list, so
   refresh/deep-link still works).
2. **FR-2 ends behavior** → disable at start/end (no wrap).
3. **FR-3 pref storage** → a server `db.settings` setting (`autoplayNext`,
   `defaultView`-style), OFF by default.
4. **FR-4 restore mechanism** → subsumed by FR-1's persistent shell (no
   separate sessionStorage/history.state design needed; see FR-4 above).
5. **FR-5 extraction** → extract-first-URL then strictly validate (accept the
   share-sheet payload; every guard unchanged).
6. **NEW (from the FR-1 pivot) — which views are SPA-swappable?** Recommend
   **all four main views**: home, watch, setup, and subscriptions (when the
   yt-dlp module is enabled). A uniform shell across every in-app view is
   simpler to reason about and test than special-casing which views opt in;
   the subscriptions view only ever appears in the swap set when the module's
   existing enabled-gate would have shown its nav link anyway, so the
   disabled-module no-op guarantee is preserved.
7. **NEW (from the FR-1 pivot) — does the docked player persist on
   setup/subscriptions, or only on home?** Recommend it **docks and keeps
   playing on setup/subscriptions too**, not just home. These pages are now
   in-shell views like home; hiding/stopping the player specifically on those
   two pages would silently violate the "keep playing across navigation" hard
   requirement the moment a user visits Settings while listening/watching, and
   would be a confusing, arbitrary exception to explain or test for.

If a genuine NEW fork surfaces during Design beyond the two above, PE should
state it with a recommendation rather than block.

## Design

### Approach

FileTube today is four independent full-document pages (`index.html`,
`watch.html`, `setup.html`, and the gated `lib/ytdlp/views/subscriptions.html`),
each duplicating the same shell (`<header>`, `<aside class="sidebar">`, the
`<nav class="bottom-nav">`/Playlists sheet) and each loading `common.js` plus one
per-page script whose whole body runs inside a single `DOMContentLoaded` IIFE.
The `<video id="media-player">` lives deep inside `watch.html`'s
`#main-content > .watch-container > .watch-main > #player-wrapper`, so any
navigation destroys it. The centerpiece change is to stop letting navigation
touch the player: hoist the player out of the swappable content area into a
**persistent player host** that is reparented between an inline slot (full) and a
fixed corner dock (docked), and drive all four views through a **vanilla
SPA-lite fetch-and-swap router** that replaces only a per-view `#view-root`
fragment while the shell — and the playing `<video>` — stay mounted.

The refactor is deliberately structured to be *mechanical and reversible per
view*, not a big-bang rewrite. Each per-page script is converted from a
`DOMContentLoaded` IIFE into a **registered view module** exposing
`init(root)`/`destroy()`; the same `init` runs both on a full page load
(progressive-enhancement fallback) and on an in-app swap, so there is exactly one
code path per view and no divergence. Every view URL still resolves to a complete
server-rendered document — a hard load / external deep link renders correctly with
zero router involvement; the SPA is strictly the enhancement layer on top. The
player is split off from the watch *view* into a **persistent player controller**
that owns the `<video>` and every player feature (skip controls, audio-bg-art,
transcode overlay + polling, resume overlay, Media Session, progress saving) for
the life of the shell — so those features are preserved across full↔docked *by
construction*: the element and its listeners are never re-created during
navigation, only its media source changes.

FR-2/FR-3/FR-4 then ride cheaply on this shell: prev/next re-derives the home
sort order on the watch view (durable on refresh) and the shell optionally scopes
it to the active home filter; autoplay-next is a server setting read by the
persistent controller's existing `'ended'` handler; and home-view/scroll
retention falls out of caching the (verifiably timer-free) home view node across
a round trip. FR-5 is an isolated server-side input-normalization pre-step in
`lib/ytdlp/url.js` that shares no surface with the client work.

### Component changes

- **Shared shell (all four HTML pages)** — Introduce a stable per-view boundary:
  wrap each page's per-view content in `<div id="view-root" data-view="home|watch
  |setup|subscriptions">`. The header/sidebar/bottom-nav/Playlists-sheet markup
  stays outside it (unchanged, still duplicated per file as today). Add two shell
  elements outside `#view-root`: a persistent `<div id="player-dock">` (fixed
  corner container, initially empty/hidden) and a single
  `<template id="player-host-template">` holding the *static* player markup
  (the current `#player-wrapper` subtree: `#audio-bg-art`, `#resume-overlay`,
  `#transcode-overlay`, `<video id="media-player" ... playsinline
  webkit-playsinline x5-playsinline>`, `#speed-badge`, `#skip-controls`,
  `#audio-visualizer`). The router clones this template **once** into a single
  live `#player-host`, so there is never a duplicate live `<video>`/duplicate-id.
  All four shells load `common.js` (router + registry), `main.js`, `watch.js`,
  `player.js`, and `setup.js` so every core view module is registered regardless
  of entry page (`subscriptions.js` stays lazy-loaded — see the no-op note).

- **`public/js/common.js` (router + registry, persistent)** — Add a tiny view
  registry (`FileTube.registerView(name, { init, destroy })`) and the SPA-lite
  router: a single delegated `document` click interceptor, `navigate(url,
  {replace})`, a `popstate` handler, per-view scroll bookkeeping in
  `history.state`, the fetch-parse-extract-`#view-root`-swap pipeline, and the
  progressive-enhancement boot (on `DOMContentLoaded`, derive the current view
  from `location` and call its `init` once — the identical path a swap uses). The
  existing yt-dlp nav-link / one-off-button injection stays where it is (it runs
  once at boot against the persistent shell and persists across swaps).

- **`public/js/watch.js` → watch *view* module** — Reduced to the view-owned,
  per-visit DOM under `#view-root`: metadata population, description toggle,
  related sidebar, comments, star rating, delete button, sidebar-folder render,
  search wiring, and the new FR-2 prev/next controls. Its `init(root)` reads
  `?v=`, fetches media data, and asks the persistent player controller to load
  that media into the full slot (idempotently — see the state machine). All of
  its listeners register through one per-view `AbortController` so `destroy()`
  removes them in one call — no leaks. The current
  `playerWrapper.innerHTML = <error>` fallback is replaced with a view-area error
  that never nukes the persistent host.

- **`public/js/player.js` (NEW — persistent player controller)** — Owns
  `#player-host` and every player feature currently in `watch.js`'s player half:
  `setupPlayer`, live-transcode (`startLiveStream`/`currentAbsTime`/`liveMode`),
  skip controls (buttons/double-tap/press-hold/keyboard), rotate-to-fullscreen,
  transcode overlay + `pollTranscodeUntilReady`, resume overlay, audio
  background-art (`.audio-mode`), Media Session (metadata/state/position +
  `visibilitychange` re-assert), and the progress saver. Exposes
  `load(id, data, { adopt })`, `getState()`, `dock()`, `expand()`, `close()`, and
  `currentId`. Player-scoped timers/listeners (progress interval, transcode poll,
  media-session throttle, the `document` keydown shortcuts, `visibilitychange`,
  `fullscreenchange`) are owned here — the keydown/skip shortcuts are active only
  in FULL state; everything is torn down only on `close()`, never on a view swap.

- **`public/js/setup.js` (NEW — extracted from `setup.html` inline script)** —
  The setup page's script becomes an external, registered view module
  (`init`/`destroy`) so it can be loaded and swapped like the others. Behavior
  unchanged; the FR-3 autoplay-next toggle is added here.

- **`public/js/main.js` → home *view* module** — Its `DOMContentLoaded` body
  becomes `init(root)`/`destroy()`. It is verifiably free of `document`-level
  listeners and timers (all listeners are on its own `#view-root` elements), which
  is why the home view node is the one safely retained for FR-4.

- **`server.js`** — FR-3 only: add `autoplayNext: false` to `DEFAULT_SETTINGS`,
  to `settingsResponse`, to the `KNOWN_KEYS` allowlist, and a
  `typeof body.autoplayNext !== 'boolean'` 400 branch — a byte-for-byte mirror of
  the `pruneMissing`/`defaultView` pattern. No routing/static change is required:
  the four view URLs already resolve to their static documents, which is exactly
  the progressive-enhancement fallback the router depends on.

- **`lib/ytdlp/url.js`** — FR-5 input-normalization pre-step inside
  `validateChannelUrl` (details below). No other server change; disabled-path
  no-op preserved.

### The docked-player state machine (FR-1)

States of the single persistent `#player-host`:

- **CLOSED** — no media; host hidden/empty. Initial state on any fresh non-watch
  load.
- **FULL** — host reparented into the current watch view's inline `#player-slot`,
  full size, in normal document flow (so it scrolls with the page exactly as
  today). Entered on watch-view init, or on expanding a docked player.
- **DOCKED** — host reparented into the fixed `#player-dock` corner, mini size,
  still playing. Entered when navigating from the watch view to any other in-shell
  view while media is loaded and not closed.

Transitions:

- **FULL → DOCKED**: watch → home/setup/subscriptions with media loaded. Reparent
  host into `#player-dock`; deactivate FULL-only shortcuts; keep playing.
- **DOCKED → FULL (same media)**: tapping the dock body pushes `/watch.html?v=
  <currentId>`; the watch view inits and, because `currentId` already matches,
  calls `load(id, data, { adopt: true })` — reparent host to the slot and rebuild
  surrounding metadata **without touching the `<video>` src/currentTime** (no
  restart).
- **FULL → FULL (different media)**: a related-card / prev-next / home-card click
  into another `/watch.html?v=` — the host stays in the slot; the controller loads
  the new source in place (no dock).
- **DOCKED → FULL (different media)**: navigating to a different video's watch
  page while docked — reparent to slot and load the new source.
- **any → CLOSED**: the dock's `[x]` control — `pause()`, clear `src` + `load()`
  to release the resource, hide host, and tear down player-scoped
  timers/listeners (no orphaned playing `<video>`).

The physical move is a `reparent` (`appendChild`) between `#player-slot` and
`#player-dock`. Reparenting a same-document media element preserves `currentTime`
and, on desktop/modern browsers, does not interrupt playback. **iOS Safari risk:**
reparenting can pause/reset on some iOS versions. Mitigation: the reparent runs
inside the user gesture that triggered it (nav click / dock tap), and immediately
after the move the controller re-asserts play if it was playing before
(`if (wasPlaying && el.paused) el.play().catch(()=>{})`) — `currentTime` is
retained, so this resumes seamlessly within the gesture chain. Documented
fallback if Dean's on-device pass shows reparent still interrupts iOS: switch the
mount mechanism to a no-DOM-move technique (keep `#player-host` permanently in the
shell and position it over the slot via measured geometry) — a change localized to
the mount helper only. This is the single biggest unknown and Dean's device is the
arbiter.

### SPA-lite navigation mechanics (FR-1)

- **Interception**: one delegated `document` click listener finds the nearest
  `<a>` and intercepts only when it is a plain left-click (no modifier keys, no
  `target=_blank`), same-origin, and its path is one of the four known view
  routes (`/`, `/index.html`, `/watch.html`, `/setup.html`, `/subscriptions`),
  including query strings (`/?root=`, `/?search=`, `/?folder=`, `/watch.html?v=`).
  Anything else (external links, `/thumbnail/*`, downloads) falls through to a
  normal browser navigation. Programmatic navigations (search submit, FR-2
  prev/next, FR-3 autoplay-next, dock-expand) call `router.navigate()` directly
  instead of assigning `window.location`.
- **`navigate(url)`**: derive target view → `fetch(url)` the full document →
  `DOMParser` → extract the new `#view-root` (+ `<title>`) → `currentView.destroy()`
  → run the player transition (dock / keep-full / stay) based on from→to → replace
  the `#view-root` node → `newView.init(root)` → `history.pushState({ view, url,
  scrollY:0 }, '', url)` → set `document.title` → scroll handling (FR-4). On a
  `fetch`/parse failure, fall back to a real `window.location.assign(url)` so
  navigation never dead-ends.
- **`popstate`**: re-derive the view from `location` and run the same swap without
  `pushState`, restoring `history.state.scrollY`.
- **No fragment-script execution**: only `#view-root` (and the title) is lifted
  from the fetched document; the `<script>` tags in it are never appended (the
  view modules are already registered from the initial page's script tags). This
  avoids re-exec/CSP issues.

### FR-2 — Prev/next (derived order)

A shared, unit-testable helper pair in `common.js`:
`deriveOrderedIds(items, sort)` (wrapping the existing `sortItems`) and
`computeNeighbors(orderedIds, currentId)` → `{ prevId, nextId }` (each `null` at
the ends). On watch-view init the ordered list is **re-derived** from
`GET /api/videos` + the persisted `filetube_sort`, so a refresh/deep-link computes
the correct position with no reliance on transient state. Prev/next controls live
in the watch `#view-root`; prev is disabled when `prevId === null`, next when
`nextId === null` (no wrap). The shell may additionally carry an in-memory
`navContext` (the active home filter + ordered ids captured when a card is
clicked) so an in-shell click from a *filtered* home view keeps prev/next within
that filter; when absent (deep link), the full-library order is the durable
fallback. Neighbors are exposed to the player controller so FR-3 reuses the exact
same logic (no divergent order).

### FR-3 — Autoplay next (design)

Server setting `autoplayNext` (boolean, default `false`), added to
`DEFAULT_SETTINGS` / `settingsResponse` / `KNOWN_KEYS` + a boolean 400 branch, and
a Settings-page toggle in `setup.js`/`setup.html` mirroring `pruneMissing`. The
persistent controller's existing `'ended'` handler, when `autoplayNext` is on and
`computeNeighbors` yields a `nextId`, calls `router.navigate('/watch.html?v=' +
nextId)` (smooth, in-shell). Because `'ended'` follows a user-initiated play, the
subsequent load/play rides the existing gesture chain (generally allowed,
including most mobile); the iOS caveat is that the chain can lapse and `.play()`
may reject — caught, leaving it paused, no error. At the end of the order
(`nextId === null`) it does nothing. The desktop-only initial-load autoplay
posture in `player.js` is unchanged.

### FR-4 — Restore home view + scroll

Recommended mechanism: **retain the home view node**. Because `main.js`/home has
no `document`-level listeners and no timers, its rendered `#view-root` node is
safe to detach and hold in a single-entry `viewCache` on navigate-away (recording
`scrollY`). On return via back/forward or in-app nav to the same home URL, the
cached node is re-attached instantly — exact grid (including a `random` shuffle
already rolled), filter, sort, and scroll, with no re-fetch, no re-render flash,
and no image-height scroll-restoration race. A fresh/deep-link/hard load (no cache
entry, or a different home filter URL) builds clean and starts at the top. Only
the home view is cached; watch is always init/destroyed (its player persists
separately) and setup/subscriptions are init/destroyed (subscriptions has its own
polling that must be aborted in `destroy()`). This keeps FR-4 genuinely thin —
one cache hook on the router, no separate persistence subsystem.

### FR-5 — Share-URL validator robustness (`lib/ytdlp/url.js`)

Insert a normalization pre-step inside `validateChannelUrl`, keeping every
existing guard byte-identical and still applied to the resulting candidate:

1. Existing non-empty-string guard stays first.
2. Existing `raw.length > MAX_URL_LENGTH` reject stays **on the original raw
   string** (so an oversized blob is rejected before any regex work — satisfies
   the mandated oversized-input test; extraction can't shrink a huge input past
   this cap).
3. `const trimmed = raw.trim();`
4. If `/\s/.test(trimmed)` (internal whitespace remains — text with an embedded
   URL), extract the first `/https?:\/\/\S+/` match as `candidate`; if no match,
   keep `trimmed` (which then fails the unchanged `FORBIDDEN_CHARS` check exactly
   as today). Otherwise `candidate = trimmed`.
5. Run the **unchanged** strict validation on `candidate`: leading-`-` reject,
   `FORBIDDEN_CHARS` (now on a whitespace-free candidate), `new URL()` parse,
   http/https-only, userinfo reject, `ALLOWED_HOSTS` allowlist, `isPlausiblePath`
   shape, and the SF5 decoded `?v=`/`?list=` id charset/length check.

Because `\S+` stops at whitespace, `click https://evil.com/x` extracts
`https://evil.com/x` → fails the host allowlist; an extracted URL carrying a shell
metachar (`https://youtu.be/x;rm`) still contains `;` → `FORBIDDEN_CHARS` rejects;
`?si=`/`&feature=`/`&pp=` pass as before and `buildWatchUrl` rebuilds a clean
canonical URL (no `?si`). `classifySingleVideo` inherits the normalization for
free (it calls `validateChannelUrl` first — single source of truth). The five
mandated security-regression tests plus the accept/classify cases map directly
onto these steps.

### Disabled-module no-op (cross-cutting)

The router's known-route set may name `/subscriptions`, but the shell never
*surfaces* it when the module is disabled: the subscriptions nav link is still
injected only on a genuine `2xx` from `/api/subscriptions/health`
(`injectSubscriptionsNavLinkIfEnabled`, unchanged), so with the module off there
is no in-app link to it and no in-app navigation ever targets it; a hard load of
`/subscriptions` when disabled still 404s server-side. `subscriptions.js` is
**lazy-loaded only on the first `/subscriptions` navigation** (which can only
happen when enabled), so it is never even fetched on a disabled install — the
disabled path stays byte-identical.

### Alternatives considered

- **True single shell document + server-side templating** (one HTML shell, views
  as pure fragments) — cleaner and DRYs the duplicated shell, but requires a
  server templating change and breaks the "every URL is a complete document" free
  progressive-enhancement fallback. Rejected for this round as higher-risk; noted
  as a future refactor to de-duplicate the shell/template.
- **Fixed-overlay player (never reparent; position over the slot via measured
  geometry)** — avoids the iOS reparent question entirely, but must track slot
  geometry on scroll/resize and fights `position: fixed` not scrolling with
  content. Kept as the *documented fallback* for the reparent approach rather than
  the primary, since reparenting keeps the player in natural document flow.
- **Narrowing SPA to home↔watch only** (setup/subscriptions stay full-load) —
  smaller blast radius, but directly contradicts the PM's resolved fork #7 (the
  docked player must persist on setup/subscriptions too) — a full-load there would
  stop the player mid-listen. Rejected; instead the risk is contained by giving
  every view a real `destroy()` (AbortController + timer teardown), which the
  subscriptions view especially needs for its status polling.
- **FR-2 order via URL id-list param** — breaks on refresh/deep-link. Rejected in
  favor of re-deriving from `/api/videos` + persisted sort (PM fork #1).
- **FR-4 via re-fetch + `history.state` scroll restore** — simpler than a view
  cache, but re-renders the grid (flash) and races lazy-image heights when
  restoring scroll. Rejected in favor of retaining the (timer-free) home node.
- **FR-3 client-only `localStorage` pref** — simpler, but per-device and not a
  first-class Setting. Rejected in favor of the server setting (PM fork #3).

### Risks and mitigations

- **Risk**: iOS Safari pauses/resets the `<video>` on reparent → **Mitigation**:
  reparent inside the triggering user gesture + immediate play re-assert with
  retained `currentTime`; documented fixed-overlay fallback; Dean's device is the
  arbiter.
- **Risk**: per-view listener/timer leaks across many swaps → **Mitigation**:
  strict `init`/`destroy` contract, one `AbortController` per view, and the
  player/view ownership split (player timers live in the persistent controller and
  are cleared only on `close()`).
- **Risk**: broken deep links / refresh → **Mitigation**: progressive
  enhancement — every URL still renders as a full server document; the router
  boots on top and shares the same `init` path.
- **Risk**: back/forward and scroll jumps → **Mitigation**: `popstate` handler +
  `history.state.scrollY` + home view-node retention (no image-height race).
- **Risk**: duplicate live `<video>`/duplicate ids → **Mitigation**: exactly one
  live `#player-host` cloned from the template; only one view mounted at a time;
  cached home node is detached while inactive.
- **Risk**: a view module isn't loaded on the entry page (e.g. `watch.js` not
  present on a home hard-load) → **Mitigation**: all four shells eagerly load the
  three core view scripts + `player.js`; `subscriptions.js` is lazy-loaded on
  demand (and only when enabled).
- **Risk (FR-5)**: extraction smuggles a hostile URL past a guard → **Mitigation**:
  every existing guard runs unchanged on the extracted candidate; length check
  stays on the raw input; adversarial reviewer + the five mandated regression
  tests.

### Performance impact

No expected impact on the RELIABILITY.md budgets (all server-side: transcode
queue, scan/prune, cache eviction — untouched). Client-side, SPA-lite adds one
same-origin `fetch` of an already-cached-friendly HTML document per in-app
navigation (cheaper than a full reload) and removes repeated player teardown/
setup; the persistent player avoids re-buffering on navigation. FR-3 adds one
boolean to the settings object; FR-5 adds a `trim` + at most one bounded regex
match on the yt-dlp add path only.

## Decision log

- 2026-07-06 — Original FR-1 scope (per the Discovery inbox and
  `.state/feature-state.json`) was native Picture-in-Picture: a feature-detected
  PiP button using `requestPictureInPicture()`/`document.pictureInPictureEnabled`
  on desktop/Android Chromium, with the iOS `webkitSetPresentationMode`
  fallback — explicitly NOT a docked in-app player or SPA refactor.
- 2026-07-06 — **Dean changed the FR-1 decision after seeing the tradeoff.**
  Native PiP was dropped from scope. FR-1 is now a docked persistent
  mini-player backed by a persistent app shell + SPA-lite client-side
  navigation, because the hard requirement ("navigate away and the video
  keeps playing smoothly, no reload") cannot be met by PiP alone in a
  multi-page app — it requires the `<video>` element to survive in-app
  navigation, which is a materially larger architectural change than a PiP
  button. FR-1 is now the heavy, two-reviewer-gated item for this round.
  FR-4 (persist home-view state) is reconciled: no longer a standalone item
  with its own persistence mechanism, since it rides on FR-1's persistent
  shell — reduced to a thin "restore on return / clean on fresh load"
  acceptance requirement. FR-2, FR-3, and FR-5 are unchanged. Two new product
  forks were introduced by this pivot (which views are SPA-swappable; whether
  the docked player persists on setup/subscriptions too) — both resolved with
  a recommendation (all four main views; yes, docks there too) rather than
  left open, pending Dean's confirmation.
- 2026-07-06 — **FR-5 AC #6 reconciled during T5 implementation.** The SDE
  found the original wording ("full `youtube.com/watch?v=<id>&si=<x>`
  extracts to a valid video") is unachievable without weakening
  `FORBIDDEN_CHARS`, which has always forbidden `&` (pre-dating this feature,
  independent of the new whitespace-extraction logic) as a shell-metachar
  guard on the yt-dlp spawn path. Decision: keep `&` forbidden — this is a
  no-auth LAN server and the spawn guard stays strict; the AC is revised to
  target the real iOS share-sheet format (`youtu.be/<id>?si=<token>`, no `&`),
  with the `&`-containing case explicitly documented as REJECTED-by-design.
  The other four FR-5 security-regression ACs are unchanged.
