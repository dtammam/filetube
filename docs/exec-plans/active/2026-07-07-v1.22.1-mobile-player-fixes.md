# v1.22.1 Mobile-Player Follow-Up (targeted bug-fix round)

## Goal

Fix five grounded on-device bugs Dean found in the v1.22.0 FR-1 responsive
player (mobile-audio fullscreen/speed/hold-2x, mobile-video's CRITICAL
missing controls, rotate-to-fullscreen, desktop-missing playback speed, and
desktop click-to-pause) with the smallest correct fix per bug — no refactor
of the FR-1 responsive-controls architecture beyond what each root cause
genuinely requires.

## Scope

Five lean FRs, one bug each, confirmed against the live code
(`public/js/player.js`, `public/css/style.css`) and the shipped v1.22.0 FR-1
design (`docs/exec-plans/completed/2026-07-08-v1.22-player-parity.md`).
Grounding below is confirmed directly against the current code (line refs as

### FR-1 — [CRITICAL, heaviest] Mobile VIDEO: no visible controls in FULL

Confirmed: `resolveControlsMode('video', true)` → `'native'`, and
`applyControlsMode()` (`player.js:520-530`) sets the `controls` attribute on
`#media-player` only while `state === STATE_FULL`; `style.css:3199-3203`
hides `.player-controls` for `#player-wrapper.ff-mobile:not(.audio-mode)`.
Both are correctly wired per the FR-1 design, yet Dean sees **neither**
surface on-device — only the `.skip-controls` double-tap ±15s skip (via
`.skip-controls`' own overlay, unaffected by either toggle) works.

Confirmed by reading `wireSkipHoldGestures` (`player.js:830-915`): for
`#media-player` (no `onSingleTap` passed, `player.js:1670`), a genuine lone
tap's `touchend` handler does **not** call `preventDefault()` — only a
double-tap classified as `'skip-back'`/`'skip-fwd'` does (`player.js:885`).
So a single tap on the video body is not, on its face, blocked from
bubbling. This narrows (but does not close) the leading hypothesis: iOS's
own tap-to-reveal-auto-hidden-native-controls behavior is not a JS event a
page can observe/replicate — something in the interaction between our
touch-gesture layer's `touchstart`/`touchmove` handling, the
`revealSkipButtons()` overlay's timing, or an iOS-specific auto-hide/
re-arm quirk on inline `<video controls playsinline>` is preventing the
native strip from ever (re)appearing. **Confirming the exact mechanism on
real iOS behavior is the Design stage's #1 job** — this FR's acceptance
criteria are written for the OUTCOME (mobile video must give Dean working
play/pause + seek + fullscreen + speed + download on a real iPhone),
deliberately leaving the mechanism to PE.

### FR-2 — Mobile AUDIO: dead fullscreen button, hold-2x not engaging, no speed

Confirmed: `#fs-btn` remains present for mobile audio (FR-1 design's "Open
forks" section already flagged this as an unresolved AC11-wording fork:
locked decision + byte-identical intent kept it, though AC11's own words
said "absent"). Tapping it runs `enterFullscreen()` (`player.js:1395`) →
`mediaPlayer.webkitEnterFullscreen()` on iOS, which no-ops for an
audio-only element (no video surface to fullscreen) → dead button, matching
Dean's report exactly. `wireSkipHoldGestures(audioBgArt, toggleArtPlayPause)`
(`player.js:1671`) is wired with hold-2x logic identical to video's; the
report that it "doesn't allow for 2x" needs on-device confirmation of
*why* it fails to engage (candidate: gesture/tap classification interacting
with `.audio-mode`'s pointer-events swap) — PE to confirm. No persistent
playback-speed affordance exists on the custom bar at all — **this is
unified with FR-4**, not duplicated here.

### FR-3 — Rotate-to-fullscreen doesn't fire (audio + video)

Confirmed: an `orientationchange` → `enterFullscreen()` listener already
exists (`player.js:1685-1711`, guarded to `state === STATE_FULL` and "not
already fullscreen"), routing through the same `enterFullscreen()` that
calls `mediaPlayer.webkitEnterFullscreen()` on iOS. **Leading root cause is
a genuine iOS platform limitation**: Safari refuses programmatic fullscreen
that is not a direct consequence of a user gesture, and `orientationchange`
does not count as user activation on iPhone. This is flagged explicitly as
an **iOS platform-limit scope call for Dean** — see below.

### FR-4 — Desktop custom bar has no persistent playback-speed control (unified with FR-2's speed gap)

Confirmed: `#player-controls` (`watch.html:344-359`, byte-identical across
all four shells) has `pp-btn`, `mute-btn`, `vol-bar`, `fs-btn`, `pip-btn` —
**no speed control**. The only speed-related element in the DOM is
`#speed-badge` (`watch.html:313`, `style.css:1115-1125`), a **transient,
non-interactive** "2× ▶▶" indicator shown only for the duration of a
press-and-hold gesture (`speedBadge.style.display` toggled in
`player.js:765/772`) — it is not a persistent, clickable control and was
never meant to be one. The v1.21 custom bar replacing native controls simply
never re-exposed a persistent speed affordance for **either** desktop or
mobile, for **either** media type. One unifying fix: add ONE persistent
speed control to `#player-controls`, reconciled with (not conflicting with)
the existing hold-2x `.speed-badge` indicator.

### FR-5 — Desktop click-on-video doesn't toggle play/pause

Confirmed: `#audio-bg-art` has a click-to-toggle path
(`scheduleArtSingleTap(toggleArtPlayPause)` wired at `player.js:1646-1654`
via the desktop `click` listener, FULL-only), but `#media-player` is wired
via `wireSkipHoldGestures(mediaPlayer)` with **no** `onSingleTap`
(`player.js:1670`) — so nothing toggles play/pause on a video click at all.
Add the same YouTube-style click-to-toggle for desktop video, isolated from
`#player-controls` bar clicks and the double-click ±15s skip.

## Design

### Overview

Five surgical fixes, no refactor of the v1.22.0 FR-1 architecture beyond what
FR-1's root cause genuinely forces. All logic lands in `public/js/player.js`;
all styling in `public/css/style.css`; one new control id (`#speed-btn`) is
added byte-identically to the four shells for FR-4. The unifying move: FR-1
**retires the mobile-video native-controls path** and routes mobile video
through the same custom `#player-controls` bar that already works for desktop
and mobile audio — which then lets FR-4's one persistent speed control cover
every case (desktop audio/video, mobile audio, mobile video) with no
double-up, and gives FR-3 its "reliable manual fullscreen button" for free.

### FR-1 — Mobile VIDEO controls (CRITICAL) — root cause + fix

#### Root-cause verdict

The one thing provable from code alone is the **architectural** fault, and it
is sufficient to drive the fix: v1.22.0 made the iOS **native control strip
the SOLE control surface for mobile video, with zero fallback**.
`applyControlsMode()` (`player.js:520-530`) sets the native `controls`
attribute only while `state === STATE_FULL`, and `style.css:3201` hides the
custom `.player-controls` bar for `#player-wrapper.ff-mobile:not(.audio-mode)`.
So if the native strip does not present for *any* reason, the user is left
with nothing — exactly Dean's report.

On the coordinator's specific hypothesis (the `.skip-controls` overlay eating
the taps iOS needs to re-reveal its auto-hidden native strip): **the CSS
partially refutes it.** `.skip-controls` is `inset: 0; z-index: 6;
pointer-events: none` (`style.css:1107-1112`) — the container passes taps
through — and the skip buttons are small left/right pills (`.skip-back
{left:12%}` / `.skip-forward {right:12%}`, `height:54px`,
`style.css:1156-1178`) that only become `pointer-events: auto` for 2.5s after
a tap (`.skip-controls.skip-visible .skip-btn`, `style.css:1183-1186`). The
**center** of the video is never covered by a tap-catching element, and a
lone tap's `touchend` does not `preventDefault()` (confirmed
`player.js:865-914`) — so a center tap does reach `#media-player`. The overlay
is therefore not a clean, provable interceptor of the reveal tap.

What remains is the genuine iOS reality that cannot be observed or driven from
JS: iOS Safari's inline `<video controls playsinline>` controls auto-hide
during playback and re-reveal only via WebKit's internal shadow-DOM tap
handling — behavior a page cannot subscribe to, replicate, or repair. Under
FileTube's non-passive `touchend` + full-surface skip overlay + gesture layer,
that reveal is demonstrably not happening on Dean's device (native strip shows
briefly, hides, never returns), while the `.skip-controls` double-tap keeps
working because it is our own overlay, not the native strip. **The v1.22.0
assumption that "gestures coexist with the native strip like pre-v1.21" was
wrong**: pre-v1.21 the `controls` attribute was hard-coded in the markup
(always present, no JS toggle, no FULL-only gating, no custom-bar-hidden
fallback state).

#### Fix — direction (B): route mobile video to the custom bar

Coordinator offered (A) restrict skip zones + tap-through-center to coax the
native strip back, or (B) give mobile video the touch-sized custom bar.
**Recommend (B).** Rationale:

- **It demonstrably works.** The custom bar is always rendered (not dependent
  on un-observable iOS shadow-DOM reveal). The identical bar already works on
  Dean's device for mobile audio and on desktop for both types.
- **It is the smaller, lower-risk change now** — it mostly *deletes*
  special-casing (the native-`controls` branch, the custom-bar-hide rule, the
  DOCKED re-assert) rather than adding skip-zone geometry math that would
  itself need on-device tuning.
- **(A) still bets on the exact behavior that is failing** (iOS re-revealing
  auto-hidden inline native controls on a center tap) — more iOS guesswork,
  against Dean's "super quick / make it work" directive.
- **It matches Dean's own stated preference:** "if we can meet parity with our
  custom bar visually I like it; if we can't, go native." The custom bar *is*
  the bar he already likes — parity is met — and native inline has
  demonstrably failed, so custom wins by his own rule.
- **Native is still available on demand and reliably:** the bar's `#fs-btn` →
  `enterFullscreen()` → `mediaPlayer.webkitEnterFullscreen()` is a genuine
  user gesture on a real `<video>`, so tapping it enters iOS's **native
  fullscreen player** with the full native chrome (scrubber, playback speed,
  AirPlay, Done). Inline: themed custom bar; fullscreen: native. Best of both.

**AC2 functional set, all delivered via the custom bar:** play/pause
(`#pp-btn`), seek/scrub (`#seek-bar`), fullscreen entry (`#fs-btn` →
`webkitEnterFullscreen`, works on tap), playback speed (FR-4's `#speed-btn`),
download (the existing v1.19 watch-page download-to-device button). **AC3

#### Mechanics

- `player.js` `applyControlsMode()` (`520-530`): remove the
  `setAttribute('controls', '')` branch; it now only toggles the `.ff-mobile`
  class (still needed by CSS for touch-sizing / volume-hiding) and always
  `removeAttribute('controls')`. The native-`controls` path is retired.
- `player.js`: `resolveControlsMode(mediaType, isMobile)` (`348-350`) becomes
  vestigial (every case is now `'custom'`). **Recommend removing it** and its
  entry in `module.exports` (`388`), and repointing its unit test to lock the
  new contract: `applyControlsMode()` never sets `controls`, and `.ff-mobile`
  is still toggled from `isMobileFormFactor()`. (`resolveMobileFormFactor` /
  `isMobileFormFactor` are unchanged — the shared mobile signal is preserved,
  still reused by FR-4/FR-5 below, per the single-helper constraint.)
- `style.css`: delete the mobile-video bar-hide rule (`3199-3203`) and the now
  unnecessary DOCKED re-assert (`3205-3212`) — with the bar no longer hidden,
  the existing `#player-dock` trim rules handle DOCKED for mobile video the
  same way they already do for mobile audio/desktop.
- `style.css` (`3166-3174`): bump mobile `.pc-btn` from `36px` to **`44px`**
  square so every custom-bar button clears the ≥44px touch-target convention
  on mobile video *and* mobile audio.
- `style.css`: extend the mobile-audio volume-hide (`3218-3220`) to also match
  mobile video (`#player-wrapper.ff-mobile #vol-bar, ... #mute-btn`) — iOS
  ignores the `volume` property, so the slider/mute are non-functional on all
  mobile; hiding them declutters the touch bar. `#fs-btn` stays visible for
  mobile video (it works); it is hidden only for mobile audio (FR-2).
- No shell markup change for FR-1: the `controls` attribute was always
  JS-driven, so the four `#player-host-template`s are untouched by this FR.

The `user-select`/`-webkit-touch-callout` guards (`style.css:3190-3197`) stay
as-is. Nothing in the dock / prev-next / autoplay-next / transcode+resume /
Media Session / progress-saving / loop / PiP paths reads `controls`, so
retiring the attribute cannot regress them (they key off
`mediaPlayer`/`host`/`state`, unchanged).

### FR-2 — Mobile AUDIO: dead fullscreen button + hold-2x

- **`#fs-btn` (AC8/AC11/AC12):** hide it for mobile audio via CSS
  (`#player-wrapper.ff-mobile.audio-mode #fs-btn { display: none }`), **not**
  by editing markup. This resolves the dangling v1.22.0 AC11 fork (button
  "absent" for mobile audio) with zero four-shell markup churn and no broken
  `fsBtn` JS references — mirroring exactly how `#vol-bar`/`#mute-btn` are
  already CSS-hidden for mobile audio. It stays present/working for desktop
  audio and mobile video.
- **Hold-2x not engaging (AC9):** leading *concrete, code-grounded*
  hypothesis — the `touchmove` handler (`player.js:852-858`) clears
  `holdTimer` whenever the finger moves more than `MOVE_TOL = 10` px
  (`player.js:748`). A press-and-hold on the cover art with normal thumb
  jitter easily exceeds 10 px before `HOLD_MS = 500` elapses, so `engageHold`
  never fires and 2× never engages. Recommended minimal fix: raise the
  hold-cancel tolerance (e.g. `MOVE_TOL` → ~16 px, or a separate
  `HOLD_MOVE_TOL`) so a stationary-ish press survives to 500 ms. This only
  loosens hold cancellation (the double-tap/skip classification uses tap
  *position*, not movement, so it is unaffected). **This is a hypothesis, not
  a confirmed root cause — Dean's on-device pass is the arbiter (AC13);** the
  change is tiny and localized to the gesture constants either way. Speed
  access itself is delivered by FR-4, not here (AC10).

### FR-3 — Rotate-to-fullscreen — iOS platform verdict

**Verdict (explicit, for Dean's sign-off — AC15): auto-rotate-to-fullscreen is
NOT achievable on iPhone Safari.** iOS Safari refuses programmatic fullscreen
(`webkitEnterFullscreen()` / `requestFullscreen()`) that is not a direct
consequence of a user gesture; an `orientationchange` / `matchMedia` change is
not user activation, so the existing listener's `enterFullscreen()` call
(`player.js:1685-1711`) is silently refused on iPhone. This is a genuine
platform limit, not a FileTube bug. **Do not ship a fake auto-rotate.**

- **Accepted resolution:** the reliable, discoverable **manual** fullscreen
  button restored by FR-1 (the mobile-video custom bar's `#fs-btn` →
  `webkitEnterFullscreen`, which *is* a user gesture and works) is the path to
  fullscreen on iPhone. For audio there is no fullscreen concept at all.
- **Code change: none.** The existing `orientationchange` listener is already
  guarded (`state === STATE_FULL`), `try/catch`-wrapped, and promise-`.catch`
  guarded, so it cannot crash or spuriously loop (AC16 satisfied as-is), and
  it still works harmlessly on platforms that *do* allow it (desktop,
  Android). Removing it would be pure churn. Leave it as best-effort.

### FR-4 — Persistent playback-speed control

**Mechanism (AC18-25): a new `#speed-btn` cycle button on `#player-controls`;
NOT an extension of `.speed-badge`; cycle, not a menu.**

- **New `#speed-btn`, separate from `.speed-badge`.** `.speed-badge` is a
  transient, `pointer-events: none` overlay indicator toggled during hold-2x
  (`style.css:1114-1129`, `player.js:765/772`); repurposing it would conflict
  with that role. `#speed-btn` is a persistent, clickable `.pc-btn` on the
  bar. The two never conflict: hold-2x reads/restores the *live*
  `playbackRate` (`engageHold` saves `prevRate = mediaPlayer.playbackRate`,
  `releaseHold` restores it — `player.js:760-773`), so whatever base rate
  `#speed-btn` set is saved and restored correctly; the badge remains the
  transient "2× ▶▶" hold indicator, `#speed-btn`'s label the persistent base
  rate.
- **Cycle, not menu:** tapping cycles `1× → 1.25 → 1.5 → 1.75 → 2× → 1×`,
  setting `mediaPlayer.playbackRate` and updating the button's `textContent`
  to the active rate (e.g. `1.5×`) — reflecting the current rate (AC19). A
  cycle button needs no popup/positioning/z-index/fullscreen-layer work — the
  smallest correct fix. As a `.pc-btn` on mobile it inherits the 44px sizing
  from FR-1's touch-target bump (AC20).
- **Pure helper (AC23):** add `nextPlaybackRate(current)` (pure lookup over
  the fixed rate list, wrapping) to the exported helpers +
  `test/unit`. The live click handler is a thin mirror that calls it and sets
  `playbackRate` + label.
- **Persistence (recommended, mirrors `ft-loop`/`ft-volume`):**
  `localStorage['ft-rate']`, applied on load so speed survives navigation.
  Optional per the ACs; recommended for feel.
- **Reconciliation with the four cases:** since FR-1 routes mobile video to
  the custom bar, `#speed-btn` is the single speed control for **all** cases —
  desktop audio/video, mobile audio, mobile video. **No double-up (AC21):**
  mobile video no longer uses native controls, so there is no native speed
  menu to collide with inline; the native speed menu only appears if the user
  explicitly enters iOS fullscreen via `#fs-btn`, which is a separate surface.
- **Four-shell sync (AC22):** `#speed-btn` is a new id in
  `#player-host-template`, so it must be added **byte-identical** to
  `public/index.html`, `public/setup.html`, `public/watch.html`, and
  `lib/ytdlp/views/subscriptions.html` — same discipline as v1.22.0's
  `#pip-btn` (all four confirmed to carry the identical block today). Suggested
  placement: immediately before `#fs-btn` in the bar. This four-shell edit is
  folded into T1 (see task plan).
- **CSS (AC24):** `#speed-btn` styles reuse `.pc-btn` and era-theme tokens
  only; the label is plain `textContent`, no hardcoded colors.

### FR-5 — Desktop click-video-to-toggle-play/pause — mechanics

Add a desktop-only `click` listener on `#media-player`, mirroring the existing
`#audio-bg-art` click handler (`player.js:1645-1656`) and reusing the SAME
`scheduleArtSingleTap` / `cancelPendingArtTap` debounce (`player.js:793-806`):

- Guard `if (state !== STATE_FULL) return;` and, for **desktop-only** scoping
  (AC29), `if (isMobileFormFactor()) return;` — reusing the shared mobile
  helper (no second signal). This keeps mobile video's tap semantics
  completely untouched (on mobile, `#pp-btn` and the gesture layer own taps).
- On a genuine single click, `scheduleArtSingleTap(togglePlayPause)` — reuse
  the existing plain `togglePlayPause()` (`player.js:1201` calls it), not
  `toggleArtPlayPause` (whose cover-art glyph is audio-only). On the second
  click of a double-click, `pendingArtTapTimer` is set, so
  `cancelPendingArtTap()` + `return` lets the existing `dblclick` ±15s skip
  win — no play/pause fires from the first click (AC28), identical to the
  audio path.
- **No double-fire from the control bar (AC27):** `#player-controls` is a
  *sibling* of `#media-player` under `.player-container` (not a descendant —
  confirmed in `watch.html:308-359`), so bar-button clicks never bubble to the
  video-surface listener. Structurally safe, no `stopPropagation` gymnastics.
- **AC30:** the reused `scheduleArtSingleTap`/`cancelPendingArtTap` timing
  helpers keep their existing `node:test` coverage; add a regression assertion
  for the new desktop-video path if a pure seam is extracted (the guard logic
  can be locked via a tiny `shouldDesktopVideoTapToggle(state, isMobile)`-style
  pure helper if the reviewers want an explicit unit lock).

### Alternatives considered

- **FR-1 direction (A) — keep native, restrict skip zones + tap-through
  center.** Rejected: it still depends on iOS re-revealing auto-hidden inline
  native controls (the failing behavior), requires on-device geometry tuning,
  and adds fragile overlay logic — larger risk and more iOS guesswork than the
  delete-heavy custom-bar route, for no benefit Dean can't already get via the
  on-demand `#fs-btn` → native fullscreen.
- **FR-4 — extend `.speed-badge` into the persistent control.** Rejected: it
  is a `pointer-events: none` transient hold indicator with overlay
  positioning; overloading it would entangle the persistent and transient
  roles. A separate `#speed-btn` keeps each single-purpose.
- **FR-4 — a speed *menu* instead of a cycle button.** Rejected for this
  round: a popup needs positioning, z-index, and fullscreen-layer handling —
  more surface than a cycle button, against the keep-it-small directive. A
  cycle button meets AC19 (real choice + reflects active rate).
- **FR-2 `#fs-btn` — delete from markup.** Rejected: touches all four shells
  and breaks `fsBtn` JS references for no gain over a scoped CSS hide.

### Risks and mitigations

- **Risk:** FR-2 hold-2x fix is hypothesis-driven (no iOS here to confirm). →
  **Mitigation:** the change is a tiny, safe tolerance loosening; Dean's
  on-device pass is the arbiter; if it is a different cause, the fix stays
  localized to the same gesture constants.
- **Risk:** removing `resolveControlsMode` disturbs the AC5 regression lock. →
  **Mitigation:** repoint the test to assert the new contract (all cases
  custom; `applyControlsMode` never sets `controls`; `.ff-mobile` still
  toggled) — an equivalent, stronger lock.
- **Risk:** `#speed-btn` four-shell drift. → **Mitigation:** byte-identical
  addition + the existing `#pip-btn`-style four-shell parity test as FR-4's
  gating check.
- **Risk:** mobile-video custom bar + gesture layer interfere (double play/
  pause, etc.). → **Mitigation:** this is exactly desktop video's proven
  today-behavior (custom bar + `wireSkipHoldGestures(mediaPlayer)`); no
  `onSingleTap` is added on mobile, so no tap-toggle conflict; Dean's iOS pass
  confirms feel.

### Performance impact

No expected impact on any `docs/RELIABILITY.md` budget: no new network
requests, no new timers on hot paths, no new dependencies; the changes are
attribute/class toggles, a handful of listeners, and CSS. If anything, mobile
video does slightly *less* work (no native-controls shadow tree churn).
