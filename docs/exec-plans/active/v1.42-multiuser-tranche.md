# v1.42–v1.44 — Multi-user tranche: SQLite foundation → users/auth → RBAC libraries

Status: Intake LOCKED (9 owner answers, 2026-07-14). Design authored at the
Fable→successor handoff and passed through an adversarial DESIGN review the
same day: initial verdict REQUEST CHANGES (2 CRITICALs — the WAL-unsafe
migration recipe and a false test-blast-radius claim — plus 6 warnings, 3
suggestions), all 11 findings applied, delta verdict APPROVE. This document
is the implementation spec AND the reviewers' spec for three sequential
releases. The implementing model should treat every "locked" item as
owner-decided and every "verify" item as a mandatory design-stage check
before writing code against it. Each RELEASE still gets its own full
two-reviewer code gate — the design review does not substitute for it.

Owner vision (Dean, verbatim intent): a private YouTube/private library he can
share with family and friends — per-user accounts, a kid-safe account for his
daughter subscribed only to kid-friendly channels, fully private book
libraries, and restore-able config. "It's just not safe to share publicly at
all… especially with no user management."

## Locked intake decisions (Dean, 2026-07-14)

1. **Storage engine: `node:sqlite` (built-in `DatabaseSync`).** Owner's bar is
   "boring plain-jane long-term stability." Verified 2026-07-14: works on both
   project Node versions (v22.23.1, v24.14.0), including `backup()`; BOTH still
   print `ExperimentalWarning` at require time. Accepted with two mitigations:
   (a) every `node:sqlite` API touch lives in ONE adapter module
   (`lib/db/sqlite.js`) with a lock test greping the rest of the tree, so an
   API change across Node majors is a one-file fix; (b) the flip trigger is
   documented — if the gate or reality refutes node:sqlite maturity, the
   fallback is `better-sqlite3` (accepting the native-addon dep + musl build).
   The one-line boot warning ships as-is (visible honesty > suppression).
2. **Sequencing: storage first.** v1.42 SQLite + instance backup/restore
   (single-user, zero behavior change) → v1.43 users/auth → v1.44 RBAC +
   scoped libraries. Rationale: retrofitting per-user scoping onto flat JSON
   and then migrating again would pay the migration tax twice; the SQLite
   schema is designed for users from day one.
3. **Auth: cookie sessions, zero new deps.** `node:crypto` scrypt password
   hashing; HMAC-SHA256-signed HttpOnly cookie (~30-day expiry) with
   per-request user-row re-check (instant revocation despite long expiry —
   tasksync's pattern). Cookie (not Bearer JWT) is load-bearing: `/video/:id`,
   `/thumbnail/:id`, `/book/:id/file` etc. are plain `src` URLs that cannot
   send Authorization headers. No self-registration; first boot with zero
   users serves a create-admin page.
4. **Roles: `admin` | `member`** (two values, like tasksync). Kid safety is
   NOT a third role: it is a member whose folder grants cover only
   kid-friendly roots, plus a per-user `can_manage_subscriptions` flag.
5. **Subscriptions: shared physical downloads, per-user subscription lists.**
   One copy on disk per channel (current dirs unchanged); each user follows a
   subset, which drives their Subscriptions view. Adding channels gated by
   admin or the per-user flag.
6. **Per-user settings:** viewing prefs (`defaultView`, `defaultSort`,
   `autoplayNext`, `backgroundAudioForVideo`, `mobileCustomPlayer`) move to
   the user record. Theme/era/icon prefs ALSO mirrored server-side on the user
   (owner: "cool if cheap to bundle" — it is: keys in `settings_json`);
   localStorage remains the device-local fast path and pre-login source.
   Instance settings (folders, scan, cache, TTS, custom logo) stay global +
   admin-gated.
7. **Backup, two layers, tasksync-shaped:** (a) admin **instance bundle** —
   schema-versioned JSON of everything incl. users + password hashes (UI must
   flag the file as sensitive); restore = wipe-and-replace inside ONE SQLite
   transaction with a self-lockout guard (the restoring admin must exist as
   admin in the bundle). (b) **per-user export/import** — one user's
   settings/likes/progress/pins/subscription-follows as portable pointers.
   Media files are never in any bundle.
8. **Exposure model:** always-on VPN today, internet-for-friends later.
   Design for HTTPS-behind-reverse-proxy now (Secure cookie when proxied,
   `trust proxy` env-gated); the hardening items (login rate limit, path
   scrubbing) are v1.44 acceptance criteria, not nice-to-haves.
9. **Migration is automatic AND non-destructive — parallel-run is a hard
   requirement.** Dean will run old-tag prod and new-tag beta SIMULTANEOUSLY
   against the same media content with different DATA_DIRs. Therefore db.json
   is a READ-ONLY input: never renamed, never rewritten, never deleted. Old
   tags must keep working against it forever.

## v1.42 — SQLite persistence + instance backup/restore

**Principle: swap the persistence layer behind the existing seam; change no
behavior.** The in-memory DB object, the `updateDatabase(mutator)` contract
(synchronous mutators, mutex chain), `getCachedDatabase`, the `ensure*`
namespace owners, and the exported-`saveDatabase` test seeding contract all
survive byte-identical in shape.

### Storage layout (SQLite, WAL mode, foreign_keys ON)

- `DATA_DIR/filetube.db`. Schema versioned via `PRAGMA user_version` with a
  forward-only migration runner in the adapter.
- **Collection tables** — per-key rows so a one-item mutation writes one row,
  not 175 KB (an actual improvement over today's whole-file rewrite):
  `doc_kv(namespace TEXT, key TEXT, json TEXT, PRIMARY KEY(namespace, key))`
  for: `metadata`, `progress`, `deleteTombstones` (drift correction #1 —
  losing it resurrects user-deleted files), `books.items`,
  `books.progress`, `books.audio`, `ytdlp.downloadMeta`,
  `ytdlp.channelAvatars` — **and `viewCounts`**: the v1.42 import EXTRACTS `metadata[id].viewCount` into
  its own namespace (review finding #8). It is the design's own
  classification — the one non-rebuildable field embedded in a rebuildable
  namespace — and it is demonstrably clobber-prone in place: the Phase-2
  scan merge (server.js:969-977) does not carry it forward, so a view
  recorded mid-scan is reverted, and a changed-file re-init
  (server.js:3129-3143) zeroes it. This is the 5-strike persist-gate class;
  v1.42 is the cheapest moment ever to fix its placement, and v1.43 is
  about to increase its write traffic. The view/read paths move to the new
  namespace in v1.42 (a small, behavior-preserving change with a dedicated
  mid-scan-view regression test).
- **Singleton docs** — `doc_single(name TEXT PRIMARY KEY, json TEXT)` for:
  `folders`, `folderSettings`, `settings`, `liked`, `books.folders`,
  `books.settings`, `books.pins`, `ytdlp.subscriptions`, `ytdlp.pins`,
  `ytdlp.allowMembersOnly` (and any future namespace scalar). These are
  small arrays/objects mutated as wholes today.
- `users` and per-user tables are CREATED in v1.42's schema (empty) so
  v1.43/44 are additive `user_version` migrations, not restructures.

### Load/save mechanics

- `loadDatabase()` (inside the write lock, as today): SELECT-all → assemble
  the current object shape **modulo two documented transforms (review F4 +
  T1 implementation note): (1) metadata items carry NO `viewCount` field;
  view counts live in the top-level `viewCounts` namespace**
  (`db.viewCounts[id]`), with the view-increment + every read site moving
  in the same task; **(2) an EMPTY doc_kv namespace (`{}`) has zero rows
  and assembles as ABSENT** — observationally identical because the
  load-time backfills / lazy `ensure*` creators re-supply `{}` before any
  consumer reads it. AC1's fidelity check is deep-equal modulo exactly
  these two transforms. Fresh object per acquisition, same as
  today's parse-from-disk. The adapter also subsumes loadDatabase's
  initial-create `fs.writeFileSync` (server.js:208) — the one write that
  currently bypasses `saveDatabase`.
- `saveDatabase(db)`: keep a per-row serialized-JSON snapshot map from the
  last commit; on save, re-serialize and diff → write ONLY changed/inserted/
  deleted rows in ONE transaction; then `dbCache = db` replace-by-reference
  in the same sync tick (coherency contract unchanged). At 175 KB total the
  diff serialization cost is sub-millisecond-class; verify with a timing
  test at the ARCHITECTURE.md projected 2000-item scale.
- Crash safety: the SQLite transaction replaces tmp+fsync+rename. The orphan
  `.tmp` sweep (`cleanupOrphanDbTmp`) is retired for the SQLite path.
- `lib/db/sqlite.js` is the ONLY module that may `require('node:sqlite')` —
  enforced by a source-lock test.
- node:sqlite API precision (verified on both binaries, review finding
  #11; retained as API record — the backup() API is NOT used in v1.42,
  see the backup/restore section for the empirical unsafety evidence):
  `backup` is the MODULE-LEVEL async `backup(db, path)` returning a
  Promise — there is no `db.backup()` method. The adapter prepares its
  statements only AFTER the `user_version` migration runner completes
  (delta review F8c — preparing against pre-migration schema is a
  stale-handle class). Nested `BEGIN` throws
  ("cannot start a transaction within a transaction") — the adapter needs a
  no-nesting guard. The restore path uses a dedicated exclusive-replace
  entry point that also REBUILDS the diff snapshot map afterwards (the diff
  is self-healing regardless, but the write accounting would lie
  otherwise).
- Test migration plan (review finding #2, re-censused per drift
  correction #4): the adapter exports a `readPersistedDatabase()` test
  helper; the **43** direct-I/O test files are migrated mechanically —
  `readFileSync(DB_FILE)` assertion sites move to the helper; the 7
  raw-`writeFileSync` seeders are audited individually (the 4
  corrupt/legacy files — database, liked, progress-coalescer,
  migrate-oneoffs — become MIGRATION-PATH tests rather than being
  deleted). `scripts/diagnose-delete.js` is handled per drift
  correction #4.

### Migration (boot order — the parallel-run contract)

1. `DATA_DIR/filetube.db` exists → use it. If db.json ALSO exists, log one
   line that it is present and ignored (it belongs to the old-tag
   instance). **Exception (review F2): if filetube.db is schema-empty
   (zero rows in every table, `user_version` current) while a non-empty
   db.json sits beside it, log LOUDLY** — that shape is the fingerprint
   of a stranded import (or an aborted manual copy: a 0-byte file opens
   as a valid empty SQLite DB).
2. Else if `db.json` exists → import via the EXACT sequence below.
   **db.json is opened read-only and is byte-for-byte untouched — assert
   this in a test.** Log an import summary (row counts per namespace).
   **Corrupt input (review F2 — CRITICAL): the importer uses a STRICT
   `JSON.parse` and NEVER `loadDatabase`'s reset-to-fresh recovery path.
   On parse failure it must NOT create filetube.db — abort boot with a
   loud, actionable message naming the file and the parse error.**
   Today's recovery semantics are per-call (a transient read failure
   heals on restart); an importer that inherited them would commit a
   valid EMPTY filetube.db once, and boot rule #1 would then ignore the
   intact db.json forever — permanent, silent library stranding.
   **Ordering (review F3): the importer stores namespaces RAW (exactly
   as parsed); the adapter's load-time assembly re-applies today's
   backfills verbatim on every load** (`withDefaultSettings`, the
   top-level key backfills, with `ensureYtdlp`/`ensureBooks` staying
   call-site-owned lazy creators, as today) — so a legacy-shape db.json
   (the real dev copy has no `liked`/`deleteTombstones`/`books`/`ytdlp`
   keys) imports and then behaves exactly like today's load+backfill,
   and future DEFAULT_SETTINGS keys keep appearing without a re-import.
3. Else → fresh empty schema.

**Import sequence (adversarial review finding #1 — the naive
tmp+fsync+rename recipe LOSES THE ENTIRE IMPORT under WAL, proven
empirically: `-wal`/`-shm` sidecars keep the tmp name across a rename, so
the renamed `.db` is an empty shell; and `new DatabaseSync(path)` creates
the file eagerly, so a crashed import leaves a partial tmp that boot-loops
a re-import):**

1. Unlink any leftover `filetube.db.tmp`, `filetube.db.tmp-wal`,
   `filetube.db.tmp-shm` from a prior crashed import.
2. Create/import into `filetube.db.tmp` inside one transaction; COMMIT.
3. `close()` the tmp DB — this checkpoints and removes the sidecars.
4. `fs.openSync` + `fsyncSync` the tmp file fd.
5. `renameSync('filetube.db.tmp', 'filetube.db')`.
6. WAL journal mode is persistent in the file: set it on first open of the
   FINAL path (or on the tmp before the clean close — either is safe
   because step 3 checkpoints). If `PRAGMA journal_mode=WAL` does not
   return `wal` (e.g. network FS without shared-memory semantics), fall
   back to `DELETE` mode with a one-line log; document that DATA_DIR is
   expected to be local storage.

A regression test must exercise the actual rename path and then SELECT the
imported rows back (fidelity through the rename, not just through the
import transaction).

**Pre-upgrade dry-run tool (owner request, 2026-07-17):**
`scripts/migrate-check.js` — runs the EXACT production import code
(importDbJson; one code path, no drift — the v1.41.7 preview/executor
lesson) against a throwaway temp dir, verifies round-trip fidelity modulo
the documented transforms, sha256-verifies db.json untouched, prints
per-namespace row counts, exits 0/1. Dean points it at his real prod
db.json BEFORE upgrading; the real migration remains the automatic
first-boot import.

Downgrade path (disclose in release notes): db.json remains valid for old
tags forever; the SQLite file is simply ignored by them. State diverges
between instances from the moment the beta first boots — expected and fine.

**Operational note for Dean's beta (broadened per review finding #7):** the
shared media/download folders are mutated by more than the scheduler —
`.ytdlp-archive.txt` and `.ytdlp-skiplist.txt` live in the download root
(appendFileSync), failed-download cleanup deletes yt-dlp intermediates in
the shared channelDir BY FILENAME SHAPE (it can delete prod's in-flight
`.part`/fragment files), and delete/move endpoints mutate prod's actual
media. So from the beta instance: **no downloads (scheduled OR one-off), no
Skip-button use, no deletes, no moves.** Read/watch/like/progress are safe.
Thumbnail/TTS caches are per-DATA_DIR; the beta re-extracts its own copies
(one-time CPU cost, disclosed, NOT a regression — the media files
themselves are shared).

### Instance backup/restore (ships here; the migrator and it are one machinery)

- `GET /api/admin/backup` → `{ schema: 'filetube-backup-v1', exportedAt,
  appVersion, <every namespace>, customLogo: { light?: {mime, b64},
  dark?: {mime, b64} }, users: [] }` (users empty until v1.43; the
  section is reserved in the format from day one; logo bytes included
  per drift correction #3). The bundle is assembled from SELECTs
  enqueued on the write chain (consistent by single-threaded
  serialization). **The `node:sqlite` module-level `backup()` API is NOT
  used anywhere in v1.42** — empirically (2026-07-17, both binaries) it
  is unsafe under concurrent same-process writes: one run silently
  produced a 2-row snapshot of a 1002-row source, another rejected with
  `ERR_SQLITE_ERROR: not an error`. If a future release wants .db-file
  export it must serialize `backup()` behind the write chain and
  re-verify.
- Logo restore ordering (review F6 — the upload route's "atomicity" is
  single-writer FIFO coherence, not crash atomicity, and file writes can
  never be transactional with SQLite): inside the exclusive restore
  section, write `.bin` files tmp+rename FIRST, unlink a variant's stale
  `.bin` when the bundle lacks it, THEN commit the SQLite transaction
  (which carries the mime keys). Crash between the two leaves new bytes
  under the old mime key for one boot — bounded, cosmetic, DISCLOSED in
  the release notes rather than claimed away.
- Restore coherency (review F5): the exclusive restore section must,
  besides rebuilding the diff snapshot map: clear `pendingProgress` AND
  `pendingBookProgress` (and cancel their timers), and invalidate
  `dbCache` — otherwise hot GETs serve the pre-restore library until the
  next unrelated write, and a pre-restore progress ping whose id exists
  in the restored set would flush OVER the just-restored position.
- `POST /api/admin/restore` → validate schema string + field-level sanity,
  then wipe-and-replace inside one transaction. v1.43 adds: bundle must
  contain the requesting admin as an admin (self-lockout guard), and the
  restore endpoint becomes admin-only. In v1.42 these endpoints are as open
  as the rest of the API (no auth exists yet) — DISCLOSED in release notes,

## v1.43 — Users + auth (everyone still sees everything)

### Schema (additive migration)

- `users(id INTEGER PK, username TEXT UNIQUE COLLATE NOCASE, display_name
  TEXT, password_hash TEXT, role TEXT CHECK(role IN ('admin','member')),
  can_manage_subscriptions INTEGER DEFAULT 0, settings_json TEXT DEFAULT
  '{}', token_version INTEGER DEFAULT 0, disabled INTEGER DEFAULT 0,
  created_at TEXT)`. The table (including `token_version`) is born complete
  in v1.42's initial schema — v1.43's migration only starts populating it.
- `password_hash` self-describing format: `scrypt$N$r$p$salt_b64$hash_b64`
  (params recorded per-hash so cost can be raised later without breaking old
  hashes; login re-hashes on success if params are below current).
- Per-user state tables: `user_progress(user_id, media_id, timestamp,
  duration, updated_at, PK(user_id, media_id))`, `user_liked(user_id,
  media_id, liked_at, PK(user_id, media_id))`, `user_book_progress`,
  `user_book_pins`, `user_channel_pins`. Global `viewCount` stays an
  instance aggregate (incremented on any user's qualifying view — the
  YouTube-like public count); per-user resume comes from `user_progress`.

### Sessions and middleware

- Session cookie = `base64url(payload).hmac_sha256(secret, payload)`,
  payload `{uid, tv, iat, exp}` (~30d). **Cookie NAME is per-instance**
  (review finding #4): cookies are host-scoped, not port-scoped, and Dean
  runs prod+beta on one host every release — a shared `ft_session` slot
  with different secrets means the two instances permanently log each
  other out. Name = `ft_session_<short hash of DATA_DIR (or PORT)>`.
  Secret: 32 random bytes auto-generated to `DATA_DIR/session-secret` on
  first boot (0600), env override `FILETUBE_SESSION_SECRET`; fail-closed on
  short/placeholder values (tasksync's boot validation). Constant-time
  compare (`crypto.timingSafeEqual`).
- `tv` = per-user `token_version` column, bumped on every password change
  (review finding #5 — without it, a stolen cookie survives a password
  reset for its remaining ~30 days; disable/re-enable would re-admit it).
- Per-request: verify sig + exp → SELECT user row → compare `tv` →
  `req.user`. Disabled, deleted, or version-bumped user → 401 immediately
  (revocation genuinely instant, including after password reset).
- ONE `app.use` gate installed after `express.json()` and BEFORE
  `express.static` + `ytdlp.registerRoutes` (server.js:3647-3684), so all
  ~77 routes + static + streams are covered by default.
- The create-admin route is **`/welcome`**, NOT `/setup` (review finding
  #6): `public/setup.html` + `js/setup.js` are the EXISTING Settings page
  served by `express.static`, one careless `startsWith('/setup')` away
  from an unauthenticated settings shell. A dedicated test asserts
  `/setup.html` 401s/redirects pre-login. `POST /api/auth/setup` works
  only while user count is 0, race-guarded in one transaction.
- Unauthenticated allowlist is EXACT-MATCH and enumerated here (the route
  census cannot see `express.static` paths, so this list is spec, not
  discovery): `/login`, `POST /api/auth/login`, `/welcome`,
  `POST /api/auth/setup`, `/logo`, `/manifest.webmanifest`, the manifest's
  icon files, `/favicon.svg`, `/favicon.ico`, `/css/style.css`,
  `/fonts/*` (woff2), and — if the login page reuses the theme/era boot —
  `/js/common.js`. None of these leak data (static client code; all data
  comes from APIs that 401), but the implementer verifies there are no
  static JSON/fixture files under `public/` before shipping. Pages
  redirect to `/login`; API and byte routes return 401 JSON.
- Cookie flags: HttpOnly, SameSite=Lax, Path=/; `Secure` when the request is
  https (via `app.set('trust proxy', ...)` gated by `FILETUBE_TRUST_PROXY`).
  Plain-HTTP LAN/VPN keeps working (locked answer 8).

### Adoption of existing single-user state

At create-admin time (`/welcome`), copy the global user-state (progress,
liked, book progress, pins) into the new admin's per-user rows in the same
transaction. Global progress rows are `{timestamp, duration}` with NO
`updatedAt` (review finding #11) — synthesize `updated_at` = setup time on
copy, so the per-user import's "newer updatedAt wins" merge is
well-defined. The global rows are retained as a frozen pre-auth record but
all reads/writes go per-user from then on — no dual-write, no read-through
fallback (review finding #6's divergence concern: a fallback that consults
global rows after adoption is a bug farm; the cutover is total and tested).
Deterministic, idempotent (setup runs once), covered by a dedicated test.

### Client

- Era-styled `/login` page (username/password, error states); logout +
  current-user chip in Settings/nav. No self-registration UI. Admin user
  management UI in Settings (create user, reset password, disable, role,
  subscriptions flag).
- Viewing prefs read/write the user record when logged in; theme/era/icon
  writes mirror to `settings_json` (localStorage stays the immediate
  source, so pre-login shell and multi-device divergence keep working).

### Backup changes

- Instance bundle gains `users` (incl. hashes — UI download dialog flags
  sensitivity); restore adds the self-lockout guard; both endpoints become
  admin-only.
- Per-user export: `GET /api/me/export` → `{ schema: 'filetube-user-v1',
  exportedAt, profile, settings, progress, liked, bookProgress, pins,
  follows: [channel URLs] }` — portable pointers, importable via `POST
  /api/me/import` (merge semantics: newer `updatedAt` wins; document them).

### v1.43 acceptance criteria (abridged — full list at implementation)

- Every route 401s/redirects without a valid cookie EXCEPT the exact
  allowlist (route-census test: enumerate registered routes at runtime,
  probe each unauthenticated, assert against the allowlist).
- Streams/thumbnails work in `<video>`/`<img>` tags with only the cookie.
- scrypt verify round-trip + params-upgrade-on-login test; timing-safe
  compare in place.

  logged (rate limit is v1.44).

## v1.44 — RBAC: scoped libraries, per-user subscriptions, hardening

- `user_folder_grants(user_id, root_path TEXT, kind TEXT CHECK(kind IN
  ('media','books')))`. Grants key on the configured root path string
  (boring; if a root is re-pathed the admin re-grants — disclosed). Admin
  implicitly has all grants.
- **Every read surface filters by grants:** list/search/related/prev-next/
  shuffle-ctx/home rows/downloads views + the byte-serving endpoints
  (media id → `item.rootFolder` ∈ grants, else 404 — not 403, don't confirm
  existence). The route census from v1.43 grows a third column:
  filtered / admin-only / public — EVERY route must be classified; the gate
  reviews the census, not the vibes.
- Per-user subscriptions: `ytdlp.subscriptions` becomes the instance channel
  registry (physical downloads unchanged); `user_follows(user_id,
  channel_id)` drives each user's Subscriptions view;
  `can_manage_subscriptions` (or admin) gates add/remove of registry
  entries. A member adding an already-registered channel just follows it.
- **Path scrubbing:** non-admin responses never contain absolute paths —
  strip/replace `filePath`, `rootFolder`, `channelDir`, config roots with
  opaque ids/display names. Test: recursive scan of every non-admin JSON
  response in the census for `/`-rooted and `\\`-rooted strings (allowlist
  for URL fields).
- Hardening (v1.44 ACs, per locked answer 8): in-memory login rate limiting
  (per IP+username token bucket), Secure-cookie-behind-proxy verified
  end-to-end, `/security-review` skill pass as part of the gate.
- Kid-account recipe in README: member + only kid-friendly roots granted +
  subscriptions flag off. (This is the owner's daughter use-case — it must
  be a documented recipe, not tribal knowledge.)

## Known risks (name them so the gate hunts them)

1. node:sqlite experimental label — adapter isolation + lock test; flip
   trigger documented above.
2. v1.42's diff-based save: a mutator that mutates the PREVIOUS committed
   object (dbCache) instead of the fresh load would fool the diff — the
   existing contract already forbids this (and a 2026-07-17 sweep found
   the codebase clean: both coalescers flush through fresh loads, no
   getCachedDatabase mutation anywhere). Dev/test guard: **a throwing
   Proxy around the cached object, NOT deep-freeze** (review F7 —
   server.js is not strict mode, so writes to a frozen object are SILENT
   no-ops that would mask the exact bug the guard exists to catch and
   make dev behavior diverge from prod).
3. The v1.44 filter sweep is the highest-leak-risk surface in the project's
   history: one missed endpoint = cross-user data exposure. The route census
   is the control; it must be generated from the live Express router, not
   hand-maintained.
4. Auth middleware ordering vs `express.static`: the SPA shell JS must load
   for the login page without leaking authed views' data (views fetch via
   APIs, which 401 — verify no static JSON/fixtures under `public/`).
5. iOS PWA cookie behavior in standalone mode (ITP) — Dean's on-device
   probe before v1.43 ships wide.
6. Simultaneous prod+beta: shared yt-dlp archive race — operational note
   above; NOT solved in code this tranche.
