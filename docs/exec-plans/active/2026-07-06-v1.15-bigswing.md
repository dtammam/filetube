# v1.15.0 "UI Polish + yt-dlp Enhancements + Infra" (Big Swing)

## Goal

Ship one cohesive v1.15.0 round that polishes the core UI (folder drag-and-drop
reordering, mobile layout fixes), extends the optional yt-dlp subscription
module (a header one-off download entry point, per-subscription Shorts
skipping, nicer on-disk filenames, one-off re-download overwrite), and
improves infra/test coverage (configurable transcode directory + opt-in
higher-CRF, a best-effort PWA icon, broadened core test coverage) — without
degrading FileTube when the yt-dlp module is disabled and without regressing
any existing behavior.

## Scope

**UI polish**
1. Folder drag-and-drop reordering (Setup folder list AND left sidebar), with
   a keyboard/tap fallback, covering both real folders (`db.folders` order)
   and the synthetic Downloads folder (`folderSettings[downloadDir].order`).
2. Mobile UI scale polish: fix the home sort-controls row (heading + sort
   `<select>` + "Shuffle again" button) overflowing/clipping on mobile, and
   tighten mobile video-card sizing.

**yt-dlp module (optional, gated on `FILETUBE_YTDLP_ENABLED`)**
3. A header download button + compact modal for one-off downloads (Dean-locked
   placement), gated exactly like the existing `/subscriptions` nav-link
   injection; reuses the existing one-off endpoint, dropdowns, and status
   polling. The existing `/subscriptions` one-off form stays as-is.
4. A per-subscription "skip Shorts" toggle, implemented via a yt-dlp
   `--match-filter` argument, default off (download everything).
5. Nicer, more readable on-disk filenames (spaces allowed) while preserving
   the SF4 path-confinement guarantee, archive-dedup uniqueness, and
   `cleanDisplayTitle`.
6. One-off re-download of an already-archived/already-downloaded URL now
   overwrites instead of being silently skipped, scoped to the one-off path
   only; subscription dedup is unchanged.

**Infra**
7. A configurable `TRANSCODE_DIR` env var (default unchanged:
   `data/transcoded`) and an opt-in higher-CRF option (default unchanged: 23)
   for smaller transcoded files, without breaking size-cap eviction,
   age-retention, or `lastServedAt` bookkeeping.
8. A best-effort PWA home-screen icon PNG, added without any new dependency;
   deferred (with a tracked note) if no clean no-dependency path exists.
9. Additive core test coverage for the highest-value untested paths
   (scan/config/transcode-eviction pure logic + HTTP smoke), including
   coverage for the new config introduced by items 4/5/6/7.

## Out of scope

- No new runtime dependencies for ANY of the 9 items (this generalizes item
  8's no-dep constraint project-wide for this round) — verified against
  `docs/CONTRIBUTING.md`'s "no extra runtime deps" standard.
- No WebSocket/SSE transport — the one-off modal's live status (item 3)
  keeps using the existing `GET /api/subscriptions/status` polling.
- No change to subscription download-archive dedup behavior — item 6's
  archive bypass is one-off-only; subscriptions still carry
  `--download-archive` and "deleted stays gone" (D3) is preserved.
- No removal of the `--restrict-filenames`-equivalent path-safety guard for
  item 5 — SF4 confinement (`resolveChannelDir`/`isPathUnder`/
  `realpathUnderChannelDir`/quarantine) stays intact regardless of which
  filename template direction the PE picks.
- No writing the synthetic Downloads folder into `db.folders` for item 1 —
  it remains display-only via `extraScanRoots`; its order persists solely via
  `folderSettings[downloadDir].order` (C3/C7/E1 invariants intact).
- No silent change to existing transcode quality for item 7 — CRF default
  stays 23; the higher-CRF option is strictly opt-in via env.
- No heavy image dependency for the PWA PNG (item 8) — attempt no-dep, else
  defer explicitly with a tracked tech-debt note (not a silent drop).
- Item 9's tests are additive only — no rewriting, relocating, or deleting
  existing tests; FFmpeg/yt-dlp binaries stay out of the automated suite.
- No global (cross-subscription) skip-Shorts default in this round (item 4
  is per-subscription only; a global toggle is explicitly deferred, not
  built).
- No de-emphasis or removal of the existing `/subscriptions` one-off form
  (item 3) — the header modal is additive/primary, the existing form stays.
- Cross-check: none of the above out-of-scope items conflict with
  `docs/CONTRIBUTING.md`'s mandatory standards (tests-ship-with-every-change,
  no new deps, vanilla DOM, lint 0) — item 9 explicitly brings test coverage
  back in scope for this round rather than deferring it, and every yt-dlp
  item (3/4/5/6) still ships its own regression tests per the Definition of
  Done.

## Constraints

- **Disabled-module byte-identical no-op is sacred (items 3, 4, 5, 6).** With
  `FILETUBE_YTDLP_ENABLED` off, FileTube stays byte-identical to today: no new
  routes, no new UI, no header button/modal markup/assets leaked, no
  skip-shorts UI. Item 3's header button is gated exactly like
  `injectSubscriptionsNavLinkIfEnabled` (`public/js/common.js:601-648`): probe
  `/api/subscriptions/health`, inject only on `ok`, fail closed on
  404/network error.
- **Path-confinement / SF4 stays intact (item 5).** `--restrict-filenames`
  (`args.js:390`) is defense-in-depth, not to be simply dropped without an
  equivalent. `resolveChannelDir` + `isPathUnder`/`realpathUnderChannelDir`/
  `quarantineEscapedDownloads` (`args.js:170-239`, `index.js` quarantine)
  stays. Any nicer-filename template must preserve path-traversal confinement
  AND keep id-based uniqueness so `--download-archive` dedup +
  `cleanDisplayTitle` still work.
- **Subscription dedup is unchanged (item 6).** The one-off archive-bypass
  affects `runOneShot` (`index.js:694`) only. Subscriptions still use
  `--download-archive` (`args.js` `buildYtdlpDownloadArgs`/
  `buildYtdlpListArgs`) and D3 ("deleted stays gone") is preserved.
- **Arg-array / no-shell spawn discipline (item 4).** Any `--match-filter`
  value that reaches argv is its own argv element via the pure builders in
  `args.js` — never concatenated, never a shell string. Follows the
  `shouldSkip`/`normalizeQuality` allowlist-and-re-assert posture; a
  `skipShorts` value is validated at the store boundary
  (`validateSubscriptionInput`/`updateSubscription`) AND re-asserted at build
  time.
- **Synthetic "Downloads" folder stays out of `db.folders` (item 1).** DnD
  reordering of that entry persists via `folderSettings[downloadDir].order`
  (the v1.13 up/down path), never by writing it into `db.folders`. Real
  folders reorder via `db.folders` positional order.
- **Transcode-cache housekeeping must not break (item 7).** A configurable
  `TRANSCODE_DIR` must keep size-cap LRU eviction, age-retention sweep, and
  `lastServedAt` logic working — they all key off the transcode dir
  (`server.js:24, 200, 307-320, 382-400, 1722-1745`). CRF is hardcoded `23` at
  two encode sites today (lazy on-disk transcode `server.js:691`; live
  desktop pipe `server.js:2038`) — a higher-CRF option must cover at least the
  cached on-disk site; PE decides whether the live-pipe site is in scope.
  Defensive: mkdir -p, writability check, fail clearly rather than crash.
- **Coding standards (`docs/CONTRIBUTING.md`):** vanilla DOM (no
  framework/bundler), CommonJS server, 2-space/semicolons/single-quotes;
  prefer `textContent` over `innerHTML` for new dynamic strings (XSS-safe —
  especially item 3's modal status + item 1's DnD labels); `node:test` for
  new/regression tests; lint 0 warnings; NO new runtime dependencies.
- **Era-theme system preserved.** Any new CSS (items 2, 3) uses existing
  CSS custom-property tokens; no hardcoded colors that break themes.
- **Additive / no regressions** to existing home sort/sidebar/setup/watch
  behavior, the existing Setup up/down folder reorder (item 1 augments, keeps
  the fallback), the existing `/subscriptions` one-off form (item 3, kept),
  and the disabled-module byte-identical guarantee.
- **Item 4's end-to-end threading gotcha:** a new `skipShorts` field must be
  threaded through `store.js` `validateSubscriptionInput` + `updateSubscription`
  + `ensureYtdlp` backfill (`undefined` -> `false`), the PATCH edit handler in
  `lib/ytdlp/index.js`, and the subscriptions UI toggle — mirroring the
  existing `allowMembersOnly` (members-only) toggle. A field not added to the
  store validators/allowlist is silently dropped.
- **Item 9:** additive tests only; do not rewrite/relocate existing tests;
  keep FFmpeg/yt-dlp binaries out of the automated `node:test` suite; target
  risky/untested CORE paths, not 100% coverage.
- **Node 22** is the toolchain standard for build-verify and CI.

## Functional requirements

**FR-1 — Folder drag-and-drop reordering.** Today: folder order has two
sources — real folders reorder via `db.folders` positional order using
up/down buttons in Setup; the synthetic Downloads folder reorders via
`folderSettings[downloadDir].order` (display-only, never in `db.folders`);
the sidebar renders client-side (`renderSidebarFolders`) reflecting that
order. Target: drag-and-drop reordering is available in both the Setup
folder list and the left sidebar, persisting through the same two existing
mechanisms (unchanged persistence model), with a keyboard/tap-accessible
fallback retained for users/devices that can't drag.

**FR-2 — Mobile UI scale polish.** Today: the home sort-controls row
(heading + "Newest first"-style `<select>` + the v1.14.0 "Shuffle again"
button) overflows the mobile viewport, cutting off the shuffle button; video
cards read large on mobile, limiting how many are visible at once. Target:
the sort row wraps/scales/shortens to fit mobile viewports without clipping
any control; cards are sized so more than one is comfortably visible on
mobile; desktop layout is unaffected.

**FR-3 — One-off download header button + modal.** Today:
`POST /api/ytdlp/download` (one-off endpoint, `index.js:967` ->
`runOneShot` `index.js:694`) exists with format/quality/filetype dropdowns on
the `/subscriptions` page's one-off form; `GET /api/subscriptions/status`
returns live activity including one-shots; the enabled-gated nav-injection
pattern lives in `public/js/common.js` (`injectSubscriptionsNavLinkIfEnabled`,
:601-648). Target: a small header download-icon button (next to Settings)
opens a compact modal (URL input + format/quality/filetype dropdowns +
Download button + live status), reusing the same endpoint/dropdowns/polling,
gated by the same enabled-probe pattern so it is absent (byte-identical) when
the module is disabled. The existing `/subscriptions` one-off form stays; the
header modal becomes the primary surface.

**FR-4 — Skip YouTube Shorts (per-subscription).** Today: subscription
records carry format/quality/filetype/maxVideos/paused/allowMembersOnly,
validated in `store.js` `validateSubscriptionInput` (:188) +
`updateSubscription` (:236) with `ensureYtdlp` backfill; gating already
follows the `shouldSkip` pattern (`rules.js`) for members-only/availability;
arg builders are pure allowlist-and-re-assert (`args.js`). There is no way to
exclude a channel's Shorts today. Target: a per-subscription `skipShorts`
toggle (default off = download everything), implemented via a yt-dlp
`--match-filter` argument (PE designs the exact filter), threaded through
validation/backfill/PATCH/UI following the `allowMembersOnly` precedent.

**FR-5 — Nicer on-disk filenames.** Today: downloads are named via
`--restrict-filenames` (`args.js:390`, ASCII-folded/underscored) +
`OUTPUT_TEMPLATE` `'%(title)s [%(id)s].%(ext)s'` (`args.js:270`), producing
odd on-disk names (e.g. `Link_Miguel_..._en_Vivo [wN4p6TKlBzQ].mp3`); UI
display is already clean via `cleanDisplayTitle`; SF4 confinement is
`resolveChannelDir` + `isPathUnder`/`realpathUnderChannelDir` + post-download
quarantine. Target: on-disk names are more readable (e.g. spaces allowed)
while preserving path-confinement, id-based uniqueness for archive dedup,
and `cleanDisplayTitle`'s cleaning behavior. PE owns the exact flag/template.

**FR-6 — One-off re-download overwrites.** Today: `runOneShot`
(`index.js:694`) calls `run.runDownload` with the same `--download-archive`
dedup as subscriptions, so a one-off request for an already-archived URL is
silently skipped. Target: a one-off request for an already-downloaded URL
re-downloads and overwrites (bypassing the shared archive for the one-off
path only, e.g. `--no-download-archive`/`--force-overwrites`); subscription
dedup and D3 ("deleted stays gone") are unchanged; the re-download still
lands in the correct folder and is indexed.

**FR-7 — Configurable transcode dir + opt-in higher CRF.** Today:
`TRANSCODE_DIR = path.join(DATA_DIR, 'transcoded')` (`server.js:24`), created
on boot (:33); size-cap LRU eviction, age-retention sweep, and `lastServedAt`
all key off it (`server.js:200, 307-320, 382-400, 1722-1745`); CRF is
hardcoded `'23'` at two encode sites (lazy on-disk `:691`, live pipe
`:2038`). Target: a `TRANSCODE_DIR` env var (default unchanged) lets the
cache live on external/roomy storage, with defensive mkdir -p + writability
checks and unbroken eviction/age/`lastServedAt` bookkeeping; a
`TRANSCODE_CRF`-style opt-in env (default unchanged: 23) allows smaller
files for at least the cached on-disk encode site.

**FR-8 — PWA home-screen icon PNG.** Today: the web manifest is wired but
lacks raster PNGs generated from the SVG (previously parked, "no rasterizer
handy"). Target: PNGs are added with no new runtime/dev dependency; if no
clean no-dependency path exists, the item is explicitly deferred with a
tracked tech-debt note rather than silently dropped.

**FR-9 — Broaden core test coverage.** Today: the yt-dlp module is
well-tested; the core app (scan/metadata/transcode-eviction logic + HTTP
endpoints) has thinner coverage. Target: additive unit tests for the
highest-value untested core pure-logic paths and additive HTTP smoke tests
for main endpoints, including coverage of the new config surfaces introduced
by items 4/5/6/7 — without touching existing tests.

## Open Questions / Forks

All 6 forks below were surfaced by the engineering-manager with a
recommendation and were **resolved by Dean prior to this Discovery pass**
(confirmed, not re-opened here — see Decision log). Restated for the record:

1. **Item 4 skip-shorts scope** — per-subscription only vs. also-global.
   **Resolved: per-subscription only** (no global default in v1.15).
2. **Item 5 filename template** — exact flag/template is the PE's call; the
   user-visible goal is readable on-disk names with spaces, safety
   preserved. **Resolved direction: `--windows-filenames`-style, keeping the
   `[id]` suffix**; PE owns the exact mechanics.

## Design

### Approach (overview)

Five items get real mechanics below (4, 5, 6, 1, 3); four are direct (2, 7, 8,
9). The yt-dlp items (4, 5, 6) all land in the pure `lib/ytdlp/args.js`
builders plus their store/threading, keeping the arg-array/no-shell discipline
and the C1 "download only targets JS-derived survivor ids" architecture
intact. Item 3 mirrors the existing gated `injectSubscriptionsNavLinkIfEnabled`
pattern in `public/js/common.js`. Item 1 is a progressive-enhancement DnD layer
over the existing up/down reorder, persisting through the unchanged
`POST /api/config` folder-order model. Items 7/9 add one env-derived constant
plus additive tests; item 8 is a no-dep raster attempt with an explicit defer
path.

Every value that could reach argv (items 4/5/6) is a fixed module constant or a
strict-boolean-gated flag — no user string is ever interpolated into a filter,
template, or archive flag. The SF4 confinement guard (`resolveChannelDir` +
`realpathUnderChannelDir` + `quarantineEscapedDownloads`) is untouched and
remains the authoritative path-traversal boundary for item 5.

### Item 4 — Skip YouTube Shorts (mechanics)

**Mechanism (two layers, JS-side is authoritative).** Because of the C1
architecture (list -> JS filter -> download only the survivor ids), the binding
Shorts exclusion is a JS filter in the survivor loop; the `--match-filter`
argument is defense-in-depth on the LIST pass. Both are gated on a strict
`sub.skipShorts === true`.

- **`lib/ytdlp/rules.js` — new pure `isShort(videoMeta)`.** Returns `true` when
  any of `videoMeta.webpage_url` / `videoMeta.original_url` / `videoMeta.url`
  is a string containing the substring `'/shorts/'`. Pure, synchronous,
  `node:test`-covered like the other rules helpers. Missing/odd fields ->
  `false` (fail-open to "not a short" = download it, matching the "default off
  = download everything" posture). Exported.
- **`lib/ytdlp/index.js` `runSubscriptionCycle` survivor loop.** After the
  existing `shouldSkip`/`shouldDeferPremiere` checks and before pushing the id,
  add: `if (sub.skipShorts === true && rules.isShort(video)) { log + continue; }`.
  A short's id is then never pushed to `survivorIds`, so it can never be a
  download target (structurally binding, exactly like members-only/premiere).
- **`lib/ytdlp/args.js` — `buildYtdlpListArgs`.** New module constant
  `const SHORTS_MATCH_FILTER = 'webpage_url!*=/shorts/';` (fixed literal, never
  interpolated). Emit `'--match-filter', SHORTS_MATCH_FILTER` (two argv
  elements) **only when** `sub && sub.skipShorts === true` (re-assert the
  boolean at build time — a truthy-but-non-`true` value emits nothing). Placed
  among the flags before the `--` separator. `buildYtdlpDownloadArgs` is NOT
  touched (the download pass targets explicit `watch?v=` URLs derived from the
  already-filtered survivor set). Export `SHORTS_MATCH_FILTER` for the test.
  Because the JS `isShort` filter is the binding guarantee, the exact
  missing-field tolerance of the match-filter is non-critical; `webpage_url` is
  reliably present on full `--dump-json` extraction so it does not over-filter.

**Store threading (`lib/ytdlp/store.js`) — mirror `paused` exactly.**

- New `validateSkipShorts(value)`: `undefined` -> `{ ok: true, value: undefined }`;
  strict boolean -> ok; anything else -> `{ ok: false, error: 'skipShorts must be a boolean' }`
  (never coerced). Same shape as `validatePaused`.
- `validateSubscriptionInput`: call it, add `skipShorts` to the returned value.
- `validateSubscriptionPatch`: `if (input.skipShorts !== undefined) { validate; value.skipShorts = ... }`.
- `addSubscription`: resolve to `false` when unset (same fail-safe posture as
  `paused`); write `skipShorts` into the new record.
- `updateSubscription`: `if (body.skipShorts !== undefined) { re-validate; sub.skipShorts = result.value; }`.
- `ensureYtdlp` backfill: in the per-sub loop, `if (sub && typeof sub.skipShorts !== 'boolean') sub.skipShorts = false;` (mirrors the `paused` backfill line — non-destructive, `undefined` -> `false`, touches no other field).
- Export `validateSkipShorts`.

**PATCH/add wiring (`lib/ytdlp/index.js`).** No route change needed — the PATCH
and POST handlers already delegate to `validateSubscriptionPatch` /
`validateSubscriptionInput`, so `skipShorts` rides through once the validators
accept it. The one-off `POST /api/ytdlp/download` path is NOT touched (Shorts
skipping is per-subscription only; one-offs download exactly the requested id).

**UI toggle.** `lib/ytdlp/views/subscriptions.html` + `lib/ytdlp/client/subscriptions.js`:
add a "Skip Shorts" checkbox per subscription, wired to `PATCH` exactly like the
existing per-sub edit controls; reflect persisted state on reload.

**Regression tests (item 9 / AC4.x).** (a) `buildYtdlpListArgs` emits
`--match-filter SHORTS_MATCH_FILTER` when `skipShorts:true`, and no
`--match-filter` when `false`/absent/`undefined`/hostile (object, string with
metacharacters). (b) `isShort` true for a `/shorts/` webpage_url, false for a
`/watch` one and for missing fields. (c) `validateSkipShorts` accepts
boolean/undefined, rejects non-boolean. (d) `ensureYtdlp` backfills
`undefined` -> `false` without altering other fields. (e) PATCH round-trip
integration (`true` -> read back `true`, `false` -> `false`).

### Item 5 — Nicer on-disk filenames (SF4-preserving)

**Change: replace `--restrict-filenames` with `--windows-filenames` in
`buildYtdlpDownloadArgs` (`args.js:390`).** `OUTPUT_TEMPLATE` is unchanged
(`'%(title)s [%(id)s].%(ext)s'`). `--windows-filenames` forces yt-dlp to strip
every Windows-invalid character (`/ \ : * ? " < > |`, plus control chars and
trailing dots/spaces) from each template field while ALLOWING spaces and
ordinary punctuation — so `Link Miguel en Vivo [wN4p6TKlBzQ].mp3` instead of
`Link_Miguel_..._en_Vivo [wN4p6TKlBzQ].mp3`.

**Why traversal is still impossible.** (1) yt-dlp sanitizes each `%(...)s` field
independently and never lets a field value introduce an OS path separator — a
hostile title `../../etc/passwd` renders its slashes as safe replacements, so
the produced name stays a single flat segment. A bare `..` title still yields
`.. [<id>].<ext>` (the literal `[id].ext` suffix means it is never the `..`
path component). (2) The `-o` template is rooted at `resolveChannelDir(...)`,
which is unchanged and still throws on any dir outside the download root.
(3) `quarantineEscapedDownloads` + `realpathUnderChannelDir` still delete, before
any scan, any produced file that resolves outside the channel dir. Item 5 only
swaps a defense-in-depth sanitizer flag; the authoritative confinement guard is
untouched. `--trim-filenames` is rejected as the alternative — it only truncates
length and does NOT sanitize dangerous characters.

**Uniqueness / archive / display all preserved.** The `[<id>]` id suffix (a
space then the bracketed id) stays, so two same-title videos never collide. `--download-archive` is keyed by
`extractor id` (not filename) and is unaffected. `cleanDisplayTitle`
(`server.js:540`) already matches both separators via
`/^(.*?)[ _]\[[A-Za-z0-9_-]{11}\]$/` and collapses `_+` -> space, so a
spaces-allowed name (`Link Miguel en Vivo [id]`) and the legacy underscored
name both clean to the identical display title.

**Regression tests (AC5.x).** (a) `buildYtdlpDownloadArgs` includes
`--windows-filenames` and NOT `--restrict-filenames`. (b) A hostile title/id
still cannot escape: existing `resolveChannelDir` / `isPathUnder` /
`realpathUnderChannelDir` confinement tests continue to pass, plus a new test
asserting a `../`- and separator- and control-char-laden title still resolves
under the channel dir (confinement helpers, no real spawn). (c) `cleanDisplayTitle`
produces the same clean title for the new spaces-shape and the legacy
underscore-shape (`server.js` unit test). (d) The id-suffix uniqueness assertion
(two distinct ids -> two distinct on-disk names).

### Item 6 — One-off re-download overwrites (archive bypass, one-off only)

**Mechanism: a `oneOff` opt threaded into the download builder.**

- **`args.js` `buildYtdlpDownloadArgs(sub, config, targetIds, opts = {})`** —
  new 4th param. `const oneOff = opts && opts.oneOff === true;`. Replace the
  current unconditional `args.push('--download-archive', archivePath, '--no-warnings');`
  with:
  - subscription (default): `args.push('--download-archive', archivePath);`
  - one-off: `args.push('--no-download-archive', '--force-overwrites');`
  - then unconditionally `args.push('--no-warnings');`.
  `resolveArchivePath(config)` is still called unconditionally (its throw is a
  cheap confinement assertion); its result is only consumed in the subscription
  branch. Nothing else in the builder changes.
- **`run.js` `runDownload(sub, config, targetIds, opts = {})`** — already takes
  `opts` (for `onProgress`); forward `opts.oneOff` into
  `buildYtdlpDownloadArgs(sub, config, targetIds, { oneOff: opts && opts.oneOff })`.
- **`index.js` `runOneShot`** — pass `oneOff: true`:
  `run.runDownload(syntheticSub, config, [videoId], { onProgress, oneOff: true })`.
  Every subscription call site (`runSubscriptionCycle`) is unchanged, so
  subscriptions keep `--download-archive` and D3 ("deleted stays gone") holds.

`--force-overwrites` makes yt-dlp overwrite the existing on-disk file (fork:
overwrite, not a `(1)` copy); `--no-download-archive` makes an already-archived
id download anyway. Confinement + quarantine + the fire-and-forget scan trigger
in `runOneShot` are unchanged, so the re-download still lands in the right
folder and is re-indexed.

**Regression tests (AC6.x).** (a) `buildYtdlpDownloadArgs(sub, cfg, ids, { oneOff: true })`
contains `--no-download-archive` and `--force-overwrites` and NO
`--download-archive`. (b) `buildYtdlpDownloadArgs(sub, cfg, ids)` (subscription,
no opts) still contains `--download-archive <archivePath>` and NEITHER new flag
— asserting the two paths diverge only in the archive flags. (c) Existing
subscription-archive tests continue to pass unmodified.

### Item 3 — One-off header button + compact modal (gated injection)

**Where it lives.** A new `injectOneOffDownloadButtonIfEnabled()` in
`public/js/common.js`, called on `DOMContentLoaded` alongside

and injects ONLY on a genuine 2xx (`shouldInjectSubscriptionsNav`-style probe);
on 404/network error it does nothing (fail closed). This is the exact accepted
gating pattern already in `common.js` — the served page HTML carries no

`public/` unconditionally that reveals the feature.

**DOM (all via `createElement` + `textContent` — never `innerHTML` with dynamic
data).**

- Modal: a backdrop `<div>` + a modal `<div>` appended to `document.body`,

**Tests (AC3.x).** A pure `shouldInjectOneOffButton(response)` (mirroring

### Item 1 — Folder drag-and-drop reorder (progressive enhancement)

**Model (unchanged persistence).** Real folders reorder via `db.folders`

mutate, then use the SAME save path. No server change (C3/C7/E1 intact).

**Setup list (`public/setup.html`).** Add `draggable="true"` + a drag-handle
affordance to each `.folder-item-row`. Vanilla HTML5 DnD handlers (no dep):
`dragstart` records the source index (dataset/closure); `dragover` calls
`preventDefault()` to allow drop and marks the insertion target;
`drop`/`dragend` move the entry within `configuredFolders`
(splice-out + splice-in) and call the existing `renderFolders()`. The existing

**Left sidebar (`public/js/main.js` `renderSidebarFolders`, + the setup-page

mirror the Setup ones but on drop: (1) compute the new order of the VISIBLE
subset, (2) rebuild the FULL `folders` order by walking the original full array
and, for each slot that held a visible folder, taking the next entry from the

positions), (3) `POST /api/config { folders: rebuiltFull, folderSettings }`
using the spellings already received from `GET /api/config` (real folders keep
their `db.folders` spelling; the synthetic root keeps its resolved path), then

logic, so the synthetic Downloads entry's order persists via
`folderSettings[downloadDir].order` and never enters `db.folders`.

**Accessibility / fallback (fork 6, AC1.3).** The Setup up/down buttons remain
the keyboard/tap-accessible reorder affordance (they cover the sidebar's order
too, since both read the same `folders` order). Sidebar DnD is a desktop-mouse
enhancement; on mobile the drawer's reorder is done via the Setup list. All
labels rendered with the existing `escapeHtml`/`textContent` helpers (XSS-safe).

**Tests (AC1.x).** Integration against `POST`/`GET /api/config`: (a) reordering

### Alternatives considered

- **Item 4 filter field.** Duration-based `--match-filter duration>60` was
  rejected (Shorts now run to ~3 min and long non-Shorts exist -> both false
  positives and negatives); `aspect_ratio<1` was rejected (ordinary vertical
  uploads exist). The `/shorts/` URL match + the JS `isShort` belt is the most
  reliable, and the JS belt is what makes it structurally binding.
- **Item 5 template.** A custom `-o` with `--replace-in-metadata` was rejected as
  more surface area for the same outcome; `--trim-filenames` rejected (no char
  sanitization). `--windows-filenames` is the minimal, sanitizing swap.
- **Item 6.** A separate `buildYtdlpOneOffDownloadArgs` function was rejected as
  a divergent second download path (drift risk); a single `oneOff` opt on the
  existing builder keeps ONE path with a two-flag difference.
- **Item 3.** Serving the button/modal from the gated `lib/ytdlp/` path (like the
  `/subscriptions` page) was considered, but the button must appear in the
  header of core pages (`index`/`watch`/`setup`) served from `public/`, so the
  established `common.js` runtime health-probe injection is the correct fit.
- **Item 1.** A drag-and-drop library was rejected (no-new-deps); native HTML5
  DnD is sufficient. Replacing the up/down buttons entirely was rejected (they
  are the accessibility/mobile fallback).

### Risks and mitigations

- **Risk:** item 5's `--windows-filenames` behaves differently across yt-dlp
  versions for edge titles. -> **Mitigation:** the SF4 confinement/quarantine
  guard is authoritative and unchanged; the regression test asserts confinement
  against a hostile title regardless of sanitizer behavior.
- **Risk:** item 4's `--match-filter` over-filters if a yt-dlp version rejects
  entries missing `webpage_url`. -> **Mitigation:** the JS `isShort` filter is
  the binding mechanism; the match-filter is defense-in-depth only, and
  `webpage_url` is reliably present on full `--dump-json` extraction.
- **Risk:** item 1 sidebar reorder mis-persists when hidden-from-sidebar folders
  are interleaved. -> **Mitigation:** the rebuild-preserving-hidden-positions
  algorithm is covered by the `POST /api/config` integration test; the up/down
  Setup path (which sees the full list) remains the fallback.
- **Risk:** item 6's archive bypass accidentally reaches a subscription cycle.
  -> **Mitigation:** `oneOff` defaults to false and is only set in `runOneShot`;
  AC6.2 asserts subscription args are unchanged.
- **Risk:** item 8's hand-drawn PNG looks poor. -> **Mitigation:** explicit
  defer path with a tracked note (no silent drop, no heavy dep).

### Performance impact

No expected impact on any RELIABILITY.md budget. Item 7's writability check is a
single boot-time `accessSync`. Item 4's extra JS filter and `--match-filter` are
negligible next to the network/spawn cost. Item 1's sidebar reorder issues one
`POST /api/config` per drop (same as an up/down save today). Item 3 polls the
existing status endpoint only while the modal is open with an active job.
