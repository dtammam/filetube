## 2. Data model changes (`db.books`)

The books namespace is feature-owned (`lib/books/store.js`), which is exactly why it has dodged the 5-strike persist-gate class: `updateDatabase` round-trips untouched keys verbatim and the media scan's Phase-2 merge never sees `db.books`. We keep that discipline.

### 2.1 KNOWN CODE CORRECTION — `db.books.audio` is NOT reserved yet

`ensureBooks` (`lib/books/store.js:31,34-42`) reserves `{ folders, items, progress, pins, settings }` — there is **no `audio`**. The Wave-2 brief's claim that it was already reserved is wrong. `readBooks` (`lib/books/store.js:54-66`) rebuilds a fresh object with ONLY those five keys and would **drop** any `audio` key on every read — and every GET route uses `readBooks`, not the raw namespace. So audio status written to disk would be invisible to the UI unless BOTH functions learn the key. This is task **T1** and it is treated with persist-gate seriousness.

### 2.2 Shapes

`db.books.settings` (existing object, currently `{}`) — TTS controls:
```
settings: {
  ttsEnabled: boolean,   // master user toggle (default true; gated further by engine availability)
  engine:     string,    // 'piper' | 'espeak-ng'  (default 'piper')
  voice:      string,    // reserved; one default for now (e.g. '' => engine default)
  rate:       number     // speaking-speed multiplier, 1.0 = normal (bounded 0.5..2.0)
}
```
`rate` is a **speed multiplier** (higher = faster), engine-agnostic; each engine maps it to its own knob (§8.3). This is the value baked into the cache key, so its semantics are frozen by `ttsRev`.

`db.books.audio` — per-book, per-chapter synthesis status (NEW):
```
audio: {
  [bookId]: {
    [spineIndex]: {
      status:      'pending' | 'processing' | 'ready' | 'failed',
      key:         string,        // sha1 cache key (see §8.4) — the .m4a / .blocks.json basename
      durationSec: number | null, // filled on 'ready' (probed from the m4a), null otherwise
      updatedAt:   string          // ISO; lets a stale 'processing' be reaped on restart
    }
  }
}
```
`spineIndex` keys are the SAME chapter address the reader's `locator.spineIndex` uses (`lib/books/opf.js:117-123`).

### 2.3 No-clobber write rules (the `audioStatus` pattern, ported)

Mirror `setAudioStatus`/`clearAudioStatus` (`server.js:1317-1358`): a status mutator inside `updateDatabase` writes only if the value actually changes, returns `false` otherwise (skips the save), and never touches sibling keys. A books-owned `setBookAudioStatus(bookId, spineIndex, patch)` and `clearBookAudioStatus(bookId, spineIndex)` live in `lib/books/store.js` and are called from the server worker. Eviction of a cache file MUST eagerly delete the matching `db.books.audio[bookId][spineIndex]` entry (the `clearAudioStatus`-on-`.m4a`-delete lesson, `server.js:1328-1358`) so a `ready` status can never outlive the file it describes.

### 2.4 Persist-gate audit (the repo's 5-checkpoint stale-snapshot lesson)

Because `db.books.audio` lives in the feature-owned namespace, the media scan cannot clobber it. The **one** cross-writer risk inside books is the book scanner's own `updateDatabase` merge (`server.js:3656-3711`). Audit points, each an explicit task acceptance criterion:

- **Prune carry**: when a book is pruned (`server.js:3700-3707`, which already `delete freshNs.progress[id]`), ALSO `delete freshNs.audio[id]` — and delete its cache files (§8.5).
- **Scan merge non-interference**: the scan merge only rewrites `items`/`progress`; it must leave `freshNs.audio` untouched for surviving books. Regression-locked in the books scanner integration test.
- **ensureBooks/readBooks round-trip**: T1's own test writes `audio` state, runs it through `ensureBooks` (no-clobber) and `readBooks` (round-trips, does not drop), asserting survival both ways.

## 4. Part B — TTS "Listen from Here" (overview)

On demand only:
1. Reader taps **Listen from here**. read.js has the live `locator.{spineIndex, blockIndex}` (persisted, `read.js:281-291`).
2. Client `POST /book/:id/tts/:spineIndex/ensure` → server enqueues if not cached.
3. Worker (single FIFO, deferring during downloads) extracts chapter XHTML via `lib/books/zip.js`, chunks it by the **exact** `READER_BLOCK_SELECTOR` rule, synthesizes WAV per block with Piper, concatenates, encodes WAV→m4a via ffmpeg, writes `blocks.json` (`blockIndex → startSec`).
4. Client polls status; on `ready` fetches `blocks.json`, looks up `blocks[blockIndex].startSec`, hands `/book/:id/tts/:spineIndex` to the **existing** player as an audio item with the book cover as artwork, seeks to that offset.
5. Worker pre-synthesizes **exactly one chapter ahead** (`spineIndex+1`), never the whole book.

## 5. The block contract lockstep (top risk — designed out)

`READER_BLOCK_SELECTOR = 'p, h1, h2, h3, h4, h5, h6, li, blockquote, pre, figure, td'` (`read.js:21`). `blockIndexForNode` (`read.js:85-95`) walks from the reading position to its **nearest ancestor** matching that selector and returns its index among `doc.querySelectorAll(READER_BLOCK_SELECTOR)` — document order.

### 5.1 Single source of truth (no drift possible by construction)

read.js exports `READER_BLOCK_SELECTOR` under its `module.exports` guard (`read.js:52-60`) and its IIFE early-returns when `window` is undefined (`read.js:63`) — safe to `require` server-side. Therefore:
- Add `READER_TTS_REV` (integer, start `1`) to read.js **alongside** `READER_BLOCK_SELECTOR`, exported the same way. Single version knob.
- The chunker (`lib/books/tts-chunk.js`, new) `require('../../public/js/read.js')` and consumes BOTH constants — never re-declares the selector. The cache key's `ttsRev` = `READER_TTS_REV`.
- Any change to the block rule ⇒ bump `READER_TTS_REV`; reader index math + server chunk boundaries move in lockstep; every cached file auto-invalidates.
- **Locked test** (`test/unit/books-tts-chunk.test.js`): asserts the chunker's active selector `=== READER_BLOCK_SELECTOR` and rev `=== READER_TTS_REV`; fails if a future edit forks the selector.

### 5.2 Chunker semantics that MUST match `blockIndexForNode`

The chunker parses chapter XHTML with the string/regex posture the books module already uses (`lib/books/opf.js` — no server-side DOM dep). It reproduces querySelectorAll document order AND the nearest-ancestor assignment:
- Enumerate block elements in document order (opening-tag order), `blockIndex = 0,1,2,…`.
- For **nested** blocks (`<blockquote><p>…</p></blockquote>`, `<li><p>…</p></li>`), the reader's nearest-ancestor walk lands on the **innermost** match. So each block's spoken text is that element's **direct** text (excluding nested-block descendants). An ancestor block containing only nested blocks contributes no spoken segment but STILL occupies its `blockIndex` slot (indices stay aligned with the reader's count); its `startSec` = the start of its first descendant block.
- Locked by a shared XHTML fixture test (T4/T7): (a) block count matches a jsdom-free reference; (b) a chosen paragraph's `blockIndex` maps to the audio offset where that paragraph's words begin.

This is the single most important correctness invariant of the wave.

| Env var | Meaning | Default |
|---|---|---|
| `FILETUBE_TTS_ENGINE` | `piper` \| `espeak-ng` | `piper` |

A books-less or TTS-less install is a total no-op. Presence probe: arg-array `piper --version` / `espeak-ng --version`, non-zero/spawn-error ⇒ unavailable, logged and degraded.

## 7. The synthesis worker (FIFO + defer-during-downloads)

### 7.1 Structure — clone the audio-extract queue
Model on `audioExtractQueue`/`processAudioExtractQueue` (`server.js:1309-1440+`): a module-level array, a single `ttsBusy` boolean, atomic `.tmp`→rename finalize, per-job status via the no-clobber mutator. One worker, one job. RAM discipline from **spawn-per-job** (piper + ffmpeg exit per chapter), not the engine.

### 7.2 Job lifecycle
`queueChapterTts(bookId, spineIndex)`: no-op if `!ttsAvailable` or already queued/processing; if cache file exists ⇒ set `ready`; else push, set `pending`, kick worker.
`processTtsQueue()`: if `ttsBusy`/empty ⇒ return; **defer check (§7.4)**; else dequeue, `processing`, run pipeline (§8), finalize (`ready`+`durationSec`, or `failed`), clear `ttsBusy`, recurse. On `ready`, enqueue `spineIndex+1` one chapter ahead (guarded: within spine bounds, not already cached).

### 7.3 Cancelation / staleness
No per-job child registry (chapters are short). On boot, any `db.books.audio[*][*].status === 'processing'` with a stale `updatedAt` (or no cache file) is reset to `pending`/cleared (transcode-reconcile posture). Cache eviction (§8.5) clears status eagerly.

### 7.4 Defer-during-downloads coordination (Dean decision #4)
Add a tiny exported accessor to `lib/ytdlp/index.js`:
```
function isHeavyJobActive() { return ytdlpQueueLength > 0; }   // exported
```
`ytdlpQueueLength` (`lib/ytdlp/index.js:2603,2615-2622`) is incremented for the lifetime of every `runExclusive` job (polls + one-shot downloads) and decremented on settle — the authoritative "download/poll in flight or queued" signal. server.js holds the module handle (`server.js:14`).

**Gate rule:** `processTtsQueue()` checks `ytdlp.isHeavyJobActive()` at each dequeue. If true, re-arm `setTimeout(processTtsQueue, TTS_DEFER_POLL_MS)` (~5 s, `.unref()`'d) and return without starting. A synth already in progress is NOT killed (chapters are short); only the NEXT dequeue defers. One-directional: downloads never wait for TTS.

**Trade-offs (disclosed):** a long poll/download starves TTS for its duration (acceptable — the point is "less spiky"; the user can still read). 5 s post-download latency is invisible next to synthesis time. The accessor is a single read-only counter (atomic under JS single-threadedness). Rejected: routing TTS through `runExclusive` (would make a long TTS job block downloads — violates one-directionality).

## 8. The synthesis pipeline (Piper → WAV → ffmpeg m4a) — WITH SOURCE CITATIONS

### 8.1 Piper CLI — verified against source
Pinned to the standalone `piper` C++ binary (`rhasspy/piper`), a single self-contained executable (the opt-in yt-dlp analogue). Verified against `src/cpp/main.cpp`:
- Text to synthesize is read **from stdin**, one utterance per line.
- `-m` / `--model` — `.onnx` voice model (required).
- `-c` / `--config` — model config JSON; **defaults to `<model>.onnx.json`** if omitted.
- `-f` / `--output_file` — write a **WAV** file; `-` streams WAV to stdout.
- `--output_dir` — with multi-line stdin, writes one WAV per line into the dir (batch mode — see Implementation note).
- `--length_scale` — **Piper's speaking-rate knob; there is NO `--rate`/`--speed` flag.** Phoneme duration: **>1 = slower, <1 = faster** (normal = model default, ~1.0).
- `--sentence_silence` — seconds of silence after each sentence.
- `-q` / `--quiet`.

Sources:
- `rhasspy/piper` `src/cpp/main.cpp` (arg parser; stdin input; `textToWavFile`): https://github.com/rhasspy/piper/blob/master/src/cpp/main.cpp
- `OHF-Voice/piper1-gpl` `docs/CLI.md` (`length_scale` speaking rate): https://github.com/OHF-Voice/piper1-gpl/blob/main/docs/CLI.md
- length_scale rate mapping (0.5 ≈ 2× faster): https://bash-prompt.net/guides/bash-piper-audio/

**Implementation note (perf refinement, verify during T5/T7):** synthesizing per-block with one piper spawn per block is functionally correct but spawns N processes per chapter. Prefer ONE piper invocation per chapter using multi-line stdin + `--output_dir` (one process, N numbered WAVs), then read each WAV's duration for `startSec`. Verify `--output_dir` naming/ordering against `main.cpp` before relying on it; fall back to per-block spawn if ordering isn't deterministic.

### 8.2 Per-chapter synthesis argv (arg-array, never a shell)
```
buildPiperArgs(modelPath, configPath, outSpec, lengthScale) =>
  ['--model', modelPath,
   '--config', configPath,              // omit if deriving <model>.json default
   '--length_scale', String(lengthScale),
   '--sentence_silence', '0.4',
   ...outSpec,                          // ['--output_file', wavPath] or ['--output_dir', dir]
   '--quiet']
```
Block text is written to the child's **stdin**, never an argv token (no injection surface — the yt-dlp spawn-security posture). `cp.spawn(PIPER_BIN, args, { stdio: ['pipe','ignore','pipe'] })`, write sanitized/whitespace-collapsed text, `stdin.end()`. Bounded stderr tail. Temp WAVs under `DATA_DIR/tts-cache/.tmp-<key>/`. `startSec` for block *i* = cumulative sum of prior blocks' WAV durations.

### 8.3 espeak-ng fallback mapping
`espeak-ng` reads stdin with `--stdin`, writes WAV with `-w`, rate knob is `-s <words-per-minute>` (default 175). Engine abstraction maps the shared `settings.rate` multiplier: `piper.length_scale = clamp(1 / rate, 0.5, 2.0)`; `espeak.wpm = round(175 * rate)`. Config-selectable fallback we wire + smoke-test; Piper is primary.

### 8.4 WAV → m4a (mirror `buildAudioExtractArgs`)
```
buildTtsEncodeArgs(concatListPath, tmpM4aPath) =>
  ['-f', 'concat', '-safe', '0', '-i', concatListPath,
   '-c:a', 'aac', '-b:a', '96k', '-ac', '1',   // mono, lower bitrate: speech
   '-movflags', '+faststart',
   '-y', tmpM4aPath]
```
Concat the per-block WAVs via ffmpeg's concat demuxer, atomic `.tmp.m4a`→rename (`server.js:1393`). Probe `durationSec` (ffprobe helper, `server.js:1800`), write `<key>.blocks.json` atomically.

### 8.5 Cache layout, keying, eviction
- Dir: `DATA_DIR/tts-cache/` (sibling of `.thumbnails`/`.bookcovers`), created on demand.
- Files: `<key>.m4a` + `<key>.blocks.json` (`[{blockIndex, startSec}, …]`).
- **Key**: `sha1(`${bookId}:${spineIndex}:${engine}:${voice}:${rate}:${ttsRev}`)` (`ttsRev = READER_TTS_REV`).
- **Eviction**: age-sweep + optional size cap (the `sweepAgedTranscodes`/`enforceCacheCap` posture, `server.js:690-811`); an evicted `.m4a` MUST also delete its `.blocks.json` and eagerly `clearBookAudioStatus`. "Clear cache now" also clears `tts-cache/`.
- **Future audiobook export**: chapters are per-spine deterministic `.m4a`s ⇒ export = concat the spine's files + chapter marks from `blocks.json`, no re-synthesis. Build nothing now.

### 8.6 Engine abstraction
`lib/books/tts-engine.js` (new, pure/leaf): `buildPiperArgs`, `buildEspeakArgs`, `mapRate(engine, rate)`, `synthesizeBlock(engine, text, opts) → wavPath`. Pure argv builders unit-tested with no binary. The chunker (`lib/books/tts-chunk.js`) is separate and pure over an XHTML string.

## 12. Risks & testability notes

1. **Block-contract drift (highest).** Structural mitigation: single source of truth in read.js (`READER_BLOCK_SELECTOR` + `READER_TTS_REV`), chunker imports both, `ttsRev` auto-invalidates the cache, §5.1/§5.2 locked tests fail on any fork. Nested-block nearest-ancestor rule is fixture-locked (T4/T7).
