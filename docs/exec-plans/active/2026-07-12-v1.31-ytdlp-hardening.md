# Execution Plan: v1.31 yt-dlp Download Hardening

- **Date:** 2026-07-12
- **Status:** APPROVED 2026-07-12 (Dean; full autonomous run to release authorized)
- **Target release:** v1.31.0
- **Branch:** from `main` (v1.30.0 merge, clean)
- **Evidence:** read-only concurrency/timeout sweep 2026-07-12 (file:line map of the
  runExclusive gate, timeout structure, and pacing reality) + Dean's production
  screenshots: ~20 subscription failures in one run, ALL with the identical reason
  `error: yt-dlp timed out and was killed` — uniform timeout kills, not bot-check or
  extractor errors.

## Goal

Downloads become boringly reliable: runs don't cascade-fail, one-shots are never
invisible or lost, hung processes are reclaimed in minutes not hours, and every
failure reason says exactly what happened and in which phase.

## Verified current reality (from the sweep; file:line in the sweep record)

- One global promise-chain FIFO (`runExclusive`/`ytdlpTail`, index.js:2044-2070);
  the ENTIRE poll run (all N channels) is a SINGLE gate job (index.js:1943-1963),
  so a queued one-shot waits behind the whole run, invisibly for API/Shortcut
  callers (202 accepted is sent before the gate join; index.js:3368 vs :3383).
- Timeouts are ABSOLUTE, never reset by progress (run.js:902-906): list pass 5 min
  (run.js:105), download 180 min (configurable). No stall/idle watchdog. A hung
  download holds the gate up to 3 h.
- NO inter-channel pacing, NO consecutive-failure awareness, NO circuit breaker in
  the poll loop (index.js:1944-1962). v1.29 pacing flags act only INSIDE one
  channel's yt-dlp invocation (args.js:581-620).
- Queue is purely in-memory: queued one-shots are lost without trace on restart
  (no runlog line is written for a never-started job; runlog writer lives inside
  runOneShot).
- Repulls essentially never fail at the HTTP layer; user-visible "repull failed" is
  the run's channels erroring (same cascade) or silent busy-coalescing.
- yt-dlp is pinned fresh (Dockerfile ARG 2026.7.4); staleness is NOT the current
  culprit but there is no version surfacing/warning as it drifts (locked decision
  D5: no runtime auto-update — keep).
- Shortcut vs in-app one-shots: SAME endpoint/handler; only the body parsing
  differs (text/plain → {url}). The reliability difference is queue timing +
  visibility, not code path.

## Prime diagnostic hypothesis (H0 — verify first)

**v1.29's `--sleep-requests 1` was added to the LIST pass whose 5-minute absolute
timeout was NOT adjusted.** A 25-video listing (multiple HTTP requests per video,
+1s each, worse under throttling) can plausibly exceed 5 minutes → every channel's
list pass times out uniformly → the exact screenshot signature (identical
"timed out and was killed" across ~20 channels; each channel burning ~5 min also
matches "everything after a certain amount of time fails"). T0 must verify this
arithmetic (request budget per list pass vs timeout) against the code and, if
confirmed, fix the budget mismatch (reduce/remove list-pass sleeps, raise/scale the
list timeout with maxVideos, or both — design decides).

## Scope

- **P0: Timeout/pacing reconciliation + diagnosable reasons.** Verify H0; fix the
  list-pass budget mismatch. Every timeout reason must name the phase and duration
  ("list pass timed out after 5m", "download stalled — killed after 10m idle",
  "download hit the 180m ceiling"). No more bare "timed out and was killed".
- **P1: Per-channel queue jobs + priority.** Decompose the monolithic poll gate job
  into one job per channel. Priority ordering: one-shot > repull > scheduled poll.
  A one-shot waits ≤1 channel; a hung channel blocks only itself; the gate's
  never-wedge property and the --download-archive single-writer invariant are
  preserved.
- **P2: Circuit breaker + inter-channel backoff.** Configurable sleep between
  channel jobs; after N consecutive channel failures (config, default ~3-5), abort
  the remainder of the run with an honest persisted status ("run paused after N
  consecutive failures; retrying at <time>") and schedule a backoff retry.
  Successes reset the counter. Aborted-not-run channels keep their cutoff (already
  the case — cutoff only advances on success/partial).
- **P3: Stall watchdog.** Progress-aware idle timer alongside the absolute ceiling:
  no stdout progress for N minutes (config, default ~10) → SIGKILL + specific
  reason + retry per policy. Applies to download pass; list pass gets the P0-fixed
  budget instead.
- **P4: Durable queue.** Persist accepted-but-not-started jobs (one-shots at
  minimum) so a restart requeues instead of silently losing them; every accepted
  job eventually produces a runlog/history line no matter its fate (downloaded /
  failed / requeued / dropped-with-reason).
- **P5: Queue visibility.** Status/chip/history show queue position and what's
  ahead ("queued — 3 ahead"); repull busy-coalescing surfaced (the v1.29 body
  fields already exist — render them everywhere relevant); one-shot activity
  entries reflect queued→running transitions with the wait visible.
- **P6: yt-dlp version surface (small).** Expose the binary version (already
  probed for health) in the UI with a staleness note; document the pin-bump
  update path. No runtime auto-update (respect locked decision D5).

### Out (explicitly)

- Parallel yt-dlp processes (stays strictly serial — the archive-file invariant
  and YouTube-friendliness both demand it).
- Runtime auto-update of yt-dlp (D5 stands).
- Cookie acquisition automation (mounting cookies is Dean's env-side lever; the
  existing configured-but-missing warning stays).
- Any UI redesign beyond the P5 visibility additions.

## Constraints

- v1.29 semantics are regression surfaces: outcome classification
  (success/partial/error/cancelled), zero-attributed⇒error, cancel-latch ordering,
  runlog schema (additive evolution only), retry affordances, AC6.3 argv
  byte-identity posture (new flags/args additive with the same injection-guard
  discipline: literals or validated values only, far from `--`).
- v1.30 semantics too: chip conformance ACs, one-shot three-surface visibility,
  BUG-2 reload-never.
- shell:false arg-array spawning; SIGKILL discipline (note tech-debt #24 — process
  GROUP kill remains out of scope unless P3 makes it trivially safe).
- Queue persistence must follow the crash-safety posture (atomic writes; bounded
  file). No new runtime deps.
- Config: all new knobs FILETUBE_YTDLP_* env vars, bounds-checked in config.js per
  the established parse* pattern.
- `node:test`; ESLint clean; Node 22 + Node 24 both green at release.
- Two-reviewer gate before acceptance; honest scoring; Dean's on-device pass (his
  real ~40-subscription instance) is the final arbiter.

## Deliverables / "done"

1. A full poll run against many channels under throttling either completes or
   aborts EARLY with an honest breaker status — never a uniform 20-channel
   timeout cascade.
2. Timeout/stall reasons name the phase and duration; "timed out and was killed"
   (bare) no longer exists in the codebase.
3. A Shortcut one-shot submitted mid-run starts within ≤1 channel's work,
   its queued state (and position) is visible in status/chip/history, and it
   survives a server restart (requeued, never silently lost).
4. A stalled download is reclaimed within the idle window (default ~10 min), not
   180 minutes, and retries per policy.
5. Repull responses' busy/started state is visible wherever repull can be
   triggered.
6. yt-dlp version visible in the UI with staleness note.
7. `npm test` + lint green on Node 22 + 24; two-reviewer gate passed; v1.31.0
   released per repo convention (merge to main, tag, push).

## Complexity signal (why plan-gated)

P1 restructures the serialization primitive every download path relies on (the
single-writer archive invariant lives there); P2-P4 add stateful failure policy to
a crash-safety-sensitive orchestrator. The gate must verify the archive
single-writer property and the never-wedge tail property survive the queue
decomposition, both directions.

T0/P0 first (diagnosis + the likely one-line-class fix for Dean's immediate pain),
then P3 (watchdog), P1 (queue decomposition), P2 (breaker), P4 (durability),
P5 (visibility), P6 (version surface). EM may re-split at Tasks.

## Requirements

Numbered functional requirements (FR1–FR7), one group per plan deliverable.
Every anchor below was re-verified against the current tree on 2026-07-12
(post the sweep the plan cites). "What/observable," not "how" — Design owns

### FR1 — Deliverable 1: honest breaker status, never a uniform cascade (P2, P0-adjacent)

- **FR1.1** The poll loop (`lib/ytdlp/index.js:1943-1963`, currently a bare
  `for` loop with no pacing/failure-awareness — `key_files_from_sweep.
  poll_loop_no_pacing`) tracks consecutive channel-job failures across the
  targeted loop and, once a configurable threshold N is reached, stops
  invoking `processSubscription` for the remaining, not-yet-attempted
  channels in that run.
- **FR1.2** On tripping, the run's persisted/live status names the trip
  explicitly (consecutive-failure count + a scheduled retry time), not a
  generic error and not silence.
- **FR1.3** A channel success occurring before the threshold is reached
  resets the consecutive-failure counter to 0 (a run does not trip after N
  failures spread across an otherwise-healthy run with successes
  interleaved — only N *consecutive* failures trip it).
- **FR1.4** Channels the breaker never reached (aborted-not-run) do not have
  their cutoff date advanced — they are retried on the next poll/backoff
  exactly as if this run had not touched them (consistent with the existing
  invariant that cutoff only advances on success/partial, noted in
  Constraints).
- **FR1.5** After tripping, a backoff retry is scheduled (mechanism is
  Design's choice — e.g. reusing/extending the existing `schedulePollRerun`
  follow-up-timer machinery at `lib/ytdlp/index.js:1979-1993` — but a retry
  must be armed, not a dead end requiring manual intervention).
- **FR1.6** The consecutive-failure threshold N and the inter-channel sleep
  between jobs (FR2/P2's pacing companion) are each independently
  configurable per the Config requirements (FR-CFG below).
- **FR1.7** A full run where every channel succeeds, and a full run where
  failures never reach N consecutive, both complete all targeted channels
  exactly as today — the breaker is a new failure path, not a new limit on
  healthy runs.

### FR2 — Deliverable 2: phase-named timeout/stall reasons; bare string eliminated (P0 + P3 naming)

- **FR2.1 (H0 verification — gating, see AC0)** Before any P0 timeout/pacing
  fix lands, the arithmetic in H0 (request budget per LIST pass — driven by
  `--sleep-requests` on the list-only pacing args, `lib/ytdlp/args.js:581-
  588` — versus `DEFAULT_LIST_TIMEOUT_MS` at `lib/ytdlp/run.js:105`, currently
  5 minutes) is checked against the actual code path and the finding
  (confirmed or falsified, with the arithmetic shown) is recorded in this exec
  plan's Progress log.
- **FR2.2** If H0 is confirmed, the list-pass budget/timeout relationship is
  corrected (reduce/remove list-pass sleep, scale/raise the list timeout with
  the effective `maxVideos` window, or both — Design's choice) so that a
  channel's list pass, under the currently configured pacing flags and a
  realistic `maxVideos` window, does not spuriously exceed its own timeout
  under normal (non-hung) conditions.
- **FR2.3** Every timeout/stall failure reason produced anywhere in the
  download pipeline (list-pass timeout, download absolute-ceiling timeout,
  the new P3 stall/idle timeout) names BOTH the phase (list pass / download)
  and the configured duration that was exceeded, using the *actual configured
  value* at the moment of failure (not a literal that can drift from
  `config.js`'s parsed value).
- **FR2.4** The bare, phase-less string `"timed out and was killed"`
  (currently emitted at `lib/ytdlp/run.js:552` and `lib/ytdlp/run.js:938`, and
  asserted by name in `test/integration/ytdlp-oneshot.test.js`,
  `test/integration/ytdlp-poll.test.js`, and
  `test/integration/ytdlp-oneshot-cancel.test.js`) no longer exists anywhere
  in `lib/`/`server.js` source (test fixtures that exercise the NEW
  phase-named strings are expected and fine — see AC2's grep criterion for the
  exact scope).
- **FR2.5** A stall/idle-triggered kill (P3) and an absolute-ceiling kill
  (existing `DEFAULT_DOWNLOAD_TIMEOUT_MS`/`config.downloadTimeoutMinutes`)
  produce two DIFFERENT, distinguishable reason strings — a human (or a test)
  can tell "the download made no progress for N minutes" apart from "the
  download hit its 180-minute ceiling despite making progress" without
  inspecting anything beyond the persisted/live status text.

### FR3 — Deliverable 3: per-channel queue decomposition, priority, durability, visibility (P1 + P4 + P5)

- **FR3.1** The monolithic single-job poll run (`lib/ytdlp/index.js:1943-
  1963`, today the ENTIRE poll — all N channels — is one `runExclusive`
  callback) is decomposed so each channel's work is its own unit of work
  through the shared FIFO gate (`ytdlpTail`/`runExclusive`,
  `lib/ytdlp/index.js:2044-2070`) — a hung/slow channel's spawn blocks only
  that channel's own job, not every channel queued behind it in the same
  poll run.
- **FR3.2** A one-shot submitted while a poll run is mid-flight starts within
  at most one already-in-flight channel job's remaining work — i.e. it waits
  behind at most the currently-running channel, never behind the whole
  remainder of the poll's channel list (contrast with today's `202` at
  `lib/ytdlp/index.js:3368` racing a gate-join at `:3384` behind the ENTIRE
  run).
- **FR3.3** Priority ordering across pending work is: one-shot > repull >
  scheduled poll. When a one-shot, a repull, and a scheduled-poll channel job
  are all pending at the same time, the one-shot is served before the repull,
  which is served before the remaining scheduled-poll channels.
- **FR3.4** The `--download-archive` single-writer invariant holds across the
  decomposed per-channel jobs: at any instant, at most one yt-dlp child
  process is writing the shared archive file — the decomposition must not
  turn the previously-monolithic gate into N independently-schedulable spawns
  that could overlap.
- **FR3.5** The `runExclusive`/`ytdlpTail` FIFO's never-wedge tail property
  (documented at `lib/ytdlp/index.js:2017-2027`: a rejecting job must never
  permanently block jobs queued behind it) continues to hold after the queue
  is decomposed into per-channel jobs — a channel job that throws, rejects,
  or times out still releases the tail so the next queued job (another
  channel, a one-shot, a repull) runs.
- **FR3.6** An accepted-but-not-yet-started job (at minimum: a one-shot
  accepted via `POST /api/ytdlp/download`, which today only lives in the
  in-memory `activity` map and the in-memory FIFO — `lib/ytdlp/index.js:3355-
  3368`) is persisted durably enough that a server restart before it starts
  requeues it rather than silently discarding it.
- **FR3.7** Every accepted job — no matter how it resolves (downloaded,
  failed, requeued-then-completed-later, or dropped with an explicit reason,
  e.g. the job's target became invalid across the restart) — eventually
  produces a runlog/history line. A job is never accepted and then simply
  vanishes with no observable trace.
- **FR3.8** The durable queue's persistence write follows the existing
  crash-safety posture already used for `db.json`/the JSONL run log: atomic
  (temp file + rename), and the persisted file is bounded (a stated, testable
  cap — no unbounded growth), per Constraints. No new runtime dependency is
  introduced.
- **FR3.9** Status, the corner download chip, and history each surface a
  pending job's queue position and what's ahead of it (e.g. "queued — 3
  ahead") — the existing `activity.setOneShot`/`setSubscription` `'queued'`
  states (`lib/ytdlp/index.js:3355-3366` for one-shots) currently carry no
  position information; this requirement adds it to all three surfaces named
  in the plan, not just one.
- **FR3.10** A one-shot's activity entry visibly transitions `queued` →
  `downloading`/`running`, with the wait time/position visible while queued
  — not merely a static "queued" label with no sense of progress toward
  starting.

### FR4 — Deliverable 4: stall watchdog reclaims a hung download in the idle window (P3)

- **FR4.1** A download-phase yt-dlp child that produces no stdout progress
  output for a configurable idle window (default ~10 minutes) is killed
  (`SIGKILL`, per the existing spawn-discipline constraint) well before the
  absolute download ceiling (`DEFAULT_DOWNLOAD_TIMEOUT_MS`/180 min default,
  `run.js:902-906` — currently the ONLY timeout on the download phase, never
  reset by progress) would otherwise fire.
- **FR4.2** A download child that DOES emit progress within the idle window
  is never killed by the stall watchdog on account of the idle timer alone —
  only the pre-existing absolute ceiling remains a backstop for a
  slow-but-progressing download.
- **FR4.3** The idle window resets on every observed progress signal (the
  existing progress parser, `lib/ytdlp/progress.js`, is the source of
  progress events — not rebuilt, only consumed by the new watchdog per the
  plan's out-of-scope boundary implied by "Progress-aware idle timer").
  the plan's out-of-scope boundary implied by "Progress-aware idle timer").
- **FR4.4** The stall watchdog applies to the download pass only; the list
  pass is governed by the P0-fixed list-pass budget (FR2), not a separate
  idle timer.
- **FR4.5** A stall-killed download is retried per the existing retry policy
  (the same policy path an absolute-ceiling timeout or any other failure
  already follows) — the stall watchdog is a faster detection path, not a
  new failure category exempt from retry.

### FR5 — Deliverable 5: repull busy/started state visible wherever repull can be triggered (P5)

- **FR5.1** Every existing client-side trigger point for a repull/re-pull
  action (currently at minimum the row-level repull call,
  `public/js/common.js:4885`, and the per-folder "Re-pull this channel now"
  button, `public/js/common.js:5193` — both today only `.catch()` network
  errors and discard the response body) reads the `started`/`reason` fields
  the server route already returns (`lib/ytdlp/index.js:3189` and `:3207`,
  the v1.29 `onDecision` discriminator) and renders a distinct, visible state
  when `started === false` (busy-coalesced) versus `started === true`.
- **FR5.2** The busy/coalesced state is visible at EVERY trigger surface
  named in FR5.1, not just one of them — a repull click from the folder-page
  button and a repull click from a subscription row both surface the same
  observable busy signal when a run is already in flight.

### FR6 — Deliverable 6: yt-dlp version surface with staleness note (P6)

### Deliverable 3 — per-channel queue: priority, durability, visibility

- [ ] **AC3.1** [FIXTURE] Given a poll run targeting ≥3 channels via a
  stubbed/fake per-channel job (controllable completion, no live network), a
  one-shot submitted while the FIRST channel's job is in flight starts (its
  spawn/fake-runner invocation begins) after at most that one in-flight
  channel job completes — never after a second, third, etc. channel job.
- [ ] **AC3.2** [FIXTURE] Given a fixture with a pending repull request and a
  pending scheduled-poll channel job both queued behind an in-flight job, and
  then a one-shot submitted, the observed start order is: current in-flight
  job finishes, then the one-shot, then the repull, then the scheduled-poll
  channel — proving one-shot > repull > scheduled poll.
- [ ] **AC3.3** [FIXTURE, INV] Given concurrent submission of multiple
  channel jobs (poll channels, a one-shot, a repull) against a fake spawn
  that records timestamps/overlap, at no point are two fake "archive
  writer" invocations observed to overlap in time — strict serialization
  holds under concurrent submission (see AC-INV-1 for the paired negative).
- [ ] **AC3.4** [FIXTURE, INV] Given a fixture where one queued channel job's
  fake runner throws/rejects/times out, the NEXT queued job (another
  channel, a one-shot, or a repull) still runs to completion — the tail is
  never wedged by a prior job's failure (see AC-INV-2 for the paired
  negative).
- [ ] **AC3.5** [FIXTURE] An accepted-but-not-started one-shot job (submitted,
  `202`-accepted, activity recorded as `queued`, but its spawn has not yet
  been invoked) survives a simulated server restart (fresh process/fresh

## Design (lean-mode resolution)

Dean switched this wave to LEAN MODE after Discovery ("step away from the full
agentic harness"): the main session implemented directly against the plan's
P0-P6 and the 61 ACs, keeping the two-reviewer gate + dual-Node verification +
release ceremony. Notable design resolutions made in-code:

- H0 CONFIRMED (AC0): the list pass carried --sleep-requests/--retries against
  a hardcoded 5-minute budget; fixed via --socket-timeout (hung sockets fail
  fast) + a configurable, pacing-scaled list budget (resolveListTimeoutMs).
- P1 decomposition: per-channel gate jobs enqueued strictly sequentially --
  one-shots interleave between channels. PRIORITY TIERS RESOLVED STRUCTURALLY:
  "repull > scheduled" is vacuous because repulls COALESCE with a running poll
  (pollBusy) and never coexist as queue entries; the gate can only ever hold
  the current channel + one-shots, which IS the intended ordering. Locked by
  the AC-INV-3 structural test in ytdlp-hardening.test.js.
- One-shot gate slots are reserved AT SUBMISSION (probe runs unlocked in
  parallel; the gate job awaits its result) -- gate-fix for the slow-probe
  position-loss the QA reviewer found.
- P3 stall watchdog deliberately re-arms on ANY child output (not only parsed
  progress events) -- see the doc comment in run.js for the false-positive
  rationale vs FR4.3's literal wording.
